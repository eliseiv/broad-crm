# 00 · Видение продукта и NFR

## Цель

CRM-система для мониторинга backend-сервисов и серверов компании. Единая точка, где администратор видит здоровье инфраструктуры в реальном времени и добавляет новые серверы под наблюдение «в один клик».

## Scope Этапа 1

В объёме Этапа 1:

1. **Двухшаговый вход администратора** (логин → пароль), проверка против `ADMIN_USER` / `ADMIN_PASSWORD` из `.env`, выдача JWT.
2. **Страница «Серверы»** — сетка карточек, по карточке на сервер.
3. **Кастомные SVG-спидометры** CPU / RAM / SSD (0–100 %) в каждой карточке, цвет дуги по нагрузке (зелёный / жёлтый / красный).
4. **Карточка «+ Добавить»** (glass/blur) → модалка (Название, IP, Пользователь, Пароль).
5. **Автоматический провижининг** при добавлении: Ansible по SSH ставит node_exporter, регистрирует таргет Prometheus, сохраняет сервер (пароль зашифрован). UI показывает прогресс/статус.
6. **Метрики из Prometheus** по PromQL; Grafana — для детального анализа (datasource-only, прямой доступ `/grafana`, без drill-down ссылки из карточки — [ADR-005, поправка](adr/ADR-005-custom-gauge-vs-grafana-embed.md#поправка-2026-06-30--удаление-drill-down-ссылки-из-карточки)).
7. **Telegram-уведомления (опционально)** — фоновый нотификатор шлёт в группу алерты **только при эскалации**: вход метрики в жёлтую/красную зону и потеря доступности сервера (`up` 1→0). Без recovery-сообщений ([modules/notifier](modules/notifier/README.md), [ADR-009](adr/ADR-009-in-backend-notifier-vs-alertmanager.md)).
8. **Страница «ИИ - ключи»** — реестр API-ключей AI-провайдеров (OpenAI/Anthropic): добавление, список, удаление; ключи шифруются (Fernet), в UI показывается маска. Фоновый монитор периодически проверяет валидность ключа (лёгкий `GET /v1/models`, токены не тратятся) и шлёт в Telegram алерт при поломке ключа (🔴) и восстановлении (🟢). Навигация между «Серверы» и «ИИ - ключи» — верхними вкладками ([modules/ai-keys](modules/ai-keys/README.md), [ADR-010](adr/ADR-010-ai-key-monitor-vnutri-backend.md)).
9. **Detail-view карточек по клику + карандаш-edit** ([ADR-035](adr/ADR-035-detail-view-secret-reveal.md)) — короткий клик по карточке (Серверы/Прокси/ИИ-ключи/Бэки) открывает **read-only detail-окно**: сразу видны **только идентификаторы**, всё прочее — в свёрнутом блоке **«Информация»**, а **пустые поля не показываются вовсе** ([ADR-046](adr/ADR-046-ui-infra-fix-pack.md)). Кнопка **«Добавить»** на карточных страницах живёт **справа от заголовка** (пунктирные карточки-плейсхолдеры упразднены); редактирование — по иконке-карандашу вверху справа (та же форма, что «Добавить», в режиме edit): сервер — только `name`; AI-ключ — `name`/`provider`/`key` (секрет не префилится, пустое поле = не менять; при смене ключа/провайдера — повторная проверка). Секретные поля (SSH-пароль сервера, пароль прокси, полный AI-ключ) показываются как `••••` с on-demand reveal под правом `<page>:edit`. AI-ключи сгруппированы по провайдерам (секции OpenAI/Anthropic).
10. **Перестановка карточек drag-and-drop** — порядок серверов, AI-ключей и прокси задаётся перетаскиванием и сохраняется на сервере (`position`). **На странице «Бэки» DnD убран** ([ADR-046](adr/ADR-046-ui-infra-fix-pack.md)): порядок задаётся сортировкой по имени. **Короткий клик = read-only detail-view** (edit — по карандашу в detail, [ADR-035](adr/ADR-035-detail-view-secret-reveal.md) амендмент [ADR-011](adr/ADR-011-poryadok-blokov-server-side-dnd-kit.md)); зажатие ~200 мс = перетаскивание. DnD-механика не меняется — короткий клик ведёт в detail-view вместо прямой edit-модалки.
11. **Страница «Почты»** — **CRM хранит почту у себя** ([ADR-044](adr/ADR-044-mail-full-merge-into-crm.md); модель «read-through-прокси без хранения» **отменена**): письма, теги, каталог ящиков, привязка ящика к команде и история Telegram-уведомлений живут **в БД CRM**; агрегатор `postapp.store` — тонкий IMAP/SMTP-connector, который **сам push'ит** новые письма в CRM (`POST /api/mail/ingest`, HMAC). Три вкладки: **Сообщения** (master-detail лента, бесконечный скролл по компаундному курсору `(internal_date, id)`, inline-reply), **Почты** (CRUD ящиков; имя ящика — два поля **«Номер»**/**«Приложение»**), **Теги** (глобальный админский каталог; **удалить можно любой тег** — признак «встроенный» упразднён, [ADR-047](adr/ADR-047-mail-fix-pack.md)). Фильтры ленты по ящику и команде (AND-комбинируемы) + клиентский тумблер тегов; полнотекстового поиска нет ([TD-024](100-known-tech-debt.md)). HTML-тело письма рендерится изолированно (sandbox-iframe без `allow-scripts`/`allow-same-origin`) и **следует теме CRM**. **Telegram-уведомления о новых письмах** шлёт сам CRM (фоновый диспетчер, 5 ботов) + **Mini App `/tg/mail`** (вход без экрана логина, по подписи Telegram `initData`) ([modules/mail](modules/mail/README.md)).
12. **Страница «Дашборд»** — обзорная страница (со Спринта B **убрана из меню**, доступна по прямому URL `/dashboard` под `dashboard:view`, [ADR-022](adr/ADR-022-teams-nav-categories.md); дефолтом после логина больше не является): сетка кликабельных карточек-разделов со счётчиками (почтовые ящики Активные/Неактивные, серверы В сети/Не в сети, AI-ключи Работает/Не работает). Счётчики собираются **клиентски** из существующих list-эндпоинтов (backend-агрегатора нет); клик по карточке = переход в раздел ([08-design-system.md](08-design-system.md#страница-дашборд), [ADR-017](adr/ADR-017-dashboard-client-aggregation-mail-server-filters.md)).
13. **Страница «Прокси»** — реестр HTTP/HTTPS/SOCKS5-прокси (отдельные поля: Название, Тип, Хост, Порт, Логин опц., Пароль опц.): добавление, список, редактирование, удаление, перестановка (единый список). Пароль шифруется (Fernet), в API не возвращается (флаг `has_password`). Фоновый монитор периодически проверяет доступность прокси (`GET` эталонного URL через сам прокси) и шлёт в Telegram алерт при недоступности (🔴) и восстановлении (🟢). Устроена по образцу «ИИ - ключи» (модель со статусом + фоновый монитор), навигация — пункт «Прокси» категории «Мониторинг» (со Спринта B — [ADR-022](adr/ADR-022-teams-nav-categories.md)) ([modules/proxies](modules/proxies/README.md), [ADR-019](adr/ADR-019-proxies-availability-monitor.md)).
14. **Пользователи + роли + RBAC (Спринт 3)** — страница «Пользователи» (admin-only): управление дополнительными пользователями (логин с поддержкой кириллицы, пароль, роль) и ролями (матрица прав «страница × действия»). Права — на все страницы (каталог: `dashboard:view`; `servers`/`ai-keys`/`proxies`/`backends`: `view/create/edit/delete`; `mail:view`; со Спринта A — `roles`/`teams`: `view/create/edit/delete`). `.env`-админ становится **несменяемым супер-админом** (bootstrap, вне БД, всегда полный доступ). RBAC обеспечивается **на сервере** (`403 forbidden`); UI-гейтинг — только UX. Пароли БД-пользователей хэшируются (bcrypt) ([ADR-021](adr/ADR-021-rbac-users-roles.md), [modules/auth](modules/auth/README.md), [08-design-system.md](08-design-system.md#страница-пользователи)).
15. **CRM-команды + категоризированная навигация (Спринт A/B)** — новая доменная сущность **«Команды»** (страница «Команды»): группировка пользователей вокруг лидера (M2M «участники»). **Команда в системе одна** — прежних mail-«команд» внешнего сервиса (`groups`, `GET /api/mail/teams`) **не существует** ([ADR-044](adr/ADR-044-mail-full-merge-into-crm.md)): почтовый ящик привязан к CRM-команде напрямую (`mail_accounts.team_id`). Пользователи += опциональный **email** и членство в 0..N командах; форма создания — Логин, Почта, Пароль, Роль, Команды. **«Роли»** выносятся в отдельную страницу (список ролей показывает число носителей `user_count`), управление ролями/командами — через матрицу прав (`roles:*`/`teams:*`); управление учётками остаётся admin-only, с серверным **инвариантом эскалации** (роль нельзя выдать права сверх своих). Навигация со Спринта B была **3 категории-дропдауна**, но со Спринта C заменена на **плоский ряд пунктов** в хэдере (без категорий, [ADR-033](adr/ADR-033-flat-nav-theme-toggle-numbers-table.md)); «Дашборд» убран из меню (доступен по прямому URL), дефолт после логина → первая доступная по правам вкладка ([ADR-022](adr/ADR-022-teams-nav-categories.md), [modules/teams](modules/teams/README.md), [modules/auth](modules/auth/README.md), [08-design-system.md](08-design-system.md#навигация-плоская-applayout)).

### Out of scope (Этап 1 / после Спринта 3)

- ~~Управление несколькими пользователями / ролями (только один админ из `.env`)~~ — **реализовано** в Спринте 3 ([ADR-021](adr/ADR-021-rbac-users-roles.md)): пользователи+роли+RBAC на все страницы; `.env`-админ — супер-админ (bootstrap).
- MFA, OAuth/SSO, refresh-токены, отзыв конкретного токена, история сессий, аудит-лог действий ([TD-001](100-known-tech-debt.md)), UI-смена пароля супер-админа (`.env`) — вне scope Спринта 3.
- Windows-серверы (только Linux + systemd + node_exporter).
- Уведомления по email/Slack (Telegram — в scope, escalation-only; recovery-сообщения, silencing, история алертов и персистентность state нотификатора — вне scope, [TD-019](100-known-tech-debt.md)).
- Исторические графики на главной (только текущие значения; история — в Grafana).
- Редактирование сервера, кроме `name`: `ip`/SSH-доступ/переустановка агента не редактируются (в scope только переименование — `PATCH name`).
- Перемещение AI-ключа между провайдер-группами перетаскиванием (провайдер меняется только через edit-модалку).
- Клавиатурная перестановка карточек (drag-and-drop с клавиатуры) — опционально ([TD-022](100-known-tech-debt.md)); edit/delete с клавиатуры доступны.
- Горизонтальное масштабирование backend (один воркер провижининга).
- Прокси: ручной триггер «проверить сейчас» и настраиваемый интервал per-proxy, windowed-детект доступности, карточка «Прокси» на «Дашборде», измерение задержки/скорости/гео, использование прокси приложением для реальных запросов ([TD-028](100-known-tech-debt.md), [modules/proxies](modules/proxies/README.md#out-of-scope-этап-1)).
- Почта: хранение писем в БД CRM, server-side **полнотекстовый поиск** ([TD-024](100-known-tech-debt.md); строгий глобальный «новые сверху» доставлен через [ADR-013](adr/ADR-013-mail-newest-first-master-detail-inline-reply.md), серверные **фильтры по ящику/команде** — через [ADR-017](adr/ADR-017-dashboard-client-aggregation-mail-server-filters.md), оба из out-of-scope сняты), составление письма «с нуля» (только reply), вложения, управление аккаунтами/тегами из CRM ([modules/mail](modules/mail/README.md#out-of-scope-этап-1)).

## Пользователи

- **Супер-администратор** (`.env`, bootstrap) — несменяемая учётка из `ADMIN_USER`/`ADMIN_PASSWORD`. Всегда полный доступ ко всем функциям (роль `admin` неявно); **учётка (логин/пароль/права) в БД не хранится**, из UI не редактируется/не удаляется ([ADR-008](adr/ADR-008-admin-iz-env.md)+[ADR-021](adr/ADR-021-rbac-users-roles.md)). С [ADR-051](adr/ADR-051-superadmin-db-anchor-personal-state.md) в `users` есть его **невидимая системная строка-якорь** (`is_system`) — чисто техническая FK-цель личного состояния (личная прочитанность писем), чтобы персональные функции работали и под консольным супер-админом; на вход, права и видимость в UI она не влияет.
- **Дополнительные пользователи** (БД) — создаются супер-админом/ролью `admin` на странице «Пользователи», получают роль с гранулярными правами на страницы (RBAC, [ADR-021](adr/ADR-021-rbac-users-roles.md)).
- **Роли** — набор прав «страница × действия»; роль `admin` даёт доступ к управлению пользователями.

## Ключевые сценарии (user stories)

- **US-1.** Как админ, я вхожу через два экрана (логин, затем пароль) и попадаю на **первую доступную мне по правам вкладку** навигации (permission-aware дефолт, [ADR-022](adr/ADR-022-teams-nav-categories.md); «Дашборд» убран из меню и больше не является стартовым маршрутом — доступен по прямому URL `/dashboard`, см. US-20).
- **US-2.** Как админ, я вижу карточки серверов с актуальной нагрузкой CPU/RAM/SSD и статусом online/offline.
- **US-3.** Как админ, я добавляю сервер (имя, IP, SSH-логин, пароль) и наблюдаю прогресс автоматической установки агента.
- **US-4.** Как админ, я открываю Grafana (`/grafana`) для детального анализа метрик (из карточки прямой ссылки нет).
- **US-5.** Как админ, я удаляю сервер из мониторинга (снятие таргета, удаление из реестра).
- **US-6.** Как админ, я получаю в Telegram-группу уведомление, когда нагрузка сервера входит в жёлтую/красную зону или сервер становится недоступен, чтобы реагировать без постоянного наблюдения за UI.
- **US-7.** Как админ, я добавляю API-ключи AI-провайдеров (OpenAI/Anthropic) на отдельной вкладке «ИИ - ключи», вижу их статус (Работает/Не работает/Проверка…) и получаю в Telegram уведомление, когда ключ перестаёт работать (отозван/заблокирован/нет средств) и когда снова начинает работать — чтобы вовремя реагировать на сбои ключей.
- **US-8.** Как админ, я кликаю по карточке сервера и переименовываю его (не трогая IP/доступ), чтобы поддерживать понятные названия.
- **US-9.** Как админ, я кликаю по карточке AI-ключа и меняю его название, провайдера или сам ключ (ротация); при вводе нового ключа/провайдера система сразу перепроверяет его валидность. Ключи вижу сгруппированными по провайдерам (OpenAI/Anthropic).
- **US-10.** Как админ, я перетаскиванием меняю порядок карточек серверов и AI-ключей (внутри группы провайдера), и этот порядок сохраняется на сервере — одинаковый на всех моих устройствах.
- **US-11.** Как админ, я открываю вкладку «Почты», просматриваю входящие письма (отправитель, тема, дата, теги, тело) и отвечаю на письмо (reply) прямо из CRM — чтобы вести переписку без перехода во внешний почтовый сервис. HTML-письма отображаются безопасно (изолированно), ключ доступа к почте хранится только на сервере.
- **US-12.** Как админ, я открываю страницу «Дашборд» (по прямой ссылке `/dashboard`, если у меня есть право `dashboard:view`; в меню её нет — [ADR-022](adr/ADR-022-teams-nav-categories.md)) и вижу сводку по разделам — сколько почтовых ящиков активно/неактивно, серверов в сети/не в сети, AI-ключей работает/не работает — и одним кликом по карточке перехожу в нужный раздел.
- **US-13.** На странице «Почты» я фильтрую ленту писем по почтовому ящику **и/или** команде (фильтры **комбинируемы** — [ADR-038](adr/ADR-038-mail-headless-integration.md)), чтобы быстро находить переписку нужного адреса/отдела; фильтр применяется ко всей ленте на стороне сервиса, а не только к загруженным письмам. Дропдаун «Почта» доступен любому держателю права `mail:view` (его лента сужена ролевым scope); дропдаун **«Команда»** — при доступе к **≥ 2 вариантам команд канала** (команды канала + «Без команды»; едино для **всех** акторов, включая admin-уровень — у него порог выполняется сам), см. US-23 ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §6; прежнее «только admin-уровень (`sees_all_mail_teams`)» **отменено**, отдельной ветки под admin-уровень **нет**).
- **US-13а.** На **обеих** вкладках страницы «Почты» ящик я выбираю **одним и тем же контролом с поиском** (`ui/Combobox`, [ADR-052](adr/ADR-052-mail-mailbox-combobox.md)): клик по полю раскрывает список **всех** доступных мне почт, ввод текста отфильтровывает их — **по Номеру, Приложению или самому адресу**. На вкладке «Сообщения» выбор почты фильтрует **ленту писем** (серверно), на вкладке «Почты» ввод фильтрует **таблицу ящиков** по всем совпадениям, а выбор из списка — быстро переходит к конкретной почте.
- **US-14.** Как админ, я добавляю прокси (название, тип http/https/socks5, хост, порт, опциональные логин/пароль) на вкладке «Прокси», вижу их статус (Работает/Не работает/Проверка…) и получаю в Telegram уведомление, когда прокси перестаёт отвечать и когда снова становится доступен — чтобы вовремя реагировать на сбои прокси. Пароль прокси хранится только зашифрованным и в интерфейсе не показывается.
- **US-15.** Как админ, я добавляю бэк-сервис (код, название, домен) на вкладке «Бэки», вижу его статус (Работает/Не работает/Проверка…) на основе автоматической проверки `GET {домен}health` (домен приводится к канону `https://<host>/`, [ADR-042](adr/ADR-042-backend-domain-canonical-https.md)) и получаю в Telegram уведомление, когда бэк перестаёт отвечать и когда снова становится доступен. Код бэка уникален (повтор запрещён), название — нет (бэки с одинаковым названием группируются, [ADR-039](adr/ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)); в доп. секции «Информация» я опционально указываю сервер, ИИ-ключ, API/ADMIN-ключи (хранятся зашифрованно, показываются по кнопке «глаз»), ссылку на Git и примечания ([ADR-040](adr/ADR-040-backend-relations-secrets-reverse-lookup.md)); есть поиск по бэкам; перетаскиванием я меняю порядок карточек, и он сохраняется на сервере.
- **US-16.** Как супер-админ, я на вкладке «Пользователи» создаю дополнительных пользователей (логин — можно кириллицей, напр. «Никита», пароль, роль) и роли с матрицей прав «страница × действия», чтобы выдавать сотрудникам ограниченный доступ; при этом я сам (учётка из `.env`) всегда сохраняю полный доступ и не могу быть удалён/изменён из интерфейса.
- **US-17.** Как пользователь с ограниченной ролью, я вижу в навигации только разрешённые мне страницы и только доступные действия (кнопки «Добавить»/«Удалить» скрыты без права), а при попытке недоступного действия получаю «Недостаточно прав» без потери сессии; изменения моих прав администратором применяются без повторного входа.
- **US-18.** Как администратор, я на вкладке «Команды» создаю CRM-команды (название, лидер, участники — мультивыбор) и вижу по каждой лидера и число участников; лидер автоматически входит в состав. При создании/редактировании пользователя я задаю ему опциональную почту и включаю его в одну или несколько команд; на вкладке «Пользователи» список сгруппирован по командам. Это отдельные CRM-команды, не связанные с «командами» на вкладке «Почты».
- **US-19.** Как администратор, я на отдельной вкладке «Роли» вижу список ролей с числом пользователей под каждой и редактирую матрицу прав; я могу делегировать управление ролями сотруднику через права `roles:*`, при этом он не может выдать роли прав больше, чем есть у него самого, и не может изменить встроенную роль `admin` (сервер отвечает «Недостаточно прав»).
- **US-20.** Как пользователь, после входа я попадаю на первую доступную мне вкладку (навигация — **плоский ряд пунктов** в хэдере без категорий, [ADR-033](adr/ADR-033-flat-nav-theme-toggle-numbers-table.md)); «Дашборд» в меню отсутствует, но доступен по прямой ссылке `/dashboard`, если у меня есть на него право.
- **US-21.** Как пользователь, я переключаю светлую/тёмную тему интерфейса иконкой в хэдере; мой выбор запоминается между сессиями. При первом входе тема следует настройке моей ОС (системная), явный выбор её переопределяет ([ADR-033](adr/ADR-033-flat-nav-theme-toggle-numbers-table.md)).
- **US-22.** Как администратор, при создании/редактировании пользователя я в **свёрнутых блоках «СМС» и «Почты»** отмечаю ему **дополнительные команды** (плюс значение «Без команды») — команда из основного блока «Команды» отмечена всегда и снятию не подлежит. Сотрудник «Команды Ивана» после этого **видит и обрабатывает** почты нескольких команд и СМС нескольких команд — наравне со своей ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md)). **Ставя галочку «Без команды», я вижу подсказку об её объёме:** в блоке «СМС» она даёт доступ **ко всему ещё не распределённому потоку номеров** (они появляются автоматически из синхронизации Twilio) — с правкой, удалением и переносом; в блоке «Почты» — к ящикам без команды, которые заводит только администратор ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §3.1).
- **US-23.** Как пользователь с доступом к **двум и более** командам канала, я вижу фильтр по команде — на `/mail` (обе вкладки), на `/sms` («Сообщения») и в **обеих Telegram Mini App** (порог одинаков в вебе и на телефоне). При доступе к одной команде фильтр не показывается ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §6).
- **US-24.** Как сотрудник в Telegram, я в Mini App почты вижу **в каждом письме**, на какой ящик и чьей команды оно пришло — пилюли **«Номер»**, **«Приложение»** и **команда** (как в Mini App СМС), не открывая CRM ([ADR-056](adr/ADR-056-mail-mini-app-mailbox-context.md)).
- **US-25.** Как оператор, добавляя почту, я по названию поля **«Код приложения»** (второе поле формы, сразу под адресом) понимаю, что нужен **пароль приложения** провайдера, а не пароль от почты ([ADR-054](adr/ADR-054-mail-app-password-field.md)).

## Нефункциональные требования (NFR)

| ID | Категория | Требование | Критерий приёмки |
|----|-----------|-----------|------------------|
| NFR-1 | Простота | Минимум компонентов; монолитный backend | Нет брокера сообщений на Этапе 1 |
| NFR-2 | Безопасность | SSH-пароли зашифрованы at-rest | В БД хранится только Fernet-ciphertext |
| NFR-3 | Безопасность | Все API (кроме `/auth/*`, `/health`) под JWT | Без валидного токена — 401 |
| NFR-4 | Производительность | Метрики карточек обновляются регулярно | UI polling по умолчанию 15 с (конфиг) |
| NFR-5 | Производительность | Запрос `GET /api/servers` (до 50 серверов) | p95 < 800 мс при тёплом Prometheus |
| NFR-6 | Надёжность | Провижининг идемпотентен | Повторный запуск плейбука не ломает сервер |
| NFR-7 | Надёжность | Сбой провижининга не теряется | `provision_status = error` + сообщение |
| NFR-8 | UX/доступность | Контраст текста ≥ WCAG AA, видимый focus-ring | Проверка axe/Lighthouse a11y ≥ 90 |
| NFR-9 | Наблюдаемость | Prometheus/Grafana не выставлены наружу без auth | Доступ только из внутренней сети/за reverse-proxy |
| NFR-10 | Совместимость | Целевые серверы — Linux x86_64 с systemd | Документировано в допущениях |

## Допущения и ограничения

- Целевые серверы — Linux (systemd), node_exporter ставится как systemd-сервис на порт `9100`.
- Polling-интервал UI и scrape-интервал Prometheus задаются конфигом (значения по умолчанию см. [02-tech-stack.md](02-tech-stack.md#значения-по-умолчанию)).
- На Этапе 1 был один администратор; с Спринта 3 — многопользовательский режим с ролями/RBAC ([ADR-021](adr/ADR-021-rbac-users-roles.md)), супер-админ по-прежнему из `.env` (bootstrap).
- Сетевая доступность: backend-хост имеет SSH-доступ к целевым серверам; Prometheus имеет сетевой доступ к `<ip>:9100`.

## Метрики успеха Этапа 1

- Админ может добавить новый Linux-сервер и в течение ~1–2 минут увидеть его метрики на главной без ручных действий на сервере.
- Главная страница визуально соответствует референсу и enterprise-стандарту качества.
