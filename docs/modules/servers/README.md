# Модуль `servers` — Реестр серверов

Статус: `implemented` · **дельта [ADR-067](../../adr/ADR-067-server-ssh-key-auth.md) (вход по SSH-ключу) реализована 2026-07-22** — backend (миграция `0031`, `app/domain/ssh_keys.py`, `app/services/server_service.py`) и frontend (переключатель «Пароль / SSH-ключ», строка «Способ входа»); **тесты по ADR-067 написаны и зелёные** (`qa`, 2026-07-23: домен `ssh_keys`, матрица `_build_auth_material`, миграция `0031` upgrade/downgrade, UI-переключатель и отсутствие reveal при `auth_method='key'`). ⚠️ **Не прогонялись** (контейнерные/внешние, тестами не заменяются): runtime-smoke стека, контейнерная проверка состава backend-образа, реальный прогон плейбука ([TD-016](../../100-known-tech-debt.md)), E2E Playwright ([TD-015](../../100-known-tech-debt.md)) · Исполнители: backend, frontend

## Scope
CRUD реестра серверов: создание (с запуском провижининга) **по паролю ИЛИ по приватному SSH-ключу** ([ADR-067](../../adr/ADR-067-server-ssh-key-auth.md)), список (с метриками), **редактирование `name`**, **перестановка порядка (drag-and-drop)**, статус, удаление. Модель — [03-data-model.md](../../03-data-model.md), контракт — [04-api.md](../../04-api.md#servers).

## Out of scope
Редактирование `ip`/`ssh_user`/`exporter_port`, **смена способа входа и ротация SSH-кредов** (`auth_method`/пароль/ключ/парольная фраза), переустановка агента (PATCH меняет **только `name`**), повторный запуск провижининга, soft-delete/аудит ([TD-001](../../100-known-tech-debt.md), [TD-003](../../100-known-tech-debt.md), [TD-072](../../100-known-tech-debt.md)). Detail-view сервера ([ADR-035](../../adr/ADR-035-detail-view-secret-reveal.md)) показывает `ip`/`ssh_user`/способ входа/пароль (просмотр + reveal), но карандаш ведёт в edit только `name` — полноценное редактирование server-кредов остаётся вне scope ([Q-UI-4](../../99-open-questions.md)). **Reveal приватного ключа и парольной фразы — вне scope НАВСЕГДА в рамках [ADR-067](../../adr/ADR-067-server-ssh-key-auth.md) §4** (write-only секреты, не «пока не сделали»).

## Backend — ТЗ

### Endpoints
- `GET /api/servers[?status=]` → список с метриками (через модуль `monitoring`) + `provision_status` + `online` + `position` + **`backend_count`** (число связанных бэков, `COUNT` по `backends.server_id` — для секции «Бэки» detail-view, [ADR-040](../../adr/ADR-040-backend-relations-secrets-reverse-lookup.md)). Сортировка `position ASC, created_at DESC, id`. Graceful degradation при недоступности Prometheus (`metrics=null`, статус `200`).
- `POST /api/servers {name,ip,ssh_user,auth_method?,ssh_password?,ssh_private_key?,ssh_key_passphrase?}` → `202` ([ADR-067](../../adr/ADR-067-server-ssh-key-auth.md)); валидация (**ровно один способ**; разбор ключа `cryptography`; лимиты), шифрование материала (Fernet, `app/infra/crypto.py`), `INSERT status=pending` (`position` = `DEFAULT 0`), запуск фоновой задачи провижининга ([modules/provisioning](../provisioning/README.md)). `auth_method` **опционален, default `password`** ⇒ прежнее тело остаётся валидным. Дубликат `ip` → `409 server_conflict`; невалидный IP → `422 unprocessable`; нарушение правил способа входа / формата ключа → `422 validation_error` с точным `details[].field`. Полный контракт и таблица исходов разбора ключа — [04-api.md](../../04-api.md#post-apiservers).
- `PATCH /api/servers/{id} {name}` (JWT) → `200`; меняет **только `name`** (1–64), обновляет `updated_at`. `ip`/SSH/провижининг не трогаются. Нет записи → `404`; пустое/длинное `name` → `400`. Контракт — [04-api.md](../../04-api.md#patch-apiserversid).
- `PATCH /api/servers/order {ids}` (JWT) → `204`; `ids` — полная перестановка множества серверов, backend в одной транзакции присваивает `position = 0..N-1`. Прецеденция кодов: битое тело → `400`; **любой несуществующий `id` → `404` (проверяется до полноты)**; только если все `id` существуют, но список не полная перестановка → `422`. Контракт и полное правило — [04-api.md](../../04-api.md#прецеденция-ошибок-валидации-нормативно-едино-для-всех-order-эндпоинтов).
- `GET /api/servers/{id}/metrics` (JWT) → текущие метрики; Prometheus down → `502 prometheus_unavailable`.
- `GET /api/servers/{id}/status` (JWT) → `{provision_status,error_message,updated_at}`.
- `DELETE /api/servers/{id}` (JWT) → `204`; удалить `targets/<id>.json`, удалить запись; повтор → `404`.
- `GET /api/servers/{id}/ssh-password` (JWT, гейт `require("servers","edit")`) → `200 SecretRevealResponse {value}`; **on-demand reveal** SSH-пароля для detail-view ([ADR-035](../../adr/ADR-035-detail-view-secret-reveal.md)): расшифровка `ssh_password_encrypted` (`decrypt_secret`) in-memory, заголовок `Cache-Control: no-store`, аудит-лог `secret_revealed` (без значения). Нет права → `403`; нет сервера → `404 server_not_found`; **сервер с `auth_method='key'` → `404 secret_not_set`** ([ADR-067](../../adr/ADR-067-server-ssh-key-auth.md)). Контракт — [04-api.md](../../04-api.md#get-apiserversidssh-password), [05-security.md](../../05-security.md#reveal-секретов-по-требованию-adr-035).
- ⛔ **`GET /api/servers/{id}/ssh-key` и `/ssh-key-passphrase` НЕ СУЩЕСТВУЮТ и вводить их запрещено** ([ADR-067](../../adr/ADR-067-server-ssh-key-auth.md) §4): приватный ключ и парольная фраза — **write-only** секреты. Обоснование — [05-security.md](../../05-security.md#почему-приватный-ключ-не-раскрывается-нормативно-adr-067-4): радиус поражения ключа = весь парк хостов; `servers:edit` правит лишь `name` ⇒ симметрии «edit и так перезаписывает секрет» нет ([Q-SEC-5](../../99-open-questions.md)).
- `GET /api/servers/{id}/backends` (JWT, гейт `require("servers","view")`) → `200 {backends: BackendRef[]}` (`{code,name,domain}`); **reverse-lookup** бэков, лежащих на сервере (`backends.server_id = {id}`) — для сворачиваемой секции «Бэки» в detail-view сервера ([ADR-040](../../adr/ADR-040-backend-relations-secrets-reverse-lookup.md)). Сортировка `position ASC, created_at DESC, id`. Нет сервера → `404 server_not_found`. Свёрнутый счётчик секции — `ServerListItem.backend_count`. Контракт — [04-api.md](../../04-api.md#get-apiserversidbackends).

### Требования
1. Слои: router → service → repository (SQLAlchemy async). Pydantic-схемы запросов/ответов = контракт.
2. Пароль/ключ/парольная фраза НИКОГДА не возвращаются в обычных list/detail-ответах и не логируются. `ssh_user` и **`auth_method`** **возвращаются** в `ServerListItem`/summary/`ServerCreatedResponse` (не секреты — [ADR-035](../../adr/ADR-035-detail-view-secret-reveal.md), [ADR-067](../../adr/ADR-067-server-ssh-key-auth.md)). Plaintext SSH-пароль — только через reveal-эндпоинт под `servers:edit`; приватный ключ и фраза — **никак**.
   - **Флагов `has_password`/`has_key` не вводить:** CHECK `ck_servers_auth_material` делает наличие материала однозначной функцией `auth_method` ([03-data-model.md](../../03-data-model.md#таблица-servers)); лишнее поле означало бы возможность рассинхрона там, где его нет по построению.
   - **Текст исключения разбора ключа не пробрасывать** ни в ответ, ни в лог (может нести фрагменты материала) — сообщения `422` фиксированы [04-api.md](../../04-api.md#post-apiservers).
3. Валидация `ip` через `IPvAnyAddress`; нормализация перед сравнением/уникальностью.
4. Обработка `UNIQUE(ip)` → `409`.
5. `updated_at` обновляется при смене статуса.
6. Recovery-hook: при старте backend «зависшие» `installing` старше `ANSIBLE_TIMEOUT_SEC` → `error` ([ADR-006](../../adr/ADR-006-async-provisioning-bez-brokera.md)).
7. **Каждая Alembic-миграция обязана иметь рабочий `downgrade()`** (основа отката релиза — [07-deployment.md](../../07-deployment.md#откат-миграций-бд), [03-data-model.md](../../03-data-model.md)).
8. **Колонка `position`** (`integer NOT NULL DEFAULT 0`) — миграция `0003_add_position` (`down_revision=0002_create_ai_keys`) с backfill по `created_at DESC` и рабочим `downgrade()` ([03-data-model.md](../../03-data-model.md#миграция-0003_add_position-концепт)). Reorder присваивает `position` в одной транзакции; валидация полной перестановки — иначе `422`.
9. Переименование сервера (`PATCH name`) не требует немедленной перезаписи file_sd-таргета (скрейп идёт по `instance`; label `name` информативный, обновится при следующей записи таргета).
10. **Два способа входа** ([ADR-067](../../adr/ADR-067-server-ssh-key-auth.md)): `auth_method ∈ {password,key}` + `ssh_private_key_encrypted`/`ssh_key_passphrase_encrypted`; `ssh_password_encrypted` становится **nullable**. Согласованность — CHECK `ck_servers_auth_material` (БД, не только сервис). Миграция **`0031_servers_ssh_key_auth`** (25 симв.): `server_default='password'` на `ADD COLUMN` ⇒ **backfill не нужен**, CHECK создаётся **последним**; `downgrade()` рабочий, но **ЛОССИ** — удаляет строки `auth_method='key'` (в старой схеме они непредставимы; заглушка-пароль запрещена) ⇒ pre-deploy `pg_dump` обязателен ([03-data-model.md](../../03-data-model.md#миграция-0031_servers_ssh_key_auth-концепт-adr-067-2)).
11. **Ветка провижининга выбирается ТОЛЬКО по `auth_method`**, а не по «что не `NULL`». Передача ключа (снятие passphrase в памяти, файл `0600` через `O_CREAT|O_EXCL`, `ansible_ssh_private_key_file`, удаление в `finally`) — [09-provisioning.md](../../09-provisioning.md#ветка-auth_methodkey-нормативно-adr-067-5), [modules/provisioning](../provisioning/README.md).

### Переиспользуемый контракт репозитория и модели (нормативно)

Объявляется здесь как единственный источник; на него опираются read-path и [modules/notifier](../notifier/README.md):

- `ServerRepository.list_online() -> list[Server]` — серверы с `provision_status == online`. Используется notifier для опроса и read-path при необходимости.
- `Server.instance` (property) = `f"{ip}:{exporter_port}"` — целевой `instance` для PromQL/Prometheus file_sd. Единственное место формирования строки `instance`; модуль `monitoring` (`fetch_for_instances`) и notifier принимают именно её.
