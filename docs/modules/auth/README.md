# Модуль `auth` — Аутентификация

Статус: `implemented` (Спринт 3, [ADR-021](../../adr/ADR-021-rbac-users-roles.md)) · Исполнители: backend, frontend

## Scope
Двухшаговый вход, защита API через JWT **и RBAC** (роли, права на все страницы, управление пользователями/ролями). Супер-админ — из `.env` ([ADR-008](../../adr/ADR-008-admin-iz-env.md) + амендмент [ADR-021](../../adr/ADR-021-rbac-users-roles.md)); дополнительные пользователи/роли — в БД. Поток входа — [ADR-002](../../adr/ADR-002-dvuhshagovyy-auth.md); RBAC — [ADR-021](../../adr/ADR-021-rbac-users-roles.md).

## Out of scope
Refresh-токены, отзыв конкретного токена, история сессий, OAuth/SSO, MFA, UI-смена пароля супер-админа (`.env`), аудит-лог действий ([TD-001](../../100-known-tech-debt.md)).

## RBAC — ТЗ (Спринт 3, [ADR-021](../../adr/ADR-021-rbac-users-roles.md))

### Модель и каталог прав
- Каталог прав — серверная константа `app/domain/permissions.py::CATALOG` (**единственный источник истины**, полный разбор — [05-security.md](../../05-security.md#каталог-прав-канон-на-сервере)): `dashboard:[view]`; `servers`/`ai-keys`/`proxies`/`backends:[view,create,edit,delete]`; **`mail:[view,create,edit,delete,sync,tags]`** ([ADR-044](../../adr/ADR-044-mail-full-merge-into-crm.md); прежняя запись «`mail:[view]`» описывала отменённую read-through-модель [ADR-012](../../adr/ADR-012-mail-read-through-proxy.md)); **`sms:[view,edit,transfer,sync,delete]`** ([ADR-030](../../adr/ADR-030-sms-module-full-merge.md), без `create`); **`roles`/`teams:[view,create,edit,delete]`** ([ADR-022](../../adr/ADR-022-teams-nav-categories.md)); **`documents:[view,create,edit,delete,share]`** ([ADR-059](../../adr/ADR-059-documents-module.md)). Порядок ключей = порядок строк матрицы: `dashboard, servers, ai-keys, proxies, backends, mail, sms, roles, teams, documents`. Страница «Пользователи» (`users`) — **вне каталога** (admin-only). **Локализация ключей — только на фронтенде** ([ADR-063](../../adr/ADR-063-documents-editor-cache-lifecycle-focus.md) §D): сервер отдаёт технические ключи, каждый обязан иметь русскую подпись в словаре ДС; действия сверх CRUD пока не выдаются через UI — [TD-068](../../100-known-tech-debt.md). Каталог отдаётся `GET /api/permissions/catalog` — гейт **`require("roles","view")`** (со Спринта A; было `require_admin`).
- Модель БД — `roles`/`users` ([03-data-model.md](../../03-data-model.md#таблицы-roles-и-users-rbac)), миграция `0008_create_users_roles` (`down_revision=0007_create_backends`, сид роли `admin`). Со Спринта A ([ADR-022](../../adr/ADR-022-teams-nav-categories.md)): `users += email` (миграция `0010_add_user_email`, `down_revision=0009_create_teams`), CRM-команды `teams`/`user_teams` (миграция `0009_create_teams`, модуль [teams](../teams/README.md)). Миграция `0010` также **обновляет seed-роль `admin`** до полного нового каталога (добавляет `roles`/`teams`). **Со Спринта авт.-правок ([ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md)):** `users.email` **заменён на `users.telegram`** и `users.password_hash` стал **nullable** (беспарольные пользователи) — миграция `0011_user_passwordless_telegram` (`down_revision=0010_add_user_email`). **Тристатус пользователя ([ADR-028](../../adr/ADR-028-user-status-first-login.md)):** `users += first_login_at timestamptz NULL` (метка первого входа) — миграция `0015_user_first_login` (`down_revision=0014_proxies_alert_grace`); производный `UserListItem.status ∈ {"pending","active","inactive"}`.

### Auth-поток и JWT (claim'ы — побуквенно)
1. Логин: **сперва** супер-админ (`.env`, constant-time) → JWT `sub=ADMIN_USER, role="admin", superadmin=true` (без `uid`); **иначе** БД-пользователь — идентификатор сопоставляется с `username` точно, иначе с нормализованным `telegram` (**вход по Логину ИЛИ Телеграму**, [ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md)); при `is_active`: парольный (`password_hash IS NOT NULL`) → `verify_password` bcrypt → JWT `sub=username, uid=users.id, role=role.name, superadmin=false`; **беспарольный** (`password_hash IS NULL`) → ответ `password_setup_required:true` + limited-scope setup-token (см. ниже). **При успешном входе парольного пользователя** сервер идемпотентно проставляет `users.first_login_at = now()`, если `NULL` (метка первого входа для тристатуса — [ADR-028](../../adr/ADR-028-user-status-first-login.md)). Беспарольная ветка метку не ставит (вход не выполнен — только setup-token).
2. Расширить `app/infra/jwt.py`: access-payload несёт `sub`, `uid?`, `role`, `superadmin`, плюс `iat`, `exp`, `type:"access"`. Добавить **setup-token** `type:"pwd_setup"` (`uid`, `sub`, `iat`, `exp`; **без** `role`/прав, TTL `PWD_SETUP_TOKEN_EXPIRES_MIN`=10 мин). `decode_access_token` возвращает полный payload. **`get_current_principal` отвергает токены с `type != "access"` → `401`** (setup-token не даёт доступа к ресурсам).
3. Пароли БД-пользователей — `app/infra/passwords.py` (`hash_password`/`verify_password`, bcrypt), [05-security.md](../../05-security.md#хэширование-паролей-bcrypt). **Пароль опционален** (`password_hash` nullable): беспарольный создаётся без хэша; задаёт пароль через `POST /api/auth/set-password` при «открытом первом входе» ([ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md)). **`set-password` также идемпотентно проставляет `users.first_login_at`** (если `NULL`) — беспарольный после установки сразу залогинен, это его первый вход ([ADR-028](../../adr/ADR-028-user-status-first-login.md)).

### Enforcement
- `get_current_principal` (в `app/api/deps.py`) декодит JWT и **грузит актуальные права из БД** каждый запрос: супер-админ → полный каталог; БД-пользователь → `role.permissions` по `uid` (если нет пользователя/`is_active=false` → `401`). Возвращает `Principal(username, role, permissions, is_superadmin, user_id)`. **`user_id` — НЕ-опциональный `uuid.UUID`** ([ADR-051](../../adr/ADR-051-superadmin-db-anchor-personal-state.md)): БД-пользователь → `users.id` из `uid`; **супер-админ → константа `SUPERADMIN_USER_ID`** (системная строка-якорь, **без запроса в БД** — fallback-инвариант [ADR-008](../../adr/ADR-008-admin-iz-env.md) сохранён). JWT супер-админа **не меняется** (`uid` в него не кладётся).
- `require(page, action)` → `Principal` или `403 forbidden()` (новый в `app/errors.py`). Применить ко **всем** ресурсным роутерам вместо `_user: CurrentUser`, а также к `/api/roles`, `/api/teams` и `GET /api/permissions/catalog` ([ADR-022](../../adr/ADR-022-teams-nav-categories.md); маппинг метод→действие — [04-api.md](../../04-api.md#rbac-и-enforcement-прав)).
- `require_admin` (`is_superadmin || role=="admin"`) → со Спринта A гейтит **только Users API** (`/api/roles`, `/api/teams`, каталог переведены на матрицу `roles:*`/`teams:*`).
- `GET /api/auth/me` — `{username, role, is_superadmin, permissions}` (permissions теперь может содержать ключи `roles`/`teams`).

### Системная строка-якорь супер-админа (нормативно, [ADR-051](../../adr/ADR-051-superadmin-db-anchor-personal-state.md))

Личное состояние с FK на `users` (личная прочитанность писем, [ADR-050](../../adr/ADR-050-mail-search-team-filter-personal-read-state.md)) невозможно для принципала без строки в `users`, а владелец работает на проде под консольным супер-админом. Решение — **системная строка-якорь**: `users.is_system = true`, константный `SUPERADMIN_USER_ID = 00000000-0000-0000-0000-000000000001`, зарезервированный `username` `superadmin@system` (символ `@` отвергается `validate_identity_name` ⇒ имя нельзя создать через API), **locked** bcrypt-хэш случайного секрета (`password_hash IS NULL` **ЗАПРЕЩЁН** — иначе «открытый первый вход» [ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md) выдал бы setup-token и дал эскалацию), роль-заглушка под FK (**правами не наделяет**). Поля — [03-data-model.md](../../03-data-model.md#системная-строка-якорь-супер-админа-adr-051); граница безопасности — [05-security.md](../../05-security.md#системный-якорь-супер-админа-нормативно-adr-051).

- **Миграция `0026_users_is_system`** — **только схема** (`users.is_system` + частичный уникальный `uq_users_system_singleton`, **объявленный в `User.__table_args__`** — миграция его зеркалит; `downgrade()` сначала `DELETE FROM users WHERE is_system`). **Строку создаёт идемпотентный bootstrap** `UserRepository.ensure_superadmin_anchor()` (`INSERT … ON CONFLICT DO NOTHING` → верификация; отсутствие → ERROR-лог `superadmin_anchor_missing`, старт не падает). **Резолв роли самодостаточен:** `admin` → самая ранняя роль → **если ролей нет вовсе, bootstrap сам создаёт `admin`** с полным каталогом.
- **Порядок вызова (нормативно):** прод — `main.py::lifespan` (миграции применяются раньше, в entrypoint: `alembic upgrade head && uvicorn …`); **тесты — фикстура обязана вызвать `ensure_superadmin_anchor(session)` сразу после `Base.metadata.create_all`** (`lifespan` в фикстурах не выполняется), иначе кейсы прочитанности супер-админа падают на FK.
- **Невидимость — choke point РЕЗОЛВА субъекта, `UserRepository`:** методы-резолверы (`list_all`, `get_by_id`, `get_with_teams`, `get_by_username`, `get_by_telegram`, `get_existing_ids`, `delete_by_id`) фильтруют `WHERE NOT users.is_system` (**`get_by_id` обязан быть переписан с `session.get()` на `select(...)`** — предикат иначе невыразим); методы уникальности (`exists_by_username`, `exists_by_telegram`) — **нет** (зеркало DB-констрейнтов). Следствия: якоря нет в `GET /api/users`; `PATCH`/`DELETE` по его `id` → `404`; лидером/участником команды → `422`; вход по БД-ветке и Telegram-SSO его **не резолвят**.
- **`users` читают и МИМО `UserRepository`** ([ADR-051](../../adr/ADR-051-superadmin-db-anchor-personal-state.md) §1.4(б)): `RoleRepository` (правится явно — см. ниже), `MailTelegramLinkRepository.team_recipients`/`sees_all_candidates`, `SmsTelegramLinkRepository.recipients_for_team`. **ИНВАРИАНТ (нормативно): у якоря нет строк в `user_teams`, `mail_telegram_links`, `sms_telegram_links`, `mail_user_settings`.** Но опираться на него как на единственную преграду **запрещено**: три fan-out-выборки **ОБЯЗАНЫ** нести явный **`WHERE NOT users.is_system`** (сегодня якорь отсекает лишь INNER JOIN на линк — неявное условие, не ловимое машинным гейтом; `sees_all_candidates` отбирает получателей **по правам роли**, а роль якоря — `admin` с полным каталогом). Замена INNER→LEFT/OUTER без фильтра остаётся запрещённой.
- **Транзакция bootstrap:** `ensure_superadmin_anchor` **сам коммитит** свою транзакцию (роль + якорь + верификация = один коммит); вызывающий (`lifespan`, тестовая фикстура) передаёт отдельную сессию.
- **Роли:** `user_count` (`list_all_with_counts`/`count_users`) якорь **исключает**; гард `is_in_use` — **включает** ⇒ роль `admin` (носитель якоря) **не удаляется**: `409 role_in_use`.
- **Смена `ADMIN_USER`/`ADMIN_PASSWORD` — no-op** (идентичность на константе, а не на логине).
- **Границы:** `POST /api/sms/telegram/link` и `GET`/`PATCH /api/mail/me/settings` супер-админу — **`403`** (гейт `principal.is_superadmin`): bootstrap-учётка остаётся **console-only**, Telegram-SSO для неё дал бы беспарольный, неотзываемый из UI путь к admin-уровню.

### Users / Roles / Teams API
- `GET/POST/PATCH/DELETE /api/users` (гейт `require_admin`) — схемы/поля/коды строго по [04-api.md](../../04-api.md#users). Со Спринта A + [ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md): `UserCreateRequest`/`UserUpdateRequest` += `telegram?` (формат телеграм-ника → `422`, дубль → `409 telegram_taken`; **заменяет `email`**) и `team_ids?` (существующие CRM-команды → иначе `422`); **`password` опционален** (беспарольный пользователь); `UserListItem` += `telegram`, `has_password`, `teams`, **`status`** (производный тристатус `"pending"|"active"|"inactive"`, [ADR-028](../../adr/ADR-028-user-status-first-login.md); `first_login_at` наружу не отдаётся). `DELETE` пользователя-лидера команды — **успешно с авто-передачей лидерства** ([ADR-026](../../adr/ADR-026-teams-optional-leader-auto-transfer.md); код `user_is_team_leader` упразднён). Пароль — только на вход; уникальность `username` → `409`.
- `GET/POST/PATCH/DELETE /api/roles` — гейт **`require("roles", <action>)`** (было `require_admin`); `RoleListItem` += `user_count` (`COUNT(users) GROUP BY role_id`). Схемы/коды — [04-api.md](../../04-api.md#roles).
- `GET/POST/PATCH/DELETE /api/teams` — гейт `require("teams", <action>)`; модуль [teams](../teams/README.md), контракт — [04-api.md](../../04-api.md#teams).

### Security-инвариант эскалации ролей (нормативно, [ADR-022](../../adr/ADR-022-teams-nav-categories.md))
Раз `/api/roles` под матрицей `roles:*`, backend ОБЯЗАН запрещать эскалацию (проверка в handler после гейта; `403 forbidden` — единственная граница):
- **(а) subset:** не-супер-админ/не-`admin` не может создать/изменить роль с `permissions ⊄ permissions актора` (по каждой `page` набор `actions` — подмножество). Нарушение → `403`.
- **(б) защита `admin`:** роль `name=="admin"` меняет/удаляет только `is_superadmin || role=="admin"` → иначе `403`.
- **(в)** назначение ролей пользователям и управление учётками остаётся под `require_admin` (Users API вне матрицы) — замыкает эскалацию.

Прецеденция `POST`/`PATCH /api/roles`: каталожная валидация (`422`) → эскалация/защита `admin` (`403`) → уникальность имени (`409`). Детали — [05-security.md](../../05-security.md#security-инвариант-эскалации-привилегий-нормативно-adr-022), [04-api.md](../../04-api.md#roles).

## Backend — ТЗ

### Endpoints (контракт — [04-api.md](../../04-api.md#auth))
- `POST /api/auth/login {username, password?}` → `200 LoginResponse` (дискриминированный по `password_setup_required`: успех `{password_setup_required:false, access_token,...}` **или** установка `{password_setup_required:true, setup_token,...}`) | `401 invalid_credentials` | `400 validation_error` | `429 rate_limited`. Идентификатор = логин **или** телеграм; беспарольный → setup-token ([ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md), [05-security.md](../../05-security.md#аутентификация-логин-и-выпуск-jwt)).
- `POST /api/auth/set-password {password}` (Bearer **setup-token** `type:"pwd_setup"`) → `200 LoginResponse` (успех, access-token) | `401 unauthorized` (нет/просрочен setup-token или предъявлен access-token) | `422` (слабый пароль) | `409 password_already_set`. Установка пароля «первого входа» беспарольным пользователем ([ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md)).
- `GET /api/auth/me` (JWT) → `200 MeResponse {username, role, is_superadmin, sees_all_sms_teams, sees_all_mail_teams, mail_teams, mail_includes_unassigned, sms_teams, sms_includes_unassigned, permissions}` | `401 unauthorized` (полная схема — [04-api.md](../../04-api.md#get-apiauthme)). **`mail_teams`/`sms_teams`** — команды канала, доступные актору ([ADR-055](../../adr/ADR-055-per-channel-teams-mail-sms.md) §5.1): не-админ — **эффективный** scope (`user_teams ∪ user_channel_teams[channel]`), **admin-уровень — ВСЕ команды системы** (**не** `[]`); **`*_includes_unassigned`** — доступ к объектам без команды (при `sees_all` → `true`). Делают `/me` **единственным** источником опций команд канала на клиенте — фильтр «Команда» наполняется **без права `teams:view`** и работает в Mini App у **любого** актора ([TD-058](../../100-known-tech-debt.md) закрыт).

### Per-channel команды пользователя ([ADR-055](../../adr/ADR-055-per-channel-teams-mail-sms.md))

- **Хранение:** таблица **`user_channel_teams(user_id, channel ∈ {mail,sms}, team_id)`** (PK-тройка, обе FK `ON DELETE CASCADE`) + колонки **`users.mail_includes_unassigned`/`sms_includes_unassigned`** (`boolean NOT NULL DEFAULT false`); миграция **`0027_user_channel_teams`**, **backfill не нужен** ([03-data-model.md](../../03-data-model.md#таблица-user_channel_teams-per-channel-добавки-adr-055)).
- **Семантика — union:** scope канала = `user_teams` **∪** добавка канала. Базовое членство входит в оба канала **всегда**; блок канала задаёт только **дополнительные** команды.
- **Инвариант нормализации — на ВСЕХ путях записи в `user_teams` (security-инвариант, [ADR-055](../../adr/ADR-055-per-channel-teams-mail-sms.md) §2.3).** В `user_channel_teams` не хранятся команды, входящие в `user_teams` того же пользователя. Путей записи **два**, и оба обязаны нормализовать добавки **в той же транзакции**:
  1. **Users CRUD** (`POST`/`PATCH /api/users`, `team_ids`; `UserRepository.set_membership`) — из `*_extra_team_ids` **вычитается** `team_ids` того же запроса (оба канала). Присланная базовая команда в добавке — **не ошибка** (просто не сохраняется).
  2. **Teams CRUD** (`POST /api/teams`, `PATCH /api/teams/{id}`, `member_ids`; `TeamRepository.create`/`replace_members` — **зона [modules/teams](../teams/README.md)**, инвариант 8) — после приведения состава команды `T` удаляются строки `user_channel_teams (user_id ∈ member_ids, оба канала, team_id = T)`.
  > Без пути 2 гарантия **path-dependent**: доп-команда `B` → пользователя добавили в `B` участником на `/teams` → затем исключили там же ⇒ строка `user_teams` удалена, **добавка осталась** ⇒ доступ к почте/СМС команды `B` **сохраняется**. Инвариант «снял команду → потерял доступ в обоих каналах» обязан быть **безусловным**.
  - Удаление команды/пользователя нормализации не требует: `user_teams` **и** `user_channel_teams` каскадят (`ON DELETE CASCADE`).
- **Users CRUD:** `UserListItem += mail_extra_teams`/`sms_extra_teams` (**только добавка**) + `*_extra_includes_unassigned`; `UserCreateRequest`/`UserUpdateRequest += *_extra_team_ids`/`*_extra_includes_unassigned` (`PATCH` — полная замена набора; несуществующая команда → `422`). Гейт — прежний `require_admin`.
- **Права:** доп-команда даёт **и видимость, и действия** (решение владельца); новых действий в каталоге прав **не вводится**. Правила enforcement — [05-security.md §Per-channel scope](../../05-security.md#per-channel-scope-команд-нормативно-adr-055).

### Требования
1. Креды из настроек (`ADMIN_USER`, `ADMIN_PASSWORD`) через pydantic-settings.
2. Сравнение логина и пароля — `secrets.compare_digest` (constant-time), оба сравнения выполняются всегда (без раннего возврата), чтобы не было timing-разницы.
3. JWT: HS256, `JWT_SECRET`, claims `sub`, `iat`, `exp`, `type:"access"`, TTL `JWT_EXPIRES_MIN` (1440 мин / 24 ч, [05-security.md](../../05-security.md#jwt)).
4. FastAPI-dependency `get_current_principal` (базовый JWT-decode) валидирует токен для всех роутеров, кроме `/api/auth/login` и `/api/health`; невалидный/просроченный/легаси-токен → `401 unauthorized`. **Защита ресурсных роутеров — через фабрику `require(page, action)`** (нет права → `403 forbidden`), Users/Roles/Permissions — через `require_admin`; прежний «любой аутентифицированный» (`get_current_user`/`_user: CurrentUser`) заменяется на `require(...)`. Полная модель и маппинг метод→действие — [RBAC — ТЗ](#rbac--тз-спринт-3-adr-021), [04-api.md](../../04-api.md#rbac-и-enforcement-прав).
5. Rate-limit на `/api/auth/login`: по IP, 10 попыток / 5 мин (in-memory на Этапе 1), превышение → `429 rate_limited`.
6. Единое сообщение об ошибке для неверного логина и/или пароля.
7. Логи аутентификации — без паролей/токенов (structlog маскирование).

## Frontend — ТЗ
1. Роуты: `/login` (двухшаговый), `/servers` (защищён).
2. Шаг 1 — поле **«Логин или Телеграм»** (label) с плейсхолдером **«Логин или @username»** ([ADR-041](../../adr/ADR-041-login-theme-session-ux.md); было «admin или @username») + «Далее» (клиентский переход, без запроса). **Идентификатор входа принимается как логин, так и Telegram-ник с/без ведущего `@`** — backend нормализует (`normalize_telegram`) перед сопоставлением с `users.telegram` ([ADR-041](../../adr/ADR-041-login-theme-session-ux.md); поведение уже реализовано в `auth_service._resolve_db_user`). Шаг 2 — показ идентификатора + «назад», поле «Пароль» (не обязательно на клиенте; **беспарольный пользователь пароль не вводит**) + «Войти» → `POST /api/auth/login`. При `password_setup_required:true` — экран **«Придумайте пароль»** ([ADR-029](../../adr/ADR-029-ui-login-password-nav-team-form.md); ранее «Задайте пароль»): поле «Новый пароль» ≥8, которое пользователь **придумывает сам** (без генератора/случайного) + «Сохранить» → `POST /api/auth/set-password` (Bearer setup-token) → залогинен ([ADR-025](../../adr/ADR-025-passwordless-users-login-identifier-open-first-login.md), [08-design-system.md](../../08-design-system.md#экран-входа-двухшаговый)).
3. **Хранение access-токена — `localStorage`** (ключ `crm.auth.token`, сопутствующие `crm.auth.*`), амендмент [ADR-041](../../adr/ADR-041-login-theme-session-ux.md) — **изменено с прежнего in-memory/`sessionStorage`**. Обоснование ([05-security.md](../../05-security.md#jwt)): сессия переживает закрытие браузера (фикс 12) и **шарится между вкладками/окнами того же origin** — открытие страницы в новой вкладке (ПКМ/прямой URL) не требует повторного входа (фикс 16). **Auth-стор (`store/auth.ts`) инициализируется из `localStorage` синхронно при загрузке SPA — ДО резолва маршрута/guard**, поэтому `ProtectedRoute` видит восстановленную сессию на первом рендере и не редиректит на `/login`. Автовыход — по истечении TTL JWT (24 ч). Security-трейдофф `localStorage` (XSS-поверхность) осознанно принят для админ-панели ([05-security.md](../../05-security.md#jwt)).
4. Все запросы к `/api/*` шлют `Authorization: Bearer`. На `401` — сброс сессии (полная очистка `crm.auth.*` в `localStorage`) и редирект на `/login`.
5. Ошибка входа → единое сообщение «Неверный логин или пароль», без раскрытия деталей; shake-анимация (учитывать `prefers-reduced-motion`).
6. UI экрана входа — [08-design-system.md](../../08-design-system.md#экран-входа-двухшаговый).
