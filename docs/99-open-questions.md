| Q-SMS-1 | Поддерживать ли операторскую Mini App (`/tg/sms`) в браузерном **Telegram Web** (`web.telegram.org`, iframe)? | — | **resolved (решение пользователя): нет** — только нативные клиенты Telegram (iOS/Android/Desktop, webview верхнего уровня); глобальная CSP `frame-ancestors 'none'`/`X-Frame-Options: DENY` не ослабляется. Поддержка Telegram Web потребовала бы выделенного nginx-`location /tg/` c `frame-ancestors https://web.telegram.org` — не делается. См. [ADR-031](adr/ADR-031-sms-operator-mini-app.md) §5 |
| Q-SMS-2 | Как оператор видит свои номера/сообщения в Mini App? | — | **resolved (решение пользователя):** через **существующие** JWT-эндпоинты `GET /api/sms/numbers`/`GET /api/sms/messages` (SMS-scope до его команд) под access-JWT, выданным беспарольным SSO. **Роль оператора обязана включать `sms:view`** (требование к провижинингу), иначе просмотр пуст. Отдельный initData-scoped read-эндпоинт не вводится. См. [ADR-031](adr/ADR-031-sms-operator-mini-app.md) §3 |
| Q-SMS-3 | Риск подмены telegram-ника при беспарольном SSO (username reuse/takeover) и политика авто-провижининга операторов | Bootstrap-сопоставление по `username` доверяет владению ником в моменте auth (initData подписан Telegram); окно риска — **до первого линка** (после — резолв по иммутабельному `telegram_user_id`). Авто-провижининг операторов — **нет** (ручное заведение админом: `users.telegram` + `sms:view` + команда). Усиление (привязка по вводимому админом `telegram_user_id` без username-bootstrap) — при необходимости | open |
| Q-UI-2 | Дефолт темы при первом визите — тёмная (бренд), по системе (`prefers-color-scheme`) или светлая? | **resolved (решение пользователя, ревизия): СВЕТЛАЯ** ([ADR-041](adr/ADR-041-login-theme-session-ux.md), заменяет прежний «системная»). При отсутствии сохранённого выбора (`localStorage['crm-theme']` пуст) активна `light` — НЕ `prefers-color-scheme`; экран логина также светлый. Явный выбор пользователя переопределяет дефолт и персистится; подписка на смену темы ОС снята. Inline-скрипт без FOUC: `localStorage.getItem('crm-theme') || 'light'`. См. [ADR-041](adr/ADR-041-login-theme-session-ux.md), [ADR-033](adr/ADR-033-flat-nav-theme-toggle-numbers-table.md) §2, [08-design-system.md](08-design-system.md#темизация-светлаятёмная-adr-033) | **resolved** |
| Q-UI-3 | Судьба **удаления** номера на странице «Номера» после уборки колонки «Действия» — сохранить в UI или убрать совсем? | **Сохранить** компактной иконкой `Trash2` (без labeled-колонки «Действия»); перенос — авто-коммитом в колонке «Команда». `sms:delete`/`sms:transfer` остаются в API/каталоге. Альтернатива «убрать удаление из UI совсем» — на подтверждение. Не блокирует — [ADR-033](adr/ADR-033-flat-nav-theme-toggle-numbers-table.md) | open |
| Q-SEC-5 | Каким правом гейтить on-demand reveal секретов (SSH-пароль сервера / пароль прокси / полный AI-ключ)? | **Дефолт (принят [ADR-035](adr/ADR-035-detail-view-secret-reveal.md)): `<page>:edit`** соответствующей страницы (`servers:edit`/`proxies:edit`/`ai-keys:edit`) + супер-админ/`admin`. Обоснование: держатель `edit` уже управляет секретом (прокси/ключ — перезаписывает через `PATCH`). **На подтверждение пользователя:** не нужен ли **строже** (admin-only/`delete`) — особенно для SSH-пароля сервера, где `servers:edit` правит лишь `name`. **⚠️ Частично снят с повестки [ADR-067](adr/ADR-067-server-ssh-key-auth.md) §4 (2026-07-22):** для **приватного SSH-ключа и его парольной фразы** вопрос «каким правом» решён радикально — **раскрытия нет вообще** (write-only, эндпоинтов не существует), именно из-за отмеченной здесь несимметричности `servers:edit`. Открытым остаётся исходный вопрос про **пароль** сервера и секреты прокси/ИИ-ключей/бэков. Не блокирует реализацию | open |
| Q-UI-4 | Нужно ли полноценное редактирование server-кредов (`ip`/`ssh_user`/способ входа и материал) с карандаша detail-view сервера, или карандаш остаётся name-only? | **Дефолт (принят [ADR-035](adr/ADR-035-detail-view-secret-reveal.md), подтверждён [ADR-067](adr/ADR-067-server-ssh-key-auth.md)): name-only.** `ip`/`ssh_user`/`auth_method`/пароль/ключ/парольная фраза неизменяемы через API — смена требует репровижининга (вне scope, [modules/servers](modules/servers/README.md#out-of-scope)); detail-view показывает `ip`/`ssh_user`/**способ входа** и пароль (reveal — только для `auth_method='password'`), edit правит только `name`. **На подтверждение:** требуется ли полноценное редактирование кредов (с репровижинингом) — тогда `PATCH /api/servers/{id}` расширяется теми же правилами валидации, что `POST` ([TD-072](100-known-tech-debt.md), связано с [TD-003](100-known-tech-debt.md)). Не блокирует | open |

## Процедура закрытия
Каждый вопрос закрывается либо решением архитектора (с обновлением соответствующего документа/ADR), либо новым ADR. При закрытии — изменить статус на `resolved` и проставить ссылку на решение.
