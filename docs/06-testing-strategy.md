# 06 · Стратегия тестирования

## Пирамида тестов

```
        /\
       /E2E\        немного: критичные пути (вход → список → добавление)
      /------\
     /  Интег \      API + БД + замоканные Prometheus/Ansible
    /----------\
   /   Unit      \   много: services, crypto, PromQL-маппинг, зоны, gauge-математика
  /--------------\
```

## Coverage gate

- Backend: строки ≥ **80 %** (`pytest --cov=app`), критичные модули (`auth`, `crypto`, `monitoring` маппинг, зоны, `notifier` функция перехода/классификации сообщений) — ≥ **90 %**.
- Frontend: строки ≥ **70 %** (Vitest), компоненты `Gauge`, логика зон — ≥ **90 %**.
- **В CI на Этапе 1 — только backend-гейты** (`ruff`/`mypy`/`pytest --cov-fail-under=80`); проверяются в GitHub Actions ([07-deployment.md §CI/CD](07-deployment.md#cicd)), падение ниже порога — красный билд.
- **Frontend вне CI на Этапе 1** (решение пользователя): линт/typecheck/сборка frontend в CI **не выполняются** — SPA собирается на прод-сервере в `docker compose build` (proxy-образ), что само по себе валидирует сборку. **Эта валидация надёжна именно из-за чистоты контекста:** прод-сборка `tsc -b && vite build` идёт без кеша `tsBuildInfo`, тогда как локальный инкрементальный `tsBuildInfoFile` (`node_modules/.tmp/*.tsbuildinfo`) **маскирует** ошибки состава composite-проекта TypeScript (в частности **TS6307** — файл вне файлового списка проекта). Эталон проверки type-сборки — чистая сборка без кеша (удалить `*.tsbuildinfo` перед `tsc -b` либо Docker-сборка); прецедент — [ADR-066](adr/ADR-066-tailwind-config-in-app-tsproject.md). Тестовые гейты frontend (`vitest` ≥70 %, Gauge/зоны ≥90 %, Playwright e2e) — целевые пороги, но на Этапе 1 **не подключены к CI** ([TD-015](100-known-tech-debt.md)); их добавляет `qa` и возвращает в CI на этапе тестирования.
- Frontend-пороги (≥70 %, Gauge/зоны ≥90 %) остаются нормативной целью качества; применяются, когда тест-стек frontend появится в кодовой базе.

## Backend (pytest + pytest-asyncio + pytest-cov)

### Unit
- **auth**: генерация/валидация JWT (срок, подпись, невалидный токен), constant-time сравнение, rate-limit счётчик.
- **crypto**: Fernet encrypt→decrypt round-trip; неверный ключ → ошибка; ciphertext ≠ plaintext.
- **monitoring**: маппинг ответов Prometheus → схема метрик; вычисление `zone` на границах (79.9→green, 80→yellow, 90→yellow, 90.1→red); деградация при пустом/ошибочном ответе.
- **provisioning**: переходы `provision_status` (pending→installing→online/error); формирование `targets/<id>.json`; имена/санитизация.
- **notifier** (≥90 %, чистая функция перехода): матрица эскалаций — `green→yellow`/`green→red`/`yellow→red` → алерт; `red→yellow`/`*→green` и `offline→online` → молча; `online→offline` → offline-алерт; первая встреча online → база без алерта; возврат `offline→online` в нагрузке → переалерт (база `green`); `online` без метрик → пропуск без алерта; `PrometheusUnavailable` → state не тронут; формат всех 3 сообщений и `int(usage_percent)` (floor); опциональность (нет токена → задача не стартует). См. [modules/notifier](modules/notifier/README.md).

### Интеграционные
- API + тестовая PostgreSQL (testcontainers или отдельная схема). Prometheus и Ansible — **замоканы** (httpx mock / fake runner).
- Сценарии:
  - `POST /api/auth/login` верные/неверные креды → 200/401; rate-limit → 429.
  - Защищённый эндпоинт без токена → 401.
  - `POST /api/servers` → 202, запись в БД (пароль зашифрован, plaintext отсутствует), запущена фоновая задача; дубликат `ip` → 409; невалидный IP → 422.
  - `GET /api/servers` с замоканным Prometheus → корректные метрики/зоны; при недоступном Prometheus → 200 с `metrics=null`.
  - `GET /api/servers/{id}/metrics` при **недоступном самом Prometheus** (соединение/таймаут) → 502.
  - `GET /api/servers/{id}/metrics` при **доступном Prometheus, но `up==0` ИЛИ отсутствии метрики** → 200 с `online=false`, `cpu`/`ram`/`ssd` `detail.value=null` (`usage_percent`/`detail` не подставляются ложными значениями), **без 502** (см. [04-api.md](04-api.md#доступность-метрик-up0--отсутствие-данных-vs-prometheus-down), согласовано с [modules/monitoring/README.md](modules/monitoring/README.md) п.6).
  - `GET /api/servers/{id}/status` отражает текущий статус.
  - `DELETE` → 204, запись удалена, `targets/<id>.json` удалён; повтор → 404.
  - **Серверы — вход по SSH-ключу ([ADR-067](adr/ADR-067-server-ssh-key-auth.md)).** Каждый кейс — отдельный тест; ключи для фикстур **генерируются в рантайме** через `cryptography` (в репозиторий приватные ключи не коммитить даже тестовые).
    - `POST /api/servers` **без** `auth_method` со `ssh_password` → `202`, в БД `auth_method='password'` (обратная совместимость старого тела — регресс-гейт).
    - `auth_method='key'` + валидный незашифрованный ключ → `202`; в БД `ssh_private_key_encrypted` **не пуст**, `ssh_password_encrypted IS NULL`, plaintext ключа в строке отсутствует.
      - **Отдельным кейсом — PEM-armor без `Proc-Type` (регресс-гейт шага 1, [ADR-067](adr/ADR-067-server-ssh-key-auth.md) §3 п.4):** незашифрованные `-----BEGIN RSA PRIVATE KEY-----` (PKCS#1, `ssh-keygen -m PEM`) и `-----BEGIN PRIVATE KEY-----` (PKCS#8) **без** passphrase → `202`, `is_encrypted=false`. Самый частый формат ввода; реализация, где catch-all ловит его раньше PEM-ветки, отдаст `422` — тест обязан это ловить.
    - `auth_method='key'` + ключ **с** passphrase + верная фраза → `202`; неверная фраза → `422`, `details[].field == "ssh_key_passphrase"`; фраза не передана → `422`, тот же `field`; фраза передана к **незашифрованному** ключу → `422`, тот же `field`. ⚠️ Последние два исхода **обязаны обеспечиваться шагом 2 процедуры** (структурная кросс-проверка `is_encrypted` ↔ наличие фразы), а не поведением `cryptography`: на 43.x лишний пароль к незашифрованному ключу может быть **молча проигнорирован** — тест на этом кейсе и есть гейт, что шаг 2 реализован.
    - **Типы ключей — по кейсу:** RSA-2048/4096, ECDSA P-256, **Ed25519** → `202` (и провижининг-мок получает `ansible_ssh_private_key_file`); **DSA-2048 → `422`, `field="ssh_private_key"`** (отвергается шагом 4 **на валидации**; без него ключ прошёл бы форму и упал уже на провижининге — регресс-гейт).
    - Оба материала сразу (`ssh_password` + `ssh_private_key`) → `422` с `field` **лишнего** поля; ни одного → `422` с `field` **недостающего**.
    - Мусор вместо ключа / **публичный** ключ (`ssh-rsa AAAA…`) / ключ с испорченной base64-серединой (заголовки целы!) → `422`, `field == "ssh_private_key"`. Последний кейс — гейт против «валидации regex по BEGIN/END».
    - Ключ длиннее `SSH_KEY_MAX_BYTES` → `422` (проверка **до** разбора).
    - **Ключ с `\r\n` и без завершающего `\n`** → `202`, а в БД после расшифровки — нормализованная форма (LF + завершающий `\n`).
    - **Ответ и логи не содержат материала:** тело `202` — только `auth_method`, plaintext ключа/фразы отсутствует в ответе **и** в перехваченных structlog-записях (включая ветки `422`: текст исключения `cryptography` наружу не идёт).
    - `GET /api/servers/{id}/ssh-password` для key-сервера → **`404 secret_not_set`**; для password-сервера → `200`.
    - **Эндпоинтов `/ssh-key` и `/ssh-key-passphrase` не существует** → `404` (контракт-гейт против «доброжелательного» добавления reveal в будущем).
    - `PATCH /api/servers/{id}` с полями способа входа игнорирует их / отвергает: `auth_method` в БД не меняется.
    - **CHECK БД работает:** прямой `UPDATE` строки в состояние «пароль + ключ» / «key без ключа» → `IntegrityError` (проверка, что граница живёт в БД, а не только в сервисе).
  - **Документы — вложения ([ADR-068](adr/ADR-068-documents-image-attachments.md)).** Каталог хранения — временный (`DOCUMENTS_ATTACHMENTS_DIR` в tmp_path).
    - Загрузка валидного png/jpeg/webp/gif → `201`; на диске появился файл по пути **из `id`+`mime`**, права `0600`, `checksum` в ответе = sha256 содержимого.
    - **Подделка типа:** SVG/HTML/ELF с `Content-Type: image/png` → **`422 document_attachment_invalid`** (гейт против доверия клиентскому `Content-Type`). Отдельный кейс — честный `image/svg+xml` → тот же `422`.
    - Превышение `DOCUMENTS_MAX_IMAGE_BYTES` → `422`; проверяется, что чтение **оборвано** (файл на диске не создан).
    - Загрузка в **папку** → `422 validation_error`; в невидимый по роли узел → `404 document_node_not_found`.
    - **Отдача под чужой ролью:** пользователь без видимости узла → **`404 document_attachment_not_found`** (не `403` и не `200`) — ключевой security-кейс. Тот же код для несуществующего `id` и для вложения soft-deleted узла (**три случая — один код**).
    - Заголовки отдачи: `Content-Type` = сохранённый `mime`; `ETag` = `checksum`; повтор с `If-None-Match` → `304`; **`Cache-Control` содержит `private` и НЕ содержит `public`** (регресс-гейт shared-кэша).
    - `POST /nodes/{id}/copy` документа с двумя картинками → у копии **новые** `id` вложений, **новые** файлы на диске, и в `content_md` копии ссылки указывают на новые `id` (старых `uuid` в тексте не осталось).
    - Soft-delete узла → файлы на диске **на месте**, отдача → `404`.
    - `DELETE /attachments/{id}` → `204`, строка и файл удалены; повтор → `404`.
    - **Внешний контур не отдаёт байты:** `GET /api/external/documents/attachments/{id}` → `404` (роут не зарегистрирован); `GET /api/external/documents/{id}` возвращает `content_md` со ссылками **как есть** и **без** поля `attachments`.
    - **Удаление автора → `409 user_in_use`, НЕ `500`** ([04-api.md](04-api.md#delete-apiusersid), [TD-077](100-known-tech-debt.md)): пользователь, загрузивший вложение (и отдельным кейсом — создавший документ), `DELETE /api/users/{id}` → `409 user_in_use`; **лидерство его команд не передано** (транзакция откачена целиком); узел в состоянии soft-delete блокирует так же, как живой. **Покрыто** (`backend/tests/integration/test_users_delete_in_use_api.py`, 2026-07-23): 17 кейсов, включая гейт по `pg_constraint` «FK **не** `DEFERRABLE`» — при отложенном констрейнте `RESTRICT` сработал бы на `commit`, перехват прошёл бы мимо и вернулся бы `500`.
  - **Mail — таймауты и причина отказа ([ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md), регресс прод-бага; агрегатор замокан `httpx`-mock'ом).** Кейсы **поэлементно**, каждый — отдельный тест.
    - **⚠️ Суб-секундные бюджеты инъектируются В КЛИЕНТА, а НЕ в `Settings` — нормативно.** Прод-числа (read 75 / overall 85 / 30) в CI **не используются**: задержка мока в десятки секунд сделала бы прогон минутным. Но **через `Settings` их подменить нельзя** — точка инъекции только одна, клиентская:
      - **Почему не `Settings`.** `Settings` — `pydantic_settings.BaseSettings` ([`config.py:15,42`](../backend/app/config.py)), и границы `Field(ge=…, le=…)` валидируются **и при явном конструировании** (kwargs проходят через тот же валидатор, что и env) — «обойти `ge/le` явным конструированием `Settings`» **невозможно**. Вдобавок все четыре поля таймаутов — **`int`** (`mail_api_timeout_sec: int = Field(default=10, ge=1, le=30)` — [`config.py:157`](../backend/app/config.py); `mail_api_deadline_sec` — [`:159`](../backend/app/config.py); `mail_api_mailserver_timeout_sec: … ge=61, le=80` — [`:163`](../backend/app/config.py); `mail_api_mailserver_deadline_sec: … ge=76, le=85` — [`:166`](../backend/app/config.py)), т.е. `0.3` не проходит и **по типу**. А на **легальном** конфиге mail-server-пары (`ge=61` / `ge=76`) кейсы «агрегатор молчит дольше read-бюджета» и «исчерпание overall-deadline» потребовали бы **реального ожидания 61+ с** — ровно того многосекундного sleep, который здесь и запрещается.
      - **Реальная точка инъекции — конструктор клиента.** Бюджеты `MailClient` приходят **параметрами конструктора** — `read_timeout_sec: float` и `deadline_sec: float` ([`mail_client.py:142-148`](../backend/app/infra/mail_client.py)), а не читаются из `Settings` внутри; суб-секундные значения там легальны. Поэтому:
        - **Интеграционные кейсы:** через `app.dependency_overrides` подменяется **`get_mail_service`** ([`deps.py:402`](../backend/app/api/deps.py), зарегистрирован как `MailServiceDep = Annotated[MailService, Depends(get_mail_service)]` — [`deps.py:458`](../backend/app/api/deps.py)) — override возвращает `MailService(session=…, client=…, settings=…, mail_server_client=…)` ([`mail_service.py:196-202`](../backend/app/services/mail_service.py)) с **обоими** клиентами, собранными на **суб-секундных** бюджетах (напр. read `0.3` с, overall `0.5` с; компенсирующая уборка — ещё меньшим). ⚠️ **Подменять в `dependency_overrides` сами фабрики `get_mail_client`/`get_mail_server_client` — НЕ работает**: `get_mail_service` вызывает их **прямым вызовом** ([`deps.py:411,413`](../backend/app/api/deps.py)), а не через `Depends`, а `dependency_overrides` перехватывает только то, что FastAPI резолвит как `Depends`. (Если провязку когда-нибудь переведут на `Depends(get_mail_client)` / `Depends(get_mail_server_client)`, override фабрик станет равноправной точкой — но по действующему коду нормативна подмена `get_mail_service`.)
        - **Unit-кейсы транспорта:** `MailClient` конструируется **напрямую** с суб-секундными `read_timeout_sec`/`deadline_sec` ([`mail_client.py:142-148`](../backend/app/infra/mail_client.py)) — `Settings` не участвует вовсе.
      - «Задержка > бюджета» моделируется **относительно инъектированного** значения (мок отвечает через `2× бюджет`). Проверяется **поведение** — `error.code` фактического ответа и число исходящих запросов, — а **не абсолютные секунды**.
      - **`ge/le` прод-конфига в тестах НЕ обходятся и обходить их не нужно:** они защищают прод-конфиг и **сами являются предметом проверки** в кейсах §«Инвариант конфига» ниже, где `Settings` конструируется обычным конструктором с невалидными числами **именно ради `ValidationError`**.
    - `POST /api/mail/mailboxes/test`, агрегатор отвечает `422 {"error":{"code":"imap_login_failed"}}` **с задержкой больше read-бюджета** (моделирует прод-случай «ответ через ~31 с при бюджете 10 с») → CRM отдаёт **`422` с `error.code == "mail_imap_failed"`** (проверять именно поле `error.code` ответа, **не** текст `message`), **а не `502`**. Это прямой регресс прод-бага. Ключевое: read-бюджет mail-server-клиента **больше** задержки мока, поэтому ответ доходит.
    - То же с `smtp_login_failed` → `422 mail_smtp_failed`; с `invalid_host` → `422 mail_invalid_host`; с нераспознанным кодом → `422 unprocessable` (fallback).
    - Агрегатор «молчит» дольше **read-бюджета инъектированного mail-server-клиента** (в тесте — суб-секундного; прод-норма — `MAIL_API_MAILSERVER_TIMEOUT_SEC`) → **`504` с `error.code == "mail_timeout"`**, **ретрая нет** (mock фиксирует **ровно 1** исходящий запрос — анти-двойная-запись).
    - **Overall-deadline инъектированного клиента** (в тесте — суб-секундный; прод-норма — `MAIL_API_MAILSERVER_DEADLINE_SEC`) **— отдельный кейс:** мок «зависает» так, что суммарное время (в т.ч. ретраи `ConnectError` + backoff) превышает deadline → **`504 mail_timeout`**, ответ отдан **в пределах deadline** (проверяется, что обработчик вернулся раньше, чем истёк бы per-phase read) — граница запроса гарантирована, а не только фазовая.
    - Агрегатор вернул `504` → `504 mail_timeout` (**и на быстром пути тоже** — `DELETE`/`sync`/`oauth-authorize`, [ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md) §2.1); вернул `503`/`429` → `502 mail_unavailable`; `ConnectError` → 3 попытки → `502 mail_unavailable`.
    - **Быстрый путь, собственный таймаут CRM** (мок молчит дольше бюджетов инъектированного **быстрого** клиента — в тесте суб-секундных; прод-норма — `MAIL_API_TIMEOUT_SEC`/`MAIL_API_DEADLINE_SEC`) → **`502 mail_unavailable`**, **не** `504` (§2.1).
    - `POST /api/mail/mailboxes` и `PATCH /api/mail/mailboxes/{id}` (смена кредов) — те же 422-коды и `504 mail_timeout` (они тоже mail-server-пути).
    - `POST /api/mail/messages/{id}/reply`: агрегатор `502 {"error":{"code":"smtp_failed"}}` → **`502 mail_send_failed`** (не `mail_unavailable`) — проверяет, что транспорт донёс `status_code`+`error_code` из **5xx**-ветки, а не схлопнул её.
    - **Источник таймаута различим (unit транспорта, [ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md) §1.3 п.2):** агрегатор ответил `504` → поднят `MailTimeout` с **`status_code == 504`**; агрегатор молчит дольше read-бюджета / исчерпан overall-deadline → `MailTimeout` с **`status_code is None`**. Без этого поля кейсы «`504` от агрегатора на быстром пути → `504`» и «собственный таймаут на быстром пути → `502`» взаимно неисполнимы.
    - **Компенсирующая уборка сироты (`POST /api/mail/mailboxes`, [ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md) §1.2.2) — поэлементно:**
      - агрегатор создал ящик (`201`), вставка в каталог CRM падает → мок фиксирует **второй** исходящий вызов `DELETE /api/external/mailboxes/{id}`, клиенту уходит **исходная** ошибка вставки (не ошибка уборки);
      - компенсирующий `DELETE` сам **таймаутит** (мок молчит) → ответ клиенту — по-прежнему **исходная** ошибка (`MailTimeout` **не** пролетает наружу и не подменяет её), в логе — `mail_create_orphan_cleanup_failed`;
      - компенсирующий `DELETE` укладывается в **свой короткий deadline** (переопределённый малым значением, как в ⚠️ выше), **меньший**, чем deadline быстрой категории — уборка не съедает полный бюджет.
    - **Сборка клиента (unit):** mail-server-клиент собран с `MAIL_API_MAILSERVER_TIMEOUT_SEC`+`MAIL_API_MAILSERVER_DEADLINE_SEC`, быстрый — с `MAIL_API_TIMEOUT_SEC`+`MAIL_API_DEADLINE_SEC`; таймаут задан как `httpx.Timeout` **по фазам** (`connect` 5 / `write` 10 / `pool` 5 / `read` = бюджет), не одиночным float; весь вызов обёрнут `asyncio.wait_for`; **пер-вызовный override overall-deadline** применяется (компенсация).
    - **Инвариант конфига (unit, [ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md) §1.3 п.7) — поэлементно, `ge/le` и кросс-полевой валидатор проверяются ОТДЕЛЬНО:**
      - `ge/le`: `MAIL_API_MAILSERVER_TIMEOUT_SEC=60` (≤ потолка агрегатора) → **ValidationError**; `MAIL_API_MAILSERVER_DEADLINE_SEC=86` (> `le=85`) → **ValidationError**; **быстрая пара**: `MAIL_API_DEADLINE_SEC=120` (> `le=60`) → **ValidationError**, `MAIL_API_TIMEOUT_SEC=31` (> `le=30`) → **ValidationError**;
      - **кросс-полевой** (`model_validator`) — случай, который `ge/le` пропускают: `MAIL_API_MAILSERVER_TIMEOUT_SEC=80` **и** `MAIL_API_MAILSERVER_DEADLINE_SEC=76` (**обе** границы соблюдены, но read > overall) → **ValidationError**; то же для быстрой пары (`MAIL_API_TIMEOUT_SEC=30`, `MAIL_API_DEADLINE_SEC=10`) → **ValidationError**;
      - **бюджет запроса**: конфиг, при котором `MAIL_API_MAILSERVER_DEADLINE_SEC + MAIL_CLEANUP_DEADLINE_SEC + CRM_REQUEST_OVERHEAD_SEC ≥ NGINX_PROXY_READ_TIMEOUT_SEC` (т.е. `… + 15 + 5 ≥ 120`) → **ValidationError**; валидный дефолт (`85 + 15 + 5 = 105 < 120`) → `Settings` собирается;
      - **единственный источник константы уборки** ([ADR-053](adr/ADR-053-mail-timeouts-error-passthrough.md) §1.2.2/§1.3 п.7б): deadline, фактически применяемый компенсирующей уборкой, **равен `config.MAIL_CLEANUP_DEADLINE_SEC`**. Кейс — **спай на клиенте**: при провале вставки уборка зовёт `delete_mailbox(mailbox_id, deadline_sec=…)` ([`mail_service.py:972`](../backend/app/services/mail_service.py)), и тест ассертит, что переданный `deadline_sec` **идентичен** импортированной из `config.py` константе (а не «равен 15» — сверять с литералом бессмысленно, литерал в `mail_service.py` [запрещён](adr/ADR-053-mail-timeouts-error-passthrough.md#121-компенсирующая-уборка--отдельный-короткий-deadline)). ⚠️ **Патчить `config.MAIL_CLEANUP_DEADLINE_SEC` бесполезно** — `mail_service.py` связывает имя на импорте (`from app.config import MAIL_CLEANUP_DEADLINE_SEC`, [`mail_service.py:30`](../backend/app/services/mail_service.py)); если подмена значения всё же нужна, патчить следует `app.services.mail_service.MAIL_CLEANUP_DEADLINE_SEC`. Без этого кейса кросс-полевой валидатор стерёг бы **своё** зеркало, пока фактический deadline уборки разошёлся бы с ним, — «машинная защита» стала бы ложной.

- **Per-channel scope команд ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md)) — поэлементно** (по каждому каналу отдельно; «работает почта» **не засчитывается** за СМС и наоборот):
  - **union-семантика:** пользователь в команде A + доп-команда B (`user_channel_teams`) → лента/каталог отдают объекты **обеих**; после `PATCH /api/users/{id}` с `team_ids` **без** A команда A исчезает из scope **обоих** каналов (копии в добавке не остаётся — инвариант вычитания, [ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §2.3);
  - **вычитание базовой команды:** `POST`/`PATCH` с `mail_extra_team_ids`, содержащим команду из `team_ids`, → **не ошибка**, в `user_channel_teams` строка **не создаётся** (проверять по ответу `mail_extra_teams` и по БД);
  - **`includes_unassigned`:** без флага — бесхозный ящик/номер (`team_id IS NULL`) в чтении **отсутствует**, мутация → **`403`**; с флагом — виден и мутируется (`204`/`200`);
  - **не разворачивается:** `POST /api/mail/mailboxes` с `team_id=null` не-админом → **`403` даже при `mail_includes_unassigned=true`**; `PATCH` со сменой `team_id` не-админом (в т.ч. в свою доп-команду) → **`403`**;
  - **`no_team`:** `GET /api/mail/messages?no_team=true` и `GET /api/sms/messages?no_team=true` — только бесхозные; **`team_id` + `no_team=true` вместе → `400 validation_error`**; `no_team=true` без флага у не-админа → **пустая страница** (не `403`);
  - **`GET /api/auth/me`:** у не-админа `mail_teams`/`sms_teams` — **эффективный** набор (базовые ∪ добавка, сортировка по `name`); **у актора admin-уровня (`sees_all_<channel>_teams=true`) — ВСЕ команды системы** (**не** `[]` — [ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §5.1) и `*_includes_unassigned=true`;
  - **миграция `0027`:** `upgrade`/`downgrade` прогоняются; **без backfill** scope существующего пользователя не меняется (регрессионный кейс: до и после миграции лента одинакова);
  - **нормализация на пути teams CRUD (SECURITY-кейс, [ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §2.3 путь 2 — обязателен, отдельно от users-пути):** `X` получает доп-команду `B` (блок «Почты») → `PATCH /api/teams/{B}` с `member_ids`, включающим `X` (**строка `user_channel_teams (X, mail, B)` обязана исчезнуть**) → `PATCH /api/teams/{B}` с `member_ids` **без** `X` ⇒ у `X` **нет** доступа к почте команды `B` (лента пуста, мутация ящика `B` → `403`). Без нормализации — «висящий» доступ. Тот же кейс для `POST /api/teams` (создание команды сразу с участником, у которого эта команда — добавка) и для канала `sms`;
  - **перенос номера ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §3.2) — по кейсу на КАЖДУЮ проверку:** (а) номер вне scope → `403`; (б) не-админ, `team_id=null` **без** `sms_includes_unassigned` → **`403`**; (в) не-админ, `team_id=null` **с** флагом → `200`, номер остаётся видимым; (г) не-админ, целевая команда **вне** scope (существующая, но чужая) → **`403`**; (д) не-админ, целевая команда **несуществующая** → **`403`** (не `404` — анти-энумерация); (е) admin-уровень, несуществующая команда → **`404 sms_team_not_found`**; (ж) не-админ, целевая команда — его **доп-команда** → `200`;
  - **каталог ящиков ([ADR-055](adr/ADR-055-per-channel-teams-mail-sms.md) §5.3):** `GET /api/mail/mailboxes` у не-админа с `mail_includes_unassigned=true` **содержит** бесхозные ящики (`team_id: null`), без флага — **не содержит**. Контракт (query/схема) при этом не менялся.
- **`MailAccountRef` ([ADR-056](adr/ADR-056-mail-mini-app-mailbox-context.md)):** `GET /api/mail/messages` отдаёт `mail_account.number`/`app_name`/`team`; ящик без команды → **`team: null`** (LEFT JOIN, письмо **не пропадает** из ленты); число SQL-запросов на страницу ленты **не растёт с числом писем** (нет N+1).

### E2E (Playwright, против поднятого стека или замоканного API)

- **E2E-2**: список серверов с метриками отрисован.

## Runtime-smoke стека (перед релизом)

- Помимо статической проверки (`docker compose config`) обязателен **рантайм-smoke**: поднять стек `docker compose up -d`, дождаться `healthy` всех контейнеров, проверить `GET /api/health` → `200`, отдачу SPA (`GET /` → `200`) и запуск `proxy` (TLS-серт сгенерирован). Ловит рантайм-баги, невидимые статической проверкой (CRLF в entrypoint → `exit 127`, отсутствие `openssl`). Детали и чек-лист — [07-deployment.md §CI/CD](07-deployment.md#cicd); автоматизация в CI — [TD-014](100-known-tech-debt.md).

- **⚠️ Права каталогов под non-root — обязательный пункт smoke (класс дефекта «ломает всё в проде, невидимо в тестах»).** Образ работает под `USER app`, а Docker монтирует пустой named-volume и `tmpfs` **под `root:root`** ⇒ ошибка в правах не ловится ни юнит-, ни интеграционными тестами (там нет контейнера) и проявляется только на реальном стеке. В поднятом стеке проверить **изнутри контейнера backend**:
  - **шаг 0 (до `up`) — `docker compose config`.** Поле `tmpfs.mode` числовое (`uint32`), поэтому значение **в кавычках** отвергается ещё на разборе: `decoding failed … 'services[backend].volumes[N].tmpfs.mode' expected type 'uint32', got unconvertible type 'string'`. Стек при этом не поднимается вовсе — отказ ловится этим шагом, до `up`, и правится заменой на `mode: 0o1777`;
  - `docker compose exec -u app backend python -c "import tempfile,os; d=tempfile.mkdtemp(dir=os.environ['ANSIBLE_PRIVATE_DATA_ROOT']); print(d)"` → путь, **без `PermissionError`** (иначе `tempfile.mkdtemp` упадёт и **любое** добавление сервера вернёт ошибку — включая парольную ветку; типовая причина — режим `tmpfs`: либо явный `mode=1700`, либо **голое `mode: 1777` в compose, которое разбирается ДЕСЯТИЧНО** и даёт `0o3361` вместо `1777`. Диагностика на месте: `docker compose exec -u app backend python -c "import os,stat;print(oct(os.stat(os.environ['ANSIBLE_PRIVATE_DATA_ROOT']).st_mode))"` → обязано быть `0o41777`; `0o43361` = тот самый десятичный разбор ⇒ править на `mode: 0o1777` — **без кавычек** (кавычки дадут отказ разбора из шага 0) ([ADR-067](adr/ADR-067-server-ssh-key-auth.md) §5, [07-deployment.md §tmpfs](07-deployment.md#tmpfs-для-приватных-данных-ansible-нормативно-adr-067-5));
  - `docker compose exec -u app backend python -c "import os; p=os.environ['DOCUMENTS_ATTACHMENTS_DIR']; os.makedirs(p+'/aa/bb', exist_ok=True); open(p+'/aa/bb/.probe','w').close()"` → успех (иначе первая же загрузка изображения упадёт на создании шард-каталога — [ADR-068](adr/ADR-068-documents-image-attachments.md), [07-deployment.md](07-deployment.md#каталоги-создаваемые-в-образе-нормативно));
  - `docker compose exec backend id app` → **`uid=999(app) gid=999(app)`**. Значение запинено в `backend/Dockerfile` и является **частью формата бэкапа** (в tar и на volume лежит число): расхождение здесь означает, что уже снятые архивы вложений после восстановления окажутся нечитаемыми для backend — [07-deployment.md §Пин uid/gid](07-deployment.md#пин-uidgid-пользователя-app-нормативно).

## Тестирование провижининга (Ansible)

- Плейбук проверяется на **идемпотентность**: повторный прогон `--check`/реальный прогон не даёт `changed` на втором запуске (NFR-6).
- Запуск против эфемерного контейнера/ВМ Linux с systemd (molecule или ручной smoke в CI-окружении с docker-in-docker) — отнесено к devops/qa, объём smoke зафиксирован в [09-provisioning.md](09-provisioning.md).
- Unit-уровень backend мокает ansible-runner (не запускает реальный SSH).
- **Happy-path провижининга нельзя проверять ТОЛЬКО моками** (усвоенный урок: моки скрыли отсутствие `sshpass` в backend-образе). Минимально обязательно:
  1. **Контейнерная проверка состава backend-образа:** в собранном образе присутствуют `ansible` (`ansible --version`), `sshpass` (`command -v sshpass`), `ssh`. Это дешёвый тест, ловящий отсутствие системной зависимости до рантайма.
  2. **При возможности — реальный прогон плейбука** против эфемерного тест-хоста (sshd + systemd) с **password-SSH** (чтобы задействовать `sshpass`-путь), проверка `:9100/metrics` и идемпотентности.
- Полноценный автоматизированный e2e-провижининг (реальный SSH-хост в CI) на Этапе 1 не автоматизирован — [TD-016](100-known-tech-debt.md). Контейнерная проверка состава образа (п.1) — обязательна и дёшева, должна быть включена как минимум.
- **Definition of done мониторинга — `up=1` (нормативно):** провижининг считается успешным ТОЛЬКО когда метрики реально текут (`up=1` в Prometheus), а не когда node_exporter «установлен». Тесты/приёмка happy-path обязаны проверять реальный скрейп: после добавления сервера в течение разумного окна `up{instance=...}==1` и `GET /api/servers/{id}/metrics` возвращает значения (`online=true`). Это покрывает классы отказов цепочки скрейпа (права file_sd `0644`/`0755`, firewall `:9100`), невидимые при проверке только установки. См. [09-provisioning.md](09-provisioning.md#definition-of-done-провижининга-нормативно).
