# ADR-056 — Mini App почты: «Номер · Приложение · Команда» в каждом письме (`MailAccountRef` расширен)

- **Статус:** accepted
- **Дата:** 2026-07-14
- **Контекст-модули:** [mail](../modules/mail/README.md), [ui](../modules/ui/README.md)

- **⚠️ Разворачивает / уточняет действующие нормы (ADR-reversal, оформлено явно):**
  - **[04-api.md §Схема `MailMessage`](../04-api.md#схема-mailmessage)** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §2): `MailAccountRef = { id, email, display_name }` → **расширяется** до `{ id, email, display_name, number, app_name, team }`. Контрактное **расширение** (аддитивное, ничего не удаляется).
  - **[08-design-system.md §Telegram Mini App почты → Нормативный словарь](../08-design-system.md#нормативный-словарь-ui-строк-mini-app-почты)** ([ADR-044 поправка 2026-07-10](ADR-044-mail-full-merge-into-crm.md)): строка **«Получено на: {display_name} <{email}>»** → **ОТМЕНЕНА** и заменена на **«Получено на: {email}»** + отдельный ряд пилюль «Номер»/«Приложение» + пилюля команды (§3). Причина: `display_name` — производная склейка `"<number> <app_name>"` ([ADR-047](ADR-047-mail-fix-pack.md) §3.3), и печатать её рядом с теми же значениями в пилюлях значит показывать одно и то же дважды.

- **НЕ затронуто (явно):** десктопная `/mail` (карточка ленты и деталь письма **не меняются** — владелец просил про Mini App; новые поля контракта там просто не используются), `MailScope`/RBAC/анти-энумерация, БД (**миграций нет** — `mail_accounts.number`/`app_name` заведены `0024`, `team_id` — `0021`), порядок/курсор ленты, sandbox-изоляция тела письма ([ADR-012](ADR-012-mail-read-through-proxy.md)), reply, личная прочитанность ([ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)).

## Контекст

Требование владельца (батч 2026-07-14, дословно):

> «в MiniApp почт необходимо добавить отображение Код + Название + Команда в каждом сообщении (как сейчас это есть у смс)»

**Образец — SMS.** `SmsMessageCard` (`frontend/src/components/SmsMessageCard.tsx:44-62`) рендерит: бейдж команды (`Pill tone="green"` c `team.name`, иначе `Pill tone="neutral"` «Команды нет», `:44-48`) и ряд пилюль **«Логин: …»** (`tone="accent"`), **«Приложение: …»** (`tone="yellow"`), **«Примечание: …»** (`tone="neutral"`) (`:58-62`); пустое значение → **`'-'`** (`orDash`, `:8-11`), чтобы строка не «прыгала». Данные берутся из вложенного объекта номера (`message.number`), который несёт `team` (`SmsNumberRef.team: SmsTeamRef | null` — `backend/app/schemas/sms.py:24-29`).

**Сопоставление «Код»/«Название» с фактической схемой (нормативно — не додумывать).** У почтового ящика полей `code`/`name` **нет**. Фактические поля каталога ящиков — `number`, `app_name`, `team_id` (`backend/app/schemas/mail.py` → `MailMailbox`; [03-data-model.md §`mail_accounts`](../03-data-model.md#таблица-mail_accounts-каталог-ящиков), колонки `number`/`app_name` — миграция `0024`, [ADR-047](ADR-047-mail-fix-pack.md) §3), и в UI они уже нормированы лейблами **«Номер»** и **«Приложение»** ([08-design-system.md §Вкладка «Почты»](../08-design-system.md#вкладка-почты-нормативно), [ADR-048](ADR-048-teams-mailbox-count-mail-row.md) §3). Отсюда сопоставление:

| Слово владельца | Поле схемы | Нормативный лейбл UI |
|---|---|---|
| **Код** | `mail_accounts.number` | **«Номер»** |
| **Название** | `mail_accounts.app_name` | **«Приложение»** |
| **Команда** | `mail_accounts.team_id` → имя команды | **«Команда»** (пилюля с именем / «Команды нет») |

Обоснование: у ящика ровно три атрибута этого класса, они попарно соответствуют SMS-образцу (`login`/`app_name`/`team` номера), а «Код»/«Название» — словарь страницы **«Бэки»** (`code`/`name`/`domain`, [ADR-020](ADR-020-backends-healthcheck-monitor.md)), перенесённый владельцем по аналогии. **Вводить в почте лейблы «Код»/«Название» ЗАПРЕЩЕНО** — это расщепило бы словарь одного и того же ящика между `/mail` («Номер»/«Приложение») и Mini App.

**Факты по коду (`claims-from-code`, на 2026-07-14):**

- **Текущий API нужных полей НЕ отдаёт:** `MailAccountRef` = `id: int`, `email: str`, `display_name: str | None` (`backend/app/schemas/mail.py:30-35`), и это **единственный** ящик-объект внутри `MailMessage` (`:65`). Ни `number`, ни `app_name`, ни команды в ленте писем нет ⇒ **требуется контрактное расширение** (frontend показать нечего).
- Mini App берёт из ящика только имя/адрес: `const accountLabel = message.mail_account.display_name || message.mail_account.email` (`frontend/src/pages/MailMiniAppPage.tsx:343`), деталь — `const { email, display_name: displayName } = message.mail_account` (`:462`).
- Имя команды **нельзя** получить на клиенте: `GET /api/teams` гейтится `teams:view`, которого у mail-оператора может не быть ([TD-058](../100-known-tech-debt.md), [ADR-055](ADR-055-per-channel-teams-mail-sms.md) §5.1) ⇒ команда обязана прийти **внутри письма**, как у SMS.
- Примитив пилюли существует: `ui/Pill` с `tone ∈ {accent, yellow, neutral, green}` ([08-design-system.md §Компонент `ui/Pill`](../08-design-system.md#компонент-uipill)); тег-чип **не** подходит (он выводит цвет из HEX тега, у ящика цвета нет).

## Решение

### 1. Контракт: `MailAccountRef` расширяется тремя полями (аддитивно)

```json
"mail_account": {
  "id": 3,
  "email": "inbox@example.com",
  "display_name": "5108 Klyro Forge",
  "number": "5108",
  "app_name": "Klyro Forge",
  "team": { "id": "d3f0…a1", "name": "Команда Ивана" }
}
```

| Поле | Тип | Примечание |
|---|---|---|
| `number` | string \| null | «Номер» ящика (`mail_accounts.number`); `null` — не задан |
| `app_name` | string \| null | «Приложение» ящика (`mail_accounts.app_name`); `null` — не задано |
| `team` | `MailTeamRef` \| null | Команда-владелец ящика; **`null` — ящик без команды** (`team_id IS NULL`) |

`MailTeamRef = { id: string(uuid), name: string }` — новая схема, зеркало `SmsTeamRef` (`backend/app/schemas/sms.py:17-22`). Отдельный тип, а не переиспользование `SmsTeamRef`: модули не импортируют схемы друг друга.

- **Источник данных — LEFT JOIN `teams`** к уже выполняемой выборке ящика письма (`team_id` nullable ⇒ именно LEFT, INNER молча потерял бы письма бесхозных ящиков). **N+1 запрещён:** имя команды подтягивается тем же запросом страницы ленты, а не отдельным вызовом на письмо.
- `display_name` **остаётся** в схеме (обратная совместимость; десктоп его использует).
- **Утечки нет:** актор и так видит только письма ящиков своего `MailScope` ([ADR-055](ADR-055-per-channel-teams-mail-sms.md) §3) ⇒ имя команды/номер/приложение **уже доступного** ему ящика новой информации не раскрывают (тот же довод, что в [ADR-048](ADR-048-teams-mailbox-count-mail-row.md) §4 и [ADR-034](ADR-034-teams-number-login-app.md): слабо-чувствительный идентифицирующий контекст). Креды/статус синка в ленту по-прежнему **не** попадают.

### 2. Где это применяется

| Экран | Что делает |
|---|---|
| **Mini App `/tg/mail`** — карточка ленты и деталь письма | **Рендерит** новые поля (§3) — предмет этого ADR |
| Десктоп `/mail` (лента + деталь) | **НЕ меняется** (владелец просил Mini App). Поля контракта доступны, но не используются — это не долг, а граница scope |

### 3. UI Mini App (нормативно, [08-design-system.md](../08-design-system.md))

Образец — `SmsMessageCard`; примитив — существующий **`ui/Pill`** (**новый примитив НЕ вводится**).

**Карточка ленты (`MailMiniAppCard`) — порядок сверху вниз:**

1. Отправитель (`from_name`/`from_addr`) + **пилюля команды** + относительное время справа (существующая строка **дополняется** пилюлей).
   - команда есть → **`Pill tone="green"`**, `label = team.name`, `title = team.name`;
   - `team === null` → **`Pill tone="neutral"`**, `label` = **«Команды нет»** (строка — та же, что в SMS).
2. Тема письма (или «(без темы)») + индикатор непрочитанного ([ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)) — **без изменений**.
3. Теги-чипы (`tags[]`) — **без изменений**.
4. **Новый ряд пилюль ящика** (`flex-wrap`, переносится, обрезка значений **запрещена**):
   - **`Pill tone="accent"`**, `label` = **«Номер: {number}»**;
   - **`Pill tone="yellow"`**, `label` = **«Приложение: {app_name}»**.
   - **Пустое значение → `'-'`** (как `orDash` в SMS, `SmsMessageCard.tsx:8-11`): пилюля рендерится всегда, строка не «прыгает». (Это **осознанное отличие** от вкладки «Почты» десктопа, где пустая пара «лейбл+значение» **не рендерится** — там строка таблицы, здесь карточка-образец SMS, и владелец просил «как у смс».)
5. Строка **«Получено на: {email}»** (без `display_name` — см. ADR-reversal выше).

**Деталь письма (Mini App):** шапка получает **те же** пилюли (команда + «Номер»/«Приложение») и ту же строку «Получено на: {email}». Прочее (полная дата, тема, теги, тело в sandbox-iframe, «Назад», пометка прочитанным при открытии) — **без изменений**.

**Тон пилюль — тот же, что в SMS** (`accent` для первого идентификатора, `yellow` для приложения, `green`/`neutral` для команды): пользователь видит один визуальный язык в обеих Mini App.

## Последствия

- **Плюс:** сотрудник в Telegram видит, **на какой ящик и чьей команды** пришло письмо, не открывая CRM — то же, что уже даёт SMS Mini App.
- **Плюс:** `MailAccountRef` перестаёт быть «беднее» `SmsNumberRef`; десктоп при желании сможет показать те же атрибуты без нового контракта.
- **Минус:** запрос ленты писем получает ещё один LEFT JOIN (`teams`) — стоимость пренебрежима (команд десятки, соединение по PK).
- **Минус:** три новых поля идут **во все** ответы ленты (в т.ч. десктопу, который их не рендерит) — небольшой рост payload; выделять отдельную «облегчённую» схему ради этого не стали (две схемы одного объекта — источник рассинхрона).

## Альтернативы (отклонены)

1. **Резолвить команду на клиенте через `GET /api/teams`** — отклонено: гейт `teams:view` у mail-оператора может отсутствовать ⇒ в Mini App имя команды не отобразилось бы вовсе (ровно корень [TD-058](../100-known-tech-debt.md)).
2. **Парсить `number`/`app_name` из `display_name`** на клиенте — отклонено: `display_name` — **производная склейка** ([TD-052](../100-known-tech-debt.md)), обратный разбор неоднозначен (пробелы в `app_name`), а данные уже есть в БД отдельными колонками.
3. **Отдельный эндпоинт «контекст ящика»** (`GET /api/mail/mailboxes/{id}` для Mini App) — отклонено: N+1 на карточку ленты и второй источник истины о ящике; вложенный объект в письме — уже существующая идиома (`SmsMessageItem.number`).
4. **Лейблы «Код»/«Название»** (дословно как у владельца) — отклонено: расщепило бы словарь одного ящика между десктопом и Mini App; сопоставление зафиксировано в §Контекст.
