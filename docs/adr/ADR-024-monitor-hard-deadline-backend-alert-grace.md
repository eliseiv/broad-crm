# ADR-024 · Устойчивость мониторинга: жёсткий overall-deadline проверки прокси/бэка (анти-зависание) + 30-минутный grace-порог алерта недоступности бэков

- Статус: accepted
- Дата: 2026-07-08

> **Амендмент [ADR-027](ADR-027-proxies-alert-grace.md) (2026-07-08).** Пункт §2 «grace-порог **только** для бэков; прокси сохраняют немедленный алерт» **отменён**: по запросу пользователя (ложные срабатывания прокси) grace-модель (30 мин, персистентные `error_since`/`alert_sent`, time-aware `evaluate_transition`) распространена и на **прокси** — параметр `PROXY_ALERT_AFTER_SEC=1800`, миграция `0014_proxies_alert_grace`. Различие immediate/grace между прокси и бэками снято; прокси и бэки унифицированы. Тело этого ADR ниже сохранено как есть; актуальная модель прокси-алерта — [ADR-027](ADR-027-proxies-alert-grace.md). Слой проверки прокси (deadline/httpx.Timeout, §1) не затронут.

## Контекст

Две правки надёжности мониторинга по прямому запросу пользователя (правки 2б и 3б батча):

1. **Прокси «зависает» при проверке.** Конкретный прокси (`138.16.43.116`) «не проходит проверку — зависла»: healthcheck прокси может застревать бесконечно, не завершаясь ни `working`, ни `error`. Текущая модель ([ADR-019](ADR-019-proxies-availability-monitor.md), [modules/proxies](../modules/proxies/README.md#проверка-доступности-прокси-нормативно)) задаёт `PROXY_CHECK_TIMEOUT_SEC` (default 10 с) как таймаут `httpx` и ограниченные ретраи, но одиночный float-таймаут `httpx` **не гарантированно** покрывает все фазы соединения (особенно для `socks5` через `socksio`: SOCKS-handshake/connect-фаза может не соблюдать read-таймаут), а зависшая фаза внутри ретраев способна затянуть проверку неопределённо долго. Нужен **жёсткий верхний предел** времени одной проверки, после которого исход — гарантированно `error` по таймауту.

2. **Ложные алерты бэков при перезагрузке.** При штатной перезагрузке бэка он недоступен 1–2 минуты. Текущая модель ([ADR-020](ADR-020-backends-healthcheck-monitor.md)) шлёт 🔴 **на первом же** переходе `pending|working → error` (немедленно), из-за чего короткая недоступность при рестарте порождает ложную пару 🔴/🟢. Нужен **порог непрерывной недоступности** (30 минут) перед отправкой уведомления: короткие перезагрузки не тревожат, а реальная длительная недоступность — уведомляет.

Ограничения: монолит, один процесс, без брокеров (NFR-1, [ADR-001](ADR-001-stack-i-monolit.md)); интервал проверки прокси/бэков — 60 с ([modules/proxies](../modules/proxies/README.md), [modules/backends](../modules/backends/README.md)).

## Решение

### 1. Жёсткий overall-таймаут (deadline) одной проверки прокси и бэка

- **Явный `httpx.Timeout` по всем фазам.** `PROXY_CHECK_TIMEOUT_SEC` (default 10.0) и `BACKEND_CHECK_TIMEOUT_SEC` (default 10.0) применяются как **явный** `httpx.Timeout(connect=<t>, read=<t>, write=<t>, pool=<t>)` — не как одиночный float (чтобы connect/pool-фазы были ограничены явно, а не по умолчанию).
- **Абсолютный per-check deadline.** Проверка **одного** прокси/бэка (включая **все** внутренние ретраи и backoff) оборачивается жёстким верхним пределом `asyncio.wait_for(check_one(...), <deadline>)`. Новые параметры:
  - `PROXY_CHECK_DEADLINE_SEC` (default **30** с) — потолок полного времени проверки одного прокси;
  - `BACKEND_CHECK_DEADLINE_SEC` (default **30** с) — то же для бэка.
  - Превышение deadline (`asyncio.TimeoutError`) → исход **`error`**, `error_message = «Таймаут подключения»` (тот же текст, что при обычном таймауте). Проверка **всегда завершается** конклюзивно — зависаний нет.
- **Значения — согласованы с существующей моделью.** `deadline` 30 с < интервал 60 с: даже зависшая проверка гарантированно завершается **в пределах одного цикла** мониторинга, не накапливая «повисшие» задачи. 30 с покрывает ~3 ретрая × 10 с per-attempt с запасом (при этом каждый attempt дополнительно ограничен `*_CHECK_TIMEOUT_SEC`). Deadline — это защита-предохранитель на случай, если библиотека не соблюла per-attempt таймаут.

### 2. Grace-порог 30 минут перед алертом недоступности бэка

- **Разделение «статус для UI» и «отправка алерта».** `backends.check_status` продолжает переходить в `error` **немедленно** при первом провале проверки — карточка сразу показывает «Не работает» (реальность не скрывается). Но **Telegram-алерт 🔴 откладывается**: отправляется только если бэк недоступен **непрерывно ≥ `BACKEND_ALERT_AFTER_SEC`** (новый параметр, default **1800 с = 30 мин**).
- **Персистентные поля эпизода недоступности** (миграция `0013_backends_alert_grace`, [03-data-model.md](../03-data-model.md#таблица-backends)):
  - `backends.error_since timestamptz NULL` — момент начала **текущего непрерывного** эпизода недоступности (устанавливается при переходе `pending|working → error`; сбрасывается в `NULL` при `working`);
  - `backends.alert_sent boolean NOT NULL DEFAULT false` — отправлен ли уже 🔴 для текущего эпизода (защита от повторной отправки).
- **Логика (чистая функция перехода, time-aware)** `evaluate_transition(prev_status, result, error_since, alert_sent, now) -> (new_status, error_message, new_error_since, new_alert_sent, alert)`:
  - `result = error`: `new_status='error'`. Если `prev ∈ {pending, working}` → `new_error_since = now`, `alert_sent` остаётся `false`. Если `prev == error` → `error_since` сохраняется. Затем: если `(now − error_since) ≥ BACKEND_ALERT_AFTER_SEC` **и** `alert_sent == false` → `alert='error'` (🔴), `new_alert_sent = true`; иначе `alert=None` (тихо — grace-окно ещё не истекло).
  - `result = working`: `new_status='working'`, `new_error_since = NULL`. Если `alert_sent == true` → `alert='recovery'` (🟢, отбой ранее отправленного алерта); `new_alert_sent = false`. Если `alert_sent == false` (эпизод не «дозрел» до алерта, напр. рестарт < 30 мин) → `alert=None` (**тихо** — раз 🔴 не слали, 🟢 не нужен).
- **Персистентность переживает рестарт backend.** `error_since`/`alert_sent` в БД → grace-отсчёт и признак отправки корректны между рестартами (сломанный бэк не «переоткрывает» отсчёт, дубль-🔴 нет).
- **Только бэки.** Порог применяется к бэкам (правка адресна). Прокси сохраняют **немедленный** алерт ([ADR-019](ADR-019-proxies-availability-monitor.md)) — там проблема была в зависании (п.1), а не в ложных срабатываниях; отдельный запрос на grace для прокси не поступал.

## Обоснование

- **Deadline-предохранитель гарантирует завершаемость.** Даже если `httpx`/`socksio` не соблюли per-attempt таймаут, `asyncio.wait_for` жёстко прерывает проверку → всегда есть исход. Это минимальное, точечное изменение (обёртка вокруг существующего `check_one`), не трогающее матрицу переходов.
- **Grace через персистентные поля** — тот же приём, что durable-состояние нотификатора ([ADR-014](ADR-014-persist-notifier-state-alert-on-first-elevated.md), [ADR-018](ADR-018-notifier-windowed-offline-recovery-alert-log.md)): состояние эпизода в БД, чистая функция перехода тестируется без сети/времени (детерминированный `now`). Проще полноценного windowed-детекта (который был отложен в [TD-029](../100-known-tech-debt.md)) и решает конкретную боль — ложные алерты при рестартах.
- **UI не обманывается.** Статус в карточке — мгновенный; откладывается только уведомление. Пользователь всегда видит реальную доступность.
- **30 минут** — покрывает штатные рестарты/деплои/миграции (обычно ≪ 30 мин), но достаточно мало, чтобы реальная поломка уведомила своевременно. Значение — env-параметр, ужесточается/смягчается без кода.

## Последствия

- (+) Проверка прокси/бэка не зависает — всегда конклюзивный исход в пределах deadline (< интервала).
- (+) Штатные перезагрузки бэков (1–2 мин) больше не порождают ложные 🔴/🟢.
- (+) Grace-состояние переживает рестарт; чистая функция перехода тестируема.
- (−) Реальная поломка бэка уведомляется с задержкой до 30 мин (осознанный trade-off против шума; UI-статус — сразу).
- (−) Две новые колонки в `backends` (миграция `0013`) + новый параметр `BACKEND_ALERT_AFTER_SEC`; два новых `*_CHECK_DEADLINE_SEC`.
- (−) Прокси и бэки теперь различаются моделью алерта (прокси — немедленно, бэки — grace) — задокументировано; при желании grace можно распространить на прокси позже.

## Альтернативы

- **Только уменьшить `PROXY_CHECK_TIMEOUT_SEC`** — отвергнуто: не устраняет зависание, если библиотека не соблюдает per-attempt таймаут в connect/handshake-фазе; deadline-обёртка надёжнее.
- **Полноценный windowed-детект (`min_over_time` по образцу [ADR-018](ADR-018-notifier-windowed-offline-recovery-alert-log.md))** — отвергнуто на Этапе 1: сложнее (нужен ряд наблюдений), а persistent `error_since` + порог решает задачу проще ([TD-029](../100-known-tech-debt.md) остаётся для будущего сглаживания).
- **Считать grace в памяти (без БД)** — отвергнуто: рестарт backend сбросил бы отсчёт → сломанный бэк «переоткрыл» бы grace-окно (никогда не дозрел бы до алерта при частых рестартах); персистентность обязательна.
- **Задерживать и сам `check_status` (не только алерт) на 30 мин** — отвергнуто: UI обязан показывать реальную недоступность немедленно; откладывается только уведомление.
