# ADR-054 — Поле пароля ящика: лейбл «Код приложения» + перестановка под «Адрес почты» (порядок полей формы нормирован)

- **Статус:** accepted
- **Дата:** 2026-07-14
- **Контекст-модули:** [mail](../modules/mail/README.md), [ui](../modules/ui/README.md)

- **⚠️ Разворачивает / уточняет действующие нормы (ADR-reversal, оформлено явно):**
  - **[08-design-system.md §Страница «Почты» → «Создание/редактирование»](../08-design-system.md#вкладка-почты-нормативно)** ([ADR-047](ADR-047-mail-fix-pack.md) §3) — норма перечисляла состав полей модалки ящика («поля IMAP/SMTP (host/port/ssl/starttls, email, **пароль**, опц. SMTP-логин/пароль), двумя полями имени…»), но **не фиксировала ни лейблы, ни порядок**. Этим ADR **порядок полей и лейбл поля пароля становятся нормативными**; фактический лейбл кода `«Пароль (IMAP)»` (`frontend/src/components/MailboxFormModal.tsx:981`) **отменяется** → **«Код приложения»**, и поле **поднимается** с последней позиции формы **сразу под «Адрес почты»**.
  - Состав полей, presence-семантика `PATCH`, подсказки режима `edit`, аккордеон «Как добавить почту?», секция Outlook — **в силе целиком** ([ADR-045](ADR-045-mail-outlook-oauth-headless-reonboarding.md), [ADR-047](ADR-047-mail-fix-pack.md) §3).

- **НЕ затронуто (явно):** контракт API — **имя поля запроса остаётся `password`** (`MailMailboxTestRequest.password`, [04-api.md](../04-api.md#схема-mailmailbox)); `smtp_password` тоже не переименовывается; БД/миграции/RBAC/`MailScope` — не меняются. Правка **чисто frontend + docs**.

## Контекст

Требование владельца (батч 2026-07-14, дословно):

> «поле "Пароль (IMAP)" переименовать в "Код приложения" и переместить выше — чтобы оно шло сразу под "Адрес почты"»

**Мотив.** Почтовые провайдеры (Gmail, Яндекс, AOL, Yahoo, iCloud) не принимают обычный пароль учётной записи — нужен **app password** (пароль приложения). Лейбл «Пароль (IMAP)» провоцирует ввести пароль от почты: пользователь читает «пароль» и вводит пароль. Нормативный вводный абзац аккордеона это уже объясняет («Нужен **«пароль приложения»** (app password)» — [08-design-system.md](../08-design-system.md#инструкция-как-добавить-почту-нормативно)), но аккордеон **свёрнут по умолчанию**, а лейбл виден всегда. Лейбл — самая дешёвая точка коммуникации.

**Факты по коду (`claims-from-code`, на 2026-07-14):**

- Лейбл поля — `label="Пароль (IMAP)"` (`frontend/src/components/MailboxFormModal.tsx:981`), поле рендерится **последним** содержательным полем формы (после fieldset'ов IMAP и SMTP; ниже — только чекбокс «Активна» в режиме `edit`, `:995-1001`).
- Фактический порядок полей формы (`MailboxFormModal.tsx:711-1001`): аккордеон «Как добавить почту?» (`:711`) → **«Адрес почты»** (`:714`) → «Номер» (`:731`) + «Приложение» (`:739`) → «Команда» (`:748`) → секция Outlook (`:762`) → разделитель (`:845`) → fieldset **IMAP** (`«IMAP-хост»` `:880`, «Порт» `:893`, «SSL/TLS» `:904`) → fieldset **SMTP** (`«SMTP-хост»` `:921`, «Порт» `:934`, «Шифрование» `:945`, «SMTP-логин (опц.)» `:956`, «SMTP-пароль (опц.)» `:967`) → **«Пароль (IMAP)»** (`:981`).
- Поле связано с полем запроса **`password`** (`buildTestPayload` — `MailboxFormModal.tsx:402`), инлайн-ошибка `400` маппится по `d.field === 'password'` (`:459`). Второе, **опциональное** поле — `smtp_password` (`:971`, плейсхолдер «По умолчанию — основной пароль»), уходит в `smtp_password` (`:403`).
- В `docs/` строка «Пароль (IMAP)» **не встречается ни разу** (репо-wide grep 2026-07-14: единственные вхождения — `MailboxFormModal.tsx:981` и три qa-теста `frontend/src/components/__tests__/*`), т.е. лейбл никогда не был нормирован. Нормативным его делает этот ADR — уже в новой редакции.

## Решение

### 1. Лейбл поля основного пароля — **«Код приложения»**

Нормативная строка (`08-design-system.md`, словарь строк «Почты»): **«Код приложения»**. Прежний `«Пароль (IMAP)»` — **не использовать**.

- **Поле остаётся `type="password"`** (маскируется), `autoComplete="new-password"` — как сейчас.
- **Подсказка под полем (нормативно, обязательна):**
  - режим `add`: **«Не пароль от почты, а пароль приложения (app password) из настроек безопасности почтового сервиса. Как его получить — в блоке «Как добавить почту?» выше.»**
  - режим `edit`: прежняя подсказка **«Оставьте пустым, чтобы не менять пароль.»** (`MailboxFormModal.tsx:642`) заменяется на **«Оставьте пустым, чтобы не менять код приложения.»**
- **Инлайн-ошибка валидации** поля (`validate()`, `MailboxFormModal.tsx:566`): **«Укажите код приложения»** (было «Укажите пароль»).

### 2. Отличие от «SMTP-пароль (опц.)» — оба поля сохраняются, смысл разведён (нормативно)

Полей пароля в форме **два**, и они **не дублируют** друг друга:

| Поле (лейбл) | Поле запроса | Обязательность | Смысл |
|---|---|---|---|
| **«Код приложения»** | `password` | **обязателен** при создании | Основной секрет ящика: им CRM/агрегатор входит **и по IMAP, и по SMTP**. Для большинства провайдеров это **app password**, а не пароль учётной записи |
| **«SMTP-пароль (опц.)»** | `smtp_password` | опционален | Нужен **только** когда SMTP требует **отдельные** креды (не совпадающие с IMAP). Пусто → агрегатор берёт `password` |

- **Лейбл «SMTP-пароль (опц.)» НЕ меняется** (он не про app password, а про «отдельный секрет для SMTP»); переименовывать его в «Код приложения (SMTP)» **запрещено** — это склеило бы два разных понятия.
- **Плейсхолдер `smtp_password` приводится к новому словарю:** «По умолчанию — основной пароль» → **«По умолчанию — Код приложения»** (`MailboxFormModal.tsx:971`).
- Коллизии с полем **«Приложение»** (`app_name`, имя приложения-владельца ящика) нет по смыслу, но **есть по звучанию** — принято осознанно (требование владельца дословно). Митигация: поля стоят в **разных зонах** формы (см. §3: «Код приложения» — вторым полем, сразу под адресом; «Приложение» — в паре с «Номером» ниже) и различаются типом ввода (маскированный vs текстовый).

### 3. Порядок полей формы ящика (нормативно — реализовать РОВНО так)

| # | Поле / блок | Режим | Примечание |
|---|---|---|---|
| 1 | Аккордеон **«Как добавить почту?»** | только `add` | Свёрнут ([ADR-045](ADR-045-mail-outlook-oauth-headless-reonboarding.md)) |
| 2 | **«Адрес почты»** (`email`) | add + edit | |
| 3 | **«Код приложения»** (`password`) | add + edit | **⬅ перемещено сюда** (было последним). Подсказка — §1 |
| 4 | **«Номер»** (`number`) + **«Приложение»** (`app_name`) | add + edit | Одна строка, как сейчас ([ADR-047](ADR-047-mail-fix-pack.md) §3) |
| 5 | **«Команда»** (`team_id`) | add + edit | Гейт переноса — [ADR-044](ADR-044-mail-full-merge-into-crm.md) §4 |
| 6 | Секция **«Outlook (OAuth)»** + разделитель «или добавьте ящик вручную (IMAP / пароль)» | только `add` | [ADR-045](ADR-045-mail-outlook-oauth-headless-reonboarding.md) |
| 7 | Fieldset **IMAP**: «IMAP-хост», «Порт», «SSL/TLS» | add + edit | |
| 8 | Fieldset **SMTP**: «SMTP-хост», «Порт», «Шифрование», «SMTP-логин (опц.)», «SMTP-пароль (опц.)» | add + edit | |
| 9 | **«Активна»** (`is_active`) | только `edit` | |

- **Блок сообщения об отказе подключения** (`connectionError`, [ADR-053](ADR-053-mail-timeouts-error-passthrough.md) §2/§4) остаётся **перед** fieldset'ом IMAP — рядом с полями подключения (позиция не меняется).
- Разделитель «или добавьте ящик вручную (IMAP / пароль)» текстуально **не меняется** (в нём «пароль» — родовое слово, а не лейбл поля).

### 4. Что НЕ меняется

- **Контракт API** — поле запроса называется **`password`** и остаётся им (`04-api.md`): переименование **UI-лейбла** не переименовывает поле DTO. Инлайн-маппинг `400 validation_error` по `details[].field == "password"` → показать под полем **«Код приложения»**.
- Валидация, presence-семантика `PATCH` («пусто = не менять»), транзит кредов в агрегатор, `Cache-Control: no-store`, таймауты ([ADR-053](ADR-053-mail-timeouts-error-passthrough.md)) — без изменений.

## Последствия

- **Плюс:** самое частое поражение при заведении ящика (ввели пароль от почты → `422 mail_imap_failed`) адресуется до отправки формы: лейбл называет то, что нужно ввести, и стоит вторым полем — прямо под адресом, до которого пользователь дочитает гарантированно.
- **Минус:** в форме соседствуют «Код приложения» (секрет) и «Приложение» (имя приложения) — созвучные лейблы. Принято по прямому требованию владельца; митигация — разнесение по зонам формы и разный тип ввода.
- **Тесты qa ломаются** — три файла обращаются к полю по старому лейблу (`getByLabelText('Пароль (IMAP)')`: `frontend/src/components/__tests__/MailboxFormModal.test.tsx:225`, `.../MailboxFormModalTimeouts.test.tsx:114`, `:216`). Это **штатный хендофф на `qa`** (зона тест-файлов), а не дефект исполнителя.

## Альтернативы (отклонены)

1. **«Пароль приложения»** вместо «Код приложения» — точнее терминологически (везде в инструкции говорится «пароль приложения»), но владелец назвал строку дословно. Отклонено: нормативный лейбл задаёт владелец.
2. **Оставить лейбл, усилить подсказку** — подсказка под полем не спасала: поле было **последним** в форме и часто заполнялось «на автомате» первым попавшимся паролем.
3. **Переименовать и `smtp_password`** — отклонено: это другое понятие (см. §2), склейка увеличила бы путаницу, а не уменьшила.
