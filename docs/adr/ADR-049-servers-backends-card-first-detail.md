# ADR-049 — UI-пакет: `/servers` — креды в главном блоке, блок «Информация» упразднён, «Бэки» переезжают на карточку; `/backends` — «Информация» на карточке, detail-модалка упразднена

- **Статус:** accepted
- **Дата:** 2026-07-13
- **Контекст-модули:** [servers](../modules/servers/README.md), [backends](../modules/backends/README.md), [ui](../modules/ui/README.md)
- **Разворачивает (супессия части нормы):** [ADR-046](ADR-046-ui-infra-fix-pack.md) **§2в** — строки таблицы «Состав видимой зоны» для **Сервера** и **Бэка** (см. §0)
- **Амендмент:** [ADR-035](ADR-035-detail-view-secret-reveal.md) (для `/backends` паттерн «клик по карточке → read-only detail-модалка → карандаш» **отменяется**; для `/servers`, `/proxies`, `/ai-keys` — **в силе**), [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) **§5** (место жительства секции «Бэки» сервера: detail-модалка → карточка; сам эндпоинт reverse-lookup и `backend_count` **не меняются**)

## Контекст

Требования владельца (батч 2026-07-13):

3. **`/servers`:** «Пользователь и пароль должны отображаться не в блоке Информация, а в главном блоке, где Название и IP. Блок Информация вообще убрать. Список бэков перенести на главную страницу: внизу каждой карточки сервера писать "Бэки: N", при раскрытии — список бэков этого сервера в формате Код + Название + Домен».
4. **`/backends`:** «Внизу карточки бэка будет поле Информация, при раскрытии которой будет писаться вся информация».

Оба требования **разворачивают действующую норму** [ADR-046](ADR-046-ui-infra-fix-pack.md) §2в: «при открытии `<Entity>DetailModal` видны только идентификаторы; всё остальное — внутри свёрнутого по умолчанию блока «Информация»», причём секция «Бэки» — **вложена внутрь** «Информации». Смысл разворота: два месяца эксплуатации показали, что для **сервера** креды (`ssh_user` + пароль) — это не «дополнительная информация», а **основной рабочий контекст** (за ними и открывают карточку), а перечень бэков нужен **до** открытия модалки, при беглом просмотре сетки. Для **бэка** вся «дополнительная» информация и так уже загружена вместе со списком (`GET /api/backends` → `BackendListItem`), поэтому промежуточный экран-модалка ничего не добавляет.

**Что уже есть в коде (сверено, `claims-from-code`):**

- **Detail-модалка сервера** — `ServerDetailModal` (`frontend/src/components/ServerDetailModal.tsx`): главный блок = `DetailRow label="Название"` (или inline-edit `Input`, [ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) + `DetailRow label="IP"`; далее свёрнутый `DetailInfoSection` (лейбл `Информация` — `frontend/src/components/DetailFields.tsx:82`), внутри которого `DetailRow label="Пользователь"` (`server.ssh_user`), `SecretRevealField label="Пароль"` и `BackendsDetailSection`.
- **Секция «Бэки»** — `BackendsDetailSection` (`frontend/src/components/BackendsDetailSection.tsx`): свёрнутая, показывает `Бэков: {count}` по `backend_count`, при раскрытии — ленивый `GET /api/servers/{id}/backends` → строки Код/Название/Домен (`BackendRefRow`).
- **`backend_count` УЖЕ в list-схеме сервера:** `ServerListItem.backend_count: int` (`backend/app/schemas/server.py:85`), считается `BackendRepository.count_by_servers` в `ServerService.list_servers`. **Нового поля/запроса для счётчика на карточке не требуется.**
- **Reverse-lookup эндпоинт УЖЕ существует:** `GET /api/servers/{server_id}/backends` → `BackendRefListResponse { backends: [BackendRef{code,name,domain}] }`, гейт `require("servers","view")` (`backend/app/api/servers.py:104-109`; схемы — `backend/app/schemas/backend.py:102-117`). Хук фронта — `useServerBackends(id, enabled)` (`frontend/src/features/servers/hooks.ts`), **ленивый** (`enabled` = секция раскрыта). **Нового эндпоинта не требуется.**
- **Секреты сервера:** `servers.ssh_user` и `servers.ssh_password_encrypted` — **оба `nullable=False`** (`backend/app/models/server.py:55-56`). Флага `has_password` у сервера **нет и не нужен** (в отличие от прокси/бэка): пароль у сервера **всегда задан**.
- **Detail-модалка бэка** — `BackendDetailModal` (`frontend/src/components/BackendDetailModal.tsx`): главный блок = Код/Название/Домен; свёрнутый `DetailInfoSection` = Сервер / ИИ-ключ / API KEY (reveal) / ADMIN API KEY (reveal) / Git / Примечания. **Все эти значения уже приходят в `BackendListItem`** (`backend/app/schemas/backend.py:68-93`: `server_name`, `ai_key_name`, `has_api_key`, `has_admin_api_key`, `git`, `note`) — модалка **не делает ни одного дополнительного запроса** за ними.
- **Карточка бэка** — `BackendCard` (`frontend/src/components/BackendCard.tsx`): иконка + `name` + статус-бейдж, `code` (моно), `domain`, ошибка, время проверки, кнопка «Удалить» (гейт `backends:delete`). Клик по карточке → `BackendDetailModal`; карандаш живёт в шапке модалки и открывает `AddBackendModal mode="edit"`.
- **DnD:** на `/servers` — **есть** (`SortableItem` в `ServersPage.tsx`, [ADR-011](ADR-011-poryadok-blokov-server-side-dnd-kit.md)); на `/backends` — **убран** ([ADR-046](ADR-046-ui-infra-fix-pack.md) §2а).
- **Готовый card-level accordion-паттерн уже есть:** `/teams` (`frontend/src/pages/TeamsPage.tsx`) — `<Card>`, чей заголовок несёт `role="button"`/`aria-expanded`/`aria-controls` + `ChevronDown` с `rotate-180`, а тело панели монтируется только при раскрытии. Общего примитива `Collapsible` в `components/ui/` **нет** — паттерн повторяется вручную (`DetailInfoSection`, `BackendsDetailSection`, `CollapsibleSection` в `TeamDetailPanel.tsx`, `MailHelpAccordion`).

## Решение

### §0. Что именно разворачивается в [ADR-046](ADR-046-ui-infra-fix-pack.md) §2в (нормативно)

Таблица «Состав видимой зоны» [ADR-046](ADR-046-ui-infra-fix-pack.md) §2в **заменяется** следующей (изменённые строки помечены):

| Модалка | Видно сразу | Внутри «Информации» |
|---------|-------------|---------------------|
| **Бэк** ⚠️ | — **detail-модалка `BackendDetailModal` УПРАЗДНЯЕТСЯ** (§3) | — (блок «Информация» переезжает **на карточку**, §3) |
| **Сервер** ⚠️ | **Название**, **IP**, **Пользователь** (`ssh_user`), **Пароль** (reveal) | — (**блока «Информация» в `ServerDetailModal` больше НЕТ**; секция «Бэки» переезжает **на карточку**, §2) |
| ИИ-ключ | **Название**, **Провайдер** | **Ключ** (`key_masked` + reveal), сворачиваемая секция **«Бэки»** — **БЕЗ ИЗМЕНЕНИЙ** |
| Прокси | **Название**, **Хост**, **Порт** | Тип, Логин, **Пароль** (reveal) — **БЕЗ ИЗМЕНЕНИЙ** |

Остальные положения [ADR-046](ADR-046-ui-infra-fix-pack.md) §2в (карандаш в шапке модалки под `<page>:edit`; reveal не ослабляется; «если внутри «Информации» не осталось строк — блок не рендерится») действуют для **ИИ-ключа** и **Прокси** без изменений. Норма [ADR-046](ADR-046-ui-infra-fix-pack.md) **§3** («пустые поля detail-view не рендерятся; прочерк «—» упразднён») **не отменяется** и распространяется на новый блок «Информация» карточки бэка (§3).

### §1. `/servers` — detail-модалка: главный блок = Название + IP + Пользователь + Пароль; блок «Информация» упразднён

- **`ServerDetailModal` показывает четыре строки сразу, без сворачивания:** **Название** (с inline-edit по карандашу — [ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md), норма не меняется), **IP** (моно), **Пользователь** (`ssh_user`, моно), **Пароль** (`SecretRevealField`, маска `••••••••` + глаз).
- **Свёрнутый блок «Информация» (`DetailInfoSection`) в `ServerDetailModal` УПРАЗДНЯЕТСЯ полностью.** Компонент `DetailInfoSection` (`frontend/src/components/DetailFields.tsx`) **НЕ удаляется** — он остаётся в употреблении у `AiKeyDetailModal` и `ProxyDetailModal` (их норма не менялась). Снимается только его использование в `ServerDetailModal`.
- **Секция «Бэки» (`BackendsDetailSection`) из `ServerDetailModal` УБИРАЕТСЯ** — переезжает на карточку (§2). Компонент `BackendsDetailSection` **НЕ удаляется**: его продолжает использовать `AiKeyDetailModal` ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §5, норма для ИИ-ключа не менялась). Дублировать список бэков и на карточке, и в модалке — **запрещено** (одна и та же информация в двух местах на одной странице).
- **Порядок строк (нормативно):** Название → IP → Пользователь → Пароль. Обе новые строки рендерятся **всегда**: `ssh_user` и `ssh_password_encrypted` — `NOT NULL` в модели (`backend/app/models/server.py:55-56`), поэтому правило [ADR-046](ADR-046-ui-infra-fix-pack.md) §3 «пустое поле не рендерится» здесь **не срабатывает никогда**. **Флаг `has_password` у сервера НЕ вводится** (у него нет смысла — пароль обязателен; это отличие от прокси/бэка, где секрет опционален).
- **Контракт API не меняется:** `ssh_user` уже отдаётся в `ServerListItem` (`backend/app/schemas/server.py:74`); пароль по-прежнему **не отдаётся** ни в каком списочном ответе — только reveal-эндпоинтом (§4).

### §2. `/servers` — «Бэки: N» внизу КАРТОЧКИ сервера, ленивый список по раскрытию

- **Внизу каждой `ServerCard`** — сворачиваемая секция-триггер со строкой **«Бэков: N»**, где `N` = **`server.backend_count`** (`ServerListItem.backend_count`, уже в ответе `GET /api/servers` — [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §4). **Дополнительного запроса на счётчик нет.**
  - **Нормативная строка — «Бэков: N»** (родительный падеж, ровно как в уже существующем `BackendsDetailSection`), **НЕ** «Бэки: N». Владелец написал «Бэки: N» описательно; берётся **уже действующая** строка словаря дизайн-системы, чтобы одна и та же секция не называлась на странице и в модалке ИИ-ключа по-разному.
- **Загрузка списка — ТОЛЬКО по раскрытию** (нормативно): при первом раскрытии секции вызывается **уже существующий** `GET /api/servers/{id}/backends` (гейт `require("servers","view")`, [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §4) — тем же ленивым хуком `useServerBackends(id, enabled)`, что сейчас питает `BackendsDetailSection`. **Преднагрузка списков бэков при рендере сетки серверов ЗАПРЕЩЕНА** — это дало бы N запросов на N карточек (ровно тот N+1, который [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) и отклонял). Свёрнутая карточка не делает ни одного запроса.
- **Раскрытый список** — строки **Код + Название + Домен** (`BackendRef{code,name,domain}`), порядок `position ASC, created_at DESC, id` (контракт [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §4 — **не меняется**). Состояния: loading (спиннер/skeleton), empty — **«Бэков нет»**, error — **«Не удалось загрузить»** + «Повторить» (строки уже существуют в `BackendsDetailSection`).
- **`backend_count = 0` → строка «Бэков: 0» рендерится, но секция НЕ раскрывается** (нет chevron, нет `role="button"`, нет запроса) — это информативный счётчик, а не пустой аккордеон. Правило [ADR-046](ADR-046-ui-infra-fix-pack.md) §3 («пустое → не рендерить») здесь **неприменимо**: оно нормативно ограничено **detail-view** (прямая цитата §3: «Правило действует **только** в detail-view»), а скрытие счётчика делало бы высоту карточек в сетке неоднородной.
- **Разведение жестов на карточке сервера (нормативно, обязательно).** На `/servers` карточка **одновременно**: (а) кликабельна целиком → `ServerDetailModal` ([ADR-035](ADR-035-detail-view-secret-reveal.md)); (б) является drag-ручкой DnD ([ADR-011](ADR-011-poryadok-blokov-server-side-dnd-kit.md)). Поэтому триггер «Бэков: N» обязан:
  1. быть **собственным `<button>`** с `aria-expanded`/`aria-controls` (паттерн `TeamsPage.tsx` / `BackendsDetailSection`), а не `div` внутри кликабельной области;
  2. **останавливать всплытие** события клика/`keydown` (`stopPropagation`) — раскрытие секции **не должно открывать `ServerDetailModal`**;
  3. **не быть частью drag-ручки** — нажатие на триггер и на строки раскрытого списка **не инициирует перетаскивание** карточки (сенсор DnD не должен захватывать этот подэлемент).
  Нарушение любого из трёх пунктов = функциональный дефект (`major`), а не косметика: пользователь не сможет ни раскрыть список, ни закрыть его, не открыв модалку.
- **Секция «Бэки» на карточке ИИ-ключа НЕ вводится** — владелец просил только про серверы; у ИИ-ключа секция остаётся в detail-модалке ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §5, норма в силе).

### §3. `/backends` — блок «Информация» внизу КАРТОЧКИ; `BackendDetailModal` упраздняется

- **Внизу каждой `BackendCard`** — сворачиваемый (свёрнут по умолчанию) блок **«Информация»** с тем же составом, что был в `BackendDetailModal` ([ADR-046](ADR-046-ui-infra-fix-pack.md) §2в, [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §5) и **в том же порядке**:
  1. **Сервер** (`server_name`)
  2. **ИИ-ключ** (`ai_key_name`)
  3. **API KEY** — `SecretRevealField`, маска + глаз, только при `has_api_key = true`
  4. **ADMIN API KEY** — `SecretRevealField`, маска + глаз, только при `has_admin_api_key = true`
  5. **Git** (`git`, ссылка)
  6. **Примечания** (`note`)
- **Пустые поля не рендерятся** ([ADR-046](ADR-046-ui-infra-fix-pack.md) §3 — **в силе, распространяется на этот блок**): `null`/пустая строка → строки нет; прочерк «—» не используется. **Если внутри «Информации» не остаётся ни одной строки** (бэк без связей, секретов, git и примечаний) — **блок «Информация» на карточке не рендерится вовсе** (ни триггера, ни счётчика; прямое следствие §2в-правила [ADR-046](ADR-046-ui-infra-fix-pack.md), перенесённое на карточку).
- **Дополнительных запросов при раскрытии НЕТ.** Все шесть значений **уже** пришли в `BackendListItem` вместе со списком (`server_name`, `ai_key_name`, `has_api_key`, `has_admin_api_key`, `git`, `note` — `backend/app/schemas/backend.py:68-93`). Раскрытие — чисто клиентское разворачивание уже имеющихся данных. **Секреты при этом НЕ преднагружаются** (см. §4).
- **`BackendDetailModal` УПРАЗДНЯЕТСЯ** (файл `frontend/src/components/BackendDetailModal.tsx` удаляется; клик по карточке больше **не** открывает detail-модалку).
  - **Почему.** После переезда «Информации» на карточку модалка показывала бы **ровно то же самое**: Код/Название/Домен уже видны на карточке, «Информация» — тоже на карточке. Оставить обе = один и тот же контент в двух местах + конфликт жестов (клик по карточке открывает модалку, а раскрытие блока требует `stopPropagation`, и пользователь не понимает, куда он кликает).
  - **Карандаш (edit) переезжает В БЛОК ДЕЙСТВИЙ КАРТОЧКИ** — рядом с уже существующей кнопкой «Удалить»: иконка `Pencil`, гейт **`backends:edit`** (без права — не рендерится), клик → `AddBackendModal mode="edit"` (та же модалка формы, что и сегодня). Гейт удаления — `backends:delete` (без изменений).
  - **Клик по телу карточки `/backends` больше не открывает ничего** (нет detail-модалки). Это **не** регресс: на `/backends` DnD уже убран ([ADR-046](ADR-046-ui-infra-fix-pack.md) §2а), поэтому разводить «клик vs зажатие» здесь не нужно и жестовых конфликтов не остаётся.
  - **⚠️ С ТЕЛА карточки СНИМАЮТСЯ все интерактивные семантики паттерна [ADR-035](ADR-035-detail-view-secret-reveal.md) (нормативно, a11y).** Тело `BackendCard` становится **обычным неинтерактивным контейнером**: снимаются `role="button"`, `tabIndex={0}`, `onClick` открытия detail, `cursor-pointer`, focus-ring карточки. Оставить их при отсутствующем действии — **дефект доступности** (`major`): скринридер анонсирует кнопку, клавиатурный фокус на неё встаёт, нажатие не даёт ничего. **Интерактивны только:** триггер блока «Информация» (`<button>` + `aria-expanded`/`aria-controls`), карандаш (`backends:edit`), «Удалить» (`backends:delete`), глаз-reveal, ссылка Git — каждый со своим focus-ring и доступный с клавиатуры. `stopPropagation` на них больше **не требуется** (обработчика на теле нет), но и не вредит.
  - **Это амендмент [ADR-035](ADR-035-detail-view-secret-reveal.md) ТОЛЬКО для `/backends`.** Паттерн «клик по карточке → read-only detail-модалка → карандаш в её шапке → edit» **остаётся в силе** для `/servers`, `/proxies`, `/ai-keys`.
- **Поиск по бэкам** ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md), плейсхолдер «Поиск по бэкам…»), **плоская сетка и сортировка по `name`** ([ADR-046](ADR-046-ui-infra-fix-pack.md) §2а), **кнопка «Добавить» в правой зоне заголовка** ([ADR-046](ADR-046-ui-infra-fix-pack.md) §2б) — **не меняются**.

### §4. Секреты: гарантии reveal НЕ ослабляются (нормативно)

Перенос строки секрета из **свёрнутого** блока в **видимую** зону (сервер, §1) и на **карточку в сетке** (бэк, §3) меняет **только место строки-маски**. Момент и условия раскрытия значения — **прежние** ([ADR-035](ADR-035-detail-view-secret-reveal.md), [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) §2):

| Гарантия | Как держится после этого ADR |
|----------|------------------------------|
| **Секрет не преднагружается** | Ни `GET /api/servers`, ни `GET /api/backends` секретов не отдают (`ServerListItem` их не содержит; `BackendListItem` отдаёт только `has_api_key`/`has_admin_api_key`). Раскрытие блока «Информация» на карточке **не делает ни одного запроса** — рендерится маска `••••••••` по флагу `has_*` |
| **Значение — только по клику на глаз** | Запрос `GET /api/servers/{id}/ssh-password` / `GET /api/backends/{id}/api-key` / `/admin-api-key` идёт **только** из обработчика клика `SecretRevealField` (`frontend/src/components/DetailFields.tsx`), по одному ресурсу за раз |
| **Гейт** | `require("servers","edit")` (`backend/app/api/servers.py:32`) / `require("backends","edit")` (`backend/app/api/backends.py:26`) — **не меняются**. Кнопка-глаз не рендерится без права (`canReveal={canEdit}`) |
| **`Cache-Control: no-store`** | Ставится обработчиком reveal-эндпоинта — **не меняется** |
| **Аудит** | `log_secret_revealed(principal, resource_type="server"\|"backend", resource_id=…)` (`backend/app/infra/audit.py`) — **не меняется**; значение не логируется |
| **Значение не кэшируется на клиенте** | Живёт только в локальном стейте `SecretRevealField`, стирается при скрытии/размонтировании — **не меняется** |

**Ключевое ограничение (нормативно): раскрытие блока «Информация» на карточке НЕ ДОЛЖНО приводить к массовой преднагрузке секретов.** Запрещено: батч-эндпоинт раскрытия, `useQueries` по всем карточкам, авто-reveal при раскрытии, «раскрыть все». Один клик по глазу = один запрос по одному ресурсу. Раскрытие «Информации» у 20 бэков подряд по-прежнему даёт **ноль** обращений к reveal-эндпоинтам.

### §5. Frontend — состав правок (ориентир, структура на усмотрение исполнителя)

- `ServerDetailModal.tsx` — убрать `DetailInfoSection` и `BackendsDetailSection`; поднять `DetailRow "Пользователь"` и `SecretRevealField "Пароль"` в главный блок.
- `ServerCard.tsx` — добавить секцию-триггер «Бэков: N» + ленивый список (переиспользовать `useServerBackends` и разметку строк `BackendRefRow`); развести жесты (§2).
- `BackendCard.tsx` — добавить свёрнутый блок «Информация» (состав §3) + карандаш `Pencil` (гейт `backends:edit`) в блок действий; снять открытие detail-модалки по клику.
- `BackendDetailModal.tsx` — **удалить файл**; снять его импорт/использование в `BackendCard.tsx`.
- **Общего примитива `ui/Collapsible` этот ADR НЕ вводит** (его в `components/ui/` нет). Переиспользуется уже существующий ручной паттерн: `<button aria-expanded aria-controls>` + `ChevronDown` с `rotate-180`, содержимое монтируется только при раскрытии (образцы — `BackendsDetailSection.tsx`, `TeamsPage.tsx`). Заведение общего примитива — отдельная задача рефакторинга, вне scope этого ADR ([TD-057](../100-known-tech-debt.md)).

### §6. Судьба потребителей удаляемых/переносимых символов (repo-wide, нормативно)

Прогнан repo-wide grep (включая зоны **вне** CI-type-check/тест-scope) по удаляемому символу `BackendDetailModal` и по переносимым `DetailInfoSection`/`BackendsDetailSection`:

| Потребитель | Зона | Судьба | Владелец |
|-------------|------|--------|----------|
| `frontend/src/components/BackendDetailModal.tsx` | prod-код | **Удалить файл** | `frontend` |
| `frontend/src/components/BackendCard.tsx` (импорт + рендер `BackendDetailModal`) | prod-код | **Переписать** по §3 | `frontend` |
| `frontend/src/components/ServerDetailModal.tsx` (`DetailInfoSection`, `BackendsDetailSection`) | prod-код | **Переписать** по §1 | `frontend` |
| `frontend/src/components/DetailFields.tsx` → `DetailInfoSection` | prod-код | **НЕ удалять** — остаётся у `AiKeyDetailModal`/`ProxyDetailModal` | — |
| `frontend/src/components/BackendsDetailSection.tsx` | prod-код | **НЕ удалять** — остаётся у `AiKeyDetailModal` | — |
| `frontend/src/components/__tests__/DetailModals.test.tsx` (ассерты на `BackendDetailModal` и на свёрнутую «Информацию» сервера) | **тесты** | **Переписать/удалить** под новый контракт — тесты на удалённую модалку не компилируются | **`qa`** (исполнитель кода тест-файлы **не трогает**) |

**Гарантия полноты снятия зависимостей — МАШИННЫЙ ГЕЙТ, а не этот перечень.** Перед сдачей обязателен зелёный прогон **всех** gating-jobs фронта ровно так, как их гоняет CI (команды — из `.github/workflows/*` / [07-deployment.md](../07-deployment.md)): lint + **type-check (`tsc -b` по всему проекту, включая тест-файлы)** + полный прогон тестов + build. **Любая висячая ссылка на удалённый `BackendDetailModal` = красный type-check = СТОП до прода.** Таблица выше — guidance по порядку и по судьбе зон вне гейта, она **не объявляется исчерпывающей**.

**Хендофф на `qa` (обязателен, [CLAUDE.md](../../CLAUDE.md) «Orphaned qa-тесты»):** `frontend` перечисляет некомпилируемые orphaned-тесты в `follow_up_for_qa` и **не правит их сам**; orchestrator направляет `qa` на их переписывание **до** финального build-гейта.

## Последствия

- **Миграций нет. Контракты API не меняются** — ни одного нового поля, эндпоинта или кода ошибки. Всё, что нужно (`ServerListItem.backend_count`, `GET /api/servers/{id}/backends` + `BackendRef`, `BackendListItem.*`, три reveal-эндпоинта), **уже существует** ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)). Изменения — **чисто frontend**.
- **Backend-агентам делать нечего** по этому ADR (кроме отсутствия правок). Задача целиком у `frontend` + `qa`.
- **Производительность `/servers` не деградирует:** свёрнутая карточка = 0 запросов; счётчик уже в списке; список бэков грузится **только** по раскрытию, по одному серверу. Сценарий «раскрыть все карточки» даёт ровно столько запросов, сколько карточек раскрыл пользователь — и это его явное действие, а не рендер страницы.
- **`/backends` теряет промежуточный экран:** одна страница, одна карточка, вся информация на месте. Число кликов до данных — с двух (клик → модалка → раскрыть «Информацию») до одного.
- **Гарантии секретов не ослаблены** (§4): преднагрузки нет, гейт `<page>:edit` на месте, `no-store` и аудит не тронуты.
- **Расхождение UX между страницами (принимается осознанно):** `/servers`, `/proxies`, `/ai-keys` сохраняют паттерн detail-модалки ([ADR-035](ADR-035-detail-view-secret-reveal.md)), `/backends` — переходит на card-first. Унификация всех четырёх страниц под card-first **не делается**: владелец просил разное, а у сервера/прокси/ключа в модалке есть то, чего нет в списке (inline-edit имени сервера, reveal-строки, у ключа — секция «Бэки»), т.е. модалка там несёт смысл. Косметическая однородность не стоит четырёх разворотов нормы. Остаточная неоднородность — [TD-057](../100-known-tech-debt.md).
- **QA:** тесты `DetailModals.test.tsx` (ассерты на `BackendDetailModal` и на свёрнутую «Информацию» сервера) требуют переписывания — см. §6.

## Альтернативы (отклонены)

1. **Оставить `BackendDetailModal` и добавить «Информацию» на карточку.** Отклонено: один и тот же контент в двух местах + жестовый конфликт (клик по карточке открывает модалку, а триггер блока обязан гасить всплытие — пользователь не понимает, куда кликает). Модалка после переезда «Информации» не показывала бы **ничего**, чего нет на карточке.
2. **Перенести список бэков сервера на карточку, но оставить его и в `ServerDetailModal`.** Отклонено по той же причине (дублирование). Владелец сказал «перенести», а не «продублировать».
3. **Преднагружать списки бэков для всех карточек сервера одним батч-эндпоинтом** (`GET /api/servers/backends?server_id=…&server_id=…`). Отклонено: новый эндпоинт + контракт ради данных, которые в 95% случаев никто не раскроет; [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) уже осознанно выбрал «дешёвый счётчик в списке + ленивый список по раскрытию» — эта модель переносится на карточку **как есть**.
4. **Ввести общий примитив `ui/Collapsible` в рамках этого ADR.** Отклонено: пять мест уже используют ручной паттерн; заведение примитива = рефакторинг пяти компонентов и их тестов, что раздувает пакет фиксов владельца. Зафиксировано долгом [TD-057](../100-known-tech-debt.md).
5. **Скрывать строку «Бэков: 0» при нулевом счётчике.** Отклонено: разная высота карточек в сетке; правило «пустое не рендерится» [ADR-046](ADR-046-ui-infra-fix-pack.md) §3 нормативно ограничено detail-view и на карточки не распространяется.
6. **Ввести `has_password` в `ServerListItem` ради симметрии с прокси/бэком.** Отклонено: `servers.ssh_password_encrypted` — `NOT NULL` (`backend/app/models/server.py:56`), пароль **всегда** есть; флаг был бы константой `true` — мёртвое поле контракта.
