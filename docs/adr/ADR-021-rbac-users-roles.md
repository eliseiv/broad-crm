# ADR-021 · Пользователи, роли и RBAC поверх `.env`-супер-админа

- Статус: accepted
- Дата: 2026-07-07
- Спринт: 3
- Расширяет / уточняет: [ADR-008](ADR-008-admin-iz-env.md) (амендмент — см. ниже), [ADR-002](ADR-002-dvuhshagovyy-auth.md) (JWT-payload)

## Контекст

До Спринта 3 система — **однопользовательская**: единственный администратор берётся из `.env` (`ADMIN_USER`/`ADMIN_PASSWORD`), сравнение plaintext через `secrets.compare_digest` ([ADR-008](ADR-008-admin-iz-env.md)); JWT несёт только `sub`; зависимость `get_current_user` возвращает голый `str` username; таблиц `users`/`roles` нет; ролей в токене нет; нет `403`/`forbidden()`; хэширования паролей нет (нет `bcrypt`). Ресурсные роутеры (`servers`/`ai-keys`/`mail`/`proxies`/`backends`) защищены пер-эндпоинт зависимостью `_user: CurrentUser` — «любой аутентифицированный».

Требуется многопользовательский режим с ролями и правами на **все** страницы, при этом:
- `.env`-админ **остаётся** и становится несменяемым **супер-админом** (bootstrap): всегда полный доступ, роль `admin` неявно, из UI не редактируется и не удаляется, в таблицу `users` не попадает;
- дополнительные пользователи и роли — в БД;
- права — на каждую страницу (каталог «страница × действия»); страница «Пользователи» — вне матрицы, доступна только роли `admin` / супер-админу;
- ники допускают **кириллицу и другие юникод-буквы** («Админ», «Никита»).

## Решение

### 1. Каталог прав (канон на сервере)

Единственный источник истины прав — серверная константа `app/domain/permissions.py::CATALOG`. Каталог «страница → допустимые действия»:

| Страница (`page`) | Действия (`actions`) |
|-------------------|----------------------|
| `dashboard` | `view` |
| `servers` | `view`, `create`, `edit`, `delete` |
| `ai-keys` | `view`, `create`, `edit`, `delete` |
| `proxies` | `view`, `create`, `edit`, `delete` |
| `backends` | `view`, `create`, `edit`, `delete` |
| `mail` | `view` |

- Ключи `page` совпадают со слагами маршрутов SPA (`ai-keys` — с дефисом).
- Страница **«Пользователи» в каталог НЕ входит** — управление пользователями/ролями гейтится отдельно (`require_admin`, см. §5), а не через матрицу.
- Каталог отдаётся UI через `GET /api/permissions/catalog` (админ-гейт) для построения матрицы; порядок страниц в ответе — как в таблице выше.

**Формат прав роли (`permissions`, jsonb):** объект `{ "<page>": ["<action>", ...], ... }`. Пустой массив = страница видна в каталоге, но доступа нет. Отсутствие ключа страницы = нет доступа к ней.

**Правило валидации `permissions` против каталога (нормативно):** валиден ⇔ каждый ключ — известная `page` из каталога (кроме `users` — запрещён); каждое действие ∈ `CATALOG[page]`; без дублей и неизвестных страниц/действий. Нарушение → `422 unprocessable`.

### 2. Модель данных (см. [03-data-model.md](../03-data-model.md#таблица-roles))

- `roles(id uuid PK, name text UNIQUE, permissions jsonb NOT NULL, created_at, updated_at)`.
- `users(id uuid PK, username text UNIQUE, password_hash text NOT NULL, role_id uuid FK→roles ON DELETE RESTRICT, is_active bool DEFAULT true, created_at, updated_at)`.
- Миграция `0008_create_users_roles`, `down_revision = "0007_create_backends"` (текущая голова). `upgrade()` создаёт обе таблицы + сидит одну роль `admin` с полными правами по каталогу; `downgrade()` — `DROP TABLE users; DROP TABLE roles` (порядок из-за FK).
- **`username` допускает кириллицу/юникод-буквы:** DB-CHECK — «свободный» инвариант (длина 1–64, без ведущих/хвостовых пробелов, без control-символов); полное правило набора символов — на уровне Pydantic (по образцу `backends.domain`). Регэксп и обоснование — [03-data-model.md](../03-data-model.md#таблица-users).

### 3. Хэширование паролей (см. [05-security.md](../05-security.md#хэширование-паролей-bcrypt))

- Библиотека — **`bcrypt`** (напрямую, без `passlib`). Новая зависимость в `pyproject`. Модуль `app/infra/passwords.py`: `hash_password(plain) -> str`, `verify_password(plain, hashed) -> bool`. Cost = дефолт bcrypt (12).
- В БД хранится **только** `password_hash`; plaintext-пароль никогда не хранится, не логируется и не возвращается в ответах API.

### 4. Auth-поток (см. [modules/auth](../modules/auth/README.md), [05-security.md](../05-security.md#аутентификация-логин-и-выпуск-jwt))

При `POST /api/auth/login` порядок проверки:
1. **Сначала `.env`-супер-админ.** Логин/пароль сравниваются constant-time с `ADMIN_USER`/`ADMIN_PASSWORD` (как в [ADR-008](ADR-008-admin-iz-env.md), plaintext-сравнение сохранено). Успех → JWT супер-админа: `sub=ADMIN_USER`, `role="admin"`, `superadmin=true` (без `uid`).
2. **Иначе — БД-пользователь.** Ищется `users.username`; проверяется `verify_password` (bcrypt) и `is_active=true`. Успех → JWT: `sub=username`, `uid=<user.id>`, `role=<role.name>`, `superadmin=false`.
3. Неудача любой ветки → единое `401 invalid_credentials` (user-enumeration не раскрывается).

**JWT payload (claim'ы — побуквенно):**

| Claim | Тип | Супер-админ | БД-пользователь |
|-------|-----|-------------|-----------------|
| `sub` | string | `ADMIN_USER` | `username` |
| `uid` | string (uuid) | **отсутствует** | `users.id` |
| `role` | string | `"admin"` | `role.name` |
| `superadmin` | boolean | `true` | `false` |
| `type` | string | `"access"` | `"access"` |
| `iat`, `exp` | int | как раньше | как раньше |

Алгоритм — прежний HS256, `JWT_SECRET`, TTL `JWT_EXPIRES_MIN` (не меняются).

### 5. Enforcement (сервер — единственная граница безопасности)

`get_current_principal` декодирует JWT и **загружает актуальные права из БД на каждый запрос**:
- `superadmin=true` → `Principal(username=sub, role="admin", permissions=<полный каталог>, is_superadmin=True)`;
- иначе (по `uid`) → загрузка `users`+`roles`; если пользователь не найден **или** `is_active=false` → `401 unauthorized` (сессия аннулируется без пере-логина); иначе `Principal(username, role=role.name, permissions=role.permissions, is_superadmin=False)`;
- токен без `superadmin=true` и без `uid` (легаси до Спринта 3) → `401` (требуется повторный вход).

Свежая загрузка из БД означает: правки прав роли применяются **без пере-логина** пользователя.

**Фабрика зависимости `require(page, action)`** (в `app/api/deps.py`) → возвращает `Principal`, если `principal.is_superadmin or action in principal.permissions.get(page, [])`, иначе `403 forbidden()` (новый в `app/errors.py`).

**Фабрика `require_admin`** → `Principal`, если `principal.is_superadmin or principal.role == "admin"`, иначе `403 forbidden()`. Гейтит Users/Roles API и `GET /api/permissions/catalog`.

**Маппинг метода → действия для ресурсных роутеров** (заменяет прежний `_user: CurrentUser`):

| Роутер | GET (list/status/metrics) | POST | PATCH `/{id}` | PATCH `/order` | DELETE |
|--------|---------------------------|------|----------------|-----------------|--------|
| `servers` | `servers:view` | `servers:create` | `servers:edit` | `servers:edit` | `servers:delete` |
| `ai-keys` | `ai-keys:view` | `ai-keys:create` | `ai-keys:edit` | `ai-keys:edit` | `ai-keys:delete` |
| `proxies` | `proxies:view` | `proxies:create` | `proxies:edit` | `proxies:edit` | `proxies:delete` |
| `backends` | `backends:view` | `backends:create` | `backends:edit` | `backends:edit` | `backends:delete` |
| `mail` | `mail:view` (messages/teams/mailboxes) | — | — | — | — |

- **Reply почты** (`POST /api/mail/messages/{id}/reply`) гейтится `mail:view` — у почты в каталоге единственное действие `view`, и чтение, и ответ требуют его (Стап-3 решение; отдельного `reply`-права нет).
- **Dashboard** backend-эндпоинтов не имеет ([ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md)); `dashboard:view` — **UI-гейт** (не серверный): гейтит и видимость вкладки, и **сам маршрут `/dashboard`** (page-level view-guard — при прямом переходе без `dashboard:view` страница рендерит заглушку «Недостаточно прав», а не контент; см. Последствия и [08-design-system.md](../08-design-system.md#гейтинг-навигации-и-действий-по-правам-rbac-нормативно)). Карточки дашборда тянут ресурсные list-эндпоинты, поэтому для секции нужен и соответствующий `*:view`; при его отсутствии карточка показывает состояние «Недостаточно прав» (принятое поведение).

### 6. Frontend (UI-гейтинг — только UX, не безопасность)

Страница «Пользователи» (маршрут `/users`, вкладка — **admin-only**, последняя): список пользователей (username/роль/статус) + модалка создания/редактирования (username, пароль, `Select` роли); список ролей + редактор роли с **матрицей прав** (страницы × действия, чекбоксы). Новый UI-примитив **`Checkbox`** (нативный стилизованный, без новой зависимости — по образцу `Select`). Гейтинг: вкладки навигации фильтруются по праву `view`; вкладка «Пользователи» — только `is_superadmin || role=="admin"`; кнопки Создать/Редактировать/Удалить скрываются по правам. Дополнительно — **page-level view-guard на каждой permission-gated странице** (прямой URL/навигация без требуемого права → заглушка «Недостаточно прав» с **page-scoped** подсказкой «нет доступа к этому разделу», не редирект; включая `/dashboard`, `/mail` и `AdminRoute` для `/users` — см. Последствия). Отдельная заглушка «нет ни одного раздела» — для случая полного отсутствия `view` (и index `/`/fallback `*`/дефолт-маршрут — permission-aware). Обе заглушки и точные строки — [08-design-system.md](../08-design-system.md#гейтинг-навигации-и-действий-по-правам-rbac-нормативно). `403` **не сбрасывает сессию** — показывается «Недостаточно прав». Детали — [08-design-system.md](../08-design-system.md#страница-пользователи).

## Амендмент к ADR-008 (не отменяет)

ADR-008 остаётся `accepted`. Уточняется:
- `.env`-учётка (`ADMIN_USER`/`ADMIN_PASSWORD`) более не «единственный пользователь», а **несменяемый супер-админ (bootstrap)**: всегда полный доступ, роль `admin` неявно (`superadmin=true` в JWT), в таблицу `users` не попадает, из UI не редактируется/не удаляется. Сравнение кредов — по-прежнему plaintext constant-time (bcrypt применяется только к БД-пользователям).
- Дополнительные пользователи и роли теперь **есть в БД** (`users`/`roles`) — прежнее «нельзя добавить второго пользователя без доработки» снято этим ADR.
- Смена пароля супер-админа — по-прежнему только через `.env`/деплой (UI-смены нет — by design bootstrap); UI-управление паролями появляется для БД-пользователей (сужает [TD-009](../100-known-tech-debt.md)).

## Обоснование

- **Супер-админ вне БД** сохраняет свойство ADR-008 «невозможно залочить себя / потерять доступ через данные»: даже при пустой/битой таблице ролей вход супер-админом работает.
- **Права загружаются из БД на каждый запрос** — правки роли мгновенны, не нужен list отзыва токенов (refresh-токенов нет). Стоимость — один SELECT на запрос; при NFR-1 (единицы пользователей, монолит) приемлемо.
- **Каталог на сервере** — единственный канон; UI строит матрицу из него, невозможны «права на несуществующую страницу».
- **RBAC обеспечивается на сервере (`403`)**; UI-гейтинг — только UX. Нельзя обойти скрытие кнопки прямым запросом.
- **bcrypt напрямую** — меньше зависимостей и совместимостных сюрпризов, чем `passlib` (известные проблемы `passlib`↔`bcrypt 4.x`).
- **Кириллица в никах** — валидация на Pydantic (unicode `\w`), DB-CHECK свободный (по образцу `backends.domain`), чтобы не зависеть от locale/collation Postgres.

## Последствия

- (+) Многопользовательский режим с гранулярными правами на все страницы; супер-админ-fallback исключает lockout.
- (+) Единый серверный enforcement; UI-гейтинг производный от прав из `/api/auth/me`.
- (=) **Page-level view-guard обязателен на КАЖДОЙ permission-gated странице (нормативно).** UI-гейтинг не выборочный: любая страница под RBAC (`dashboard`/`mail`/`servers`/`ai-keys`/`proxies`/`backends`, а также `/users` по `is_superadmin||role=="admin"`), открытая по прямому URL или через навигацию/редирект без требуемого права, рендерит заглушку «Недостаточно прав» с **page-scoped** подсказкой «нет доступа к этому разделу» — единообразно для всех, включая Дашборд и Почты, а не только ресурсные страницы и `/users`. `AdminRoute` для `/users` показывает эту заглушку, а не редиректит на `/dashboard`. Отдельная заглушка «нет ни одного раздела» (иной текст подсказки) — только для случая полного отсутствия `view`; index `/`, fallback `*` и дефолт-маршрут после логина — **permission-aware** (резолвятся в первую доступную по правам вкладку, а не безусловно в `/dashboard`). Серверный `403 forbidden` остаётся единственной границей безопасности; page-level view-guard — только UX. Нормативные строки обеих заглушек (единый источник) и правило — [08-design-system.md §Гейтинг навигации](../08-design-system.md#гейтинг-навигации-и-действий-по-правам-rbac-нормативно).
- (−) +1 SELECT (users+roles) на каждый защищённый запрос БД-пользователя. Кэш прав per-request не вводится (простота); при росте нагрузки — будущая оптимизация.
- (−) На деплой Спринта 3 действующие сессии (токены без новых claim'ов) требуют повторного входа (легаси-токен → `401`).
- (−) Роль `admin` — зарезервированное имя (гейт «Пользователи» по `role=="admin"`); переименование сид-роли `admin` лишит её носителей доступа к «Пользователям» (супер-админ всегда может починить). Хард-защита имени в scope Спринта 3 не вводится.
- Аудит-лог действий пользователей — вне scope ([TD-001](../100-known-tech-debt.md)).

## Альтернативы

- **Супер-админ в таблице `users` (сид на миграции)** — отвергнуто: теряется fallback-инвариант ADR-008, появляется риск удалить/деактивировать себя и залочить систему.
- **Права в JWT (без загрузки из БД)** — отвергнуто: правки роли не применялись бы без пере-логина/отзыва токена; при TTL 24 ч это дыра управляемости.
- **`passlib[bcrypt]`** — отвергнуто в пользу `bcrypt` напрямую (меньше слоёв, нет известных compat-warning).
- **Radix `@radix-ui/react-checkbox`** — отвергнуто: нативный стилизованный чекбокс достаточен (как нативный `Select`), без новой зависимости (NFR-1).
- **Отдельное право `mail:reply`** — отложено: у почты одно действие `view` (read-through-прокси, Стап-3).
