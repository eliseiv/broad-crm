# ADR-009 · In-backend Telegram-нотификатор vs Alertmanager

- Статус: accepted
- Дата: 2026-06-30

## Контекст

Требуются Telegram-уведомления в группу при эскалации нагрузки серверов (вход метрики в жёлтую/красную зону) и при потере доступности (`up` 1→0). Варианты реализации:

1. **Prometheus Alertmanager** + alerting rules в Prometheus (PromQL-выражения, маршрутизация в Telegram-receiver).
2. **Фоновая asyncio-задача внутри backend**, переиспользующая `MonitoringService` и пороги `usage_to_zone()`.

Ограничения проекта: монолит, команда 1–2 человека (NFR-1, [ADR-001](ADR-001-stack-i-monolit.md)); Alertmanager в стеке **отсутствует**; пороги зон уже заданы в коде ([thresholds](../04-api.md#пороги-зон)) и обязаны совпадать с UI; формат сообщений и семантика «алерт только при повышении зоны» — продуктовые и app-специфичные.

## Решение

**In-backend asyncio-нотификатор.** Фоновая задача в процессе backend опрашивает online-серверы через тот же `MonitoringService` (с общим TTL-кэшем/single-flight read-path), сравнивает зоны через `usage_to_zone()` и шлёт сообщения в Telegram по HTTP API бота. Alertmanager **не вводится**. Спецификация — [modules/notifier](../modules/notifier/README.md).

## Обоснование

- **Единый источник порогов.** Зоны/пороги уже в `thresholds.py` и должны быть идентичны UI; дублировать их в alerting-rules PromQL — риск рассинхрона.
- **Без новой инфраструктуры.** Нет ещё одного контейнера/конфига (Alertmanager + receivers + routing) — соответствует NFR-1 (простота).
- **Продуктовая семантика проще на Python.** «Алерт только при повышении зоны», базовая фиксация при первой встрече, «переалерт» после возврата, точный русскоязычный формат сообщений и группировка по зоне выражаются прямо в коде; в Alertmanager это сложнее и менее наглядно.
- **Переиспользование read-path.** Та же защита от лавины (кэш/семафор/ретраи) — нотификатор не добавляет отдельной нагрузки на Prometheus.
- **Опциональность.** Активируется только при заданных `TELEGRAM_BOT_TOKEN`+`TELEGRAM_CHAT_ID`; иначе backend работает как прежде.

## Последствия

- (+) Один источник порогов, нет нового инфраструктурного компонента, точный контроль формата и анти-спама.
- (+) Консистентность данных бота и карточек (один `MonitoringService`).
- (−) **Состояние in-memory** — рестарт backend сбрасывает state ([TD-019](../100-known-tech-debt.md)): «горячие» серверы не переалертятся до следующего повышения зоны.
- (−) **Best-effort доставка с ограниченными ретраями (семантика at-least-once)**: транзиентные ошибки Telegram (`429`/`5xx`/таймаут/сеть) ретраятся в пределах бюджета попыток (до 3, backoff 0.2/0.5 с) → потеря алерта маловероятна, но trade-off — **редкий дубликат** вместо потери; при исчерпании бюджета попыток уведомление **пропускается** с warning-логом (без секретов), цикл опроса не падает. Детали — раздел [«Доставка в Telegram»](../modules/notifier/README.md#доставка-в-telegram).
- (−) Не горизонтально масштабируется (один процесс — то же ограничение, что у провижининга, [ADR-006](ADR-006-async-provisioning-bez-brokera.md)).
- (−) Нет «из коробки» истории алертов, silencing и группировки, которые даёт Alertmanager.

## Альтернативы

- **Alertmanager + alerting rules** — отвергнуто: новый инфраструктурный компонент, дублирование определения порогов в PromQL, более тяжёлая настройка маршрутизации/формата для Этапа 1.
- **Отдельный сервис-бот (внешний процесс/контейнер)** — отвергнуто: ещё один деплой-юнит и копия логики порогов; противоречит монолиту (NFR-1).

## Будущее

- Персистентность state ([TD-019](../100-known-tech-debt.md)); при росте требований (история/silencing/масштаб) — пересмотр в пользу Alertmanager или вынесенного воркера с очередью ([TD-004](../100-known-tech-debt.md)).
