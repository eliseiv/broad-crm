# ADR-041 · Login/theme/session UX-пакет: дефолт темы СВЕТЛАЯ + плейсхолдер/приём `@`-идентификатора входа + персистентная 24-часовая сессия

- Статус: accepted
- Дата: 2026-07-09
- Затрагивает: [modules/ui](../modules/ui/README.md), [modules/auth](../modules/auth/README.md)
- Амендмент: [ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md) (дефолт темы), [ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md) (идентификатор входа), [ADR-002](ADR-002-dvuhshagovyy-auth.md)/[ADR-021](ADR-021-rbac-users-roles.md) (JWT/сессия)

## Контекст

Три UX-правки входа/темы/сессии, объединённые в пакет. Контракты БД не затрагиваются; меняется дефолт темы, плейсхолдер поля входа и модель хранения токена на фронте.

1. **Тема (фикс 10).** По [ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md) дефолт темы при первом визите — **системная** (`prefers-color-scheme`), а страница логина фактически отображается тёмной. Пользователь требует дефолт **СВЕТЛАЯ** (включая экран логина и при отсутствии сохранённого выбора). Возможность переключить на тёмную сохраняется (выбор персистится и приоритетен).
2. **Идентификатор входа (фикс 11).** Плейсхолдер поля входа — «admin или @username»; пользователь просит «**Логин или @username**». Дополнительно: Telegram-ник при входе должен приниматься **и с `@`, и без**.
3. **Сессия (фиксы 12 и 16).** Токен хранится в `sessionStorage` → **(12)** при закрытии браузера стирается, требуется повторный вход; **(16)** `sessionStorage` изолирован по вкладке → открытие страницы в новой вкладке (ПКМ → «Открыть в новой вкладке») стартует без токена → редирект на `/login`. Пользователь хочет, чтобы сессия жила по TTL JWT (24 ч), переживала закрытие браузера и **шарилась между вкладками**.

## Решение

### 1. Дефолт темы — СВЕТЛАЯ (разворот дефолта [ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md) §2)

- **При отсутствии сохранённого выбора** (`localStorage['crm-theme']` пуст) активная тема — **`light`**, а НЕ `prefers-color-scheme`. Экран логина (`/login`, вне `AppLayout`) следует этому дефолту → **светлый**.
- **Явный выбор пользователя по-прежнему приоритетен и персистится** (`localStorage['crm-theme']` ∈ {`light`,`dark`}): сохранённая тёмная тема применяется и на логине, и в приложении.
- **Следование за `prefers-color-scheme` отменяется как дефолт и как подписка.** Приложение больше не подписывается на `matchMedia` change для авто-переключения — при отсутствии выбора всегда `light`. No-FOUC inline-скрипт: `const t = localStorage.getItem('crm-theme') || 'light'; document.documentElement.dataset.theme = t;`.
- **Причина «логин был тёмным».** Прежний дефолт брал `prefers-color-scheme` (у пользователя — тёмная ОС) до/на логине → тёмный экран. После разворота дефолт `light` устраняет это без хардкода темы на странице логина: логин следует общему правилу темы.
- **Cross-ref (нет дефекта):** разворот дефолта на светлую тему **не ломает** цветовые тег-чипы модуля «Почты» — они **тема-зависимы** (`color-mix` + таблица контраста светлой темы, [08-design-system.md §Тег-чип](../08-design-system.md#вкладка-теги-нормативно)), поэтому корректно читаются и в светлой теме по умолчанию. Заметка для навигации; фактического изменения тег-чипов не требуется.

### 2. Идентификатор входа: плейсхолдер + приём `@`/без `@` (амендмент [ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md))

- **Нормативный плейсхолдер** поля идентификатора входа (шаг 1) — **«Логин или @username»** (label остаётся «Логин или Телеграм»). Словарь — [08-design-system.md](../08-design-system.md#страница-роли) (раздел экрана входа).
- **Приём обеих форм Telegram-ника.** Вход по Telegram принимает и `@username`, и `username`: сервис нормализует идентификатор (`app/domain/telegram.normalize_telegram` — снимает ведущий `@`, lower-case) перед сопоставлением с `users.telegram`. **Это уже реализовано** (`auth_service._resolve_db_user` → `get_by_telegram(normalize_telegram(identifier))`) — ADR фиксирует поведение как нормативное; изменений backend не требуется. Контракт `POST /api/auth/login` не меняется (тело `{username, password?}`, где `username` — идентификатор: логин ИЛИ Telegram с/без `@`).

### 3. Персистентная сессия 24 ч + мульти-вкладка — `localStorage` (фиксы 12 и 16; амендмент [ADR-002](ADR-002-dvuhshagovyy-auth.md)/[ADR-021](ADR-021-rbac-users-roles.md))

- **Access-JWT хранится в `localStorage`** (ключ `crm.auth.token`; сопутствующие `crm.auth.*` — username/role/superadmin/permissions/seesAllSmsTeams — тоже), **а не в `sessionStorage`**. Токен переживает закрытие/повторное открытие браузера **и шарится между вкладками/окнами того же origin** (`sessionStorage` изолирован по вкладке — отсюда прежний баг).
- **Фикс 12 (персистентная сессия):** в пределах TTL JWT (24 ч, `JWT_EXPIRES_MIN=1440`, [05-security.md](../05-security.md#jwt)) повторный вход не требуется; автовыход остаётся по истечении токена (сервер `401`).
- **Фикс 16 (новая вкладка не требует релогина):** открытие страницы в новой вкладке (ПКМ → «Открыть в новой вкладке») или прямой ввод URL страницы читает токен из общего `localStorage` → пользователь остаётся авторизован, `ProtectedRoute` **не** редиректит на `/login`. **Требование к фронту:** auth-стор (`store/auth.ts`) инициализируется из `localStorage` **синхронно при загрузке SPA (регидрация) — ДО резолва маршрута/guard** (уже так устроен: `create()` читает `crm.auth.*` в начальный стейт), поэтому `ProtectedRoute` видит восстановленную сессию на первом рендере. Фиксы 12 и 16 покрываются **одной** правкой auth-персиста (`sessionStorage → localStorage`); отдельной логики для 16 не требуется — общий origin-storage + существующая синхронная регидрация.
- **Автовыход/logout:** `401`/logout полностью очищают `crm.auth.*` (во всех вкладках — через `localStorage`). Refresh-токенов нет.
- **Security-трейдофф (осознанно принят).** `localStorage` доступен JS всего origin → чуть шире XSS-поверхность кражи токена, чем у in-memory/`sessionStorage`. Принято для админ-панели: строгая CSP (`default-src 'self'`, [05-security.md](../05-security.md#content-security-policy-spa-location-)), отсутствие сторонних скриптов, экранирование React и **24-часовой TTL** ограничивают окно. Удобство «не логиниться при каждом открытии браузера/вкладки» перевешивает при данной модели угроз (внутренняя одно-/малопользовательская панель, NFR-1). Ужесточение (httpOnly-cookie + CSRF, refresh-ротация) — при появлении недоверенного окружения.

- `docs/modules/auth/README.md`, `docs/modules/ui/README.md`: auth-flow фронта — токен в `localStorage`, синхронная регидрация до guard, мульти-вкладочная сессия.

## Альтернативы (отклонены)

- **Оставить дефолт по системе, но хардкодить светлый логин.** Отклонено: рассинхрон логин↔приложение и дубль правил; проще развернуть общий дефолт на `light`.
- **httpOnly-cookie для сессии.** Отклонено на Этапе 1: требует CSRF-защиты и серверной сессии/refresh; избыточно для внутренней панели. `localStorage`+24ч JWT — минимальное изменение с приемлемым риском.
- **sessionStorage + «Запомнить меня».** Отклонено: усложняет UX; пользователь хочет персист по умолчанию.
