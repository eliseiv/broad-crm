# ADR-016 · Оценка зоны нотификатором по максимуму метрики за окно опроса (ловля транзиентных всплесков)

- Статус: accepted
- Дата: 2026-07-06
- Связанные: [ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md) (in-backend нотификатор), [ADR-014](ADR-014-persist-notifier-state-alert-on-first-elevated.md) (персист состояния + alert-on-first-elevated — не меняется, усиливается этим ADR), [ADR-003](ADR-003-prometheus-istochnik-metrik.md) (Prometheus — источник метрик)

## Контекст

Нотификатор ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)) на каждой итерации берёт **мгновенное** значение CPU/RAM/SSD через `MonitoringService.fetch_for_instances(...)` (тот же метод, что и UI-карточки) и оценивает зону от этого мгновенного значения. Опрос идёт раз в `NOTIFIER_POLL_INTERVAL_SEC` (default 60 с).

**Подтверждённый прод-дефект (Prometheus).** Сервер «Бэки Елисея» (`87.239.135.154`): `max_over_time(cpu[2h])` = **98.6 %** (red-зона `>90`), тогда как мгновенное «сейчас» = 13 %; RAM/SSD не поднимались. Всплеск CPU был **коротким (~минуты)** и **не совпал** ни с одним моментом опроса (плюс рестарты backend при деплоях создают окна, когда нотификатор вообще не опрашивал). Итог: нотификатор ни разу не сэмплировал red → эскалации `green→red` не было → алерт не отправлен; `notifier_server_state` для сервера = `green` (наблюдался только green).

ADR-014 чинил **другой** класс дефекта — потерю in-memory состояния при рестарте и ре-базлайн уже-«горячих» серверов (alert-on-first-elevated). Он **не покрывает** пропуск всплеска, случившегося **между** моментами опроса: сколько бы состояние ни персистилось, если мгновенный сэмпл ни разу не попал в red, эскалации не возникнет.

Мгновенный сэмплинг раз в 60 с принципиально «слеп» к пикам короче интервала: доля наблюдаемого времени ≈ (длительность пика / poll_interval), для минутного пика при 60-с опросе — единицы процентов вероятности поймать.

Пороги зон (`app.domain.thresholds.usage_to_zone()`: `<80 green` / `80..90 yellow` / `>90 red`) — единый источник для UI/backend/нотификатора и **не подлежат изменению**.

## Решение

**Нотификатор оценивает зону по МАКСИМУМУ метрики за окно опроса, а не по мгновенному значению.** Для каждой метрики CPU/RAM/SSD usage_percent считается как `max_over_time(<usage_expr>[<окно>])` вместо мгновенного значения того же выражения. Зона выводится из этого максимума через **неизменённый** `usage_to_zone()`. Всё остальное — `evaluate()`, персист, дедуп, alert-on-first-elevated, offline↔online, деэскалация — **без изменений**: на вход подаётся max-за-окно зона вместо мгновенной.

**1. Разделение UI vs notifier (критично).** UI-карточки (`GET /api/servers`, `GET /api/servers/{id}/metrics`) продолжают показывать **текущее (мгновенное)** значение — их поведение и контракт **не меняются**. Max-over-window применяется **только** на пути нотификатора. Разведено через **опциональный параметр окна** в переиспользуемом методе monitoring:

```
MonitoringService.fetch_for_instances(instances, window_sec: int | None = None)
```

- `window_sec is None` (**default**) → мгновенные запросы (текущее поведение UI/read-path) — **без изменений**.
- `window_sec = <секунды>` → CPU/RAM/SSD `usage_percent` берётся как max_over_time за окно; `zone` = `usage_to_zone(max)`. Notifier вызывает с `window_sec = NOTIFIER_METRIC_WINDOW_SEC`.

Тип возврата (`dict[instance -> InstanceMetrics]`), маппинг, `usage_to_zone` и пороги — идентичны. Меняется **только источник числа** `usage_percent` для CPU/RAM/SSD в windowed-режиме. Поля `online` (мгновенный `up`), `uptime_seconds`, `detail` (мгновенные gauge/cores/GB) в windowed-режиме сохраняют **мгновенную** семантику — max-over-window касается **только** usage_percent метрик нагрузки (offline-детект и абсолютные значения не входят в этот класс дефекта). TTL-кэш ключуется с учётом окна: `key = (tuple(sorted(instances)), window_sec)` — windowed-результат нотификатора и мгновенный результат UI **не смешиваются** даже при совпадении набора instances.

**2. Выбор окна.** `NOTIFIER_METRIC_WINDOW_SEC` (env, default **90 с**), нормативно **≥ `NOTIFIER_POLL_INTERVAL_SEC`**. Обоснование: окно должно быть **не меньше реального интервала между опросами**, чтобы соседние окна `[t-W, t]` и `[t+Δ-W, t+Δ]` **перекрывались** и любой момент времени попадал хотя бы в одно окно опроса. Реальный интервал = `poll_interval` + длительность итерации + джиттер планировщика, поэтому окно = `poll_interval` + запас (~50 %). При 60-с опросе default = 90 с даёт ~30-с перекрытие соседних окон — любой ~минутный пик гарантированно виден хотя бы одному опросу. Если задано `< poll_interval` — между окнами остаются «слепые» зазоры (дефект не устраняется); backend при старте валидирует/поднимает окно до `poll_interval` и пишет warning-лог.

**3. Баланс «слишком широкого окна».** Слишком большое окно даёт **«залипание» алерта**: max держит повышенное значение всю длину окна после спада пика, задерживая деэскалацию и «съедая» способность переалертить на следующий пик, пока окно не очистится. При окне ≈ `poll_interval + запас` пик виден 1–2 соседним опросам: первый после пика опрос → red → **один** алерт (если база green); второй (если ещё в окне) → red → дедуп (повтора нет — зона уже персистнута); затем окно очищается → max возвращается в green → **молчаливая** деэскалация; следующий пик снова алертит (полностью согласуется с дедупом ADR-014 по персистнутой зоне). Широкие окна (десятки минут/часы) отвергнуты именно из-за залипания.

**4. Взаимодействие с ADR-014 (без изменений).** Max-over-window **усиливает наблюдение** (нотификатор перестаёт «просыпать» пики между опросами), но не трогает механику: `evaluate(prev, im)`, персист в `notifier_server_state`, дедуп `rank(cur) > rank(base)`, alert-on-first-elevated (`prev is None ≡ green×3 baseline`), offline↔online, молчаливая деэскалация — всё как есть. Просто `im.metrics.*.zone` теперь = зона max-за-окно, а не мгновенная.

Нового API/endpoint нет; [04-api.md](../04-api.md) **не затрагивается** (UI-контракт карточек не меняется). Пороги и `usage_to_zone()` не меняются.

## Обоснование

- **Закрывает корневую причину** — «слепоту» мгновенного сэмплинга к пикам короче интервала опроса. Пик 98.6 %, случившийся между опросами, теперь виден как red через `max_over_time` → эскалация → один алерт.
- **Нулевой blast radius для UI.** Опциональный параметр с `None`-default сохраняет весь read-path и контракт карточек побайтово; изменение локализовано в windowed-ветке маппинга и в вызове нотификатора.
- **Переиспользование порогов и `evaluate`.** Меняется только число на входе; вся проверенная логика эскалации/дедупа/персиста ADR-014 работает без правок.
- **Простота (NFR-1).** Ни нового компонента, ни recording-rules, ни Alertmanager. Одна обёртка `max_over_time` в PromQL + один опциональный параметр + один env.

## Последствия

- (+) Транзиентные ~минутные всплески CPU/RAM/SSD между опросами больше не теряются — эскалация и алерт срабатывают.
- (+) UI-карточки не меняются: продолжают показывать текущее мгновенное значение.
- (+) Согласуется с ADR-014: спад max → молчаливая деэскалация → следующий пик снова алертит (дедуп по персистнутой зоне).
- (−) **Остаточная оговорка (принимается):** пики короче разрешения scrape/`rate` (~15 с scrape, `rate[1m]` для CPU) всё равно сглаживаются усреднением внутри `rate`/scrape и могут не достичь порога. Мы **сознательно** алертим на **устойчивые ~минутные** пики (как этот 98.6 %), а не на микро-спайки; это приемлемо и не заводит TD.
- (−) **Окна простоя нотификатора** (рестарт backend при деплое дольше окна) max-over-window закрыть не может: пока процесс не опрашивает, окно не наблюдается. Это ограничение одного процесса ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)), не цель данного ADR; частично смягчается тем, что окно ≥ интервала перекрывает короткие рестарты между соседними опросами.
- (−) Небольшой рост стоимости запроса: subquery `[W:step]` вычисляет выражение на нескольких шагах вместо одной точки. На ≤ десятков серверов и общем семафоре конкуренции ([monitoring](../modules/monitoring/README.md#устойчивость-read-path-нормативно)) — незначительно; windowed-запросы делает только нотификатор (не UI).
- (−) Windowed-результаты кэшируются отдельным ключом (с `window_sec`) — +1 класс кэш-записей; на объёмах проекта незначимо.

## Альтернативы

- **Мгновенный сэмплинг (текущий)** — **отвергнут:** принципиально слеп к пикам короче интервала опроса (см. Контекст) — это и есть исходный дефект.
- **Более частый опрос** (`poll_interval` вниз до секунд) — **отвергнут:** не решает проблему, а маскирует — пик короче нового интервала всё равно теряется; линейно повышает нагрузку на Prometheus (против [«Устойчивость read-path»](../modules/monitoring/README.md#устойчивость-read-path-нормативно)) и не даёт гарантии покрытия. Max-over-window даёт покрытие **без** роста частоты.
- **Alertmanager + recording/alerting-rules на стороне Prometheus** (`max_over_time`/`ALERTS`) — **отвергнут для Этапа 1** (переоценка [ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)): даёт корректную серверную оценку окна и естественный `for:`-дедуп, но вводит новый компонент/канал доставки и дублирует форматирование сообщений/RBAC группы. Отмечен как **осознанная будущая альтернатива**, если появится потребность в силенсинге/группировке/эскалационных политиках; пока in-backend max-over-window закрывает дефект дешевле (NFR-1).
- **Хранить и агрегировать историю сэмплов в backend** (свой rolling-max между итерациями) — **отвергнут:** дублирует то, что Prometheus уже делает через `max_over_time`; добавляет состояние/память и не переживает рестарт (в отличие от запроса к TSDB, который видит историю независимо от аптайма нотификатора).
