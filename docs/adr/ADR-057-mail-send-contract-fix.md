# ADR-057 — Контракт отправки письма после cut-over: `sent_id` выдаёт CRM, агрегатор возвращает только `smtp_message_id`; восстановление reply на проде

| | |
| --- | --- |
| Статус | accepted (§5–§7 добавлены 2026-07-14 амендментом по находке `backend-reviewer`) |
| Дата | 2026-07-14 |

**Амендмент** [ADR-044](ADR-044-mail-full-merge-into-crm.md) §8 (контракт reply/send). Парный ADR агрегатора — `ADR-0048` (репозиторий `D:\BA\mail-agregator`), он же амендирует их `ADR-0043` §3 и `ADR-0044` §2/§4 (Фаза A2). Разрешает **живой прод-баг**: отправка ответа из CRM сломана с момента cut-over.

## Context

### 1. Прод-баг (подтверждён кодом и прямым запросом к проду)

CRM зовёт **`POST /api/external/mailboxes/{id}/send`** (`backend/app/infra/mail_client.py:225`). В агрегаторе такого маршрута **нет** — в его `backend/app/external/router.py` реализованы только pull (`GET /messages`, `GET /mailboxes`), mailbox-write (`POST /mailboxes`, `test`, `PATCH`, `DELETE`, `sync`), OAuth (`authorize`/`callback`) и **message-scoped reply** `POST /api/external/messages/{id}/reply` (`:280`). Прод: `404` на `.../mailboxes/1/send`, `401` на `.../messages/1/reply`.

⇒ **Ответ на письмо из CRM не работает с cut-over (10.07.2026).** Пользователь видит `404 mail_message_not_found` — CRM маппит внешний `404` именно так (`backend/app/services/mail_service.py:1123-1124`), то есть ошибка ещё и вводит в заблуждение (письмо в CRM есть).

Разрыв прожил незамеченным, потому что кросс-репозиторный контракт отправки **не покрыт ни одним контрактным тестом/смоуком** → [TD-062](../100-known-tech-debt.md).

### 2. Связанный разрыв: источник `sent_id` не определён нигде

- Агрегаторский `ADR-0043` §3 задавал ответ `200 { sent_id, smtp_message_id }`.
- Их `ADR-0044` §4 (Фаза A2) требует реализовать send **без записи в `sent_messages`**, а §1 — дропнуть эту таблицу.
- Единственный производитель `sent_id` у них — `INSERT` в `sent_messages` (их `backend/app/send/service.py:460`).

Со стороны CRM поле **обязательное**: `backend/app/schemas/mail.py:128-129` (`sent_id: int`, `smtp_message_id: str`), и этой же схемой CRM парсит **сырой** ответ агрегатора (`backend/app/services/mail_service.py:783` — `self._parse(MailReplyResponse, raw)`). То есть публичная схема CRM и схема ответа внешнего сервиса — **одна и та же**, чего быть не должно.

## Decision

### §1. Публичный контракт CRM `POST /api/mail/messages/{id}/reply` (амендмент ADR-044 §8)

Путь, гейт (`mail:view`), тело запроса и нормы валидации — **без изменений** (`MailReplyRequest`, ADR-044 §8; [04-api.md](../04-api.md#post-apimailmessagesidreply)).

**Меняется только ответ:**

```json
{ "sent_id": "9f1c2e40-6a1b-4f3e-9d2a-7b8c0d1e2f34", "smtp_message_id": "<abc123@example.com>" }
```

- **`sent_id` — `string (uuid)`, а не `int`.** Это `mail_sent_messages.id` **самой CRM** (`backend/app/models/mail_sent_message.py:35-37` — `UUID`, `server_default gen_random_uuid()`), то есть идентификатор строки, которая реально существует и адресуема. Прежний `int` приходил из агрегаторской `sent_messages` — таблицы, которая по агрегаторскому `ADR-0043` §4 / `ADR-0044` §1 **дропается**; после демонтажа у агрегатора источника целого `sent_id` не остаётся вовсе.
- **`smtp_message_id` — `string | null`**, из ответа агрегатора (Message-ID отправленного письма). `null` — агрегатор ответил `200`, но идентификатор не прислал: письмо отправлено и записано в историю, идентификатора у CRM нет (§5). Нормальный случай — строка.

**Ломающее изменение типа — оценено:** единственный потребитель — типизация SPA (`frontend/src/types/api.ts:350`, `sent_id: number`); **значение поля нигде не читается** (grep по `frontend/src` — только объявление типа и мок в тесте). Правка SPA — смена типа поля.

**Альтернатива «убрать `sent_id` из ответа CRM вовсе»** отклонена: id записи отправки — законный хендл истории/аудита, и у CRM он **есть**; выкидывать существующий идентификатор ради упрощения ответа незачем.

### §2. Контракт CRM → агрегатор: ответ агрегатора сужен до `{ smtp_message_id }`

Нормативно (парно с агрегаторским `ADR-0048` §1):

| | |
| --- | --- |
| Метод / путь | `POST /api/external/mailboxes/{id}/send` (`{id}` = `mail_accounts.id`) — **путь остаётся тем, что уже строит `mail_client.py:225`** |
| Запрос | `{ to, cc?, subject?, body_text, in_reply_to?, refs? }` — **ровно то, что CRM уже шлёт** (`mail_client.py:218-226`); правок клиента в части запроса нет |
| Ответ 200 | **`{ smtp_message_id: string }`** — `sent_id` в ответе агрегатора **отсутствует**. Агрегатор **обязан** прислать `smtp_message_id` (их `ADR-0048` §1); CRM на приёме **толерантна** к его отсутствию — пишет факт отправки и warning в лог, а не `502` (§5) |

**Правка backend CRM (обязательна, иначе `_parse` упадёт на отсутствующем `sent_id`):**
1. Ввести **отдельную** внутреннюю схему ответа агрегатора (напр. `MailServerSendResult { smtp_message_id: str | None }` — поле **опционально**, §5) — публичная `MailReplyResponse` больше **не** используется для парсинга сырого ответа внешнего сервиса (это и было корнем сцепки).
2. `MailService.reply` (`backend/app/services/mail_service.py:773-795`): распарсить ответ агрегатора внутренней схемой → создать строку `mail_sent_messages` (`self._sent.create(...)`, репозиторий возвращает объект — `backend/app/repositories/mail_sent_message_repository.py:22-45`) → собрать ответ `MailReplyResponse(sent_id=sent.id, smtp_message_id=result.smtp_message_id)`.
3. Порядок сохраняется: сначала подтверждённая SMTP-отправка агрегатором, затем запись факта отправки. Провал записи после успешной отправки — прежняя семантика (транзакция запроса), новых гарантий не вводится.

### §3. `404` от send = «ЯЩИК не найден», а не «письма нет»

В обобщённом send письма нет в контракте: `404` означает, что **ящика `{id}` нет в агрегаторе** (агрегаторский `ADR-0048` §4). CRM резолвит письмо из **своей** БД, значит внешний `404` на send = **рассинхрон каталога ящиков**, а не отсутствие письма.

**Норма:** `_map_reply_rejected` (`mail_service.py:1114-1130`) маппит внешний `404` в **`404 mail_mailbox_not_found`** (`backend/app/errors.py:497-500`), а не в `mail_message_not_found`. Код `404 mail_message_not_found` остаётся за собственной проверкой CRM: письма нет или оно вне `MailScope` (анти-энумерация, ADR-044 §8 / [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)). Остальные ветки маппинга (`400`/`409`/`422` + `502 smtp_failed` → `mail_send_failed`, `504 mail_timeout`) — **без изменений** ([ADR-053](ADR-053-mail-timeouts-error-passthrough.md) §2).

### §4. Восстановление на проде — fix-forward, а не заплатка

Отвергнут временный вариант «CRM зовёт существующий `/api/external/messages/{id}/reply`» — он **опасен**, а не просто временен (детальное обоснование — агрегаторский `ADR-0048` §5):

1. **`id` писем CRM ≠ `id` писем агрегатора.** `preserve id` действовал только на ETL ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §10). Новые письма CRM вставляет **своим** `BIGSERIAL` (`backend/app/models/mail_message.py:45`) через `INSERT … ON CONFLICT DO NOTHING` (`backend/app/repositories/mail_message_repository.py:66`); Postgres расходует `nextval` **и на конфликтных** вставках, а повторные push'и штатны (агрегаторский `crm_push_recovery`) → последовательности расходятся. Ответ мог бы уйти **по чужому письму** (чужой адресат, чужой ящик).
2. **Ретенция агрегатора — 30 дней** (их `worker/app/cleanup.py:34-40`, `RETENTION_DAYS=30`): по письму старше месяца оригинала там нет → reply невозможен, хотя в CRM письмо есть.
3. Их reply гейтится `EXTERNAL_REPLY_ENABLED` (default `false`) и принимает поле `body`, а не `body_text` — правка CRM под старый контракт по объёму сопоставима с правкой под целевой, но её пришлось бы снимать.

⇒ **Восстановление = аддитивный релиз агрегатора «Фаза A2.1»** (реализовать `send`, ничего не удаляя — их `ADR-0048` §3) **+ парный минимальный релиз CRM** (§2/§3 выше). Оба — вне зависимости от остального демонтажа. До деплоя reply на проде не работает — [TD-062](../100-known-tech-debt.md).

### §5. Внутренняя схема ответа агрегатора **толерантна** к отсутствию `smtp_message_id`: аудит уже отправленного письма важнее строгости

**Проблема (найдена backend-reviewer'ом, подтверждена кодом).** После §2 ответ агрегатора состоит из ОДНОГО поля, и оно объявлено обязательным: `MailServerSendResult.smtp_message_id: str` (`backend/app/schemas/mail.py:149`). Парсинг идёт **после** фактической SMTP-отправки (`mail_service.py:788`, вызов агрегатора — `:776`). Если агрегатор ответит `200` без поля (или с `null`), `_parse` бросит `502 mail_unavailable` (`mail_service.py:1174-1176`) **до** `self._sent.create(...)` (`:789`) ⇒ письмо ушло, а строки `mail_sent_messages` нет вовсе. При этом сама модель данных поля НЕ требует: колонка `smtp_message_id` **nullable** (`backend/app/models/mail_sent_message.py:53`; [03-data-model.md](../03-data-model.md) §`mail_sent_messages`). Норма схемы оказалась **строже модели данных** — и цена этой строгости платится необратимым действием.

**Решение — толерантность (нормативно):**

1. **`MailServerSendResult.smtp_message_id` — опциональное** (`str | None`, default `None`). Тело `200` без этого поля/с `null` **НЕ** является несовместимым ответом и **НЕ** даёт `502`.
2. **Строка `mail_sent_messages` создаётся ВСЕГДА после подтверждённого `200`** — при отсутствии значения `smtp_message_id=None` (колонка nullable, репозиторий уже принимает `str | None` — `backend/app/repositories/mail_sent_message_repository.py:32`). Факт отправки не теряется никогда.
3. **Публичный `MailReplyResponse.smtp_message_id` — `string | null`** (было `string`). Ответ CRM остаётся `200` (отправка состоялась), `sent_id` присутствует всегда. Расширение типа безопасно: значение поля **нигде не читается** (grep по `frontend/src`: только объявление типа `frontend/src/types/api.ts:356` и мок в тесте `features/mail/__tests__/api.test.ts:96`); SPA правит тип на `string | null`.
4. **Отсутствие поля — наблюдаемое событие, а не тишина:** `logger.warning("mail_send_missing_smtp_message_id", ...)` с полями по §6.
5. **Обязанность агрегатора возвращать `smtp_message_id` НЕ снимается** (их `ADR-0048` §1 — контракт прежний). Толерантность CRM — принцип Postel'а на приёме, а не ослабление их контракта: нарушение видно в логе (п. 4), а не в потерянном письме.

**Обоснование выбора (строгость vs аудит).** SMTP-отправка **необратима**. Строгая схема защищает от «мусорного» ответа, но платит за это потерей факта уже совершённого необратимого действия — и вдобавок провоцирует пользователя отправить письмо **повторно** (адресат получает дубль), потому что в UI операция выглядит проваленной. Обратная ошибка (записали отправку с пустым `smtp_message_id`) стоит лишь отсутствия одного справочного идентификатора в истории — величина ущерба несопоставима. Строгость остаётся там, где она **бесплатна** — до необратимого side-effect (см. §7).

**Общий принцип (шире reply):** схема, которой парсится ответ, приходящий **ПОСЛЕ необратимого side-effect**, обязана быть **не строже**, чем модель данных, куда этот факт записывается. Валидация не должна отменять запись о том, что уже произошло.

### §6. Обязательный контекст логов reply

Логи reply — единственный след для разбора рассинхрона каталога и неполного ответа; без контекста они бесполезны (сейчас `logger.warning("mail_reply_mailbox_missing_in_aggregator")` пишется **без единого поля** — `mail_service.py:1135`). Нормативно:

| Событие | Обязательные поля |
| --- | --- |
| `mail_reply_mailbox_missing_in_aggregator` (внешний `404`, §3) | `mail_account_id`, `message_id` |
| `mail_send_missing_smtp_message_id` (`200` без `smtp_message_id`, §5) | `mail_account_id`, `message_id`, `sent_id` |

Способ проброса контекста в `_map_reply_rejected` (параметры метода / логирование на месте вызова) — **на усмотрение `backend`**; нормативен только **набор полей**. Тела письма, адресов и любых кредов в логах быть не должно ([05-security.md](../05-security.md)).

### §7. Аудит: где ещё норма ADR-057 строже модели данных / мешает восстановлению после сбоя

Проверены все места ADR-057, где валидация встречается с необратимым действием:

1. **`MailServerSendResult` (парсинг после отправки)** — было строже модели данных. **Устранено §5.**
2. **`MailReplyResponse` (публичный ответ)** — требовал `smtp_message_id: string`, которого может не быть. **Устранено §5.3** (`string | null`).
3. **`MailMailboxTestResponse`** (`mail_service.py:507`, схема `schemas/mail.py:204-208`) — строгость **сохраняется**: `test` идемпотентен и необратимого следа не оставляет, `502` на битом ответе ничего не теряет. Правило §5 сюда не распространяется.
4. **Провал записи `mail_sent_messages` ПОСЛЕ успешной отправки** (падение БД / откат транзакции запроса) — остаточное окно потери аудита, **осознанно принимается** (§2.3, семантика не менялась). Устранение требует outbox/2PC — диспропорционально объёму модуля; семантика для пользователя та же, что у `504 mail_timeout` ([ADR-053](ADR-053-mail-timeouts-error-passthrough.md) §2): отправка могла состояться, подтверждения нет, автоповтора нет. Отличие от дефекта §5 принципиальное: там запись не создавалась из-за **валидации** (устранимо бесплатно), здесь — из-за **отказа инфраструктуры** (не устранимо без распределённой транзакции).

Иных мест, где норма ADR-057 жёстче модели данных или препятствует восстановлению после сбоя, не найдено.

## Consequences

- Reply восстанавливается парным релизом; временного кода и заплаточного TD не появляется.
- Неполный ответ агрегатора (`200` без `smtp_message_id`) больше не уничтожает запись об уже отправленном письме: факт отправки фиксируется всегда, отсутствие идентификатора — warning в логе, а не `502` (§5). Публичный `smtp_message_id` становится nullable — расширение типа без фактических потребителей значения.
- Логи reply получают обязательный контекст (`mail_account_id`/`message_id`/`sent_id`) — рассинхрон каталога CRM↔агрегатор становится диагностируемым по логу, а не только по жалобе пользователя (§6).
- Публичный ответ CRM меняет тип `sent_id` (`int` → `uuid`) — ломающее изменение с нулевым числом фактических потребителей значения; SPA правит тип.
- Разрывается сцепка «публичная схема CRM = схема ответа внешнего сервиса»: у ответа агрегатора появляется отдельная внутренняя схема. Это устраняет класс дефекта, при котором изменение внешнего контракта молча ломает публичный.
- `404` на отправке перестаёт лгать про «письма нет» и называет истинную причину (ящика нет в почтовом сервисе).
- Заведён [TD-062](../100-known-tech-debt.md): нет контрактной проверки CRM ↔ агрегатор (из-за чего разрыв путей жил на проде 4 дня) + reply не работает до деплоя.

## Alternatives considered

- **Агрегатор продолжает вести `sent_messages` и выдавать `int sent_id`.** Отклонено: два источника истории отправки; противоречит их `ADR-0043` §4 (durable-лог — в CRM) и `ADR-0044` §1 (таблица под drop). CRM это число всё равно не использует.
- **Убрать `sent_id` из публичного ответа CRM.** Отклонено (§1): у CRM есть настоящий id записи — незачем терять хендл.
- **Оставить `sent_id: int`, синтезируя его в CRM** (напр. счётчик). Отклонено: `mail_sent_messages.id` — `uuid` (`models/mail_sent_message.py:35-37`); вводить второй, «числовой», id ради обратной совместимости типа, у которого нет потребителей, — чистый долг.
- **Временный вызов `/messages/{id}/reply`.** Отклонено, §4.
- **Оставить `smtp_message_id` обязательным во внутренней схеме** («строгость контракта с агрегатором»). Отклонено (§5): строгость здесь оплачивается потерей аудита **уже отправленного** письма и провокацией дубля у адресата, а нарушение контракта агрегатором всё равно фиксируется — логом. Строгость, которая уничтожает запись о необратимом действии, защищает контракт ценой данных.
- **Отдавать наружу `smtp_message_id: ""` / вовсе опускать поле при его отсутствии.** Отклонено: пустая строка — ложь про несуществующий Message-ID, а опускание поля даёт клиенту два разных «формата» `200`. `null` честен и совпадает с nullable-колонкой БД (`models/mail_sent_message.py:53`).
