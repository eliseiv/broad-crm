# ADR-012 · Модуль «Почты» — read-through-прокси без хранения

- Статус: **`partially superseded by` [ADR-044](ADR-044-mail-full-merge-into-crm.md)**

> ⚠️ **ЧАСТИЧНО ОТМЕНЁН [ADR-044](ADR-044-mail-full-merge-into-crm.md) (2026-07-10).** Модель **«read-through-прокси БЕЗ ХРАНЕНИЯ»** (нет БД/миграций под почту; лента/теги/ящики проксируются во внешний API) **более НЕ действует**: письма/теги/каталог ящиков/Telegram-уведомления **хранятся в БД CRM**, агрегатор — тонкий IMAP/SMTP-connector с push. Действующая архитектура — [ADR-044](ADR-044-mail-full-merge-into-crm.md); контракт — [04-api.md §Mail](../04-api.md#mail); модуль — [modules/mail](../modules/mail/README.md).
>
> **ЧТО ИЗ ЭТОГО ADR ДЕЙСТВУЕТ (инварианты пережили супессию, НЕ ослаблены):**
> 1. **`MAIL_API_KEY` — только на backend**, в заголовке `X-API-Key` исходящего запроса; никогда не в ответах CRM/логах/SPA/URL.
> 2. **HTML-тело письма рендерится ТОЛЬКО в sandbox-iframe** — `srcDoc` + `sandbox=""` **без** `allow-scripts` и **без** `allow-same-origin`, `referrerPolicy="no-referrer"`, без DOMPurify. Смена фона/цвета тела под тему CRM ([ADR-047](ADR-047-mail-fix-pack.md) §6) — чисто стилевая инъекция, изоляцию **не ослабляет**.
> 3. **JWT** на всех пользовательских mail-эндпоинтах.

## Контекст

Пользователь запросил новую страницу «Почты» в CRM: просмотр писем, приходящих во внешний почтовый сервис `postapp.store`, и ответ (reply) на письмо. Внешний сервис предоставляет **external-API**:

- `GET https://postapp.store/api/external/messages?since_id=&limit=` (auth `X-API-Key`) — лента писем. Пагинация **keyset вперёд** по `id ASC` от `since_id`; `limit` 1..200 (дефолт 50); **без** server-side фильтров/поиска; ретенция писем ~30 дней; rate-limit 120 запросов/мин на IP. Ответ: `{ messages: [...], next_since_id, has_more }`.
- `POST https://postapp.store/api/external/messages/{id}/reply` (auth `X-API-Key`) — ответ на письмо (endpoint добавляется параллельно на стороне mail-агрегатора; его нормативный контракт фиксирует architect mail-агрегатора, CRM его проксирует).

Ключевые ограничения-требования: простота (NFR-1, монолит), внешний ключ почты — **системный секрет, только на backend** (инвариант проекта, как для AI-провайдеров и Telegram: секрет не попадает в SPA/логи/URL — [05-security.md](../05-security.md), [ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)), фронт наружу не ходит. Тело письма может содержать произвольный `body_html` от третьих лиц — потенциальный вектор XSS.

Нужно решить: (1) как CRM получает письма — синхронизировать во внутреннюю БД или читать «на пролёте»; (2) где живёт ключ; (3) как безопасно рендерить HTML письма.

## Решение

### 1. Read-through-прокси без хранения (без БД в CRM)

CRM **не хранит** письма и не заводит таблиц/миграций под почту. Backend выставляет два тонких прокси-эндпоинта под JWT, которые синхронно проксируют запрос во внешний сервис, подставляя `X-API-Key`, и возвращают ответ фронту 1:1 (нормализованный в свои Pydantic-схемы):

- `GET /api/mail/messages?since_id=&limit=` → внешний `GET /api/external/messages` → схема `MailListResponse`.
- `POST /api/mail/messages/{id}/reply` → внешний `POST /api/external/messages/{id}/reply` → схема `MailReplyResponse`.

Контракт CRM — [04-api.md#mail](../04-api.md#mail), ТЗ — [modules/mail](../modules/mail/README.md).

**Почему read-through, а не синхронизация в БД (отклонённая альтернатива):** пользователь явно выбрал read-through. Синхронизация во внутреннюю БД дала бы server-side поиск/фильтры/строгий глобальный порядок «новые сверху», но требует: фоновый sync-воркер, таблицы + миграции, дедупликацию, инвалидацию, обработку ретенции (внешний сервис удаляет письма ~через 30 дней). Это существенное усложнение (против NFR-1) ради возможностей, не запрошенных на данном этапе. Read-through даёт рабочую ленту и reply минимальными средствами; цена — отсутствие server-side поиска/фильтров и строгого глобального newest-first (ограничено keyset-only внешнего API), зафиксировано как [TD-024](../100-known-tech-debt.md). Возврат к sync-в-БД — путь закрытия TD-024, если поиск/фильтры/newest-first станут требованием.

### 2. Ключ почты — системный секрет только на backend

Ключ внешнего API — `MAIL_API_KEY` (env, секрет), задаётся администратором развёртывания (не через UI, в отличие от AI-ключей). Подставляется backend'ом **только** в заголовок `X-API-Key` исходящего запроса к `postapp.store`. Ключ **никогда** не отдаётся в ответах CRM API, не логируется (structlog-фильтр секретов), не передаётся в SPA и не попадает в query-строку/URL. Базовый URL — `MAIL_API_BASE` (default `https://postapp.store`), таймаут — `MAIL_API_TIMEOUT_SEC` (default 10). **⚠️ ЕДИНЫЙ таймаут ОТМЕНЁН [ADR-053](ADR-053-mail-timeouts-error-passthrough.md) (2026-07-14):** `MAIL_API_TIMEOUT_SEC=10` (read) действует только для **быстрых** путей (их overall-deadline — `MAIL_API_DEADLINE_SEC=30`); пути с удалённым IMAP/SMTP (`test`/create/`PATCH`/reply) — `MAIL_API_MAILSERVER_TIMEOUT_SEC=75` (read) при overall-deadline `MAIL_API_MAILSERVER_DEADLINE_SEC=85`. Это тот же инвариант, что для ключей AI-провайдеров и `TELEGRAM_*` ([05-security.md](../05-security.md), [ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)). Egress backend → `postapp.store` (HTTPS) фиксируется в [07-deployment.md](../07-deployment.md).

**Гейт активации:** почта включена, только если задан `MAIL_API_KEY` (`settings.mail_enabled`). При выключенной почте оба эндпоинта отвечают единым кодом `503 mail_not_configured` (аналог опциональности Telegram-нотификатора), фронт показывает «Сервис почт не настроен».

### 3. Изоляция HTML-тела письма — sandbox-iframe (`srcDoc` + атрибут `sandbox`), без новой зависимости

`body_html` письма — недоверенный контент третьих лиц. Он рендерится **только** внутри изолированного `<iframe srcDoc={body_html} sandbox>` с максимально строгим `sandbox` (без `allow-scripts` и без `allow-same-origin` → скрипты письма не исполняются, доступа к origin/куки/DOM CRM нет). Это исключает XSS и утечку JWT/сессии из письма без добавления зависимостей.

**Почему sandbox-iframe, а не DOMPurify (отклонённая альтернатива):** DOMPurify санитизирует HTML, но добавляет frontend-зависимость и требует поддержания в актуальном состоянии; при ошибке конфигурации возможен обход. Sandbox-iframe даёт изоляцию на уровне браузера без парсинга и без зависимостей (NFR-1) и согласуется с CSP SPA (`frame-ancestors 'none'`, `script-src 'self'` — контент письма исполняться не может). Если письмо содержит только `body_text` (или `body_html=null`) — рендерится текст (моношрифт/пробелы), iframe не создаётся. Внешние ресурсы в письме (картинки/шрифты по URL) блокируются CSP страницы; на Этапе 1 письма отображаются без подгрузки внешних ресурсов — осознанное упрощение (см. [modules/mail](../modules/mail/README.md#изоляция-html-тела-нормативно)).

## Последствия

**Плюсы:**
- Минимум движущихся частей: без БД, миграций, фонового sync-воркера (NFR-1).
- Секрет почты не покидает backend; фронт наружу не ходит — единый инвариант с AI/Telegram.
- HTML письма изолирован средствами браузера, без зависимостей и без риска XSS/кражи JWT.
- Reply реализуется тонким проксированием — CRM не дублирует SMTP-логику.

**Минусы / принятый долг:**
- Нет server-side поиска/фильтров и строгого глобального «новые сверху» (keyset-only внешнего API) — [TD-024](../100-known-tech-debt.md). Пагинация — «Загрузить ещё» вперёд по `since_id`; в пределах батча письма показываются новыми сверху.
- CRM зависит от доступности `postapp.store`: при недоступности — `502 mail_unavailable` (не деградация с частичными данными — хранилища нет). **⚠️ Уточнено [ADR-053](ADR-053-mail-timeouts-error-passthrough.md):** **таймаут** (агрегатор доступен, но не успел) — это **`504 mail_timeout`**, а не `502`.
- Контракт reply-эндпоинта — внешняя зависимость от architect mail-агрегатора; при его изменении CRM-прокси синхронизируется (расхождение docs↔контракт эскалируется как обычно).
- Внешние ресурсы в HTML-письме (картинки по URL) на Этапе 1 не подгружаются (CSP + sandbox) — приемлемо для служебной ленты.

## Альтернативы (отклонены)
- **Синхронизация писем в БД CRM** — даёт поиск/фильтры/newest-first, но требует sync-воркер + таблицы + миграции + обработку ретенции; против NFR-1 и прямого выбора пользователя. Зафиксировано как путь закрытия [TD-024](../100-known-tech-debt.md).
- **DOMPurify для санитизации HTML** — добавляет зависимость и риск обхода при мисконфиге; sandbox-iframe изолирует надёжнее и без зависимости.
- **Ключ почты в SPA / прямой вызов `postapp.store` из фронта** — нарушает инвариант «секрет только на backend, фронт наружу не ходит» ([05-security.md](../05-security.md)); отклонено безусловно.
