# ADR-032 · Видимость SMS: «все команды» для admin-уровня по полному каталогу прав

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Амендмент:** [ADR-030](ADR-030-sms-module-full-merge.md) §6 (видимость по командам); опирается на RBAC-модель [ADR-021](ADR-021-rbac-users-roles.md) / [ADR-022](ADR-022-teams-nav-categories.md).

## Контекст

Модуль «СМС» ([ADR-030](ADR-030-sms-module-full-merge.md) §6) задавал видимость сообщений/номеров так: **«всё видит только `is_superadmin`»** (консольный `.env`-супер-админ), а **любой БД-пользователь** — только SMS/номера своих команд (`user_teams` → `sms_phone_numbers.team_id ∈ team_ids`), включая БД-пользователя с админ-ролью.

Требование пользователя — восстановить поведение исходного SMS-агрегатора: **администратор видит сообщения всех команд**, остальные роли — только своих. В прежней системе `super_admin` видел все команды; `group_leader`/`group_member` — только свои. В CRM это «администратор системы», а не только консольный `.env`-админ: БД-пользователь с админ-ролью (напр. роль «Админ») сейчас ошибочно ограничен своими командами.

Правило симметрично для **обоих** read-эндпоинтов страницы «СМС»: `GET /api/sms/messages` и `GET /api/sms/numbers` (оба под `SmsScopeDep`).

## Проблема идентификации «admin-уровня»

«Видит все команды» должно стать: `is_superadmin` **ИЛИ роль имеет admin-уровень доступа**. Нужен устойчивый, RBAC-native признак «admin-уровня». Рассмотрены варианты:

- **A. По имени роли (`name == "Админ"`/`"admin"`).** Отклонено: имя роли — **редактируемые данные** (ADR-021 допускает переименование и кириллические имена, роль создаётся/переименовывается через Ua). Security-решение на точном строковом совпадении хрупко: переименование роли молча ломает видимость. Кроме того, канонической в CRM является англоязычная seed-роль `admin`, а кастомная «Админ» — отдельная роль → однозначного имени нет.
- **C. Новое право/action «видеть SMS всех команд» (напр. `sms:view_all`).** Отклонено на Этапе как источник риска недоставки и лишней сложности: матрица прав в UI (`ACTION_ORDER = view/create/edit/delete`) **не рендерит** sms-специфичные действия (`transfer`/`sync` уже не грантуются через матрицу), поэтому новый `view_all` не был бы грантуем через существующий UI без его расширения; плюс потребовались бы миграция seed-роли и **backfill** права на уже существующую кастомную роль «Админ» — иначе «как раньше» не заработало бы автоматически.
- **B. По признаку прав роли — роль владеет ПОЛНЫМ каталогом прав.** Выбрано.

## Решение

**«Видит SMS всех команд» ⇔ `is_superadmin` ИЛИ роль владеет полным каталогом прав.**

Нормативное условие (для `get_sms_scope` в `deps.py`):

```
sees_all_teams = principal.is_superadmin \
                 or permissions_subset(full_catalog_permissions(), principal.permissions)
```

где `permissions_subset(child, parent)` (уже существует в `domain/permissions.py`) истинно ⇔ для каждой страницы `child` набор действий — подмножество действий `parent`. Подстановка `child = full_catalog_permissions()` означает: **`principal.permissions` покрывает КАЖДОЕ действие КАЖДОЙ страницы каталога** (`CATALOG`), т.е. роль имеет полный доступ.

- **Прочие роли** (неполный каталог: PM, «Пользователь» и т.п.) — `sees_all_teams = False` → видимость **только по своим командам** (`team_ids` из `user_teams`), поведение **не меняется**.
- Правило применяется **одинаково** к сообщениям и номерам (симметрия read/list).
- **Мутации** (`PATCH`/`transfer`/`DELETE`) — прежняя семантика scope сохраняется (номер вне scope → `403`); `sees_all_teams` одинаково снимает ограничение и на мутации (admin-уровень оперирует любым номером), как ранее делал `is_super_admin`.

### Поле `SmsScope`

Флаг `SmsScope.is_super_admin` **переименовывается в `sees_all_teams`** (его значение — «видит все команды», а не «консольный супер-админ»): держать флаг с именем `is_super_admin`, истинный для БД-роли, — семантическая ловушка на security-чувствительном признаке. Сервисы (`sms_message_service`, `sms_number_service`) читают `scope.sees_all_teams`.

## Обоснование выбора B

- **Устойчивость к переименованию.** Признак не зависит от редактируемого имени роли — переименование «Админ»→«Administrator» не ломает видимость.
- **RBAC-native, без миграций и нового UI.** Переиспользует существующие чистые функции `full_catalog_permissions()` + `permissions_subset()`; каталог прав **не меняется**, миграции не нужны, матрицу прав в UI расширять не нужно.
- **Семантически корректно.** Роль с полным каталогом может редактировать/удалять всё (серверы, роли, команды, SMS-номера) — это однозначно системный администратор; логично, что он видит и все SMS. Совпадает с тем, что консольному супер-админу в рантайме выдаётся ровно `full_catalog_permissions()`.
- **«Просто работает» для admin-ролей.** Seed-роль `admin` уже имеет полный каталог (миграция 0010, [ADR-022](ADR-022-teams-nav-categories.md) §65) → попадает под правило автоматически. Кастомная админ-роль «Админ» с полным доступом — тоже, без ручного тоггла и backfill.

## Последствия

**Плюсы:** восстановлено ожидаемое поведение (admin видит все команды); решение устойчиво к переименованию ролей; ноль миграций и изменений каталога/UI-матрицы; переиспользование существующей RBAC-логики.

**Минусы / операционное условие:**
- Признак admin-уровня = **полный каталог прав**. Если админ-роль (напр. «Админ») по какой-то странице неполна, она НЕ считается admin-уровнем и увидит только свои команды. **Операционное требование:** роль, которая должна видеть все команды, обязана владеть **полным каталогом прав** (все действия всех страниц). Это осознанная, аудируемая граница, а не хрупкий строковый матч.
- Изменение расширяет круг «видящих всё» с одного консольного супер-админа до всех обладателей полного каталога — это соответствует требованию и модели RBAC (полный каталог = администратор).

## Альтернативы

- **A (имя роли):** отклонено — хрупко (редактируемое имя, нет единого канонического имени).
- **C (новое право `sms:view_all`):** отклонено на данном этапе — не грантуемо через текущую UI-матрицу, требует миграции + backfill, риск недоставки «как раньше». Может быть пересмотрено, если понадобится **раздельная** от полного каталога кросс-командная видимость SMS (тогда — отдельным ADR с расширением `ACTION_ORDER` и миграцией).
