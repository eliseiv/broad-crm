## Context

**A. Docs врут (prerequisite, блокирующий).** [ADR-044](ADR-044-mail-full-merge-into-crm.md) перенёс почту **полностью в БД CRM** (таблицы `mail_accounts`/`mail_messages`/`mail_tags`/… , CRUD прямо в БД, агрегатор = IMAP/SMTP-коннектор + push), но синхронизацию нормативных docs отложил в §12 (чтобы не затереть незакоммиченные хунки параллельного агента). Follow-up **не выполнен**: `docs/modules/mail/README.md` и `docs/04-api.md §Mail` до сих пор описывают `group_id`, `MailTeam`, `GET /api/mail/teams`, проксирование тегов в `GET /api/external/tags`, «без БД, без миграций». Код — источник истины: `app/api/mail.py`, `app/services/mail_service.py`, `app/schemas/mail.py`, `app/models/mail_*.py`. Исполнители, читающие docs, писали бы код против несуществующего контракта. Синхронизация — обязательный prerequisite фиксов ниже.

**B. Шесть фиксов владельца:**

5. **Любой тег можно удалить.** Признак «встроенный» (`mail_tags.is_builtin`) не нужен; сейчас backend отдаёт `409` на удаление builtin (`mail_service.py`), а UI пишет «встроенный» (`MailTagRow.tsx`). Хуже: даже удали мы 409, при рестарте приложения удалённый тег **воскреснет** — `seed_builtin_mail_tags` сеет каталог в lifespan (`main.py` → `mail_bootstrap.py` → `mail_tag_repository.py`).
6. **Лейблы типов правил** («Тема содержит»/«Тело содержит»/…) — владелец даёт свой словарь; четвёртый тип `sender_exact` из UI создания правила убирается.
7. **Поля ящика.** «Отображаемое имя» — одно поле, в которое руками пишут «`5108 Klyro Forge (Codex)`». Владелец хочет два поля: **Номер** и **Приложение**.
8. **Колонка «Команда»** на вкладке «Почты» узкая — значение не читается целиком.
9. **Строка ящика** должна выглядеть по референсу `screen/1.jpg`: статус + «Номер» + жирное значение + «Приложение» + значение пилюлей; вторая строка — адрес почты.
10. **Тело письма игнорирует тему.** srcDoc sandbox-iframe несёт **хардкод** тёмного фона: `background:#161A22;color:#E6E9EF` — в двух местах, `MailDetail.tsx` и `MailMiniAppPage.tsx`. В светлой теме тело остаётся тёмным.

Долг — [TD-051](../100-known-tech-debt.md).
- **Мёртвая фабрика ошибки `mail_group_not_found`** (`app/errors.py`) — не вызывается ни одним путём; удаляется из таблицы кодов `04-api.md` и из `errors.py` (dead code) вместе с §1.

### §1. Фикс 5 — `is_builtin` упраздняется, удалять можно ЛЮБОЙ тег

- **БД:** `mail_tags.is_builtin` — **DROP COLUMN** (миграция `0023`).
- **API:** поле `is_builtin` уходит из схемы **`MailTagFull`**. Ветка **`409` при удалении встроенного тега — снимается** (`DELETE /api/mail/tags/{id}`: остаются `204` / `404 mail_tag_not_found` / `403` / `401`). Код `409 mail_conflict` в модуле сохраняется **только** для занятого имени тега (`POST`/`PATCH /api/mail/tags`) и конфликта ящика (email занят).
- **Seed при старте приложения УБИРАЕТСЯ.** `seed_builtin_mail_tags` (вызов `main.py`), `app/services/mail_bootstrap.py`, `app/domain/mail_builtin_tags.py`, `MailTagRepository.seed_builtin` (`mail_tag_repository.py`) — **удаляются**. Иначе удалённый владельцем тег воскресает при ближайшем рестарте/деплое — это и есть корень фикса, а не 409.
- **Первичный сев для ЧИСТЫХ инсталляций переносится в data-миграцию.** Миграция `0023` (до drop-колонки) вставляет канонические **10 тегов + их правила** идемпотентно (`INSERT ... ON CONFLICT (name) DO NOTHING` по `uq_mail_tags_name`; правила — только для впервые созданных тегов). Набор — тот, что сейчас в `app/domain/mail_builtin_tags.py` (`DPLA.PLA`, `Отменить подписку`, `Продление аккаунта`, `Диспут`, `Бан Аккаунта`, `Релиз`, `Реджект`, `Ревью`, `Ждет Ревью`, `Нужна замена реквизитов`). **Данные вшиты в тело миграции** (миграция не импортирует код приложения — иначе удаление модуля сломает миграцию задним числом). На проде теги уже есть → `ON CONFLICT` но-опит; на чистой инсталляции — создаются один раз и **больше никогда не пересеваются**.
- **UI:** слово «встроенный» (`MailTagRow.tsx`) удаляется; кнопка удаления активна для любого тега.
- **Следствие (принимается):** после `0023` каталог тегов полностью управляется пользователем. Удалил все теги → они не вернутся (это и требовалось). Восстановление — только вручную.
- **Судьба одноразового ETL-скрипта `backend/scripts/migrate_mail_data.py` (решение, нормативно): УДАЛЯЕТСЯ.**
  - **Факт:** скрипт импортирует `app.domain.mail_builtin_tags` (`scripts/migrate_mail_data.py`) и пишет в колонку `is_builtin` (`:753-754`, `:836`, `:846`, `:859`) — оба символа этим ADR **удаляются**. После `0023` скрипт **неработоспособен** (падает на импорте и на INSERT в несуществующую колонку). CI это не ловит: mypy/тесты гоняются по `app`, не по `scripts/`.
  - **Решение — удалить файл.** Обоснование: (а) это **одноразовый cut-over-ETL**, свою работу он **выполнил** — миграция прода исполнена 2026-07-10 ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §10, 2874 письма, доставка подтверждена end-to-end); (б) повторный прогон невозможен **по существу**, а не только из-за `is_builtin`: агрегатор демонтирован ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §10 шаг 9), источник данных ETL больше не существует; (в) мёртвый код, обращающийся к **несуществующей схеме**, опаснее его отсутствия — он выглядит рабочим инструментом и будет запущен кем-то «на всякий случай»; (г) артефакт **не теряется**: он остаётся в истории git, а сама процедура ETL нормативно описана в [ADR-044](ADR-044-mail-full-merge-into-crm.md) §10 (маппинги, идемпотентность, порядок cut-over) — при гипотетическом restore-from-backup скрипт всё равно пришлось бы **переписывать** под новую схему (`number`/`app_name`, без `is_builtin`).
  - **Отклонено:** «законсервировать с пометкой» (мёртвый код с обращением к дропнутой колонке — приманка для запуска; пометка в docstring не защищает) и «адаптировать под новую схему» (адаптировать нечего — источник данных демонтирован; это была бы работа впустую).
  - **Не заводится TD:** долга не остаётся — удаление полное и осознанное.

### §3. Фикс 7 — «Номер» и «Приложение» вместо «Отображаемого имени»

#### §3.1. Данные (миграция `0024`)

`mail_accounts` **`+= number TEXT NULL`, `+= app_name TEXT NULL`**. Колонка **`display_name` СОХРАНЯЕТСЯ** (см. §3.3).

**Backfill из `display_name` (нормативно, детерминированный разбор).** Правило: **ведущая числовая часть (включая перечисления через запятую) → `number`; остаток текста → `app_name`.**

- Регулярное выражение разбора: `^\s*(\d+(?:\s*,\s*\d+)*)\s*(.*)$` над `display_name`.
  - Совпало → `number` = группа 1, **нормализованная** к разделителю «запятая + пробел» (`", ".join(цифровые токены)`); `app_name` = группа 2 после `strip()`, пустая строка → `NULL`.
  - Не совпало (нет ведущих цифр) → `number = NULL`, `app_name` = `display_name.strip()`, пустая строка → `NULL`.
- `display_name IS NULL` → обе колонки `NULL`.

**Нормативные тест-кейсы (примеры владельца — обязаны воспроизводиться побуквенно):**

| `display_name` (до) | `number` | `app_name` |
|---------------------|----------|------------|
| `5108 Klyro Forge (Codex)` | `5108` | `Klyro Forge (Codex)` |
| `173, 57, 104` | `173, 57, 104` | `NULL` (пусто) |
| `WIU` | `NULL` (пусто) | `WIU` |

#### §3.2. Контракт (нормативно, точные схемы — [04-api.md §Mail](../04-api.md#mail))

- **`MailMailbox` (ответ)** `+= number: string|null`, `+= app_name: string|null`. Поле `display_name` в ответе **сохраняется** (производное, §3.3).
- **`MailMailboxCreateRequest`**: поле `display_name` **удаляется** из тела запроса, вместо него — `number: string|null?`, `app_name: string|null?` (оба опц.). Остальное (поля `MailMailboxTestRequest` + `team_id`) без изменений.
- **`MailMailboxUpdateRequest`**: `display_name` **удаляется**, добавляются `number: string|null?`, `app_name: string|null?` (presence-семантика сохраняется).
- **`display_name` клиентом НЕ принимается** — сервер вычисляет его сам (§3.3). Это устраняет двоемыслие «кто хозяин имени».
- ~~`TeamMailboxItem` (`GET /api/teams/{id}/mailboxes`) — **не меняется** (там `display_name`, производное значение; клиенту detail-панели `/teams` этого достаточно).~~ **⚠️ РАЗВЁРНУТО [ADR-048](ADR-048-teams-mailbox-count-mail-row.md) §2 (2026-07-13):** `TeamMailboxItem` **расширен** полями `number`/`app_name`, а `display_name` в строке detail-панели `/teams` **не рендерится** (строка почты получила тот же визуальный язык, что строка ящика на `/mail`: крупный жирный «Номер» + «Приложение» пилюлей `ui/Pill tone="accent"`). Действующий контракт — [04-api.md](../04-api.md#get-apiteamsidmailboxes).

#### §3.3. `display_name` — производное поле (нормативно)

- **Вычисление:** `display_name = " ".join(part for part in (number, app_name) if part and part.strip())`; обе части пусты → `display_name = NULL`. Пересчитывается **сервером** при **каждом** create/update ящика.
- **Зачем сохраняем:** это **единственная** форма имени, которую понимает агрегатор (внешний контракт `POST/PATCH /api/external/mailboxes` — поле `display_name`). Держать её в БД CRM производной дешевле, чем склеивать на каждом исходящем вызове и чем менять внешний контракт. Долг «денормализованное производное поле» — [TD-052](../100-known-tech-debt.md).

#### §3.4. Что уходит в агрегатор (ловушка кода — закрывается нормативно)

`MailService.create_mailbox` **до этого фикса** делал `payload.model_dump(exclude={"team_id"})` (`mail_service.py`) и шлёт **всё остальное** в агрегатор. Если просто добавить поля в схему — `number`/`app_name` **утекут** в агрегатор, который их не знает.

**Нормативно:**

- В агрегатор при **создании** уходит `payload.model_dump(exclude={"team_id", "number", "app_name"})` **плюс** явно подставленный **вычисленный `display_name`** (§3.3). `number`/`app_name` в агрегатор **не уходят никогда**.
- В агрегатор при **правке** уходит по-прежнему только whitelist `_MAILBOX_AGGREGATOR_FIELDS` (`mail_service.py`) — **whitelist-подход сохраняется** (он-то и защищает `team_id`). `number`/`app_name` в whitelist **не добавляются**; вместо этого: если в `PATCH` присутствует `number` **или** `app_name`, сервис пересчитывает `display_name` и кладёт **его** в агрегаторный payload (ключ `display_name` уже в whitelist).
- Отсюда общее правило (нормативно): **исходящий payload в агрегатор строится белым списком, а не «всё, кроме…»**. `create_mailbox` приводится к тому же принципу (явный whitelist кредов + `email` + `display_name`), чтобы следующее новое поле схемы снова не утекло наружу.

#### §3.5. Идентификаторы миграции: revision id ≤ 32 символов (нормативно, для ВСЕХ будущих миграций)

Служебная таблица Alembic хранит идентификатор в колонке **`alembic_version.version_num` типа `VARCHAR(32)`**. Идентификатор длиннее 32 символов **физически неприменим**: `alembic upgrade` падает с `StringDataRightTruncationError` (воспроизведено на `0024_mail_accounts_number_app_name` — 34 символа).

**Норматив:**
- **`revision` / `down_revision` — не длиннее 32 символов.** Проверять при создании КАЖДОЙ миграции.
- **Имя ФАЙЛА миграции длиной не ограничено** и остаётся описательным. Расхождение «файл длиннее id» — **допустимо и предпочтительно** укорачиванию читаемого имени файла.
- **Причина укорочения фиксируется в docstring самой миграции** (чтобы следующий разработчик не «исправил» id обратно на имя файла).
- **Фактические значения этой миграции (проверены по коду `backend/alembic/versions/0024_mail_accounts_number_app_name.py`):** файл `0024_mail_accounts_number_app_name.py`; `revision = "0024_mail_accounts_num_app_name"` (31 симв.); `down_revision = "0023_mail_tags_drop_is_builtin"` (30 симв.).
- Прежняя формулировка docs, называвшая revision id по имени файла (34 симв.), **отменена** — она была нереализуема.

#### §3.7. OAuth-ingest: `number`/`app_name` разбираются из `display_name` агрегатора (нормативно, решение architect 2026-07-11)

**Пробел, который закрывается.** Путь `POST /api/mail/oauth/ingest` ([ADR-045](ADR-045-mail-outlook-oauth-headless-reonboarding.md) §3) заводит Outlook-ящик **в обход** `POST /api/mail/mailboxes`: агрегатор присылает готовый `display_name`, а `MailAccountRepository.upsert_catalog` пишет его как есть. Исходный текст §3 этот путь **не покрывал** → у OAuth-ящиков `display_name` непуст, а `number`/`app_name` = `NULL`, т.е. **инвариант §3.3 («`display_name` — производное от `number`/`app_name`») нарушался**. Видимое следствие: в рендере строки ящика (§5) у такого ящика в первом ряду видны только индикатор статуса и e-mail — ни «Номера», ни «Приложения». Реализация backend отклонением от ТЗ **не была** — ТЗ молчало.

**Решение — вариант (б): разбирать `display_name`, пришедший от агрегатора, ТЕМ ЖЕ правилом §3.1.** Инвариант §3.3 обязан держаться на **всех** путях записи каталога, а не только на `POST /api/mail/mailboxes`.

**Норматив:**

1. **INSERT-ветка `upsert_catalog` (новый OAuth-ящик):** `(number, app_name) = parse_display_name(<display_name агрегатора>)` — **побуквенно то же правило разбора**, что в §3.1 (regex `^\s*(\d+(?:\s*,\s*\d+)*)\s*(.*)$`, нормализация разделителя к «запятая + пробел», пустой остаток → `NULL`, нет ведущих цифр → `number = NULL`). `display_name` сохраняется **канонически производным** — как `build_display_name(number, app_name)` от разобранных частей (для входов, покрываемых правилом, это тот же текст; расхождение возможно лишь в нормализации пробелов/разделителей — и тогда канон CRM **важнее** сырой строки агрегатора).
2. **`ON CONFLICT (id) DO UPDATE` (re-consent уже существующего ящика):** поля имени — **`number`, `app_name`, `display_name` — НЕ перезаписываются.** После создания **CRM — источник истины имени ящика** (админ мог отредактировать «Номер»/«Приложение» через `PATCH /api/mail/mailboxes/{id}`), а агрегатор его не редактирует — он лишь эхо-возвращает то, что CRM ему когда-то и отдала. Перезапись затёрла бы правку админа. Обновляются, как и прежде, **`email`, `is_active`, `team_id`** (последний — детерминированно из `crm_state`, [ADR-045](ADR-045-mail-outlook-oauth-headless-reonboarding.md) §3). Поля синка не трогаются (их ведёт status-канал).
3. **Правило разбора — ОДНА чистая функция** `parse_display_name(value) -> tuple[str | None, str | None]` в `app/domain/mail.py` (рядом с уже существующей `build_display_name`, `app/domain/mail.py`). Её переиспользуют `upsert_catalog` и — при необходимости — любые будущие пути импорта каталога.
   - **Миграция `0024` свою копию правила СОХРАНЯЕТ** (Alembic-миграции не импортируют код приложения — иначе удаление модуля сломало бы миграцию задним числом, ср. §1). Две реализации одного правила — **осознанно**; **источник истины правила — §3.1 настоящего ADR**, обе обязаны ему соответствовать. Нормативные тест-кейсы §3.1 (`5108 Klyro Forge (Codex)` / `173, 57, 104` / `WIU`) применяются к **обеим**.
4. **Ручное дозаполнение не требуется** — админ может, но не обязан править «Номер»/«Приложение» у OAuth-ящика: они заполняются автоматически.

**Отклонено — вариант (а) «оставить `NULL`».** Он оставил бы инвариант §3.3 нарушенным на живом пути, а рендер строки ящика (§5) — деградированным (у Outlook-ящиков пропадали бы «Номер» и «Приложение»), переложив дозаполнение на ручную работу админа по каждому подключённому ящику. Разбор детерминирован, правило уже существует и покрыто тест-кейсами — цена решения близка к нулю.

#### §3.6. UI

Форма ящика (`MailboxFormModal`): поле **«Отображаемое имя»** заменяется двумя — **«Номер»** и **«Приложение»** (оба опциональны, обычные `Input`). Поля `display_name` в форме нет.

### §4. Фикс 8 — колонка «Команда» расширяется

- Значение команды на вкладке «Почты» обязано читаться **полностью**. **`truncate` / `overflow-hidden` / любое клиппирование значения команды — ЗАПРЕЩЕНЫ** (правило CLAUDE.md: переполнение решается размером, а не скрытием контента).
- Решение — **размером**: контрол команды — **`w-full min-w-[14rem]`** внутри колонки **`w-[22rem]`** (было `w-40`); статичное (не-admin) значение команды переносится (`break-words`), а не обрезается. Точная спецификация — [08-design-system.md](../08-design-system.md#вкладка-почты-нормативно).
- **Уточнение нормы для нативного `<select>` (решение architect 2026-07-11, по итогам реализации).** Контрол — существующий `ui/Select` на нативном `<option>`; нативный `<select>` **физически не переносит** текст выбранной опции: имя команды длиннее **≈35 символов** упирается в край контрола средствами браузера. Это **не** запрещённая обрезка (`truncate`/`overflow-hidden` в коде **отсутствуют** — правило CLAUDE.md не нарушено), а конструктивное ограничение элемента. **Норма «значимый контент читается полностью» считается выполненной**, если: (1) контрол растёт вместе с колонкой; (2) на нём стоит **`title` с полным именем команды**; (3) **раскрытый список опций** браузер показывает **полностью** (обрезается лишь «схлопнутое» отображение выбранного значения) — значение остаётся доступным пользователю.
- **Кастомный (не-нативный) dropdown НЕ вводится — отклонено.** Ради краевого кейса (>35 символов; на проде имён такой длины нет) потребовался бы **новый примитив дизайн-системы** с собственным a11y-контрактом (клавиатура, ARIA-роли, focus-trap, портал) и замена `ui/Select` на всех страницах ради консистентности. Цена несоразмерна. Остаточное ограничение — [TD-056](../100-known-tech-debt.md); при появлении длинных имён — отдельная задача frontend на примитив `ui/Combobox`.

### §5. Фикс 9 — новый рендер строки ящика (референс `screen/1.jpg`)

Таблица вкладки «Почты» сохраняется; перестраивается **идентификационная ячейка** ящика (и отдельная колонка статуса упраздняется — кружок переезжает внутрь):

- **Строка 1:** индикатор статуса (кружок; та же логика `healthy`, что сейчас — `is_active` + отсутствие ошибок синка) → лейбл **«Номер»** (вторичный цвет) + значение `number` (**крупно, полужирно**) → лейбл **«Приложение»** (вторичный цвет) + значение `app_name` **подсвеченной пилюлей**.
- **Строка 2:** адрес почты (`email`).
- **Пилюля «Приложение» — существующий примитив `ui/Pill`, `tone="accent"`** ([08 §Компонент `ui/Pill`](../08-design-system.md#компонент-uipill)). **Новый примитив не вводится.** Почему `ui/Pill`, а не тег-чип (`MailTagChip`): тег-чип выводит все три цветовые составляющие из **HEX-цвета тега** (`tag.color`) — у «Приложения» цвета нет, а `ui/Pill` — ровно тот существующий примитив пилюли, который работает на tone-токенах темы (им же нарисованы пилюли «Логин»/«Приложение» на `/sms`). Референс `screen/1.jpg` показывает индиго-заливку = `--accent`.
- Пустое `number`/`app_name` → соответствующая пара «лейбл + значение» **не рендерится** (лейбл без значения не показывается). Оба пусты → строка 1 содержит только индикатор статуса.
- Точная структура и словарь — [08-design-system.md](../08-design-system.md#вкладка-почты-нормативно).

### §6. Фикс 10 — тело письма следует теме CRM; единый источник srcDoc

- **Фон и цвет текста sandbox-iframe тела письма следуют текущей теме CRM.** Хардкод `background:#161A22;color:#E6E9EF` снимается.
- **Единый источник (нормативно):** билдер srcDoc **обязан быть один** — общий модуль (напр. `features/mail/mailBody.ts`), импортируемый и `MailDetail.tsx`, и `MailMiniAppPage.tsx`. Дублирование билдера в двух файлах (текущее состояние: `MailDetail.tsx` и `MailMiniAppPage.tsx`) **запрещено** — именно оно и породило фикс, применённый в одном месте и забытый в другом.
- **Значения (нормативно; iframe — собственный документ, CSS-переменные родителя в него НЕ наследуются, поэтому цвета подставляются литералами по текущей теме):**

  | Тема | `background` | `color` |
  |------|--------------|---------|
  | `dark` | `#161A22` (`--surface-2` тёмной) | `#E6E9EF` (`--text-primary` тёмной) |
  | `light` | `#F7F8FA` (`--surface-2` светлой) | `#111827` (`--text-primary` светлой) |

  Литералы обязаны совпадать с токенами `index.css` ([08 §Цветовые токены](../08-design-system.md#цветовые-токены-css-custom-properties--tailwind-theme)); при смене токенов — синхронизировать.
- Билдер принимает тему параметром (`buildMailBodySrcDoc(bodyHtml, theme)`), тема берётся из существующего механизма (`data-theme` / `useTheme()`); смена темы перерисовывает iframe.
- **Инвариант изоляции НЕ ослабляется:** `sandbox=""` **без** `allow-scripts` и **без** `allow-same-origin`, `referrerPolicy="no-referrer"`, CSP не трогается ([ADR-012](ADR-012-mail-read-through-proxy.md), [ADR-015](ADR-015-csp-img-src-remote-mail-images.md)). Инъекция остаётся чисто стилевой (`<style>html,body{…}</style>` перед телом), best-effort для писем с собственным фоном.
- **Mini App `/tg/mail`** — тема Telegram (`themeParams`, [08 §Темизация](../08-design-system.md#темизация-светлаятёмная)); билдер тот же, тему для него резолвит Mini App (светлая/тёмная по `colorScheme` Telegram).

## Consequences

- **Docs↔код расхождение по модулю «Почты» закрыто** (follow-up ADR-044 §12 исполнен) — исполнители наконец читают действующий контракт.
- **Теги полностью во власти пользователя:** нет «встроенных», нет воскрешения при рестарте. Цена: на чистой инсталляции каталог создаётся один раз миграцией; удалил — не вернётся (это требование, а не дефект).
- **Имя ящика структурировано** (`number`/`app_name`) → возможен новый рендер строки (фикс 9) и осмысленная сортировка/поиск в будущем. Цена: производный `display_name` — денормализация ([TD-052](../100-known-tech-debt.md)) ради неизменности внешнего контракта агрегатора.
- **Устранён класс ошибки «новое поле схемы утекло в агрегатор»** — исходящий payload теперь строится белым списком в обоих путях (create и update).
- **Тело письма читаемо в обеих темах**, изоляция sandbox не ослаблена; билдер один — следующий фикс не разъедется по двум файлам.
- **`sender_exact` остаётся в БД/контракте, но не создаётся из UI** ([TD-055](../100-known-tech-debt.md)) — существующие правила прода не ломаются.
- **Требуются две миграции** (`0023`, `0024`); обе — на живом проде с 121 ящиком и 16 тегами. `0023` идемпотентна (`ON CONFLICT`), `0024` — чистый backfill без потери данных (`display_name` остаётся на месте, пересчитывается только при следующей правке ящика).

## Alternatives considered

- **Оставить `is_builtin`, но разрешить удаление builtin-тегов** (снять только 409). Отклонён: seed в lifespan **воскресит** удалённый тег при ближайшем рестарте — фикс был бы иллюзорным. Корень — сев при старте, а не проверка при удалении.
- **Оставить seed в lifespan, но сделать его «однократным» через флаг в БД.** Отклонён: новая таблица/флаг ради того, что естественно выражается data-миграцией (миграция и есть «однократное действие с отметкой в `alembic_version`»).
- **Удалить `sender_exact` из БД/контракта.** Отклонён: на проде существуют правила этого типа (напр. builtin «Диспут» = `sender_exact AppStoreNotices@apple.com`) — удаление типа сломало бы CHECK и разметку. Владелец просил убрать из **UI**, не из модели.
- **Заменить `display_name` на `number`/`app_name` полностью (drop колонки).** Отклонён: `display_name` — поле **внешнего** контракта агрегатора (`POST/PATCH /api/external/mailboxes`); его снятие требует правки чужого сервиса. Производное поле дешевле.
- **Передавать `number`/`app_name` в агрегатор как есть** (пусть игнорирует лишние поля). Отклонён: молчаливая зависимость от снисходительности чужой схемы; при `extra=forbid` на той стороне создание ящика упало бы `422`. Явная склейка — предсказуема.
- **Оставить два независимых билдера srcDoc, поправив оба.** Отклонён владельцем по существу («потребуй единый источник») и по факту: именно дублирование даёт «починили в одном месте».
- **Читать цвета темы в iframe через CSS-переменные родителя.** Технически невозможно: `srcDoc`-документ не наследует CSS родительского документа (и не должен — это и есть изоляция). Отсюда литералы, синхронизированные с токенами.
- **Решить фикс 8 обрезкой длинных имён команд (`truncate` + `title`).** Отклонён: прямое нарушение правила CLAUDE.md — переполнение решается размером, а не скрытием значимого контента.
