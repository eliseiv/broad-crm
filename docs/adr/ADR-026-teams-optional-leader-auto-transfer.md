# ADR-026 · Команды без лидера: nullable `leader_id`, авто-назначение и авто-передача лидерства

- Статус: accepted
- Дата: 2026-07-08

## Контекст

Правка 5 батча — прямые решения пользователя по CRM-командам (амендмент к [ADR-022](ADR-022-teams-nav-categories.md), где лидер был обязателен и `teams.leader_id NOT NULL`):

1. **Команду можно создавать без лидера и без участников** (в т.ч. пустую).
2. **Первый добавленный участник автоматически становится лидером** (если лидера нет).
3. **Если текущий лидер покидает команду** (удалён/исключён из состава) → **авто-передача лидерства следующему участнику по дате добавления**; если участников не осталось → команда **без лидера**.
4. **Пересмотр инварианта «лидер ∈ участники»:** теперь «**если лидер есть — он ∈ участники**», но лидера может **не быть**.
5. **Пересмотр `409 user_is_team_leader`:** при авто-передаче удаление пользователя-лидера больше не должно блокироваться так же.

Действующая модель ([ADR-022](ADR-022-teams-nav-categories.md), [03-data-model.md](../03-data-model.md#таблицы-teams-и-user_teams-crm-команды)): `teams.leader_id NOT NULL`, FK `ON DELETE RESTRICT` (удаление лидера-пользователя → `409 user_is_team_leader`), инвариант «лидер ∈ участники», `user_teams` — M2M `(user_id, team_id)` без временной метки.

## Решение

### 1. `teams.leader_id` — nullable, FK `ON DELETE SET NULL`

- Колонка `teams.leader_id uuid **NULL**` (миграция `0012_teams_optional_leader`, [03-data-model.md](../03-data-model.md#таблицы-teams-и-user_teams-crm-команды)). `NULL` = команда **без лидера**.
- FK `teams.leader_id → users(id)` меняется с `ON DELETE RESTRICT` на **`ON DELETE SET NULL`** (предохранитель уровня БД: удаление пользователя никогда не блокируется командой). Прикладная **авто-передача** (см. §4) выполняется до/в транзакции удаления и обычно проставляет нового лидера раньше, чем сработал бы `SET NULL`.

### 2. Порядок участников: `user_teams.created_at`

- В `user_teams` добавляется `created_at timestamptz NOT NULL DEFAULT now()` (миграция `0012`) — «дата добавления» участника, нужна для детерминированного выбора «следующего участника» при авто-передаче и «первого» при авто-назначении.
- **Порядок для авто-лидера/авто-передачи (нормативно):** по возрастанию `(created_at ASC, user_id ASC)`. Для участников, добавленных **одной** операцией (POST/PATCH с `member_ids`), при равном `created_at` порядок соответствует их позиции в массиве `member_ids` (сервис вставляет их в этом порядке; тай-брейк — `user_id`).

### 3. Опциональные `leader_id`/`member_ids` + авто-назначение при создании

- `TeamCreateRequest`: `leader_id?` (опц.), `member_ids?` (опц., default `[]`). Возможна **пустая команда** без лидера и участников. Контракт — [04-api.md](../04-api.md#teams).
- **Авто-назначение лидера при `POST` (нормативно):**
  - `leader_id` задан → он лидер и добавляется в участники (как раньше).
  - `leader_id` не задан, `member_ids` непуст → лидером становится **первый** участник (по правилу §2 — фактически `member_ids[0]`).
  - оба не заданы/пусты → команда создаётся **без лидера** (`leader_id = NULL`), пустой состав.

### 4. Авто-передача лидерства при выбытии лидера

Единое правило на всех путях, где лидер покидает состав:
- **`PATCH /api/teams/{id}` (замена состава `member_ids`) / смена `leader_id`:**
  - `leader_id` задан → он лидер, добавляется в участники.
  - `leader_id` не задан, но текущий лидер **отсутствует** в новом `member_ids` (лидер исключён) → лидерство **авто-передаётся** первому из оставшихся участников (правило §2); если участников не осталось → `leader_id = NULL` (команда без лидера).
  - если у команды **не было** лидера и добавлены участники → первый добавленный становится лидером.
- **`DELETE /api/users/{id}` — удаление пользователя-лидера больше не блокируется.** Для каждой команды, где удаляемый — лидер: лидерство авто-передаётся следующему участнику (правило §2); если других участников нет → команда становится без лидера (`leader_id = NULL`). Затем пользователь удаляется (членства `user_teams` снимаются `ON DELETE CASCADE`). **Код `409 user_is_team_leader` упраздняется** — удаление пользователя-лидера завершается успешно (`204`) с авто-передачей.
- **`PATCH /api/users/{id}` (замена `team_ids`, исключение пользователя из команды):** если исключаемый — лидер этой команды, лидерство авто-передаётся (правило §2) или команда становится без лидера. Прежнее поведение «лидер не исключается / остаётся участником» ([ADR-022](ADR-022-teams-nav-categories.md)) **отменяется**.

### 5. Инвариант и агрегаты

- **Новый инвариант:** «**если `leader_id` задан — существует строка `(leader_id, team_id)` в `user_teams`**» (лидер ∈ участники, когда лидер есть). При `leader_id = NULL` инвариант тривиально выполнен.
- `TeamListItem`: `leader_id: string(uuid) | null`, `leader_username: string | null` (`null` — команда без лидера); `member_count = members.length` (может быть `0`); `members` может быть пустым. Контракт — [04-api.md](../04-api.md#teams).

## Обоснование

- **Nullable `leader_id` + `SET NULL`** — прямое выражение «команда без лидера»; `SET NULL` гарантирует, что команда никогда не блокирует удаление пользователя (устраняет источник `409 user_is_team_leader`), а прикладная авто-передача даёт осмысленного нового лидера вместо «просто NULL», когда есть кем заменить.
- **`user_teams.created_at`** — минимальное расширение M2M для детерминированного «следующего/первого по дате»; альтернатива (полагаться на порядок вставки без метки) недетерминирована при выборках.
- **Авто-назначение/авто-передача в сервисе** — единственная точка записи состава (как инвариант «лидер ∈ участники» в [ADR-022](ADR-022-teams-nav-categories.md)); БД-триггеров нет, логика тестируема чистыми функциями.
- **Упразднение `409 user_is_team_leader`** — прямое следствие требования: удаление лидера должно проходить с авто-передачей, а не блокироваться.

## Последствия

- (+) Гибкий жизненный цикл команды: пустая → наполняется (первый = лидер) → лидер меняется/выбывает (авто-передача) → без лидера, если пусто.
- (+) Удаление пользователя больше не блокируется его лидерством — меньше ручных шагов у админа.
- (+) Модель без лидера явна в контракте (`leader_id`/`leader_username` nullable).
- (−) Миграция `0012`: `leader_id` nullable + смена FK на `SET NULL` + новая колонка `user_teams.created_at`.
- (−) Упразднён код `409 user_is_team_leader` (амендмент к [ADR-022](ADR-022-teams-nav-categories.md); историческая запись ADR-022 сохраняется как есть).
- (−) UI «Команды» должен корректно отображать «Без лидера» и не фиксировать лидера жёстко в участниках.

## Альтернативы

- **Оставить лидера обязательным (RESTRICT + блокировка удаления)** — отвергнуто: прямое требование разрешить команды без лидера и авто-передачу.
- **`ON DELETE SET NULL` без прикладной авто-передачи** — отвергнуто как единственный механизм: терял бы осмысленного лидера, когда есть кем заменить; авто-передача даёт лучший UX. `SET NULL` оставлен как предохранитель.
- **Авто-передача по алфавиту/`username`** — отвергнуто: требование — «по дате добавления»; нужна временная метка (`user_teams.created_at`).
- **Хранить лидера как флаг в `user_teams` (без `teams.leader_id`)** — отвергнуто: ломает существующий контракт (`leader_id`/`leader_username` в `TeamListItem`, FK-денормализацию) без выгоды; nullable `leader_id` минимальнее.

## Амендмент (2026-07-08): ретроактивный backfill лидеров (миграция `0016`)

> Пометка-амендмент; тело решения выше не изменяется.

Форвард-поведение авто-назначения (§3) заработало **после** того, как на проде уже были созданы CRM-команды с участниками, но `leader_id IS NULL` (stale-данные: «Команда Ивана» — 2 участника без лидера, «Команда Мухамеда» — 1 участник без лидера). Новые команды и `PATCH` назначают лидера корректно, но исторические строки остаются «Без лидера» → пользователь видит незакрытый баг.

**Решение амендмента:** one-time backfill-миграция **`0016_backfill_team_leaders`** (`down_revision = 0015_user_first_login`) ретроактивно применяет то же правило авто-назначения (§2–§3) к существующим командам: для команд с `leader_id IS NULL` **и** непустым составом лидером становится первый участник по `(user_teams.created_at ASC, user_teams.user_id ASC)`. Команды без участников остаются без лидера (легитимный кейс §3). Инвариант §5 сохраняется (лидер берётся из состава). Правило идемпотентно; `downgrade()` — no-op (data-fix необратим). Рантайм-контракт и поведение эндпоинтов **не меняются** — это только коррекция данных. Концепт и SQL — [03-data-model.md «Миграция `0016_backfill_team_leaders`»](../03-data-model.md#миграция-0016_backfill_team_leaders-концепт).
