# ADR-068 · Изображения в документах: файлы на volume + таблица метаданных + авторизованная отдача

- **Статус:** accepted
- **Дата:** 2026-07-22
- **Контекст-модуль:** [modules/documents](../modules/documents/README.md)
- **Амендирует:** [ADR-059](ADR-059-documents-module.md) (модель модуля — добавляется таблица), [ADR-060](ADR-060-documents-external-readonly-api-key.md) (внешний контур — граница уточняется), [ADR-062](ADR-062-documents-wysiwyg-tiptap.md) (граница TipTap расширяется `@tiptap/extension-image`), [ADR-015](ADR-015-csp-img-src-remote-mail-images.md) (`img-src` расширяется `blob:`)
- **Закрывает частично:** [TD-065](../100-known-tech-debt.md) (вложения/изображения в документах)

## Контекст

Модуль «Документы» ([ADR-059](ADR-059-documents-module.md)) хранит **только markdown-текст** в `document_nodes.content_md`; вложения были явно вне scope ([TD-065](../100-known-tech-debt.md)), загрузка ограничена `.md`. Владелец требует **вставку изображений** в WYSIWYG-редактор ([ADR-062](ADR-062-documents-wysiwyg-tiptap.md)) двумя жестами: **кнопкой тулбара** и **вставкой из буфера обмена (Ctrl+V)**. Drag-and-drop — **не требуется** (решение владельца).

**Решение владельца по хранению — принято и не пересматривается:** файлы **на диске** (named volume, по аналогии с `file_sd`-таргетами) + **отдельная таблица метаданных**; в markdown — ссылка вида `![alt](/api/documents/attachments/{id})`.

Инварианты, которые нельзя нарушить:

1. **Видимость по ролям — per-node** ([ADR-059](ADR-059-documents-module.md)): картинка узла, невидимого пользователю, обязана быть недоступна **так же**, как сам узел. Публичная (безавторизационная) раздача файлов исключена.
2. **Анти-энумерация:** невидимое неотличимо от несуществующего (`404`, не `403`).
3. **`content_md` — каноничный markdown**, его потребляет внешний RAG-контур ([ADR-060](ADR-060-documents-external-readonly-api-key.md)); формат менять нельзя.
4. **CSP `script-src 'self'` не ослабляется ни при каких обстоятельствах** ([05-security.md](../05-security.md#csp)).
5. Soft-delete узла ([ADR-059](ADR-059-documents-module.md)) и «Создать копию» (рекурсивная копия поддерева) обязаны иметь определённое поведение для вложений.

## Решение

### 1. Таблица `document_attachments` (метаданные) + файл на volume (байты)

Строка метаданных — источник истины о вложении; файл — байты. Поля: `id`, `document_node_id` (FK `ON DELETE CASCADE`), `filename`, `mime`, `size_bytes`, `checksum` (sha256 hex), `created_by` (FK `users`, `ON DELETE RESTRICT` — как `document_nodes.owner_id`), `created_at`. DDL и индексы — [03-data-model.md](../03-data-model.md#таблица-document_attachments-adr-068).

**Путь на диске строится ТОЛЬКО из машинных значений:**

```
${DOCUMENTS_ATTACHMENTS_DIR}/<id[0:2]>/<id[2:4]>/<id>.<ext>
```

- `<id>` — UUID вложения (канонический lowercase-hex с дефисами);
- `<ext>` — **производное от `mime`** по константному whitelist-маппингу (`image/png→png`, `image/jpeg→jpg`, `image/webp→webp`, `image/gif→gif`);
- **пользовательская строка `filename` в пути не участвует вовсе** — path traversal невозможен **конструктивно**, а не за счёт санитайзинга (санитайзинг имён — исторический источник обходов: `..%2f`, NUL, юникод-нормализация, длинные имена);
- два уровня шардинга (`aa/bb`, 256×256) — чтобы каталог не вырождался в десятки тысяч файлов в одной директории.

Дополнительно — **defensive-проверка на отдаче**: `os.path.realpath(candidate)` обязан лежать **внутри** `os.path.realpath(DOCUMENTS_ATTACHMENTS_DIR)`, иначе `404` (страховка от симлинка/битой конфигурации, а не основная защита).

`checksum` — **sha256 содержимого**; используется как **`ETag`** отдачи и для проверки целостности после восстановления volume. UNIQUE по `checksum` **не вводится**: одинаковый файл в двух узлах хранится дважды (дедупликация с refcount усложнила бы копирование и GC ради экономии на масштабе, которого нет).

**Права файлов — `0600`, каталогов — `0700`** (владелец — uid backend). Это **намеренно строже**, чем `0644`/`0755` у `file_sd` ([09-provisioning.md](../09-provisioning.md#регистрация-таргета-file_sd)): у target-файлов есть **второй читатель под другим uid** (Prometheus), у вложений читатель ровно один — backend. Норму зафиксировать явно, чтобы «унификация прав» не сделала приватные картинки world-readable внутри контейнера.

### 2. Эндпоинты

| Метод / путь | Гейт | Назначение |
|--------------|------|-----------|
| `POST /api/documents/nodes/{id}/attachments` | `documents:edit` + видимость узла | Загрузка (multipart `file`) |
| `GET /api/documents/attachments/{id}` | `documents:view` + видимость узла-владельца | Отдача байтов |
| `DELETE /api/documents/attachments/{id}` | `documents:edit` + видимость узла-владельца | Удаление строки + файла |

**Загрузка — `documents:edit`, а не `create`** (требование владельца, и оно согласовано с моделью): вложение — часть **контента существующего** узла, а `create` в этом модуле означает «создать узел» ([04-api.md §RBAC](../04-api.md#rbac-и-enforcement-прав)). Узел обязан быть `node_type='document'` (папка контента не хранит — CHECK [ADR-059](ADR-059-documents-module.md)) → иначе `422 validation_error`.

**Отдача — гейт `documents:view` + тот же рекурсивный CTE видимости узла-владельца.** Все негативные исходы (вложения нет / узел невидим по роли / узел soft-deleted) дают **один** код **`404 document_attachment_not_found`** — иначе различие кодов сообщало бы о существовании невидимого узла (нарушение инварианта 2). Валидация — **по узлу-владельцу**, а не по «кто загрузил»: enforcement в модуле permission-based, не owner-based ([ADR-059](ADR-059-documents-module.md)).

**Валидация загрузки (нормативно, прецеденция сверху вниз):**

1. **Размер** — ≤ `DOCUMENTS_MAX_IMAGE_BYTES` (env, default `5242880` = 5 МБ). Проверяется **по мере чтения потока** с обрывом на превышении — файл целиком в память до проверки не читается.
2. **Тип — по фактическому содержимому (magic bytes), а не по `Content-Type` клиента:** `\x89PNG\r\n\x1a\n` (png), `\xFF\xD8\xFF` (jpeg), `RIFF....WEBP` (webp), `GIF87a`/`GIF89a` (gif). Заявленный `Content-Type` вне whitelist **или** расхождение заявленного и фактического → отказ. Причина: клиентский `Content-Type` подделывается тривиально, а сохранённый `mime` управляет заголовком отдачи.
3. Whitelist — **ровно четыре** типа: `image/png`, `image/jpeg`, `image/webp`, `image/gif`.
   **SVG в whitelist НЕ входит — нормативно и навсегда в рамках этого ADR:** SVG — активный документ (скрипты, `<foreignObject>`, внешние ссылки), отдаваемый с нашего origin ⇒ это XSS-вектор на собственном домене, который не закрывается ни `nosniff`, ни CSP страницы (файл открывается напрямую по своему URL). Владелец просил «фотографии» — растровых форматов достаточно.
4. Любой отказ выше → **`422 document_attachment_invalid`** (единый код: детализация «слишком большой» vs «не тот тип» ничего не даёт атакующему и не нужна UI сверх текста сообщения).

**Порядок записи (нормативно; редакция 2).** Первая формулировка («`INSERT` → `flush` → запись файла») была неисполнима: `checksum` и `size_bytes` — `NOT NULL`, а они известны только **после** вычитывания потока; и «temp в том же каталоге» невозможен до получения `id` (шард-каталог из него и выводится). Действующий порядок:

1. **Стриминг во временный файл в КОРНЕ `DOCUMENTS_ATTACHMENTS_DIR`** (`tempfile.mkstemp(dir=<корень>, prefix=".upload-", suffix=".tmp")`) с одновременным подсчётом **sha256** и **размера** и обрывом на превышении лимита. Корень выбран намеренно: он гарантированно на **той же файловой системе**, что финальный шард-путь ⇒ последующий `os.replace` останется атомарным (переименование через границу ФС упало бы с `EXDEV`).
2. **Проверки** (magic bytes по уже прочитанной «голове», итоговый размер) — до всякой записи в БД.
3. **`INSERT`** метаданных с **готовыми** `checksum`/`size_bytes` → **`flush`** ⇒ получен `id` ⇒ известен шард-путь.
4. `mkdir -p` шард-каталогов (`0700`) → **`os.replace(tmp, final)`** → **`commit`**.
5. Любой сбой до `commit` → `rollback` **и** `os.unlink(tmp)` в `finally`.

**Строки без файла не бывает** (файл кладётся на место до `commit`). Обратная аномалия — файл без строки (падение между `replace` и `commit`, либо брошенный `.tmp` при жёстком kill) — возможна и безвредна: осиротевший файл никем не адресуем, убирается GC ([TD-076](../100-known-tech-debt.md)); GC обязан подчищать и `.tmp`-остатки в корне по возрасту.

**Удаление:** строка удаляется в транзакции, файл — **после успешного `commit`** (best-effort; сбой → лог + GC). Обратный порядок дал бы строку, указывающую в пустоту. Ссылка в `content_md` **не переписывается автоматически** — редактор удаляет узел изображения сам; висячая ссылка деградирует до alt-текста (плашка «Изображение недоступно»), а не ломает документ.

**Ответ загрузки (`201`):** `{ id, document_node_id, filename, mime, size_bytes, checksum, url, created_at }`, где `url = "/api/documents/attachments/<id>"` — **сервер сам сообщает канонический URL**, клиент его не конструирует (иначе форма ссылки размазалась бы по фронту и стала бы неявным контрактом).

### 3. Отдача картинки: авторизованный `fetch` + `blob:`-URL (а не `<img src>` напрямую)

**Проблема, которую нельзя обойти:** SPA хранит JWT в `localStorage` ([ADR-041](ADR-041-login-theme-session-ux.md)), а браузер отправляет `<img src="/api/…">` **без** заголовка `Authorization`. Значит наивный `<img>` на защищённый эндпоинт получит `401` — либо, если «починить» это анонимным доступом, **картинка утечёт в обход видимости узла** (прямой запрет требования).

**Решение:** markdown хранит канонический `![alt](/api/documents/attachments/{id})`, а **клиент резолвит его сам**: компонент изображения делает `fetch` с `Authorization: Bearer` → получает `Blob` → `URL.createObjectURL(blob)` → подставляет в `src`; на размонтировании — **обязательный `URL.revokeObjectURL`** (иначе blob'ы копятся в памяти вкладки на всё время сессии). Ошибка/`404` → плашка «Изображение недоступно» + alt-текст, документ не ломается.

Цена — **CSP `img-src` дополняется `blob:`**: `img-src 'self' data: https: blob:`. Расширение узкое и пассивное: `blob:`-URL создаётся **нашим же** JS из данных, уже полученных под JWT, живёт в origin документа и не является внешним источником; `script-src 'self'` **не трогается** (инвариант 4 соблюдён). Значение CSP обязано побайтово совпасть в [05-security.md](../05-security.md#csp) и **в обоих** SPA-`location` nginx ([07-deployment.md](../07-deployment.md#reverse-proxy-nginx--требования)) — правило дублирования [ADR-046](ADR-046-ui-infra-fix-pack.md) §4.4.

**Отклонённые альтернативы отдачи:**

| Вариант | Почему отклонён |
|---------|-----------------|
| Анонимный `GET` по неугадываемому UUID (security through obscurity) | Прямо нарушает требование «не утечь в обход видимости»: UUID попадает в `content_md`, историю браузера, referer-цепочки, копии документа; отзыв доступа невозможен |
| Подписанный короткоживущий токен в query (`?t=<hmac>`) | Секрет в URL ⇒ в access-логах nginx и в истории — прямой запрет [05-security.md](../05-security.md#reveal-секретов-по-требованию-adr-035) («секрет в теле, не в URL»); плюс второй, параллельный JWT механизм аутентификации |
| Cookie-сессия для `/api/documents/attachments/*` | Ввод cookie-auth в проект без него = CSRF-контур, `SameSite`, отдельный жизненный цикл — несопоставимая цена (NFR-1) |
| `data:`-URI вместо `blob:` (CSP не менять) | `data:` уже разрешён, но: +33 % к памяти на base64, синхронная конверсия крупного изображения в UI-потоке; выигрыш — только «не трогать одну директиву CSP» |

### 4. `data:`-изображения в `content_md` запрещены

Расширение `@tiptap/extension-image` регистрируется с **`allowBase64: false`**. Причина: вставка из буфера в TipTap по умолчанию охотно превращает картинку в `data:`-URI прямо в документе — тогда 1 МБ-лимит `DOCUMENTS_MAX_MD_BYTES` расходуется на base64, хранилище вложений обходится, а RAG получает документ, забитый бинарным мусором вместо текста. **Любая картинка проходит через upload** — иных путей появления изображения в `content_md` нет.

Там же расширение регистрируется с **`inline: false`**: изображение — **блочный узел** ProseMirror, а не инлайн внутри абзаца. Причина: `tiptap-markdown` сериализует блочный узел в отдельную строку `![alt](/api/documents/attachments/{id})`, тогда как инлайн-вариант вклеивает картинку в текст абзаца — и round-trip markdown↔ProseMirror перестаёт быть устойчивым (переносы строк и соседний текст «прилипают» к ссылке). Блочная форма также совпадает с тем, как изображение отображается в read-only рендере документа и как его читает внешний RAG-контур.

### 5. Soft-delete узла и «Создать копию»

**Soft-delete узла** ([ADR-059](ADR-059-documents-module.md)): вложения **не трогаются физически** — ни строки, ни файлы (undelete отсутствует, но tombstone обязан оставаться консистентным для RAG). Отдача `GET /attachments/{id}` для вложения удалённого узла → **`404 document_attachment_not_found`** (узел исключён из всех внутренних выборок ⇒ и его вложения). Физическая уборка — вместе с будущим GC tombstones ([TD-067](../100-known-tech-debt.md), расширяется на файлы вложений).

**«Создать копию»** (`POST /nodes/{id}/copy`): вложения **копируются физически** — новые `id`, новые файлы (побайтовая копия), и **в `content_md` копии ссылки переписываются** со старых `id` на новые. Замена — **точечная, по литеральной подстроке** `/api/documents/attachments/<old_uuid>` → `/api/documents/attachments/<new_uuid>` для каждой пары из карты копирования; **regex-разбор markdown запрещён** (ложные срабатывания в коде/цитатах). Всё — в той же транзакции, что и копия поддерева.

Разделение файлов между копиями (общий файл + refcount) **отклонено**: копия перестала бы быть независимой (удаление оригинала обрывало бы картинки копии либо требовало бы счётчика ссылок и его согласованности при каждом soft-delete/GC). Цена решения — дублирование байтов при копировании, приемлемая при масштабе модуля.

**Ручной перенос ссылки между узлами — допустимое недокументированное состояние (зафиксировано осознанно).** Пользователь может скопировать строку `![alt](/api/documents/attachments/{id})` из одного документа в другой; `document_attachments.document_node_id` при этом **не меняется** (сервер не парсит markdown и о переносе не знает). Следствия, принятые как есть:

- **видимость картинки продолжает следовать узлу-ВЛАДЕЛЬЦУ**, а не тому, куда её вставили ⇒ у части читателей нового узла изображение молча не отрисуется (`404` → плашка «Изображение недоступно»), хотя сам текст они видят;
- `DELETE /attachments/{id}` (или удаление узла-владельца с последующим GC) **рвёт ссылку везде**, включая узлы, куда её перенесли.

Автоматически это не чинится: «переносить владение по факту появления ссылки» означало бы парсить markdown на каждом `PATCH` и решать конфликт при ссылке из двух узлов сразу, а «клонировать вложение при вставке» — угадывать намерение пользователя. Штатный путь получить независимую картинку в другом документе — **загрузить её туда** (или скопировать узел целиком — тогда вложения клонируются и ссылки переписываются, см. выше). Это не дефект реализации, а граница модели «вложение принадлежит ровно одному узлу».

### 6. Внешний RAG-контур ([ADR-060](ADR-060-documents-external-readonly-api-key.md)) — не расширяется

- Бинарные эндпоинты наружу **не вводятся**: `/api/external/documents/attachments/{id}` **не существует**. Внешний контур остаётся **текстовым** read-only контуром.
- `ExternalDocumentNode` **не расширяется** полем `attachments[]`. Обоснование: вложения **иммутабельны** — перезаписи файла нет, «замена картинки» = новая загрузка = новый `id` = изменение `content_md` = `content_version += 1`. Значит `content_version` и водяной знак `updated_at` **уже полностью** описывают состав узла для RAG, и отдельный список метаданных не даёт ему ни одного нового решения.
- Ссылки `![alt](/api/documents/attachments/{id})` остаются в `content_md` **как есть**; RAG индексирует alt-текст и видит факт наличия изображения. Это **явно зафиксированное ограничение**, а не недосмотр: контур read-only-текстовый ([TD-074](../100-known-tech-debt.md) — выдача бинарников наружу при появлении требования, отдельным ADR).

### 7. Кэширование отдачи

- **`Cache-Control: private, max-age=300, must-revalidate`** + **`ETag: "<checksum>"`**; `If-None-Match` совпал → `304`.
- **`public` ЗАПРЕЩЕНО:** ответ зависит от прав запрашивающего (per-node видимость) ⇒ shared-кэш прокси отдал бы картинку постороннему. Это тот же класс ошибки, что кэширование secret-ответов, только «наоборот».
- `Content-Disposition: inline; filename*=UTF-8''<pct-encoded filename>`; `Content-Type` — **из БД** (`mime`, проверенный по содержимому при загрузке), не из имени файла. `X-Content-Type-Options: nosniff` уже ставится backend-middleware на весь `/api` ([05-security.md](../05-security.md#http-заголовки-безопасности-нормативно)) — отдельно дублировать не нужно.

### 8. Развёртывание

Новый named volume **`documents-attachments`** (`DOCUMENTS_ATTACHMENTS_DIR`, default `/var/lib/crm/documents/attachments`), монтируется **только в `backend`** (в отличие от `file_sd`, который читает ещё и Prometheus). `client_max_body_size 50m` на `location /api` уже перекрывает 5 МБ-лимит — менять не нужно.

**Бэкап становится двухкомпонентным:** до сих пор единственным состоянием был `pgdata` ([07-deployment.md §Бэкапы](../07-deployment.md#бэкапы-и-данные)); теперь `pgdata` и `documents-attachments` обязаны сниматься **согласованно** — иначе восстановление даёт строки без файлов (битые картинки) или файлы-сироты.

## Последствия

**Положительные.** Требование владельца закрыто обоими жестами (тулбар + Ctrl+V). Хранение бинарников вне БД — БД не пухнет, дампы остаются лёгкими. Видимость картинки **автоматически** следует видимости узла (один и тот же CTE). Path traversal невозможен конструктивно. `content_md` остаётся каноничным markdown ⇒ RAG-контракт цел.

**Отрицательные / принятые.**
- Второй объект состояния (volume) и требование согласованного бэкапа.
- CSP `img-src` += `blob:` — правка нормативного значения в трёх местах (docs + два `location` nginx).
- Клиентский резолвер вместо «просто `<img>`»: обязательный `revokeObjectURL`, состояние загрузки, плашка ошибки.
- Возможны файлы-сироты (падение между записью и commit; сбой удаления) — нужен GC ([TD-076](../100-known-tech-debt.md)).
- Копия узла дублирует байты вложений (§5).
- Новая frontend-зависимость `@tiptap/extension-image` (официальное расширение линии TipTap 2.x — тот же класс, что `@tiptap/extension-link` в поправке [ADR-062](ADR-062-documents-wysiwyg-tiptap.md) §2). **При реализации (2026-07-22) в манифест добавлен и `@tiptap/pm` 2.27.2** — не новая библиотека в дереве (ProseMirror-бандл, и раньше присутствовавший транзитивно как peer `@tiptap/core`; версия в `frontend/package-lock.json` та же), а явный пин под **прямые** импорты `@tiptap/pm/{markdown,model,state,view}` в `frontend/src/features/documents/imageExtension.ts` и `imageUploadPlaceholder.ts`. Состав TipTap стал **шесть пакетов** — врезка-амендмент в [ADR-062](ADR-062-documents-wysiwyg-tiptap.md), таблица [02-tech-stack.md §Frontend](../02-tech-stack.md#frontend).
- SVG не поддерживается (§2.3) — осознанно.
- Внешний RAG не получает байты картинок ([TD-074](../100-known-tech-debt.md)).

## Альтернативы (хранение)

| Вариант | Почему отклонён |
|---------|-----------------|
| Байты в БД (`bytea` в `document_attachments`) | Решение владельца — файлы на диске. Плюс: раздувание дампов, TOAST-нагрузка, потоковая отдача из БД сложнее |
| S3/MinIO-совместимое хранилище | Новый инфраструктурный сервис ради нескольких МБ — прямое нарушение NFR-1; прецедент отказа — демонтаж MinIO агрегатора ([ADR-044](ADR-044-mail-full-merge-into-crm.md)) |
| `data:`-URI прямо в `content_md` (без хранилища) | Лимит 1 МБ на документ, base64-раздувание, мусор в RAG-корпусе — §4 |
| Переиспользовать `file_sd`-volume | Смешение двух несвязанных состояний с **разными правами** (`0644` world-readable vs `0600`) и разными потребителями |
