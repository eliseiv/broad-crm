# ADR-067 · Вход на целевой сервер по приватному SSH-ключу (второй способ рядом с паролем)

- **Статус:** accepted
- **Дата:** 2026-07-22
- **Контекст-модули:** [modules/servers](../modules/servers/README.md), [modules/provisioning](../modules/provisioning/README.md)
- **Амендирует:** [ADR-007](ADR-007-shifrovanie-fernet.md) (Fernet — расширяется на новый класс материала), [ADR-035](ADR-035-detail-view-secret-reveal.md) (контур reveal — **сужается** для key-серверов), [ADR-049](ADR-049-servers-backends-card-first-detail.md) §1 (состав detail-модалки сервера)

## Контекст

Сегодня сервер добавляется **только с паролем**: `POST /api/servers {name, ip, ssh_user, ssh_password}`, пароль шифруется Fernet (`app/infra/crypto.py`), колонка `servers.ssh_password_encrypted` — **`NOT NULL`**, провижининг кладёт расшифрованное значение в `ansible_password` inventory (`app/infra/ansible.py`), а `sshpass` в backend-образе обязателен именно ради password-SSH ([09-provisioning.md](../09-provisioning.md)).

Требование владельца (подтверждено): **парольный вход остаётся**, рядом добавляется **вход по приватному SSH-ключу** с **опциональной парольной фразой**; в форме добавления сервера — переключатель «Пароль / SSH-ключ». Значительная часть хостинг-провайдеров выдаёт машины вообще без парольного SSH (`PasswordAuthentication no`) — сегодня такой сервер в CRM добавить нельзя.

Ограничения, которые нельзя нарушить:

1. Секрет at-rest — **только** Fernet-ciphertext (`FERNET_KEY` из env, [ADR-007](ADR-007-shifrovanie-fernet.md)); plaintext не хранится, не логируется, не попадает в обычные ответы ([05-security.md](../05-security.md#защита-ssh-кредов-целевых-серверов)).
2. Провижининг — блокирующий `ansible-runner` во временном `private_data_dir`, удаляемом в `finally` ([09-provisioning.md](../09-provisioning.md#передача-кредов-безопасно)).
3. Каждая Alembic-миграция со схемными изменениями обязана иметь **рабочий** `downgrade()` ([03-data-model.md §2](../03-data-model.md#2-рабочий-downgrade--обязателен-карв-аут-необратимые-чисто-dml-миграции)).
4. Существующие записи серверов (парольные) обязаны продолжать работать **без ручного вмешательства**.

## Решение

### 1. Модель: дискриминатор `auth_method` + два новых зашифрованных поля

`servers` получает:

| Колонка | Тип | Ограничения |
|---------|-----|-------------|
| `auth_method` | `text` | `NOT NULL`, `DEFAULT 'password'`, CHECK `IN ('password','key')` |
| `ssh_private_key_encrypted` | `bytea` | `NULL` — Fernet-ciphertext приватного ключа |
| `ssh_key_passphrase_encrypted` | `bytea` | `NULL` — Fernet-ciphertext парольной фразы (опциональна даже в key-режиме) |
| `ssh_password_encrypted` | `bytea` | **`NOT NULL` → `NULL`** |

Согласованность материала гарантирует **CHECK на уровне БД** (`ck_servers_auth_material`), а не только сервис:

```sql
CHECK (
  (auth_method = 'password'
     AND ssh_password_encrypted     IS NOT NULL
     AND ssh_private_key_encrypted  IS NULL
     AND ssh_key_passphrase_encrypted IS NULL)
  OR
  (auth_method = 'key'
     AND ssh_private_key_encrypted  IS NOT NULL
     AND ssh_password_encrypted     IS NULL)
)
```

Смысл: **ровно один способ входа на запись**, «половинчатых» строк (key без ключа, password без пароля, пароль+ключ одновременно) в БД быть не может даже при ошибке сервиса или ручном `UPDATE`. Passphrase допустима только вместе с ключом.

**Отклонено — отдельная таблица `server_credentials`.** 1:1-расширение ради двух nullable-колонок = лишний JOIN на каждом чтении списка + собственный жизненный цикл; дискриминатор в той же строке проще и покрывается одним CHECK (NFR-1).

**Отклонено — хранить ключ файлом на volume (как file_sd/вложения документов).** SSH-креды — секрет класса «Fernet at-rest в БД» ([05-security.md](../05-security.md#управление-секретами)); вынос на диск создал бы второй, несогласованный с БД носитель секретов и второй объект бэкапа.

### 2. Миграция `0031_servers_ssh_key_auth` (25 симв. ≤ 32)

`down_revision` = фактическая голова цепочки на момент реализации (сейчас — `0030_document_node_roles`).

**`upgrade()`** — `ADD COLUMN auth_method` **со `server_default='password'`** → `ADD COLUMN` двух bytea → `ALTER COLUMN ssh_password_encrypted DROP NOT NULL` → два CHECK.

**Backfill данных не требуется — и это следствие порядка операций, а не совпадение:** `server_default` проставляет `'password'` **всем существующим строкам в момент `ADD COLUMN`**, а `ssh_password_encrypted` у них уже `NOT NULL` ⇒ на момент создания `ck_servers_auth_material` каждая существующая строка ему **удовлетворяет**. Отдельного `UPDATE` нет; CHECK добавляется **последним** (до него строки временно без ограничения — допустимо внутри одной транзакции миграции). Совместимость со старыми записями полная: старый сервер = `auth_method='password'` и ведёт себя ровно как прежде.

**`downgrade()` — рабочий (не `pass`), карв-аут §2 неприменим (есть DDL):**

1. `DROP CONSTRAINT ck_servers_auth_material`, `ck_servers_auth_method`;
2. **`DELETE FROM servers WHERE auth_method = 'key'`** — key-серверы в старой схеме **непредставимы** (`ssh_password_encrypted NOT NULL`, а пароля у них нет и взять его неоткуда). Подстановка фиктивного пароля-заглушки **запрещена**: провижининг после отката пошёл бы на хост с заведомо неверным паролем, а оператор увидел бы «пароль есть» вместо честного отсутствия сервера. Каскады срабатывают штатно (`notifier_server_state` — `CASCADE`, `notifier_alert_log`/`backends.server_id` — `SET NULL`);
3. `ALTER COLUMN ssh_password_encrypted SET NOT NULL`;
4. `DROP COLUMN` трёх новых колонок.

**Цена отката зафиксирована явно:** `downgrade` **теряет строки** key-серверов (не только колонки). Поэтому откат этой миграции — операция уровня 1 политики отката **с обязательным pre-deploy `pg_dump`** ([07-deployment.md §Откат миграций БД](../07-deployment.md#откат-миграций-бд)); файлы `targets/<id>.json` удалённых серверов подчищаются регенерацией file_sd из БД. Требование к миграции: перед `DELETE` — `logger`-строка/`RAISE NOTICE` с числом удаляемых строк (без секретов), чтобы потеря не была молчаливой.

### 3. Контракт `POST /api/servers` — дискриминированное тело

```jsonc
{ "name": "...", "ip": "...", "ssh_user": "...",
  "auth_method": "password" | "key",      // опционально, default "password"
  "ssh_password": "...",                   // ровно при auth_method=password
  "ssh_private_key": "-----BEGIN ...",     // ровно при auth_method=key
  "ssh_key_passphrase": "..." }            // опционально, только при auth_method=key
```

**`auth_method` опционален с дефолтом `password`** — старое тело `{name,ip,ssh_user,ssh_password}` остаётся валидным; ломающего изменения контракта нет.

Валидация (нормативно, прецеденция — сверху вниз):

1. **Ровно один способ.** Поле «чужого» режима присутствует (даже `null`/`""`) → `422 validation_error` с `details[].field` = имя лишнего поля. Обязательное поле режима отсутствует/пустое → `422`, `field` = имя недостающего.
2. **Лимиты размера.** `ssh_password` 1–256 (как сейчас); `ssh_key_passphrase` 1–256; `ssh_private_key` — **1–`SSH_KEY_MAX_BYTES`** (env, default `16384` = 16 КБ; покрывает RSA-4096 PEM с запасом). Проверка размера — **до** парсинга (анти-DoS: не отдавать многомегабайтную строку в разбор).
3. **Нормализация перед проверкой и шифрованием:** `CRLF → LF`, срез хвостовых пробелов, гарантированный завершающий `\n` (OpenSSH отвергает ключ без newline в конце файла). Хранится **нормализованная** форма.
4. **Формат ключа — реальный разбор, а не regex по заголовку.** Разбор выполняет уже установленный `cryptography` ([ADR-007](ADR-007-shifrovanie-fernet.md) — новой зависимости нет). Regex-проверка заголовка отклонена как **достаточная**: она пропускает битую base64-середину, и ошибка всплыла бы только при провижининге, уже под `status=error`.

   **⚠️ Процедура (нормативно; редакция 2 — первая была неисполнима).** Наивное `load_ssh_private_key(data, password=passphrase)` **не даёт** требуемых исходов: на `cryptography` 43.x (пин проекта — `>=43,<44`) все негативные ветки приходят **одним `ValueError`**, различить «битый ключ» / «нужна фраза» / «неверная фраза» **по типу исключения нельзя**, а разбирать его **текст** запрещено (не контракт библиотеки, ломается от версии к версии и может нести фрагменты материала). Поэтому исход задаётся **нашей** проверкой, а не поведением библиотеки:

   **Шаг 1 — определить `is_encrypted` СТРУКТУРНО** (по документированным полям формата, без криптоопераций). **Ветки проверяются СВЕРХУ ВНИЗ, первое совпадение выигрывает; последняя — catch-all, и ни один ввод не может «провалиться» мимо всех:**
   - PEM-armor `-----BEGIN ENCRYPTED PRIVATE KEY-----` → `true` (PKCS#8 encrypted);
   - PEM-armor `… PRIVATE KEY` + заголовок `Proc-Type: 4,ENCRYPTED` (и/или `DEK-Info:`) → `true` (legacy PEM);
   - **PEM-armor `-----BEGIN PRIVATE KEY-----` / `-----BEGIN RSA PRIVATE KEY-----` / `-----BEGIN EC PRIVATE KEY-----` / `-----BEGIN DSA PRIVATE KEY-----` БЕЗ `Proc-Type`/`DEK-Info` → `is_encrypted = false`** — это **незашифрованный** PKCS#8 (`PRIVATE KEY`) или PKCS#1/SEC1 (`RSA`/`EC`/`DSA PRIVATE KEY`); загрузка на шаге 3 — `load_pem_private_key(data, password=None)`. **Ветка обязательна и является самым частым случаем** (`ssh-keygen -m PEM`, ключи от облачных провайдеров, экспорт из `openssl`): без неё такой ключ не покрывался бы ни одной из веток выше и при буквальном чтении catch-all ошибочно уехал бы в `422`. `DSA PRIVATE KEY` здесь **не отвергается** — тип ключа проверяется на **шаге 4** (там же `422` «Тип ключа не поддерживается»), чтобы у пользователя было корректное сообщение вместо «не удалось разобрать»;
   - armor `-----BEGIN OPENSSH PRIVATE KEY-----` → base64-декодировать тело, проверить магию `openssh-key-v1\0`, прочитать **первое** length-prefixed поле `ciphername`: `"none"` → `false`, иначе → `true`;
   - **всё остальное** (armor отсутствует / это не `*PRIVATE KEY`-armor / у OPENSSH-armor база64 не декодируется или нет магии) → **сразу `422`, `field="ssh_private_key"`** («Не удалось разобрать приватный SSH-ключ»). Сюда попадают **публичный** ключ (`ssh-rsa AAAA…`), сертификат, `known_hosts`-строка, `-----BEGIN CERTIFICATE-----`, произвольный текст. ⚠️ **Эта ветка НЕ должна ловить PEM-armor `… PRIVATE KEY` без `Proc-Type`** — он уже разобран веткой выше.

   **Шаг 2 — кросс-проверка с вводом пользователя (ДО загрузки).** Именно здесь появляются исходы по `ssh_key_passphrase`, и они больше **не зависят** от того, ругается ли библиотека на лишний/недостающий пароль (на 43.x для незашифрованного ключа лишний пароль может быть **молча проигнорирован**):
   - `is_encrypted = false`, а `ssh_key_passphrase` задана → `422`, `field="ssh_key_passphrase"` («Ключ не защищён парольной фразой — уберите её»);
   - `is_encrypted = true`, а `ssh_key_passphrase` не задана → `422`, `field="ssh_key_passphrase"` («Ключ защищён парольной фразой — укажите её»).

   **Шаг 3 — загрузка** соответствующим загрузчиком (`load_ssh_private_key` для OpenSSH-armor, `load_pem_private_key` для PEM) с `password = passphrase or None`. Ветка отказа выбирается по **уже известному** `is_encrypted`, а не по типу/тексту исключения:
   - успех → OK;
   - исключение при `is_encrypted = true` → `422`, `field="ssh_key_passphrase"` («Неверная парольная фраза»);
   - исключение при `is_encrypted = false` → `422`, `field="ssh_private_key"` («Не удалось разобрать приватный SSH-ключ»).

   **Шаг 4 — тип ключа.** Загруженный объект обязан быть **RSA**, **ECDSA** (NIST P-256/384/521) или **Ed25519**. **DSA не поддерживается** и отвергается здесь же — `422`, `field="ssh_private_key"` («Тип ключа не поддерживается»): в `cryptography` 43 DSA помечен deprecated, а OpenSSH ≥ 7.0 отключил `ssh-dss` по умолчанию ⇒ пере-сериализация в OpenSSH-формат (§5) на DSA >1024 бит падает. Без явной проверки такой ключ **прошёл бы валидацию и упал на провижининге** — ровно тот класс отложенной ошибки, ради устранения которого выбран реальный разбор.

   **Границы правила о логировании.** Внутреннее чтение исключения (тип/атрибуты) **не запрещено**, но по процедуре выше **не требуется** — ветвление детерминировано `is_encrypted`. Запрет остаётся на **проброс наружу и в лог**: текст исключения не попадает ни в ответ, ни в structlog; сообщения фиксированы выше. Разбор **локальный, без сети**.

   **Известная неточность (принята):** зашифрованный ключ с верной фразой, но повреждённым содержимым, отдаст «Неверная парольная фраза». Отличить это без разбора текста исключения нельзя; цена — неточная подсказка в редком случае, выгода — детерминизм, не зависящий от версии библиотеки.
5. Шифрование — `encrypt_secret` тем же `FERNET_KEY` (**один примитив на все три материала**; отдельного ключа для SSH-ключей не вводится — [ADR-007](ADR-007-shifrovanie-fernet.md), ротация — [TD-006](../100-known-tech-debt.md)).

Ответы `ServerCreatedResponse`/`ServerSummaryResponse`/`ServerListItem` расширяются полем **`auth_method`** (не секрет: способ входа, а не материал). Полей `has_password`/`has_key` **не вводится** — `auth_method` их и так однозначно определяет (гарантия CHECK из §1).

`PATCH /api/servers/{id}` — **без изменений** (по-прежнему только `name`). Смена способа входа/ротация кредов существующего сервера требует репровижининга и остаётся вне scope ([Q-UI-4](../99-open-questions.md), [TD-072](../100-known-tech-debt.md)).

### 4. Reveal: приватный ключ и парольная фраза НЕ раскрываются (сужение ADR-035)

**Решение:** новых reveal-эндпоинтов **не вводится**. `GET /api/servers/{id}/ssh-key` и `/ssh-key-passphrase` **не существуют**; приватный ключ и passphrase — **write-only** секреты. Существующий `GET /api/servers/{id}/ssh-password` сохраняется без изменений для `auth_method='password'`, а при `auth_method='key'` отдаёт **`404 secret_not_set`** — тот же код и та же семантика, что у прокси без пароля ([04-api.md](../04-api.md#reveal-секретов-по-требованию-adr-035)).

**Обоснование (почему здесь разворачивается логика ADR-035):**

- **Радиус поражения несопоставим.** Обоснование [ADR-035](ADR-035-detail-view-secret-reveal.md) — «секрет относится к одному ресурсу, держатель `edit` им уже управляет». Приватный SSH-ключ — **переиспользуемый** креденшл: один и тот же ключ обычно открывает **весь парк** хостов, включая машины вне CRM. Утечка пароля = один сервер; утечка ключа = всё, куда положен его публичный ключ. Уравнивать их гейтом `servers:edit` нельзя.
- **Гейт `servers:edit` для сервера и без того слабейший в наборе.** У прокси/ИИ-ключа/бэка держатель `edit` **может перезаписать** секрет через `PATCH` ⇒ раскрытие ему симметрично. У сервера `PATCH` правит **только `name`** — симметрии нет; это прямо отмечено открытым [Q-SEC-5](../99-open-questions.md). Расширять на несимметричный секрет с бо́льшим радиусом — движение в неверную сторону.
- **Рабочего сценария нет.** Пароль оператор вводит руками и может забыть → reveal закрывает реальную потребность. Приватный ключ оператор **не набирает** — у него есть локальный файл; «достать ключ из CRM» — сценарий экспорта/эксфильтрации, а не эксплуатации.
- **Half-reveal бессмыслен.** Раскрывать только passphrase — либо бесполезно (без ключа), либо опасно (в паре с утёкшим ключом снимает последний барьер).

Следствия: CRM — **не хранилище ключей**; потеря локальной копии ключа означает пересоздание сервера ([TD-073](../100-known-tech-debt.md)). Аудит `secret_revealed` для key-серверов просто не порождается (раскрывать нечего). Строгий вариант «reveal ключа под admin-only» отклонён: он вводил бы канал эксфильтрации ради несуществующей потребности.

### 5. Передача ключа в ansible-runner: снятая passphrase + временный файл `0600`, удаляемый в `finally`

Провижининг для `auth_method='key'`:

1. `decrypt_secret(ssh_private_key_encrypted)` (+ passphrase) **в памяти**;
2. **passphrase снимается в памяти**: ключ загружается `cryptography` и **пере-сериализуется в незашифрованный OpenSSH-PEM** (`PrivateFormat.OpenSSH` + `NoEncryption()`);
3. результат пишется в файл **внутри временного `private_data_dir`** — файл создаётся с `os.open(..., O_CREAT|O_EXCL|O_WRONLY, 0o600)` (не `open()` + `chmod` — иначе есть окно, когда файл существует с umask-правами);
4. в inventory host_vars — `ansible_ssh_private_key_file: "<путь>"` (+ `ansible_user`, `ansible_connection: ssh`); `ansible_password` в key-режиме **не задаётся вовсе**;
5. `finally` — существующий `shutil.rmtree(private_data_dir, ignore_errors=True)` удаляет и каталог, и ключ. Дополнительно: явный `os.remove` файла ключа **до** `rmtree` (чтобы промах `ignore_errors` по каталогу не оставил именно ключ).

**Где живёт `private_data_dir` (нормативно; поправка к первой редакции §5).** Каталог создаётся **не** безадресным `tempfile.mkdtemp()` (сегодня — `app/infra/ansible.py`, уходит в `/tmp`), а `tempfile.mkdtemp(dir=ANSIBLE_PRIVATE_DATA_ROOT, prefix="ansible-runner-")`, где `ANSIBLE_PRIVATE_DATA_ROOT` (env, default `/var/run/crm/ansible`) — **выделенный** каталог, созданный в backend-образе с `chown app:app` и `0700` и перекрытый в проде **отдельным `tmpfs`-mount'ом**. Полный сниппет Dockerfile/compose — [07-deployment.md](../07-deployment.md#tmpfs-для-приватных-данных-ansible-нормативно-adr-067-5). Два требования, каждое из которых уже ловило бы прод-инцидент:

- **⛔ `tmpfs` с `mode=1700` — ОШИБКА, ломающая провижининг ЦЕЛИКОМ, включая существующую парольную ветку.** Backend работает под **non-root** (`USER app` в `backend/Dockerfile`), а Docker монтирует `tmpfs` под `root:root`; при `1700` процесс `app` не создаст в нём каталог, `tempfile.mkdtemp` упадёт с `PermissionError`, и **любое** добавление сервера завершится ошибкой — не только по ключу. Нормативен эффективный режим **`0o1777`** (семантика системного `/tmp`: sticky-бит защищает от чужого удаления, а сам ключ закрывают права **вложенного** `mkdtemp`-каталога `0700` и файла `0600`). Опции `uid=`/`gid=` для tmpfs в compose не выражаются — поэтому именно `0o1777`.
- **⛔ Нормативная форма записи режима — `mode: 0o1777`.** Поле `tmpfs.mode` в compose — **числовое** (`uint32`), отсюда: **(а)** любое значение **в кавычках** отвергается на разборе — падают `docker compose config` и `up`, стек не поднимается целиком; **(б)** голое `mode: 1777` разбирается **десятично** ⇒ режим `0o3361` (`stat` → `0o43361`) — без sticky и без `w` для others, т.е. `mkdtemp` под `app` падает с `PermissionError`, тот же дефект, что и `1700`; **(в)** корректны только `0o1777` и **некавыченное** `01777` (go-yaml читает ведущий ноль как восьмеричный литерал YAML 1.1 — целое, не строку). Комментарий «восьмерично» рядом со значением парсер не читает. Фактические прогоны, docker 29.6.1 / compose v5.1.4 (`alpine:3.20`, long-syntax mount): `mode: "01777"` → `decoding failed … 'services[a].volumes[0].tmpfs.mode' expected type 'uint32', got unconvertible type 'string'`; `mode: 0o1777` → `1777 drwxrwxrwt`; `mode: 01777` → `1777 drwxrwxrwt`; `mode: 1777` → `3361 d-wxrwS--t`. Под `user: "1000:1000"` `mkdir` в точке монтирования проходит при `0o1777` и даёт `Permission denied` при `1777`. Сниппет и предупреждение — [07-deployment.md §tmpfs](../07-deployment.md#tmpfs-для-приватных-данных-ansible-нормативно-adr-067-5).
- **⛔ Монтировать `tmpfs` на `/tmp` не следует** (предписание первой редакции **отменено**): помимо той же проблемы прав, `/tmp` — место, куда Starlette/`python-multipart` спуливают `UploadFile` больше ~1 МБ, т.е. загружаемые изображения документов до 5 МБ ([ADR-068](ADR-068-documents-image-attachments.md)) конкурировали бы за тот же маленький раздел с артефактами ansible-runner. Каталоги **разведены**: `/tmp` остаётся обычным, Ansible работает в своём tmpfs.

Без tmpfs схема остаётся **работоспособной** (каталог существует в образе с `0700`/`app`) — ключ лишь окажется в пишущем слое контейнера, что повышает остаточный риск [TD-073](../100-known-tech-debt.md), но ничего не ломает.

**Почему passphrase снимается, а не передаётся Ansible.** OpenSSH не принимает парольную фразу из переменной окружения или файла — единственный путь это `ssh-agent` + `ssh-add` с интерактивным вводом или `SSH_ASKPASS`-обвязкой. Штатный параметр `ansible_runner.run(ssh_key=...)` именно так и работает (пишет ключ + поднимает `ssh-agent`) и на **зашифрованном** ключе повиснет на приглашении ввода фразы до `job_timeout` — то есть добавление сервера молча превращалось бы в `provisioning timeout`. Снятие фразы в памяти убирает интерактив целиком: passphrase **никогда не покидает процесс backend** и не попадает ни в inventory, ни в env, ни в лог.

**Остаточный риск (принят осознанно, [TD-073](../100-known-tech-debt.md)):** на время прогона плейбука расшифрованный ключ существует файлом. Митигации: каталог `mkdtemp` — `0700`, файл — `0600`; корневой каталог `ANSIBLE_PRIVATE_DATA_ROOT` в проде — **`tmpfs`** (требование к devops), т.е. ключ не касается постоянного диска и не попадает в бэкапы/volume'ы; удаление в `finally`; время жизни ограничено `ANSIBLE_TIMEOUT_SEC`.

**Логи не меняются:** inventory в key-режиме содержит только **путь** к файлу — печатать его безопасно; сам материал в inventory/extravars/логи не попадает. `no_log: true` на тасках с паролем сохраняется. `sshpass` остаётся обязательным (password-режим никуда не делся), в key-режиме не используется.

**Классификация ошибок** дополняется одной строкой: ключ расшифровался, но не загружается `cryptography` (например, после ротации `FERNET_KEY` или ручной правки строки) → `status=error`, `error_message="SSH key unusable"` — отдельно от `"SSH connection failed"`, иначе оператор чинил бы сеть вместо кредов.

### 6. UI

- **`AddServerModal` (режим `add`)** — сегментированный переключатель **«Пароль / SSH-ключ»** (radio-группа) под полем «Пользователь». `password` → поле «Пароль» (как сейчас). `key` → `Textarea` **«Приватный ключ»** (моношрифт) + `Input type=password` **«Парольная фраза (опц.)»**. Переключение режима **очищает** поля другого режима — тело запроса всегда несёт материал ровно одного способа (иначе сервер вернёт `422` по §3.1). Режим `edit` — **без изменений** (только «Название»).
- **`ServerDetailModal`** — после строки «Пользователь» появляется строка **«Способ входа»** (`Пароль` / `SSH-ключ`). При `password` — существующая строка «Пароль» с reveal (гарантии [ADR-049](ADR-049-servers-backends-card-first-detail.md) §4 не ослаблены). При `key` — строка **«SSH-ключ»** со значением `••••••••` и **без кнопки-глаза**: маска показывает, что материал задан, но раскрытия нет (§4). Это первый случай маски **без** reveal — правило [ADR-035](ADR-035-detail-view-secret-reveal.md) «кнопка-глаз рендерится при `<page>:edit`» дополняется: для не-раскрываемого секрета глаз **не рендерится ни при каком праве**.
- **`ServerCard`** — не меняется: способ входа не выносится на карточку (карточка — про метрики и статус).

## Последствия

**Положительные.** Добавляются серверы с `PasswordAuthentication no`. Материал ключа шифруется тем же проверенным примитивом. Битый ключ/неверная фраза ловятся **на форме** (`422`), а не спустя минуты `status=error`. Существующие серверы и старое тело запроса продолжают работать без действий оператора.

**Отрицательные / принятые.**
- `downgrade` миграции **удаляет** key-серверы — откат требует бэкапа (§2).
- Ключ **невосстановим** из CRM ([TD-073](../100-known-tech-debt.md)) — осознанная цена решения §4.
- Расшифрованный ключ кратковременно материализуется файлом (§5), безопасность опирается на `tmpfs` + права + `finally`.
- Смена способа входа существующего сервера невозможна ([TD-072](../100-known-tech-debt.md)).
- В форме появляется ветвление — покрывается тестами обеих веток (qa).

## Альтернативы

| Вариант | Почему отклонён |
|---------|-----------------|
| Заменить пароль ключом (один способ) | Прямо противоречит требованию владельца; сломал бы все существующие серверы |
| `ansible_runner.run(ssh_key=...)` (штатный ssh-agent) | Виснет на зашифрованном ключе до `job_timeout` — добавление сервера превращается в `provisioning timeout` (§5) |
| `sshpass -P passphrase … ssh-add` вокруг agent'а | Тот же результат, что снятие фразы, но с внешним процессом, парсингом приглашения и фразой в argv (видна в `ps`) |
| Хранить ключ в файле-volume, в БД — только путь | Второй носитель секретов, рассинхрон БД↔диск, новый объект бэкапа; противоречит [05-security.md §Управление секретами](../05-security.md#управление-секретами) |
| Reveal ключа под `servers:edit` | Радиус поражения и отсутствие симметрии с `PATCH` (§4); гейт `servers:edit` для сервера — слабейший ([Q-SEC-5](../99-open-questions.md)) |
| Валидация ключа regex по BEGIN/END | Пропускает битую base64-середину — ошибка всплывает уже в `status=error` (§3.4) |
