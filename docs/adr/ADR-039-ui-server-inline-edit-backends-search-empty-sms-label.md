# ADR-039 · UI-пакет: инлайн-редактирование сервера в detail-view + поиск и пустое состояние страницы «Бэки» + лейбл «СМС» в матрице прав `/roles`

- Статус: accepted
- Дата: 2026-07-09
- Затрагивает: [modules/ui](../modules/ui/README.md), [modules/servers](../modules/servers/README.md), [modules/backends](../modules/backends/README.md)
- Амендмент: [ADR-035](ADR-035-detail-view-secret-reveal.md) (detail-view/editable-scope сервера), [ADR-020](ADR-020-backends-healthcheck-monitor.md) (страница «Бэки»), [ADR-030](ADR-030-sms-module-full-merge.md) (каталог прав `sms`)

## Контекст

Четыре не связанных между собой UI-правки, объединённых в один пакет (по образцу [ADR-023](ADR-023-ui-nav-dropdown-proxy-ip-single-delete.md)/[ADR-029](ADR-029-ui-login-password-nav-team-form.md)/[ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md)). Контракты API и БД **не затрагиваются** — правки чисто фронтовые/нормативно-словарные.

1. **Редактирование сервера — отдельным окном.** По [ADR-035](ADR-035-detail-view-secret-reveal.md) карандаш в `ServerDetailModal` **закрывает** detail и открывает **отдельную** edit-модалку `AddServerModal mode='edit'` (правит только `name`). Пользователь требует, чтобы редактирование происходило **непосредственно на странице просмотра** сервера (инлайн), без нового окна.
2. **Пустое состояние `/backends` перегружено текстом.** При отсутствии бэков (у держателя `backends:create`) показывается блок «Пока нет бэков» + «Добавьте первый бэк» **вместе** с карточкой добавления. Пользователь просит убрать текст-подсказку, оставив только карточку добавления.
3. **Нет поиска на `/backends`.** Список бэков растёт; поиска нет (в отличие от вкладки «Номера» `/sms`).
4. **Лейбл `sms` в матрице прав `/roles`.** Каталог прав ([ADR-030](ADR-030-sms-module-full-merge.md)) содержит страницу `sms`, поэтому редактор роли рендерит для неё строку. Локализованной подписи `sms` во фронт-словаре (`features/users/labels.ts::PAGE_LABEL`) **нет** → строка падает на фолбэк технического ключа (`sms`). Нужна русская подпись **«СМС»** (согласована с пунктом навигации и вкладкой, [ADR-030](ADR-030-sms-module-full-merge.md)/[ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md)).

## Решение

### 1. Сервер — инлайн-редактирование прямо в `ServerDetailModal` (амендмент [ADR-035](ADR-035-detail-view-secret-reveal.md) §6)

- Карандаш `Pencil` (гейт `servers:edit`) в `ServerDetailModal` **больше не открывает** `AddServerModal mode='edit'`. Вместо этого он **переключает саму detail-модалку в inline-edit-режим**: поле **«Название»** становится редактируемым (`Input`, префил текущим `name`) прямо в теле detail-view; появляются кнопки **«Сохранить»** (primary) и **«Отмена»** (ghost). Прочие поля (`IP`/`Пользователь`/`Пароль`) остаются read-only (репровижининг вне scope — [modules/servers](../modules/servers/README.md#out-of-scope)).
- **«Сохранить»** → `PATCH /api/servers/{id} {name}` (существующий контракт, не меняется) → при успехе модалка возвращается в read-only-режим с обновлённым именем; toast «Сервер обновлён». **«Отмена»** / Esc → возврат в read-only без запроса. Ошибки (`422`/`409`/сеть) — инлайн под полем/toast.
- **`AddServerModal mode='edit'` для сервера больше не используется** и выводится из употребления на странице «Серверы» (режим `add` `AddServerModal` — создание сервера — сохраняется без изменений). Reveal SSH-пароля в detail-view ([ADR-035](ADR-035-detail-view-secret-reveal.md)) не меняется и в inline-edit-режиме кнопка-глаз остаётся доступной.
- **Прочие сущности (Прокси / ИИ-ключ / Бэк) — карандаш по-прежнему открывает существующую `Add<Entity>Modal mode='edit'`** (у них многополевая форма и/или секреты, модалка уместна). Разворот на инлайн затрагивает **только сервер** (единственное тривиальное редактируемое поле `name` — инлайн проще и не требует окна). Механика разведения жестов ([ADR-011](ADR-011-poryadok-blokov-server-side-dnd-kit.md)) и короткий-клик→detail не меняются.

### 2. Пустое состояние `/backends` — без текста-подсказки (амендмент [ADR-020](ADR-020-backends-healthcheck-monitor.md))

- В варианте «нет бэков, есть `backends:create`» рендерится **только карточка `AddBackendCard`** — заголовок «Пока нет бэков» и подсказка «Добавьте первый бэк» **убираются**. Кнопка добавления остаётся.
- Read-only-вариант (есть `view`, нет `create`) — без изменений (по-прежнему «Список бэков пуст», карточка добавления не рендерится): при отсутствии и текста, и кнопки страница иначе была бы полностью пустой.

### 3. Поиск на `/backends` — клиентский фильтр (амендмент [ADR-020](ADR-020-backends-healthcheck-monitor.md))

- Над сеткой карточек — поле поиска (`Input`, плейсхолдер **«Поиск по бэкам…»**), по образцу поиска вкладки «Номера» `/sms`.
- Фильтрация **клиентская** по уже загруженному `GET /api/backends` (бэков немного, NFR-1): совпадение по подстроке (регистронезависимо) в **любом** из трёх полей — `code`, `name`, `domain`. Серверный поиск **не** вводится.
- При непустом запросе без совпадений — состояние **«Ничего не найдено»**; `AddBackendCard` при активном поиске может скрываться (как на `/sms`). Drag-and-drop-перестановка при активном фильтре не выполняется (порядок применяется к полному списку).

### 4. Лейбл `sms → «СМС»` в матрице прав `/roles` (амендмент [ADR-030](ADR-030-sms-module-full-merge.md))

- Во фронт-словарь подписей страниц матрицы прав (`features/users/labels.ts::PAGE_LABEL`, нормативный источник — [08-design-system.md](../08-design-system.md#страница-роли)) добавляется маппинг **`sms → «СМС»`**. Матрица рендерит строку `sms` из `GET /api/permissions/catalog` (каталог не меняется — `sms` там уже есть); подпись строки — «СМС».
- Каталог прав `sms` содержит действия `view/edit/transfer/sync/delete`, но матрица рендерит только столбцы `view/create/edit/delete` — действия `transfer`/`sync` через матрицу не управляются (существующее ограничение, симметрично `mail:sync/tags`; вне scope этой правки — см. [Последствия](#последствия)).

### 5. Группировка бэков по одинаковому `name` на `/backends` (амендмент [ADR-020](ADR-020-backends-healthcheck-monitor.md))

Модель бэка: `code` — **UNIQUE**, `name` — **НЕ уникален** (обычный `text NOT NULL`, без `UNIQUE` — [03-data-model.md](../03-data-model.md#таблица-backends)), поэтому несколько бэков могут иметь одинаковое название. Требование — показывать бэки с **полностью совпадающим** `name` **сгруппированно вместе**.

- **Группировка — клиентская** по уже загруженному `GET /api/backends` (нового эндпоинта нет, NFR-1). Ключ группы — **точное совпадение** `name` (case-sensitive, «100 %»).
- **Порядок:** список приходит отсортированным по `position ASC, created_at DESC, id`. Группы формируются стабильно поверх этого порядка: сорт-ключ группы = **минимальный `position`** среди её членов (группа появляется на месте своего первого по порядку бэка); внутри группы — по `position`.
- **Рендер:**
  - Группа из **≥2 бэков** с одинаковым `name` — **визуальный кластер**: контейнер группы (`--surface-2`/тонкая граница, скругление) с компактным заголовком «`<name>` · N» (название + счётчик) и вложенными `BackendCard` членов группы.
  - **Одиночный** бэк (его `name` уникален в текущем наборе) — обычная standalone-`BackendCard` **без** заголовка/контейнера (чтобы не плодить заголовки на каждую карточку).
- **Согласование с поиском (§3):** фильтр применяется к карточкам (по `code`/`name`/`domain`), **затем** отфильтрованный набор перегруппировывается (группа может сократиться до 1 члена → рендерится как standalone). Нет совпадений → «Ничего не найдено».
- **Согласование с DnD:** drag-and-drop-перестановка сохраняется и оперирует **плоским `position`-списком** (`PATCH /api/backends/order` не меняется). Группировка — **презентационный слой поверх `position`-порядка**: после перестановки кластеры пересобираются по `name` (группа привязана к позиции первого члена). Осознанный трейдофф (NFR-1): `position` — грубый порядок, первичный организующий принцип на странице — кластеризация по названию; чтобы свести бэки одной группы вместе вручную, их перетаскивают друг к другу. Группировка **как единица** перетаскивания (drag всего кластера) — отклонена ради простоты (см. [Альтернативы](#альтернативы-отклонены)).
- `AddBackendCard` — вне групп (standalone, в конце); при активном поиске может скрываться (как на `/sms`).

## Последствия

- `docs/08-design-system.md`: detail-view сервера — inline-edit-режим (карандаш → редактирование `name` в модалке, Сохранить/Отмена) вместо `AddServerModal`; страница «Бэки» — поле поиска + пустое состояние без текста + **группировка карточек по одинаковому `name`** (кластер с заголовком «`name` · N» для дублей, singletons — обычные карточки); матрица `/roles` — строка «СМС» и словарная запись `sms → «СМС»`.
- `docs/modules/ui/README.md`: `ServerDetailModal` inline-edit; `AddServerModal mode='edit'` для сервера не используется.
- `docs/modules/backends/README.md`: поиск (клиентский, `code`/`name`/`domain`), пустое состояние без текста, группировка по `name`.
- Frontend: `ServerDetailModal` (inline-edit state), `BackendsPage` (search input + filter, empty-state без текста), `PAGE_LABEL += sms:'СМС'`. Контракты/БД не трогаются.
- **Не в scope (существующее ограничение):** матрица прав не показывает столбцы `transfer`/`sync`/`tags` — управление этими действиями `sms`/`mail` через `/roles` недоступно; выделено как наблюдение, отдельным решением не закрывается в этом ADR.

## Альтернативы (отклонены)

- **Инлайн-редактирование для всех четырёх сущностей.** Отклонено: у прокси/ИИ-ключа/бэка многополевые формы с секретами — модалка остаётся уместной; у сервера редактируется единственное поле `name`, инлайн проще. Разворот точечный.
- **Серверный поиск бэков** (`GET /api/backends?q=`). Отклонено: объём мал (NFR-1), клиентский фильтр по загруженному списку достаточен и не добавляет эндпоинт (симметрично поиску `/sms`).
- **Убрать строку `sms` из матрицы** вместо добавления подписи. Отклонено: `sms` — легитимная страница каталога прав; ролям нужно назначать `sms:*`. Проблема была в отсутствующей подписи, не в наличии строки.
- **Группа-как-единица DnD** (перетаскивание всего кластера одинаковых `name`). Отклонено: смешение drag группы и drag карточки усложняет @dnd-kit-модель; при NFR-1 достаточно кластеризации поверх `position`.
- **Серверная группировка** (`GET /api/backends?group_by=name`). Отклонено: объём мал, группировка чисто презентационная — считается на клиенте по загруженному списку.
- **Заголовок-группа на КАЖДЫЙ бэк** (в т.ч. singletons). Отклонено: заголовок на каждую уникальную карточку — визуальный шум; кластер только при ≥2 одинаковых `name`.
