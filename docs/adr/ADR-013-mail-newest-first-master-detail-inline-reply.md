# ADR-013 · Почты — newest-first backward-пагинация, master-detail и inline-reply

- Статус: **`partially superseded by` [ADR-044](ADR-044-mail-full-merge-into-crm.md)**

> ⚠️ **ЧАСТИЧНО ОТМЕНЁН [ADR-044](ADR-044-mail-full-merge-into-crm.md) (2026-07-10).** **Механика пагинации отменена:** режимов `order=asc|desc`, курсоров `since_id`/`before_id` и полей ответа `next_since_id`/`next_before_id`/`has_more` в действующем контракте **НЕТ**. Лента читается **из БД CRM**, сортируется по **`internal_date DESC, id DESC`** (по истинной дате письма, **не** по `id` — `id` отражает порядок прихода push'а) и пагинируется **компаундным opaque-курсором `(internal_date, id)`**: параметр `before` → поле ответа `next_cursor`. Актуальный контракт — [04-api.md §GET /api/mail/messages](../04-api.md#get-apimailmessages).
>
> **ЧТО ДЕЙСТВУЕТ:** UX-решения — **master-detail** (список ~30% / тело ~70%), **авто-выбор самого свежего** письма, **бесконечная лента newest-first** (`IntersectionObserver`, без кнопки «Загрузить ещё», дедуп по `id`), **inline-reply** (`Textarea` + «Ответить», без модалки), теги-чипы.
- Дата: 2026-07-04
- Контекст модулей: [mail](../modules/mail/README.md)
- Связанные: [ADR-012](ADR-012-mail-read-through-proxy.md) (read-through-прокси без хранения — **не отменяется**, расширяется), [05-security.md](../05-security.md), внешний контракт mail-агрегатора **ADR-0036** (backward-пагинация external-API)

## Контекст

Модуль «Почты» ([ADR-012](ADR-012-mail-read-through-proxy.md)) на старте был спроектирован под **keyset вперёд** (`since_id`, `id ASC`), так как внешний external-API `postapp.store` умел только это. Следствия, зафиксированные как [TD-024](../100-known-tech-debt.md): нет строгого глобального «новые сверху» (только в пределах батча), пагинация — кнопка «Загрузить ещё».

Внешний mail-агрегатор финализировал контракт **ADR-0036**, добавивший в `GET /api/external/messages` **backward-режим**:

- `order` (enum `asc|desc`, default `asc`); `before_id` (int `ge=1`, **только** при `order=desc`).
- `order=desc` без `before_id` → последние N писем (`id DESC`, самые свежие); с `before_id` → письма `id < before_id` по `id DESC`.
- Ответ desc-режима: `next_before_id` (`int|null` = `min(id)` батча, `null` если старее нет) + `has_more`.
- asc-режим прежний (`since_id`/`next_since_id`). Взаимоисключение режимов → `400`. `tags[]` возвращаются в обоих режимах без изменений.

Параллельно пользователь запросил переработку UX страницы «Почты» (5 требований): вкладка «Почты» первой + дефолтный маршрут `/mail`; master-detail (список слева / тело справа) с авто-выбором самого свежего письма; inline-ответ в стиле чата вместо модалки `ReplyModal`; бесконечная лента (новейшие сверху, догрузка старых при скролле, без кнопки «Загрузить ещё»); цветные пилюли тегов.

Нужно решить: (1) как CRM-прокси использует backward-режим и остаётся стабильным контрактом; (2) какой режим — основной для страницы; (3) целевую UX-модель страницы, не отменяя инвариантов [ADR-012](ADR-012-mail-read-through-proxy.md).

## Решение

### 1. CRM-прокси пробрасывает backward-режим; `desc` — основной режим страницы

`GET /api/mail/messages` расширяется параметрами `order` (`asc|desc`) и `before_id`, проксируемыми во внешний `GET /api/external/messages`. В ответ добавляется `next_before_id` (`int|null`); `next_since_id` и `has_more` сохраняются. Схема `MailListResponse` держит **оба** курсора как `int|null`: заполнен курсор запрошенного режима, второй — `null`. Точные поля/правила — [04-api.md#mail](../04-api.md#get-apimailmessages).

- **Основной режим страницы — `desc`** (newest-first). Frontend всегда шлёт `order=desc` явно; CRM-прокси **всегда передаёт `order` во внешний API явно** (не полагается на внешний default `asc`). Default CRM-эндпоинта — `desc` (отражает основной сценарий страницы; отличие от внешнего default `asc` осознанно и задокументировано, см. [04-api.md#mail](../04-api.md#get-apimailmessages)). asc-режим (`since_id`) сохранён для обратной совместимости и как супертип.
- **Взаимоисключение режимов** (`before_id` при `order=asc`; `since_id` при `order=desc`) → `400 validation_error` — согласно текущей политике модуля (валидация параметров запроса, как `limit` вне диапазона). CRM валидирует локально до вызова внешнего API; внешний `400` (взаимоисключение) также маппится в `400 validation_error` как страховка.

**Почему `desc` основной, а не оставить asc:** страница теперь newest-first (свежее письмо — сверху и авто-выбрано). desc-backward даёт **строгий глобальный** порядок «новые сверху» с догрузкой старых — то, чего не хватало при keyset-вперёд. Это снимает newest-first-часть [TD-024](../100-known-tech-debt.md).

### 2. Бесконечная лента вместо «Загрузить ещё»

Страница грузит новейшие `limit=20` (`order=desc`), при скролле вниз догружает старые (`before_id = next_before_id`, `limit=20`) через `IntersectionObserver` на sentinel в конце списка. Остановка при `has_more=false`. Дедуп по `id`. Кнопка «Загрузить ещё» удаляется. Опционально (v1) — фоновый poll `order=desc` с малым `limit` для prepend новых писем (`id > текущего max`), без нарушения порядка. Детали — [08-design-system.md](../08-design-system.md#бесконечная-лента-infinite-scroll) и [modules/mail](../modules/mail/README.md#пагинация-нормативно).

### 3. Master-detail layout + inline-reply вместо модалки

- **Master-detail:** список писем слева (~30%), тело выбранного письма справа (~70%). По умолчанию выбрано и показано **самое свежее** письмо (первое в desc-ленте). На узких вьюпортах — одноколоночный режим: список → по тапу full-width деталь с кнопкой «Назад» (значимый контент — тело, значения — **не скрывается и не обрезается**, см. правило CLAUDE.md; тело скроллится внутри своего контейнера).
- **Inline-reply:** под телом письма в правой панели — многострочный `Textarea` (`body`, обязателен) + кнопка «Ответить» рядом (chat-like). Форма ответа = **только** `Textarea` + «Ответить», **без** полей `to`/`cc`/`subject` (см. [Поправку 2026-07-04](#поправка-2026-07-04--ux-доводка-страницы)); ответ шлётся телом `{body}`, дефолты (`to`=отправитель, `subject`=`Re:`) подставляет внешний сервис (`to`/`cc`/`subject` опциональны в контракте). Успех → toast «Ответ отправлен» + очистка поля. Прежняя модалка `ReplyModal` **удаляется/заменяется**.

**Почему master-detail + inline:** служебная лента одного оператора — быстрый просмотр «читаю сверху → сразу отвечаю» удобнее модалки; двухпанельный layout соответствует enterprise-эстетике (Linear/Superhuman) и токенам проекта.

### 4. Инварианты [ADR-012](ADR-012-mail-read-through-proxy.md) сохранены

Read-through без хранения (нет БД/миграций); `MAIL_API_KEY` только в заголовке `X-API-Key` исходящего запроса backend (не в SPA/логах/URL); `body_html` — **только** в sandbox-iframe (без `allow-scripts`/`allow-same-origin`); JWT на обоих эндпоинтах; идемпотентность ретраев (`reply` POST не ретраит read-timeout/`5xx`). Ничего из этого ADR-013 не меняет.

## Последствия

**Плюсы:**
- Строгий глобальный newest-first + бесконечная лента (снята newest-first-часть [TD-024](../100-known-tech-debt.md)).
- UX «читаю → отвечаю» без модального контекст-свитча; master-detail компактнее.
- Минимальное расширение контракта: проброс двух параметров + одно новое поле ответа; ADR-012 не отменён.

**Минусы / принятый долг:**
- Server-side **поиск/фильтры** по-прежнему недоступны (внешний API их не отдаёт) — [TD-024](../100-known-tech-debt.md) **сужается** до поиска/фильтров (newest-first снят).
- `MailListResponse` несёт оба курсора (`next_since_id`, `next_before_id`), один всегда `null` — небольшая избыточность ради стабильной схемы для обоих режимов.
- Зависимость от доступности `postapp.store` и внешнего rate-limit 120 req/min сохраняется; фоновый poll (v1) должен быть неагрессивным (`env.pollIntervalMs`).

## Альтернативы (отклонены)

- **Оставить asc + «Загрузить ещё»** — не даёт строгого глобального newest-first (претензия пользователя), backward-режим внешнего API остался бы неиспользованным.
- **Отдельный response-тип под desc-режим** — усложняет frontend (две схемы). Единый `MailListResponse` с двумя nullable-курсорами проще и стабильнее.
- **Модалка reply (как было)** — противоречит требованию «написал и сразу отправил»; контекст-свитч на модалку медленнее inline.
- **Синхронизация писем в БД CRM** (для поиска/фильтров) — против NFR-1 и [ADR-012](ADR-012-mail-read-through-proxy.md); остаётся путём закрытия остатка [TD-024](../100-known-tech-debt.md).

## Поправка (2026-07-04) — UX-доводка страницы

По итогам ревью UI пользователь запросил доводки страницы «Почты». Изменения — **UI/UX-уровня, без изменения контракта API** (`04-api.md#mail`, `MailReplyRequest`/`MailListResponse` не меняются) и **без нарушения инвариантов [ADR-012](ADR-012-mail-read-through-proxy.md)** (read-through без хранения; `MAIL_API_KEY` только на backend; `body_html` только в sandbox-iframe без `allow-scripts`/`allow-same-origin`; JWT). Затрагиваются только `08-design-system.md` и `modules/mail/README.md`.

1. **Full-bleed `/mail`.** Страница занимает всю ширину и примыкает к sticky-хэдеру (без `max-w-[1400px]`/`py-8`). Реализация — **условный контейнер `<main>` в `AppLayout` по активному маршруту** (`/mail` → `w-full` без паддингов; прочие маршруты — прежний `mx-auto max-w-[1400px] px-6 py-8`), чтобы не задеть вёрстку «Серверы»/«ИИ-ключи». Эквивалент — per-route layout. Детали — [08](../08-design-system.md#full-bleed-layout-нормативно).
2. **Единый серый фон тела письма.** Контейнер тела — единый серый `--surface-2` (`#161A22`) для text и html. Для sandbox-iframe серый достигается инъекцией базового стиля `html,body{background:#161A22;…}` в **обёртку `srcDoc`** (sandbox **не ослабляется**); произвольный HTML с собственным фоном — **best-effort**. Детали — [08](../08-design-system.md#деталь-письма-правая-панель).
3. **Клиентский фильтр «Только с тегами».** Тумблер над списком фильтрует загруженный набор по непустому `tags[]`; догрузка старых по скроллу продолжается. Детали — [08](../08-design-system.md#фильтры-ленты-с-тегами--почта--команда). (Переименован в **«С тегами»** и дополнен серверными фильтрами по ящику/команде — [ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md).)
4. **Выравнивание кнопки «Ответить».** Кнопка — штатной высоты ДС, выровнена по центру высоты `Textarea` (`flex items-center`, `Textarea flex-1`), не растягивается на всю высоту.
5. **Удаление «Расширенно».** Блок disclosure и поля `to`/`cc`/`subject` убраны из формы reply. Ответ отправляется телом `{body}`; дефолты (`to`=`from_addr`, `subject`=`Re: <subject>`) подставляет внешний сервис (поля опциональны в `MailReplyRequest`). **Заменяет** исходную формулировку §3 про свёрнутые «расширенные поля».
6. **«Получено на:» — имя + адрес.** В детали письма (`MailDetail`) подпись аккаунта-получателя показывает **и `display_name`, и `email`**: «Получено на: {display_name} <{email}>»; при пустом `display_name` — только `{email}`. Оба значения видны полностью (правило CLAUDE.md — не обрезать; длинный адрес переносится, не `truncate`).

> **Смежная правка (страница «Серверы», вне модуля mail).** В том же спек-проходе убрана ручная кнопка **«Обновить»** со страницы «Серверы» (шапка `ServersPage`): данные обновляются штатным polling'ом/`refetch`, без явной кнопки. Прочее на «Серверы» не меняется. См. [08-design-system.md · Страница «Серверы»](../08-design-system.md#страница-серверы).
