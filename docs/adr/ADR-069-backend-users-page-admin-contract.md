# ADR-069: Страница «Пользователи бэков» и универсальный CRM Admin API contract v1

- **Статус:** accepted
- **Дата:** 2026-07-23

## Контекст

Бэки в реестре CRM мониторятся по health-check ([ADR-020](ADR-020-backends-healthcheck-monitor.md)),
а их admin-ключи уже хранятся зашифрованными ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)),
но управлять пользователями бэков (баланс токенов, подписка) можно только вручную
через curl к каждому бэку. У бэков исторически два варианта префикса admin-эндпоинтов:
`/api/billing/admin` и `/v1/admin`; единого контракта нет (эталон 232-claude-backend
имеет только два write-эндпоинта и ни одного admin-чтения).

## Решение

1. **Универсальный контракт, а не интеграция под конкретный бэк.** Зафиксирован
   **CRM Admin API contract v1** (эталонный текст — у владельца в
   `BA/crm-admin-api-contract.txt`, сводка — [modules/backend-users](../modules/backend-users/README.md)):
   auth `X-Admin-Key`, 6 read- и 2 write-эндпоинта, ISO 8601 UTC, пагинация
   `limit/offset`, сортировка `registered_at DESC` (нормативна — на ней построен
   merge), опциональные блоки (`revenue`/`media_stats`) = `null`, а не ошибка.
   Страница в CRM написана один раз и работает с любым бэком, реализующим контракт.
2. **Оба префикса поддерживаются автоопределением** (404 на кандидате → следующий;
   рабочий префикс кэшируется в памяти процесса) — без нового поля в БД и без
   ручной настройки. Цена: до одного лишнего запроса после рестарта процесса.
3. **CRM — прокси без хранилища.** Собственных таблиц/миграций нет; данные читаются
   на лету, admin-ключ расшифровывается в памяти и не покидает backend CRM.
   Отклонено: синк пользователей в БД CRM (дублирование источника истины,
   устаревание, объём) — при необходимости отчётности можно вернуться (TD не заводится).
4. **Режим «Все приложения»** — конкурентный fan-out (семафор 5, паттерн монитора)
   + merge-пагинация окном ≤ 1000; упавший источник не роняет ответ, а попадает в
   `errors[]` (partial-data warning в UI). При явном `backend_id` ошибка источника
   пробрасывается точным кодом.
5. **RBAC:** новая страница каталога прав `backend-users` с действиями
   `view`/`edit` ([ADR-021](ADR-021-rbac-users-roles.md)); admin-операции — только
   под `edit` + аудит-событие `backend_admin_action` (structlog, ADR-035-паттерн,
   без значений секретов).
6. **Идемпотентность операций:** «Начислить токены» не идемпотентна по контракту —
   UI блокирует двойной сабмит; «Установить план» идемпотентна по `grant_id`,
   который генерируется при открытии модалки (безопасный ретрай; `applied=false`
   распознанного повтора показывается отдельным тостом).
7. **Ошибки контракта** транслируются в отдельные коды 04-api.md
   (`backend_admin_rejected` / `backend_admin_not_supported` /
   `backend_admin_unavailable` / `backend_admin_key_not_set` /
   `backend_user_not_found` / `backend_admin_bad_request`) — UI различает
   «нет ключа», «ключ не принят», «контракт не реализован» и «бэк лежит».

## Последствия

- Новые файлы: `infra/backend_admin_client.py`, `services/backend_user_service.py`,
  `schemas/backend_user.py`, `api/backend_users.py`; frontend —
  `features/backend-users/`, две страницы, модалки действий; маршруты
  `/backend-users`, `/backend-users/:backendId/:userId`, пункт навигации.
- 232-claude-backend (и остальные бэки) должны реализовать контракт v1 отдельными
  задачами в своих репозиториях; до этого страница показывает их ошибкой
  `backend_admin_not_supported` (или бэк не участвует в fan-out без admin-ключа).
- Глубокая пагинация «Все приложения» дороже (дочитывание страниц источников),
  ограничена окном 1000.
