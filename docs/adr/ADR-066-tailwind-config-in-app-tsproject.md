# ADR-066 — Включение `tailwind.config.ts` в файловый список app-проекта (`tsconfig.app.json`): фикс TS6307 прод-сборки от токен-контракт-теста ADR-064

- **Статус:** accepted
- **Дата:** 2026-07-21
- **Область:** frontend / build-конфигурация (`frontend/tsconfig.app.json`)
- **Связи:** логическое продолжение [ADR-064](ADR-064-design-tokens-channel-format.md) (его токен-контракт-тест — импортёр, вызвавший TS6307). **Формат токенов и логика [ADR-064](ADR-064-design-tokens-channel-format.md)/[ADR-065](ADR-065-users-flat-list-team-chips.md) не меняются** — это правка исключительно состава composite-проекта TypeScript, не кода и не тестов.

## Контекст

[ADR-064](ADR-064-design-tokens-channel-format.md) ввёл channel-формат цветовых токенов. В рамках его проверки появился токен-контракт-тест `frontend/src/components/ui/__tests__/Button.tokens.test.ts`, который **статически импортирует** `frontend/tailwind.config.ts`, чтобы **non-vacuously** компилировать реальный Tailwind-конфиг (иначе тест сверял бы только собственные строковые ожидания, не касаясь фактического конфига).

Прод-сборка фронтенда — `npm run build` = **`tsc -b && vite build`** ([frontend/Dockerfile](../02-tech-stack.md), [07-deployment.md](../07-deployment.md)). Ключевой момент: `tsc -b` — это **build-режим composite-проектов**, а не `tsc --noEmit` (последний используется в `npm run typecheck`, [02-tech-stack.md](../02-tech-stack.md)).

Механика бага:

1. Тест лежит под `src` → входит в composite-проект `tsconfig.app.json` (`include: ["src"]`).
2. Тест импортирует `tailwind.config.ts`, который находится **вне `src`** (в корне `frontend/`) и **не входил в файловый список ни одного** composite-проекта (`tsconfig.app.json` / `tsconfig.node.json`).
3. В **build-режиме** `tsc -b` требует, чтобы каждый импортируемый `.ts`-файл принадлежал файловому списку проекта-импортёра (или проекта, на который тот ссылается через `references`). Нарушение даёт:
   ```
   src/components/ui/__tests__/Button.tokens.test.ts(6,24): error TS6307:
   File '/app/tailwind.config.ts' is not listed within the file list of project '/app/tsconfig.app.json'.
   ```
4. **Локально ошибка маскировалась** инкрементальным кешем `tsBuildInfoFile` (`node_modules/.tmp/tsconfig.app.tsbuildinfo`): при инкрементальной пересборке TS не переоценивал состав проекта, и TS6307 не всплывал. **Чистая Docker-сборка** (пустой контекст, кеша нет) ошибку поймала — прод-сборка упала.

## Рассматриваемые варианты

- **(A) Включить `tailwind.config.ts` в `include` app-проекта** (`tsconfig.app.json`). Одна строка, разрешает зависимость в том же проекте, где живёт импортёр-тест. **Принят.**
- **(B) Убрать статический импорт конфига из теста** (вернуть тест к строковым ожиданиям без импорта реального конфига). Отклонён: разрушает цель токен-контракт-теста из [ADR-064](ADR-064-design-tokens-channel-format.md) — non-vacuous сверку **фактического** `tailwind.config.ts`; тест снова стал бы vacuous.
- **(C) Отдельный composite-проект для корневых конфигов** (`tsconfig.config.json` + `references`). Отклонён: оверинжиниринг ради одного файла — лишний проект, лишний `tsBuildInfo`, усложнение графа сборки без выгоды.
- **(D) Добавить `tailwind.config.ts` в `tsconfig.node.json`** (проект vite/eslint-конфигов). Отклонён: `tsconfig.node.json` — **не** проект-импортёр теста; TS6307 требует, чтобы файл был в списке проекта, который его импортирует (app), поэтому добавление в node-проект ошибку **не сняло бы**.

## Решение

Принят **вариант (A).** В `frontend/tsconfig.app.json` поле `include`: `["src"]` → **`["src", "tailwind.config.ts"]`**. Единственная правка; логика/тесты не тронуты.

### Обоснование выбора именно `tsconfig.app.json` (а не node-проекта)

- **Импортёр уже принадлежит app-проекту.** Тест лежит под `src`, то есть в `tsconfig.app.json`. TS6307 снимается включением импортируемого файла в файловый список **проекта-импортёра** — это app, а не node. Добавление в node-проект (вариант D) ошибку не устранило бы (см. выше).
- **`tailwind.config.ts` не добавляет в app-проект новых типов/ошибок.** Конфиг тянет только `import type { Config } from 'tailwindcss'` (**type-only** импорт) и экспортирует конфиг-объект; в `noEmit`-режиме app-проекта он лишь type-check'ится под `strict`/`noUnusedLocals`/`noUnusedParameters` и проходит без побочных ошибок. Компиляция app-проекта расширяется на один тривиальный файл, бандл не затрагивается (`noEmit: true`; сам Tailwind обрабатывает конфиг отдельно от `tsc`).

## Последствия

**Плюсы:**
- Чистая прод-сборка (`tsc -b && vite build` в Docker) проходит полностью.
- Токен-контракт-тест [ADR-064](ADR-064-design-tokens-channel-format.md) остаётся **non-vacuous** (реальный конфиг компилируется).
- `tailwind.config.ts` теперь под type-check'ом app-проекта (`strict`) — желаемый побочный эффект.
- Правка минимальна и обратима (одна запись в `include`).

**Минусы / риски (незначительны):**
- Конфиг попадает в граф зависимостей app-проекта. `noEmit: true` → на бандл влияния нет; риск только в том, что будущие type-ошибки в самом конфиге теперь провалят `tsc -b` (это желаемо, а не дефект).

### §Урок (нормативно): эталон проверки type-сборки — ЧИСТАЯ сборка без кеша

Инкрементальный `tsBuildInfoFile` (`node_modules/.tmp/*.tsbuildinfo`) **маскирует локально** ошибки состава composite-проекта — в первую очередь **TS6307** (файл вне файлового списка проекта): при инкрементальной пересборке TS не переоценивает членство файлов, и ошибка не воспроизводится, пока кеш «тёплый». Поэтому:

- **Локальный `npm run typecheck`/`tsc -b` с сохранённым `tsBuildInfo` — НЕ эталон** прохождения type-сборки. Он может быть зелёным при реально сломанной чистой сборке.
- **Эталон** — сборка **без кеша**: удалить `node_modules/.tmp/*.tsbuildinfo` перед `tsc -b`, **либо** полагаться на **чистую Docker-сборку** (`docker compose build` proxy-образа стартует с пустого контекста — кеша `tsBuildInfo` в нём нет).
- Это согласуется с [06-testing-strategy.md](../06-testing-strategy.md): «SPA собирается на прод-сервере в `docker compose build`, что само по себе валидирует сборку» — валидатором её делает именно **чистота** контекста (см. там же кросс-ссылку на этот ADR).
