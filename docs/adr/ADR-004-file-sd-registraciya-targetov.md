# ADR-004 · file_sd для динамической регистрации таргетов

- Статус: accepted
- Дата: 2026-06-28

## Контекст

При добавлении сервера нужно динамически зарегистрировать его как scrape-таргет Prometheus без ручного редактирования конфига и без рестарта Prometheus.

## Решение

Использовать **file-based service discovery (file_sd)** Prometheus. Backend пишет файл `${FILE_SD_DIR}/<id>.json` (общий volume с Prometheus) с таргетом `<ip>:<exporter_port>` и метками `server_id`/`name`. Prometheus сконфигурирован на `file_sd_configs` с `refresh_interval: 30s` и перечитывает каталог без рестарта.

Запись атомарна (temp-файл + `os.replace`). Удаление сервера удаляет файл. Формат — [09-provisioning.md](../09-provisioning.md#регистрация-таргета-file_sd).

## Обоснование

- file_sd — штатный, простой механизм Prometheus, не требует внешних систем.
- Нет рестартов Prometheus, нет гонок при редактировании одного большого конфига (по файлу на сервер).
- Файлы восстановимы из БД (реестр серверов).

## Последствия

- (+) Простая интеграция, отсутствие доп. инфраструктуры (NFR-1).
- (+) Изоляция: один файл — один сервер, удаление тривиально.
- (−) Нужен общий volume backend↔Prometheus и атомарная запись.
- (−) Задержка появления таргета до `refresh_interval` (≤30 с) — приемлемо.

## Альтернативы

- **HTTP SD / Consul / прямое редактирование `prometheus.yml`** — отвергнуто: лишняя инфраструктура или рестарты, сложнее для Этапа 1.
- **Pushgateway** — неприменимо: node_exporter — pull-модель.
