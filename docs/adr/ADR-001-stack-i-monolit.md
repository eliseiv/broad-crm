# ADR-001 · Стек проекта и монолитная архитектура

- Статус: accepted
- Дата: 2026-06-28

## Контекст

Этап 1 CRM-мониторинга разрабатывается небольшой командой (1–2 человека). Базовый стек зафиксирован пользователем: Python+FastAPI, React+Vite, PostgreSQL, Prometheus + node_exporter + Grafana, Ansible, кастомные SVG-спидометры. NFR-1 требует простоты.

## Решение

1. **Backend — монолит на FastAPI** (Python 3.12), слоистая архитектура (api → services → repositories → infra). Модули auth/servers/monitoring/provisioning внутри одного процесса.
2. **Frontend — SPA на React 18 + Vite + TypeScript**, Tailwind + Radix/shadcn-подход.
3. **БД — PostgreSQL 16** (async SQLAlchemy 2 + asyncpg + Alembic).
4. **Мониторинг — Prometheus + node_exporter + Grafana** (как зафиксировано пользователем).
5. **Провижининг — Ansible** через ansible-runner из backend.
6. Развёртывание — единый docker-compose. Конкретные версии — [02-tech-stack.md](../02-tech-stack.md).

Микросервисы, брокеры сообщений, отдельные воркеры — НЕ вводятся на Этапе 1.

## Последствия

- (+) Простой деплой и отладка, низкий оверхед для малой команды.
- (+) Единая транзакционность, общий конфиг.
- (−) Долгие операции (провижининг) живут в том же процессе → ограничение масштабирования воркеров ([ADR-006](ADR-006-async-provisioning-bez-brokera.md), [TD-004](../100-known-tech-debt.md)).
- При росте нагрузки/команды возможен вынос провижининга в отдельный сервис (новый ADR).

## Альтернативы

- **Микросервисы** — отвергнуто: избыточно для Этапа 1, противоречит NFR-1.
- **Другой язык/фреймворк** — вне выбора: стек зафиксирован пользователем.
