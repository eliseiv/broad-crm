# ADR-014 · Персистентность состояния нотификатора + alert-on-first-elevated

- Статус: accepted
- Дата: 2026-07-04
- Связанные: [ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md) (in-backend нотификатор), [TD-019](../100-known-tech-debt.md) (закрывается этим ADR)

## Контекст

Telegram-нотификатор ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)) держал state-машину **в памяти процесса** (`dict[server_id -> ServerState]`). Два следствия этого дизайна дали реальный дефект:

1. **Рестарт/деплой сбрасывает состояние** ([TD-019](../100-known-tech-debt.md)). После перезапуска backend память пуста.
2. **Правило «первая встреча — молча»:** сервер, впервые увиденный уже в повышенной зоне (`yellow`/`red`), фиксировался как база **без алерта** (защита от шторма на старте).

Комбинация: после каждого деплоя нотификатор ре-базлайнит уже-«жёлтый» сервер как норму и не шлёт предупреждение, пока зона не поднимется **ещё выше**. Реальный кейс — сервер «Фотобудка» (SSD ~81 %, `yellow`): после серии деплоев алерт так и не пришёл, предупреждение потеряно.

Пороги зон (`app.domain.thresholds.usage_to_zone()`, `<80 green` / `80..90 yellow` / `>90 red`) — единый источник для UI/backend/нотификатора и **не подлежат изменению** в рамках этой задачи.

## Решение

**1. Персистить состояние нотификатора в PostgreSQL** — новая таблица `notifier_server_state` (1:1 к `servers`, `ON DELETE CASCADE`): `online` + зеркало последних наблюдённых зон `zone_cpu/zone_ram/zone_ssd` (green/yellow/red или `NULL`) + `updated_at`. БД — **источник истины** состояния: `prev` читается из БД в начале каждой итерации, `new_state` записывается UPSERT'ом в конце. Схема — [03-data-model.md](../03-data-model.md#таблица-notifier_server_state).

**2. Alert-on-first-elevated** — отсутствие персистнутой строки (`prev is None`: новый сервер или **первый прогон после выката фичи**) трактуется чистой функцией `evaluate()` как **здоровый baseline** `{online: True, zones: green×3}`. Тогда существующая логика эскалации `rank(cur) > rank(base)` сама срабатывает `green→elevated` и шлёт **ровно один** алерт; дедуп обеспечивается тем, что после алерта повышенная зона персистится (следующая итерация видит `base == cur` → повтора нет). Правило «первая встреча — молча» **упраздняется** — единое правило baseline заменяет спец-ветку.

**3. Выкат** — миграция `0004_create_notifier_state` создаёт таблицу **пустой** (backfill намеренно не выполняется). Поэтому первый после-деплойный опрос трактует каждый сервер против здорового baseline: currently-elevated/offline серверы («Фотобудка») получают **ровно один** catch-up-алерт, здоровые — молча персистятся. Деэскалация/восстановление остаются молчаливыми, но теперь **персистятся** (падение в `green` обновляет базу → повторный рост снова алертит).

Нового API/endpoint не вводится — это фоновая логика ([04-api.md](../04-api.md) не затрагивается). Пороги и `usage_to_zone()` не меняются.

## Обоснование

- **Закрывает корневую причину TD-019.** Состояние переживает рестарт/деплой → ре-базлайн уже-«жёлтых» серверов исключён.
- **Зеркалирование зон, а не «last-alerted-zone».** Хранение фактических последних зон — прямое зеркало in-memory `ServerState`; дедуп получается из уже существующей логики `rank(cur) > rank(base)` без нового поля и без изменения чистой функции. Отдельная «последняя алертнутая зона» была бы избыточна и рисковала разойтись с семантикой `evaluate()`.
- **Единое правило baseline.** `prev is None ≡ {online, green×3}` убирает спец-ветку «первая встреча — молча», делает `evaluate()` проще и одновременно даёт нужный catch-up-алерт. Симметрично работает и для offline-first (`online→offline`).
- **Минимальный blast radius.** Одна таблица (1:1, CASCADE), одна миграция без backfill, изменение только в трактовке `prev=None` и в источнике состояния (БД вместо памяти). Формат сообщений, доставка, опциональность — без изменений.
- **Простота (NFR-1).** PostgreSQL уже в стеке; Redis не вводится (в отличие от гипотезы в TD-019) — лишний компонент не нужен для ≤ десятков строк.

## Последствия

- (+) [TD-019](../100-known-tech-debt.md) закрыт: состояние переживает рестарт/деплой; «горячие» серверы больше не теряют предупреждения.
- (+) Currently-elevated серверы получают ровно один catch-up-алерт при выкате (без ручного вмешательства).
- (+) Деэскалация/восстановление персистятся → повторный рост корректно переалертивается после рестарта.
- (−) **Одноразовый burst при выкате:** все сейчас-повышенные/offline серверы алертнут по разу. Ограничен реально проблемными серверами (здоровые молчат) и числом серверов (NFR-1) — приемлемо и является целевым «catch-up».
- (−) **Поведенческое изменение:** сервер, впервые наблюдаемый offline (`up==0`), теперь даёт offline-алерт (раньше — молчаливая база). Семантически корректно (провижининг-online сервер, который `down`, — это инцидент); дедуп ограничивает одним алертом.
- (−) +1 таблица и +1 запрос чтения / UPSERT за итерацию (незначительно на ≤ десятков строк).
- (−) Всё ещё один процесс (не горизонтально масштабируется) — то же ограничение, что у [ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)/[ADR-006](ADR-006-async-provisioning-bez-brokera.md); не является целью этой задачи.

## Альтернативы

- **Только персист без alert-on-first-elevated** (сохранить правило «первая встреча — молча») — **отвергнуто:** после выката currently-«жёлтые» серверы («Фотобудка») остались бы без алерта до следующего повышения зоны — ровно тот дефект, который закрывается. Персист без catch-up не решает исходную жалобу.
- **Периодическое напоминание** (повторный алерт каждые N минут, пока сервер в повышенной зоне) — **отложено:** решает потерю алерта, но вводит анти-спам-политику/расписание напоминаний, чего Этап 1 избегает; ортогонально дедупу по зоне. Кандидат в отдельный TD при появлении требования.
- **Redis для состояния** (гипотеза из [TD-019](../100-known-tech-debt.md)) — **отвергнуто:** лишний инфраструктурный компонент ради ≤ десятков строк; PostgreSQL уже в стеке и даёт транзакционность + CASCADE «бесплатно» (NFR-1).
- **«last-alerted-zone» вместо зеркала зон** — **отвергнуто:** усложняет чистую функцию и рискует рассинхроном с логикой эскалации; зеркало текущих зон переиспользует существующую семантику без изменений.

## Будущее

- Периодические напоминания об устойчивой нагрузке (при появлении требования) — отдельный TD.
- Горизонтальное масштабирование нотификатора — вместе с масштабированием backend ([TD-004](../100-known-tech-debt.md)); потребует блокировок per-server, но состояние уже в БД.
