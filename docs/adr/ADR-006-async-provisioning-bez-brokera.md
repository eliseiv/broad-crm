# ADR-006 · Асинхронный провижининг без брокера сообщений

- Статус: accepted
- Дата: 2026-06-28

## Контекст

Установка node_exporter через Ansible занимает десятки секунд–минуты. `POST /api/servers` не должен блокироваться до завершения; UI отслеживает прогресс через `provision_status`. Нужен механизм фоновой обработки. Команда малая, NFR-1 (простота).

## Решение

Провижининг выполняется **асинхронно в том же backend-процессе**, без внешнего брокера (Celery/RQ/Redis). `POST /api/servers` создаёт запись (`status=pending`), отвечает `202`, затем запускает фоновую задачу, которая:
1. переводит `status=installing`,
2. запускает блокирующий **ansible-runner** в thread-executor (чтобы не блокировать event loop),
3. по результату пишет `online` / `error` и регистрирует file_sd.

Статус читается через `GET /api/servers/{id}/status`.

## Обоснование

- Нет внешних зависимостей (брокер/воркеры) — соответствует NFR-1.
- FastAPI background task + executor достаточно для одного администратора и редких добавлений.
- Состояние задачи персистентно в БД (`provision_status`), переживает завершение запроса.

## Последствия

- (+) Минимум инфраструктуры, простой деплой.
- (−) Привязка к одному процессу: при рестарте backend во время `installing` задача теряется → запись «зависает» в `installing`. Митигация: при старте backend помечать «зависшие» `installing` старше таймаута как `error` (recovery-hook) — требование к backend.
- (−) Не масштабируется на несколько воркеров без координации ([TD-004](../100-known-tech-debt.md)).
- При росте объёма добавлений — вынести в очередь задач (новый ADR).

## Альтернативы

- **Celery/RQ + Redis** — отвергнуто на Этапе 1: лишняя инфраструктура.
- **Синхронный provisioning в запросе** — отвергнуто: долгий блокирующий запрос, плохой UX, таймауты.
