# ADR-046 — UI/infra-пакет: перечень бэков в алертах, плоская сетка «Бэки» + кнопка в заголовке, свёрнутая «Информация» в detail, пустые поля не рендерятся, фикс тёмной темы на проде (CSP + no-FOUC + токены + кэш HTML)

Статус: `accepted` · Дата: 2026-07-11

Пакет из четырёх фиксов владельца (1, 2, 3, 4 из батча 2026-07-11). Затрагивает Telegram-алерты (`domain/notifications.py`), страницы карточек (`/backends`, `/servers`, `/ai-keys`, `/proxies`), detail-модалки, `index.html`/`index.css`/theme-хук и nginx-раздачу SPA. **БД не меняется, контракты API не меняются** (кроме явно оговорённого — их нет). Mail-часть батча (фиксы 5–10) — [ADR-047](ADR-047-mail-fix-pack.md).

> **Ссылки на код — по ИМЕНАМ СИМВОЛОВ, без номеров строк (нормативно).** Номера строк дрейфуют при первой же реализации и превращают доказательную базу ADR в дезинформацию. Указываются файл + имя символа (`app/domain/notifications.py` → `_backend_block`), поиск — по символу.

## Context

Владелец дал батч фиксов по результатам эксплуатации:

1. **Алерты не дают контекста.** Приходит «🔴 Сервер "X" не доступен» — какие бэки на нём лежат, приходится вспоминать вручную. Связи уже есть в БД (`backends.server_id` / `backends.ai_key_id`, [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)), но в текст сообщения не попадают.
2. **Страница «Бэки» перегружена.** Группировка дублей `name` в контейнеры-кластеры «Имя · N» ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) визуально шумит, ручной DnD-порядок не нужен (порядок «сам собой» задаётся именем). Пунктирная карточка «+ Добавить» занимает ячейку сетки. Detail-модалки вываливают сразу все поля, включая пустые.
3. **Баг: на проде каждая перезагрузка возвращает тёмную тему**, несмотря на дефолт `light` ([ADR-041](ADR-041-login-theme-session-ux.md)).
4. **Прочерк «—» в detail** — визуальный мусор для незаполненных полей.

**Root cause бага темы (фикс 3) — установлен по коду, три независимых слоя:**

- **CSP блокирует no-FOUC-скрипт.** `frontend/nginx/default.conf` отдаёт `script-src 'self'` **без** `'unsafe-inline'`/nonce/hash, а no-FOUC-скрипт живёт **инлайном** в `frontend/index.html`. На проде браузер его **не исполняет** → `data-theme` на `<html>` **не проставляется** вовсе. (В dev-режиме Vite CSP нет — баг не воспроизводится, отсюда и «работает у меня».)
- **Токены темы «фейлят» в тёмную.** `frontend/src/index.css` — селектор `:root, [data-theme='dark']` несёт **тёмные** значения; светлые — только под `[data-theme='light']` (`:45`). Отсутствие атрибута ⇒ действует голый `:root` ⇒ **тёмная тема**. Т.е. провал скрипта детерминированно даёт тёмный экран, а не дефолтный светлый.
- **Хук не самолечится.** `frontend/src/lib/theme.ts` — `useTheme()` инициализируется **из** `document.documentElement.dataset.theme` и, если атрибута нет, ничего не применяет (`resolveTheme()` при старте не вызывается).

Плюс **инфраструктурный усилитель:** у `location /` (раздача SPA) **не было ни `Cache-Control`, ни `expires`** → браузер эвристически кэшировал `index.html`, и даже после выката исправленного HTML клиент какое-то время получал старый.

## Decision

### §1. Фикс 1 — перечень бэков в Telegram-алертах об ОШИБКАХ

**Что расширяется (нормативно):** сообщения об **ошибках** сервера (`warning` / `critical` / `offline`) и **ошибке ИИ-ключа** (`key error`) получают блок с перечнем бэков, которые используют этот сервер / этот ключ.

**Что НЕ расширяется:** **recovery-сообщения** (`build_recovered`, `build_key_recovery`, `build_backend_recovery`, `build_proxy_recovery`) — перечнем **не дополняются** (решение владельца). Алерты **бэков** и **прокси** — не расширяются (бэк сам себя не перечисляет; у прокси связи с бэками в модели нет — `backends` имеет только `server_id`/`ai_key_id`, `app/models/service_backend.py`).

**Источник перечня:** `backends WHERE server_id = :server_id` (для серверных алертов) / `backends WHERE ai_key_id = :ai_key_id` (для алертов ключа). Порядок — детерминированный: **`position ASC, code ASC`**.

**⚠️ Два разных порядка сортировки бэков сосуществуют НАМЕРЕННО (нормативно):**

| Потребитель | Порядок | Где задан |
|-------------|---------|-----------|
| **API reverse-lookup** — `GET /api/servers/{id}/backends`, `GET /api/ai-keys/{id}/backends` (и секция «Бэки» в detail-модалках) | **`position ASC, created_at DESC, id ASC`** | `BackendRepository.list_by_server` / `list_by_ai_key`; контракт [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) — **НЕ меняется** |
| **Перечень бэков в тексте Telegram-алерта** (этот ADR) | **`position ASC, code ASC`** | настоящий §1 |

- **Почему алерту нужен свой порядок.** `code` **UNIQUE** ⇒ `(position, code)` даёт **тотальный, детерминированный** порядок: тай-брейк однозначен даже при равных `position` и равных `created_at`. Это условие **побайтовой воспроизводимости** текста сообщения (qa сверяет формат посимвольно). `created_at DESC` такого гарантирует не даёт (у бэков, созданных в одной транзакции/секунде, порядок неустойчив) и человеку в алерте ничего не сообщает.
- **Почему НЕ унифицируем.** Смена `ORDER BY` в `list_by_server`/`list_by_ai_key` изменила бы **публичный контракт** reverse-lookup ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)) и порядок строк в UI-секции «Бэки» — ради косметики в алерте. Цена не оправдана.
- **Как реализуется.** Переупорядочение под алерт делается **in-memory поверх результата репозитория** (отдельный хелпер сортировки), а **не** новым запросом/сменой `ORDER BY`. Перечень мал (≤ десятки бэков), алерты редки — стоимость нулевая.
- **⚠️ Сортировка — по КОРТЕЖУ `(position, code)` в Python; перенос тай-брейка в SQL `ORDER BY` ЗАПРЕЩЁН (нормативно).** Запись «`position ASC, code ASC`» задаёт **требуемый порядок**, а не способ его получения. Python сортирует `code` по **кодпойнтам**; PostgreSQL `ORDER BY` — по **коллации БД** (`en_US.UTF-8` игнорирует регистр и пунктуацию), поэтому для кодов со **смешанным регистром** два порядка **расходятся**, а результат ещё и зависит от настроек локали инстанса. Побайтовая воспроизводимость текста алерта (qa сверяет посимвольно) требует именно кодпойнтов. Это **вторая** причина, по которой `ORDER BY` репозитория не трогаем — помимо неизменности контракта reverse-lookup.

**Формат строки бэка — ПЕРЕИСПОЛЬЗУЕТСЯ существующий нормативный блок** [modules/backends §Формат сообщений Telegram](../modules/backends/README.md#формат-сообщений-telegram-точно-нормативно--источник-истины) (`_backend_block`, `app/domain/notifications.py`), побуквенно:

```
Бэк "<name>" [<code>] <domain>
```

**Блок перечня (нормативно):**

```
<пустая строка>
Бэки:
Бэк "<name1>" [<code1>] <domain1>
Бэк "<name2>" [<code2>] <domain2>
```

- **Пустой перечень → блок НЕ добавляется вовсе** (ни заголовка «Бэки:», ни пустой строки) — сообщение побайтово равно нынешнему.
- **Лимит (нормативно): `MAX_ALERT_BACKENDS = 10`.** При числе бэков `N > 10` печатаются **первые 10** (в том же порядке `position ASC, code ASC`), последней строкой блока добавляется:

  ```
  … и ещё <N-10>
  ```

  (символ `…` = U+2026, единственный на строке перед « и ещё»). Обоснование лимита: Telegram режет `sendMessage` на 4096 символов — 10 строк бэка (~60–90 символов каждая) с запасом укладываются вместе с шапкой алерта; молча терять хвост нельзя, поэтому остаток объявляется числом. Это **не** сокрытие значимого контента (правило CLAUDE.md о `truncate`) — полный список всегда доступен в UI (detail-модалка сервера/ключа, секция «Бэки», [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)); в алерте перечень — вспомогательный контекст. Долг «перечень обрезается на 10» — [TD-053](../100-known-tech-debt.md).

**Нормативные сигнатуры билдеров** (`app/domain/notifications.py`; расширение обратно-совместимое — новый параметр опционален и по умолчанию пуст):

```python
BackendRef = tuple[str, str, str]          # (code, name, domain)
MAX_ALERT_BACKENDS = 10

def _backends_block(backends: Sequence[BackendRef]) -> str: ...
    # "" при пустом; иначе "\n\nБэки:\n" + строки (≤10) + опц. "\n… и ещё <N-10>"

def build_warning(name: str, ip: str, items: Sequence[MetricItem],
                  backends: Sequence[BackendRef] = ()) -> str: ...
def build_critical_load(name: str, ip: str, items: Sequence[MetricItem],
                        backends: Sequence[BackendRef] = ()) -> str: ...
def build_offline(name: str, ip: str, backends: Sequence[BackendRef] = ()) -> str: ...
def build_key_error(name: str, last4: str | None, reason: str,
                    backends: Sequence[BackendRef] = ()) -> str: ...
```

Побуквенные шаблоны итоговых сообщений — [modules/notifier §Типы сообщений и формат](../modules/notifier/README.md#типы-сообщений-и-формат-точно) и [modules/ai-keys §Формат сообщений Telegram](../modules/ai-keys/README.md#формат-сообщений-telegram-точно); **оба раздела обязаны совпадать побуквенно** (они дублируют друг друга — источник истины по формату бэк-строки остаётся [modules/backends](../modules/backends/README.md#формат-сообщений-telegram-точно-нормативно--источник-истины)).

**Ограничение (нормативно): CHECK `ck_notifier_alert_log_kind` НЕ меняется.** Допустимые `kind` остаются ровно `('offline','recovered','warning','critical')` (`app/models/notifier_alert_log.py`, миграция `0005`). Новых видов алертов не вводится — меняется **только текст** `notifier_alert_log.message` (в него пишется отправленная строка целиком, включая блок «Бэки:»). Миграции этот фикс не требует.

### §2. Фикс 2 — страница «Бэки», кнопка «+ Добавить», detail-модалки

#### §2а. Плоская сетка «Бэки», сортировка по имени, DnD убран

- **Группировка дублей `name` в контейнеры-кластеры «`<name>` · N» ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) УПРАЗДНЯЕТСЯ.** `/backends` — **одна плоская сетка** карточек (`grid-cols-1 md:grid-cols-2 xl:grid-cols-3`, gap 24px), как «Серверы»/«Прокси».
- **Порядок — стабильная сортировка по имени** (клиентская, над загруженным `GET /api/backends`): `name` (регистронезависимо, `localeCompare` с локалью `ru`), tie-break — `code` (UNIQUE ⇒ порядок детерминирован). Одинаковые имена стоят рядом — это и заменяет прежнюю группировку.
- **DnD-перетаскивание карточек на `/backends` УБИРАЕТСЯ** (решение владельца: порядок задаётся сортировкой, а не рукой). Снимаются `DndContext`/`SortableContext`/`SortableItem`/`PointerSensor`/`reorderMutation` на этой странице. Разведение жестов «короткий клик vs зажатие» ([ADR-011](ADR-011-poryadok-blokov-server-side-dnd-kit.md)) на `/backends` больше не требуется: клик по карточке → detail-модалка немедленно.
- **Судьба `position` и reorder-эндпоинта (нормативно, решение):** колонка **`backends.position` СОХРАНЯЕТСЯ** в БД (`app/models/service_backend.py`), эндпоинт **`PATCH /api/backends/order` СОХРАНЯЕТСЯ** в API (`app/api/backends.py`) — **но UI `/backends` их больше не использует**. Обоснование: удаление колонки+эндпоинта — необратимая правка контракта ради нулевой выгоды; `position` продолжает задавать порядок ответа `GET /api/backends` (backend не меняется) и остаётся ключом сортировки для **перечня бэков в алертах** (§1). Долг «неиспользуемый UI-путь reorder» — [TD-054](../100-known-tech-debt.md). DnD на `/servers` и `/ai-keys` — **НЕ трогаем** ([ADR-011](ADR-011-poryadok-blokov-server-side-dnd-kit.md) в силе для них).
- **Клиентский поиск по бэкам** ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) — **сохраняется** без изменений (фильтр применяется до сортировки).

#### §2б. Кнопка «+ Добавить» — вправо от заголовка страницы

- **Паттерн «заголовок страницы + правая зона действий» (нормативно, вводится в [08-design-system.md](../08-design-system.md#заголовок-страницы-и-правая-зона-действий-нормативно)).** Готовый образец — уже существующая правая кнопка «Обновить» на `/ai-keys` (`AiKeysPage.tsx`): строка-флекс `justify-between`, слева `h1` (+ опц. подпись), справа зона действий.
- Кнопка **«Добавить»** (иконка `Plus`) переезжает в правую зону заголовка на страницах **Бэки / Серверы / ИИ-ключи / Прокси**. Гейт — `<page>:create` (без права кнопка не рендерится).
- **Пунктирные карточки-плейсхолдеры `AddBackendCard`/`AddServerCard`/`AddAiKeyCard`/`AddProxyCard` УДАЛЯЮТСЯ** — из сетки **и из пустого состояния**. Раздел [08 §Карточка «+ Добавить»](../08-design-system.md#карточка--добавить-glass--blur) упраздняется.
- **Пустое состояние** больше не «только пунктирная карточка без текста» ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md) — эта норма **отменяется**): показывается текстовая строка («Бэков пока нет» / «Серверов пока нет» / «Ключей пока нет» / «Прокси пока нет»), а добавление доступно кнопкой в шапке. Нормативные строки — [08-design-system.md](../08-design-system.md).

#### §2в. Detail-модалки: только идентификаторы, остальное — в свёрнутой «Информации»

При открытии `<Entity>DetailModal` видны **только идентификаторы**; всё остальное — внутри **свёрнутого по умолчанию** блока **«Информация»** (паттерн уже существующей сворачиваемой секции — `BackendsDetailSection.tsx`: кнопка-триггер, `aria-expanded`/`aria-controls`, `ChevronDown` с `rotate-180` при раскрытии).

**Состав видимой зоны (решение владельца, нормативно):**

| Модалка | Видно сразу (идентификаторы) | Внутри «Информации» |
|---------|------------------------------|---------------------|
| Бэк | **Код**, **Название**, **Домен** | Сервер, ИИ-ключ, **API KEY** (reveal), **ADMIN API KEY** (reveal), Git, Примечания |
| Сервер | **Название**, **IP** | Пользователь (`ssh_user`), **Пароль** (reveal), сворачиваемая секция **«Бэки»** |
| ИИ-ключ | **Название**, **Провайдер** | **Ключ** (`key_masked` + reveal), сворачиваемая секция **«Бэки»** |
| Прокси | **Название**, **Хост**, **Порт** | Тип, Логин, **Пароль** (reveal) |

- Кнопка-карандаш `Pencil` остаётся **в шапке модалки** (вне «Информации»), гейт `<page>:edit` — [ADR-035](ADR-035-detail-view-secret-reveal.md) не отменяется. Inline-edit «Название» сервера ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) работает в **видимой** зоне.
- Секция **«Бэки»** ([ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md)) — **вложена внутрь** «Информации» (сворачиваемая внутри сворачиваемой; поведение самой секции не меняется).
- Reveal секретов ([ADR-035](ADR-035-detail-view-secret-reveal.md)) не ослабляется: значение живёт только в локальном стейте модалки, гейт `<page>:edit`, `Cache-Control: no-store` на reveal-эндпоинтах.
- **Если внутри «Информации» не остаётся ни одной строки и ни одной секции — блок «Информация» не рендерится** (следствие §3-фикса 4; практически достижимо только у бэка без связей/секретов/git/note).

### §3. Фикс 4 — пустые поля в detail-view не рендерятся

- **Строка detail-view с пустым значением (`null` / `undefined` / пустая строка) НЕ рендерится вовсе.** Прочерк **«—» в detail-view упраздняется** (`BackendDetailModal.tsx` `DASH`, `ProxyDetailModal.tsx` — оба способа снимаются).
- **Секретные поля с reveal — исключение частичное:**
  - секрет **есть** (`has_api_key` / `has_admin_api_key` / `has_password` / `key_masked` непуст) → строка рендерится **всегда**, значение — маска `••••••••` + глаз (наличие секрета — по флагам `has_*`, не по значению; значение backend не отдаёт);
  - секрет **не задан** → строка **не рендерится** (раньше показывалась как «Пароль: —» / «API KEY: —»). Нормативная строка словаря «**Пароль: —**» для прокси без пароля — **упраздняется**.
- Правило действует **только в detail-view**. Табличные представления (`/sms` «Номера», вкладка «Почты») сохраняют свои плейсхолдеры (`-`), чтобы строки таблицы не «прыгали» — [08 §Цветовые пилюли](../08-design-system.md#цветовые-пилюли-логинприложениепримечание-нормативно) не меняется.

### §4. Фикс 3 — баг тёмной темы на проде: три слоя + кэш HTML

Каждый из трёх слоёв самостоятельно достаточен, чтобы баг воспроизводился; чиним **все три** (defense-in-depth), **не ослабляя CSP**.

#### §4.1. no-FOUC-скрипт — из inline в отдельный статический файл

- Скрипт выносится из `<script>…</script>` в `index.html` в **отдельный статический файл своего origin** (напр. `frontend/public/theme-init.js`) и подключается в `<head>` **синхронным** тегом до бандла: `<script src="/theme-init.js"></script>`.
- **CSP НЕ ослабляется:** `script-src` остаётся **`'self'`**; `'unsafe-inline'` **не вводится**, nonce/hash **не вводятся**. Нормативное значение CSP ([05-security.md](../05-security.md#content-security-policy-spa-location-)) **не меняется ни на байт**. Тот же приём уже применён к self-hosted `telegram-web-app.js` ([05-security.md](../05-security.md#операторская-mini-app-tgsms-adr-031)).
- Логика скрипта не меняется: `localStorage['crm-theme']` ∈ {`light`,`dark`} → иначе `light`; проставить `document.documentElement.dataset.theme` + `meta[name=theme-color]`.
- **Отклонено:** `'unsafe-inline'` в `script-src` (открывает XSS-поверхность SPA ради косметики), nonce (требует динамической генерации HTML — nginx отдаёт статику), hash (ломается при любой правке скрипта, включая пробел).

#### §4.2. Токены: светлая тема — дефолт на голом `:root`

- В `index.css`: **светлые** значения переменных переезжают на **`:root`** (и дублируются под `[data-theme='light']`); **тёмные** — **только** под `[data-theme='dark']`. `color-scheme` — соответственно (`:root { color-scheme: light }`, `[data-theme='dark'] { color-scheme: dark }`).
- **Инвариант:** отсутствие атрибута `data-theme` больше **не означает «тёмная»** — оно означает **дефолт светлую** ([ADR-041](ADR-041-login-theme-session-ux.md)). Даже если скрипт снова не отработает (CSP/JS выключен/ошибка), пользователь увидит **корректный дефолт**, а не «тёмную вспышку навсегда».
- Это **разворот** технической формулировки [ADR-033](ADR-033-flat-nav-theme-toggle-numbers-table.md)/[08 §Темизация](../08-design-system.md#темизация-светлаятёмная) («тёмные значения — на `:root`»). **Продуктовое поведение не меняется** — дефолт был и остаётся `light` ([ADR-041](ADR-041-login-theme-session-ux.md)); менялся только носитель дефолта в CSS.

#### §4.3. Self-heal в theme-хуке

- `useTheme()` (`lib/theme.ts`) при монтировании проверяет `document.documentElement.dataset.theme`. Если значение **не** `'light'` и **не** `'dark'` (нет/мусор) — вызывает `applyTheme(resolveTheme())` (`resolveTheme()` = сохранённый выбор → иначе `DEFAULT_THEME='light'`) и берёт результат за начальное состояние. Идемпотентно; при штатно отработавшем скрипте — no-op.

#### §4.4. devops: кэширование раздачи SPA (nginx)

- **`index.html` (и прочие НЕхешированные корневые ассеты, включая `/theme-init.js`) — `Cache-Control: no-cache`.** До этого фикса у `location /` **не было ни `Cache-Control`, ни `expires`** → браузер кэшировал HTML эвристически и после выката какое-то время исполнял старый документ (в т.ч. со старым inline-скриптом). Здесь `always` **уместен и сохраняется**: `no-cache` на не-2xx безвреден (лишь принуждает к ревалидации, ничего «на год» не залипает).
- **Хешированные ассеты сборки (`/assets/*` — Vite добавляет content-hash в имя) — `Cache-Control: public, max-age=31536000, immutable`.**
- **⚠️⚠️ `Cache-Control` в `location /assets/` — БЕЗ модификатора `always` (нормативно; security-заголовки и CSP в том же блоке `always` СОХРАНЯЮТ).**
  - `always` выставляет заголовок и на **неуспешных** ответах, включая **`404`**. Битый/неполный выкат (ассеты не скопированы; rolling-deploy со старой репликой) → клиент получает `404` на `/assets/index-<hash>.js` **с `max-age=31536000, immutable`** и кэширует **сам `404` на год**; `immutable` подавляет ревалидацию даже при reload. Повторная сборка кладёт файл под **тем же** именем (content-hash детерминирован), но пострадавший клиент читает закэшированный `404` — SPA у него сломана, и **откат его не чинит** (только hard-refresh пользователем).
  - Без `always` заголовок применяется **только к успешным кодам** — ровно к отдаче существующего хешированного ассета. `404` при этом уезжает **без freshness-заголовков** и потому **не кэшируется агрессивно; ревалидация остаётся возможной** (формально `404` — cacheable-by-default статус, но без `max-age`/`immutable` его эвристическая свежесть практически нулевая). Следующий запрос после починки выката попадёт на сервер — в отличие от варианта с `always`, где `immutable` подавляет ревалидацию на год.
  - **Асимметрия внутри блока намеренна и НЕ подлежит «унификации» при рефакторинге:** security-заголовки обязаны покрывать **любой** ответ (`always`), вечный immutable-кэш — **только успешный** (без `always`).
- **`try_files $uri =404;` в `location /assets/` — нормативно** (явный контракт: несуществующий ассет → `404`, а **не** SPA-fallback `index.html`; иначе браузер выполнит HTML как JS). Поведенчески no-op на текущем конфиге (соседние `location`-блоки, fallback не наследуется) — но фиксирует намерение против будущего вложения блоков.
- **⚠️ Нормативное предупреждение (nginx-ловушка):** `add_header` во **вложенном** `location` **отменяет все унаследованные** `add_header` родительского блока (наследование в nginx — не аддитивное, а «замещающее на уровне блока»). Следовательно, в каждом вложенном `location`, где появляется `Cache-Control`, **обязаны быть продублированы ВСЕ security-заголовки и CSP** родителя — **побайтово** совпадающие с нормативным значением ([05-security.md](../05-security.md#content-security-policy-spa-location-)). Пропуск дубля молча снимет CSP/HSTS/`X-Frame-Options` с этих ответов. Требования и точный сниппет — [07-deployment.md §Reverse-proxy](../07-deployment.md#reverse-proxy-nginx--требования).

## Consequences

- **Алерты становятся actionable:** по 🔴/🟡 сразу видно, какие бэки задеты. Цена — по одному дешёвому индексируемому `SELECT` на алертящий сервер/ключ (алерты редки; перечень резолвится только когда сообщение реально формируется). CHECK `kind` не тронут → миграций нет.
- **`/backends` проще:** одна сетка, детерминированный порядок, нет ручного DnD и нет кластер-контейнеров. Регресс: терять ручной порядок бэков — осознанно (владелец); `position`/reorder остаются в контракте ([TD-054](../100-known-tech-debt.md)).
- **Единый паттерн шапки страницы** (заголовок + правая зона действий) на 4 карточных страницах; сетка карточек больше не содержит «нефункциональную» ячейку.
- **Detail-модалки читаемы:** идентификатор виден сразу, шум (пустые поля, связи, секреты, git/note) — под «Информацией». Прочерки «—» исчезают.
- **Баг темы устраняется тремя независимыми слоями без ослабления CSP.** Даже деградация любого одного слоя не возвращает баг. `Cache-Control: no-cache` на `index.html` устраняет «выкатили фикс, а у пользователя старый HTML».
- **Риск, принятый явно:** правка `location /` в nginx требует дублировать security-заголовки во вложенных блоках — при небрежности можно молча потерять CSP. Поэтому требование вынесено в `07-deployment.md` как нормативное, а значение CSP — как побайтовый эталон в `05-security.md`; проверка — обязанность `devops-reviewer`.

## Alternatives considered

- **Разрешить `'unsafe-inline'` в `script-src`** (самый быстрый способ «починить» тему). Отклонён: ослабляет главный анти-XSS-барьер SPA, который держит и изоляцию HTML-тела письма ([ADR-012](ADR-012-mail-read-through-proxy.md)/[ADR-015](ADR-015-csp-img-src-remote-mail-images.md)), ради косметического скрипта. Вынос в файл даёт тот же результат при `script-src 'self'`.
- **CSP-hash inline-скрипта.** Отклонён: хеш инвалидируется при любой правке скрипта (даже пробела) и требует синхронного обновления nginx-конфига — скрытая мина при рефакторинге.
- **Чинить только токены (§4.2), не трогая CSP-блокировку.** Отклонён: FOUC вернётся (без скрипта тема применится только после гидрации React → вспышка), а сохранённый выбор `dark` не применится до маунта.
- **Оставить DnD на `/backends`, добавив сортировку по имени опцией.** Отклонён владельцем: два конкурирующих источника порядка (ручной `position` и сортировка) — источник путаницы.
- **Удалить `backends.position` и `PATCH /api/backends/order` вместе с DnD.** Отклонён: необратимое изменение контракта ради нулевой выгоды; `position` продолжает использоваться как ключ порядка ответа `GET /api/backends` и порядка перечня бэков в алертах (§1).
- **Оставить «—» для пустых полей в detail.** Отклонён владельцем (визуальный мусор). Для секретов «нет секрета» теперь = отсутствие строки; факт «секрет не задан» по-прежнему виден в форме edit.
- **Показывать в алерте все бэки без лимита.** Отклонён: лимит Telegram 4096 символов — при большом числе бэков сообщение обрежет сам Telegram (молча, посередине строки). Явный `… и ещё N` честнее.
