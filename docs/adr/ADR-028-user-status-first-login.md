# ADR-028 · Тристатус пользователя: «Активен» только после первого входа (`first_login_at`, производное поле `status`)

- Статус: accepted
- Дата: 2026-07-08

## Контекст

Правка 3 батча — прямое решение пользователя по отображению статуса пользователей на странице «Пользователи».

Действующая модель ([ADR-021](ADR-021-rbac-users-roles.md), [ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)): статус пользователя в UI — бинарный по `users.is_active` («Активен»/«Неактивен»). С введением беспарольных пользователей и «открытого первого входа» ([ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)) появился класс учёток, которые уже активны (`is_active=true`), но **ещё ни разу не входили** (беспарольный не завершил первый вход, либо парольный ещё не логинился). Отображать их как «Активен» неточно: пользователь не подтверждён входом.

Требование пользователя — **тристатус**:
- `is_active=false` → **«Неактивен»**;
- `is_active=true` И пользователь ещё ни разу не входил → **«Ожидает входа»**;
- иначе (активен И хотя бы раз входил) → **«Активен»**.

Существующий индикатор «Без пароля» (`has_password=false`, [ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)) отражает **беспарольность**, а не факт входа, и не заменяет статус: парольный пользователь, ещё не входивший, тоже «Ожидает входа». Нужен отдельный сигнал «был ли первый вход».

## Решение

### 1. Данные: `users.first_login_at`

- Колонка `users.first_login_at timestamptz NULL` (миграция `0015_users_first_login`, [03-data-model.md](../03-data-model.md#таблицы-roles-и-users-rbac)). `NULL` = пользователь **ещё ни разу не входил**; непустой timestamptz = момент **первого** успешного входа.
- **Проставляется при ПЕРВОМ успешном входе, идемпотентно** (`if first_login_at is None: first_login_at = now()`; повторные входы значение **не меняют**):
  - **парольная ветка `POST /api/auth/login`** — после успешной проверки bcrypt-пароля БД-пользователя, до выдачи access-JWT;
  - **`POST /api/auth/set-password`** — при установке пароля «открытого первого входа»: беспарольный пользователь после set-password **сразу залогинен** ([ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)), поэтому это и есть его первый вход → `first_login_at` проставляется в той же операции.
- **Супер-админ (`.env`)** в таблице `users` не хранится → `first_login_at` к нему неприменимо; его статус на странице «Пользователи» не отображается (его нет в списке).

### 2. Контракт: производное поле `status` в `UserListItem`

- `UserListItem` += **`status`** — производное строковое поле со значениями (**нормативно, точные строки**):
  - `"inactive"` — если `is_active == false`;
  - `"pending"` — если `is_active == true` И `first_login_at IS NULL`;
  - `"active"` — если `is_active == true` И `first_login_at IS NOT NULL`.
- **Правило вычисления (нормативно, приоритет `is_active`):** сперва `is_active` (false → `"inactive"`), затем факт входа (`first_login_at`). Деактивированный пользователь — всегда `"inactive"` независимо от `first_login_at`.
- `status` вычисляется на сервере в схеме ответа; `first_login_at` в API **не отдаётся** (внутреннее поле — наружу только производный `status`). Поле `is_active` в `UserListItem` **сохраняется** (используется формой редактирования как тумблер «Активен»); `status` — для отображения бейджа. Контракт — [04-api.md](../04-api.md#схема-userlistitem).
- **UI-лейблы** (frontend, словарь [08-design-system.md](../08-design-system.md#страница-пользователи)): `"inactive"` → **«Неактивен»**, `"pending"` → **«Ожидает входа»**, `"active"` → **«Активен»**.

## Обоснование

- **`first_login_at timestamptz` вместо `boolean has_logged_in`** — timestamptz несёт больше информации (когда именно), обратно совместим с идемпотентной установкой (`IS NULL`-гейт), и консистентен с прочими временными метками (`created_at`/`last_checked_at`).
- **Производный `status` в контракте, а не сырой `first_login_at`** — правило вычисления централизуется на сервере (единственный источник истины), UI не дублирует бизнес-логику «активен ⇔ активен И входил»; наружу не утекает внутренняя метка. По образцу производных `has_password` ([ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)) и `member_count`.
- **Установка в двух ветках входа** (`login` парольная + `set-password`) покрывает оба сценария первого входа: парольный пользователь логинится напрямую; беспарольный «первый вход» завершается через set-password (после чего сразу залогинен). Беспарольная ветка `login` **не** проставляет метку — там вход ещё не выполнен (выдаётся только setup-token, [ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)).
- **Идемпотентность (`if None`)** — «первый вход» фиксируется однократно; повторные входы метку не переписывают, семантика «когда впервые вошёл» сохраняется.

## Последствия

- (+) Точный статус: «Ожидает входа» отличает заведённую, но не подтверждённую входом учётку от реально работающей.
- (+) Правило статуса централизовано на сервере (производный `status`); UI только маппит три строки на лейблы.
- (−) Новая колонка `users.first_login_at` (миграция `0015_users_first_login`) + производное поле `status` в `UserListItem`.
- (−) Логика входа (`login` парольная ветка + `set-password`) обязана идемпотентно проставлять `first_login_at` — новый инвариант в auth-потоке.
- (−) Существующие пользователи на момент миграции получают `first_login_at = NULL` → отображаются «Ожидает входа», пока не войдут снова (на проде БД-пользователей 0 — эффекта нет; для будущих — корректная семантика: метка ставится при следующем входе).

## Альтернативы

- **Бинарный статус по `is_active` (как сейчас)** — отвергнуто: не отражает «заведён, но не входил».
- **Использовать `has_password` как прокси «входил ли»** — отвергнуто: беспарольность ≠ факт входа; парольный, ещё не входивший, был бы ложно «Активен».
- **`boolean users.has_logged_in`** — отвергнуто: `timestamptz` информативнее при той же простоте идемпотентной установки.
- **Отдавать `first_login_at` наружу и вычислять статус на клиенте** — отвергнуто: правило «активен ⇔ активен И входил» должно быть на сервере (единственный источник), клиент не дублирует бизнес-логику; внутренняя метка наружу не нужна.
- **Проставлять метку и в беспарольной ветке `login`** — отвергнуто: там вход не выполнен (только setup-token); первый вход беспарольного завершается в `set-password`.
