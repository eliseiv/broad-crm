# ADR-025 · Беспарольные пользователи, вход по Логину или Телеграму, модель «открытого первого входа» (set-password flow)

- Статус: accepted
- Дата: 2026-07-08
- **Амендирован:** [ADR-041](ADR-041-login-theme-session-ux.md) — плейсхолдер поля входа **«Логин или @username»** (label «Логин или Телеграм» не меняется) и явно зафиксирован приём Telegram-идентификатора **с `@` и без** (сервис нормализует ведущий `@` перед сопоставлением с `users.telegram`; поведение уже реализовано). Контракт `login` не изменён.

> **Амендмент [ADR-029](ADR-029-ui-login-password-nav-team-form.md) (2026-07-08).** Заголовок экрана установки пароля первого входа переименован **«Задайте пароль» → «Придумайте пароль»**; уточнён UX: беспарольный пользователь не вводит пароль на шаге логина, экран не предлагает генератор/случайный пароль — пользователь придумывает пароль сам (≥ 8). Контракт `login`/`set-password` **не изменён**. Тело этого ADR ниже сохранено как есть (в т.ч. историческое «Задайте пароль»); актуальный заголовок экрана и UX — [ADR-029](ADR-029-ui-login-password-nav-team-form.md), [08-design-system.md](../08-design-system.md#экран-входа-двухшаговый). Статусный аспект «Активен только после первого входа» (`first_login_at`) — [ADR-028](ADR-028-user-status-first-login.md).

## Контекст

Правка 4 батча — прямые решения пользователя по управлению доступом БД-пользователей (сверх RBAC [ADR-021](ADR-021-rbac-users-roles.md) / [ADR-022](ADR-022-teams-nav-categories.md)):

1. **Пароль при создании пользователя — необязателен.** Админ может завести пользователя **без пароля**.
2. **Вход по Логину ИЛИ Телеграму.** Сейчас вход только по `username`. Требование — идентификатор входа = **логин или телеграм-ник** (поле `users.email`, введённое Спринтом A, **заменяется** на `users.telegram` — см. ниже и [ADR-022](ADR-022-teams-nav-categories.md) амендмент).
3. **Модель «открытого первого входа».** Если у пользователя нет пароля, он вводит Логин/Телеграм **без пароля** → система распознаёт «пароль не задан» → показывает окно **«Задайте пароль»** → пользователь **сам вводит** свой пароль (≥ 8 символов, как текущее правило) → после установки вход только по паролю.

Замена контакта email→telegram (уточнение пользователя): продукт ориентирован на Telegram (нотификатор уже на Telegram, [modules/notifier](../modules/notifier/README.md)); контактный идентификатор пользователя — **телеграм-ник**, не email. На проде пользователей 0 (данных нет) → чистая замена колонки.

Действующая модель: супер-админ из `.env` (constant-time, всегда с паролем — [ADR-008](ADR-008-admin-iz-env.md)); БД-пользователи — bcrypt-хэш `users.password_hash NOT NULL`, вход по `username` ([ADR-021](ADR-021-rbac-users-roles.md), [05-security.md](../05-security.md)).

## Решение

### 1. Контакт пользователя: `email` → `telegram`

- Колонка `users.email` **заменяется** на `users.telegram text NULL` (миграция `0011_user_passwordless_telegram`, [03-data-model.md](../03-data-model.md#таблицы-roles-и-users-rbac)). Частичный уникальный индекс — теперь `uq_users_telegram (telegram) WHERE telegram IS NOT NULL` (был `uq_users_email`).
- **Формат телеграм-ника (нормативно).** Опциональный ведущий `@`, затем **5–32 символа** из `[A-Za-z0-9_]`. Хранится **нормализованным**: ведущий `@` снят, lower-case (Telegram-ники регистронезависимы) → канон `[a-z0-9_]{5,32}`. Невалидный формат → `422 unprocessable` (`details[].field="telegram"`). Уникален среди заданных → дубликат → **`409 telegram_taken`** (заменяет `409 email_taken`). `NULL` — телеграм не задан. Не секрет.
- `UserCreateRequest`/`UserUpdateRequest`: поле `email` → `telegram` (опц.); `UserListItem` += `telegram` (вместо `email`). Контракт — [04-api.md](../04-api.md#users).

### 2. Пароль необязателен → `password_hash` nullable

- `users.password_hash` становится **`NULL`-разрешённым** (миграция `0011`). `NULL` = **беспарольный** пользователь (пароль ещё не задан). Непустой — bcrypt-хэш, как раньше.
- `POST /api/users`: поле `password` — **опционально** (8–128 при наличии). Отсутствует/`null`/`""` → пользователь создаётся **беспарольным** (`password_hash = NULL`). Роль/логин — по-прежнему обязательны.
- **Супер-админ (`.env`) беспарольным не бывает** — у него всегда есть `ADMIN_PASSWORD`; его ветка входа не затрагивается.

### 3. Вход по Логину или Телеграму (`POST /api/auth/login`)

- Поле запроса — по-прежнему `username`, но его **семантика расширена**: это **идентификатор** = логин **или** телеграм-ник. `password` в запросе теперь **опционален**.
- Порядок резолвинга (нормативно, [05-security.md](../05-security.md#аутентификация-логин-и-выпуск-jwt)):
  1. **Супер-админ (`.env`)** — constant-time против `ADMIN_USER`/`ADMIN_PASSWORD` (только по логину; телеграма у супер-админа нет). Успех → обычный access-JWT.
  2. **БД-пользователь** — поиск по `username` **точно**, иначе по `telegram` (нормализованному). Если найден и `is_active`:
     - `password_hash IS NOT NULL` (парольный): проверка `password` bcrypt. Успех → access-JWT. Неверный/отсутствующий пароль → `401 invalid_credentials`.
     - `password_hash IS NULL` (беспарольный): **вход не выполняется** — возвращается сигнал **«требуется установка пароля»** (`password` игнорируется, см. §4).
  3. Не найден / `is_active=false` / любая неудача парольной ветки → единое `401 invalid_credentials`.

### 4. Открытый первый вход: сигнал + limited-scope setup-token + `set-password`

- **Дискриминированный ответ `POST /api/auth/login` (200).** Ответ несёт флаг `password_setup_required`:
  - **успех** (`false`): `{ "password_setup_required": false, "access_token", "token_type": "bearer", "expires_in" }` — как раньше.
  - **требуется установка** (`true`): `{ "password_setup_required": true, "setup_token", "token_type": "bearer", "expires_in": <ttl setup-токена> }`.
- **Setup-token — токен ограниченной области.** JWT с `type:"pwd_setup"`, `uid`, `sub`, `iat`, `exp`; **без** `role`/`superadmin`/прав. TTL — `PWD_SETUP_TOKEN_EXPIRES_MIN` (default **10 мин**). `get_current_principal` **отвергает** любой токен с `type != "access"` → `401` на всех ресурсных/Users/Roles/Teams-эндпоинтах. Setup-token принимает **только** `POST /api/auth/set-password` (выделенная зависимость).
- **`POST /api/auth/set-password`** (Authorization: Bearer <setup_token>): тело `{ "password": <8–128> }`. Если пользователь всё ещё беспарольный (`password_hash IS NULL`) и активен → устанавливается bcrypt-хэш, возвращается **обычный успех** (`{ password_setup_required: false, access_token, ... }`) — пользователь сразу залогинен. Если пароль уже задан (гонка/повтор) → **`409 password_already_set`**. Слабый пароль → `422`. Невалидный/просроченный setup-token → `401 unauthorized`.
- После установки `password_hash != NULL` → пользователь входит **только по паролю** (беспарольная ветка больше не срабатывает).

### 5. Осознанный риск открытого первого входа (нормативно, [05-security.md](../05-security.md))

Открытый первый вход — **осознанное решение пользователя**, а не пробел:
- **Окно уязвимости.** С момента создания беспарольного пользователя и до установки им пароля **любой**, кто знает его Логин/Телеграм, может ввести идентификатор без пароля и **задать пароль первым**, захватив учётку. Это принято осознанно (простой онбординг: админ заводит учётку, сотрудник сам задаёт пароль при первом входе).
- **Границы беспарольного принципала.** До установки пароля принципал **не получает access-токен** и **не имеет доступа ни к одному ресурсу**: setup-token — limited-scope (`type:"pwd_setup"`), проходит **только** `set-password`; ни RBAC-права, ни Users/Roles/Teams, ни ресурсные эндпоинты им недоступны (`401`).
- **Взаимодействие с супер-админом и RBAC.** Супер-админ (`.env`) всегда парольный — модель его не касается. Роль пользователя существует, но не даёт доступа, пока не выпущен access-токен (т.е. пока пароль не задан). Деактивированный (`is_active=false`) беспарольный пользователь пароль задать не может (`401`).
- **Энумерация.** Для **парольных** пользователей вход сохраняет единое `401` (без раскрытия). Для **беспарольных** ответ `password_setup_required` раскрывает и существование, и беспарольность идентификатора — **осознанный** побочный эффект модели открытого первого входа.
- Митигация: админ сообщает Логин/Телеграм адресату оперативно; окно беспарольности держат коротким. Рекомендация — не оставлять беспарольные учётки надолго.

## Обоснование

- **Discriminated login-ответ + limited-scope токен** — минимальный контрактный сдвиг: успех сохраняет прежнюю форму (плюс явный `password_setup_required:false`), а новая ветка не смешивает setup-токен с access-токеном (разные `type`, разные права) → сервер остаётся единственной границей безопасности.
- **`password_hash` nullable** — прямое выражение «беспарольности»; отдельная таблица/флаг не нужны.
- **email→telegram** — 0 строк на проде → чистая замена колонки (rename + swap partial-unique); формат телеграм-ника проще email, консистентен с Telegram-ориентацией продукта.
- **Вход по логину/телеграму** через расширение семантики поля `username` не ломает существующий фронт (поле то же), лишь добавляет альтернативный матч.

## Последствия

- (+) Онбординг без раздачи паролей: сотрудник сам задаёт пароль при первом входе.
- (+) Вход по удобному идентификатору (логин или @телеграм).
- (+) Access-токен не выдаётся до установки пароля — беспарольный принципал ничего не может, кроме set-password.
- (−) Окно уязвимости открытого первого входа (осознанный риск; митигируется оперативной выдачей и коротким окном).
- (−) Энумерация беспарольных идентификаторов (осознанный побочный эффект).
- (−) Новый эндпоинт `POST /api/auth/set-password`, новый `type:"pwd_setup"`-токен, новые коды `password_already_set` (409) и `telegram_taken` (409, замена `email_taken`); миграция `0011` (nullable `password_hash` + email→telegram).
- (−) `get_current_principal` обязан отвергать не-`access` токены (иначе setup-token дал бы доступ) — критичный инвариант.

## Альтернативы

- **Сервер генерирует и присылает временный пароль** — отвергнуто пользователем: выбрана модель «пользователь вводит свой пароль сам» (открытый первый вход).
- **Обязательный пароль при создании (как сейчас)** — отвергнуто: требование — пароль опционален.
- **Одноразовая ссылка/токен приглашения по каналу связи** — отвергнуто на Этапе 1: нет канала доставки (email заменён на telegram-ник как идентификатор, не как транспорт); открытый первый вход проще и принят пользователем.
- **Оставить email как контакт/идентификатор** — отвергнуто: продукт на Telegram; идентификатор — телеграм-ник.
- **Setup-token = обычный access-token** — отвергнуто: дал бы беспарольному принципалу доступ к ресурсам до установки пароля (эскалация); нужен именно limited-scope `type:"pwd_setup"`.
