# ADR-018 · Нотификатор: windowed offline-детект + recovery-уведомления + durable-лог отправленных алертов

- Статус: accepted
- Дата: 2026-07-07
- Связанные: [ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md) (in-backend нотификатор), [ADR-014](ADR-014-persist-notifier-state-alert-on-first-elevated.md) (персист состояния + alert-on-first-elevated), [ADR-016](ADR-016-notifier-max-over-window-zone.md) (max-over-window для зон), [ADR-003](ADR-003-prometheus-istochnik-metrik.md) (Prometheus — источник метрик)

## Контекст

Три связанных дефекта нотификатора, вскрытых прод-инцидентом.

**Инцидент.** Сервер «Фотобудка» был недоступен ~4 часа (`up == 0`), но ни одного offline-алерта не пришло, и после возврата в сеть уведомления о восстановлении тоже не было. Разбор выявил три независимые причины:

1. **Мгновенный offline-детект.** ADR-016 обернул usage CPU/RAM/SSD в `max_over_time` за окно опроса, но **`online`/`up` остался мгновенным** (`online = up_value == 1.0`, сэмпл раз в `NOTIFIER_POLL_INTERVAL_SEC`, default 60 с). Короткое падение доступности **между** моментами опроса теряется — ровно та же «слепота» мгновенного сэмплинга, что ADR-016 закрыл для зон нагрузки, но для `up` осталась. Доля наблюдаемого времени ≈ (длительность падения / poll_interval); минутные падения ловятся с вероятностью в единицы процентов.

2. **Восстановление не уведомляется.** По ADR-009/ADR-014 переход `offline→online` — **молчаливый** (recovery-сообщений для серверов нет, out of scope Этапа 1). Оператор видит «🔴 Сервер не доступен», но не узнаёт из Telegram, что сервер вернулся, — приходится проверять UI вручную.

3. **Факт отправки алертов недоказуем.** Отправки best-effort ([modules/notifier](../modules/notifier/README.md#доставка-в-telegram)), лог отправок пишется только в stdout-логи приложения, которые **ротируются**. Нельзя постфактум ответить на вопрос «слался ли offline-алерт по этому серверу и дошёл ли он», — нет durable-записи.

Пороги зон (`app.domain.thresholds.usage_to_zone()`: `<80 green` / `80..90 yellow` / `>90 red`) — единый источник для UI/backend/нотификатора и **не подлежат изменению**.

## Решение

Три изменения, каждое переиспользует уже существующие механизмы (окно ADR-016, чистую функцию `evaluate()` ADR-014, доставку ADR-009).

### 1. Windowed offline-детект (`min_over_time` для `up`)

В **windowed-режиме** нотификатора (`window_sec` задан) доступность определяется по минимуму `up` за окно опроса:

```
online = min_over_time(up{matcher}[<W>s]) == 1
```

Сервер считается «падавшим» (`online = False`), если `up` был `0` **в любой точке** окна. Это симметрично `max_over_time` для зон (ADR-016): вместо «поймать пик нагрузки» — «поймать провал доступности» между опросами.

- **`up` — сырая серия**, поэтому используется прямой range-vector `up{matcher}[<W>s]` (без subquery/step, в отличие от вычисляемых usage-выражений ADR-016): min берётся по фактическим scrape-сэмплам окна — точнее, чем resampling через `[:step]`, и дешевле.
- **UI/read-path не меняется.** При `window_sec is None` (`GET /api/servers`, `/metrics`) `up` остаётся **мгновенным** (`up{matcher}`, `online = up_value == 1.0`). Windowed min применяется **только** на пути нотификатора. Разведение — тем же опциональным параметром `window_sec`, что уже введён ADR-016.
- **Взаимодействие с существующей windowed-веткой.** В windowed-режиме теперь оборачиваются **обе группы**: usage CPU/RAM/SSD → `max_over_time` (ADR-016), `up` → `min_over_time` (этот ADR). `uptime`/`detail` остаются мгновенными (нотификатор их не использует для решений).

### 2. Выбор окна — переиспользовать `NOTIFIER_METRIC_WINDOW_SEC` (нового env НЕ вводим)

Offline-окно = **то же `NOTIFIER_METRIC_WINDOW_SEC`** (default 90 с, эффективное `notifier_metric_window_effective_sec` после клампа к `poll_interval`), что уже используется для зон.

**Обоснование выбора против отдельного `NOTIFIER_OFFLINE_WINDOW_SEC`:**
- **Идентичный инвариант.** Требование к offline-окну — ровно то же, что к окну зон ADR-016: `W ≥ poll_interval`, чтобы соседние окна `[t-W, t]` перекрывались и любой момент попал хотя бы в одно окно опроса. Отдельный env дал бы **тот же** default и **тот же** кламп — дублирование без функциональной разницы.
- **Простота (NFR-1).** Меньше поверхности конфигурации; окно уже валидируется/клампится в `config.py` (`model_post_init`) — переиспользуется как есть.
- **Единая семантика «окна наблюдения нотификатора».** И зона, и доступность оцениваются на одном и том же окне опроса — концептуально это «что нотификатор видел с прошлого опроса».
- YAGNI: нет требования тюнить чувствительность offline независимо от зон. Если оно появится — env-разделение вводится отдельным ADR (окно уже вынесено в env, разделить дёшево).

**Trade-off «залипание recovery» (принимается).** `min_over_time` держит `online=False` до тех пор, пока хотя бы один `0`-сэмпл остаётся в окне: после фактического возврата сервера recovery-алерт (см. п. 3) задерживается **до ~W** (≤ 90 с) + один `poll_interval`. Это приемлемо и даже полезно — **дебаунсит флаппинг**: сервер, который прыгает down/up/down, не спамит recovery, окно сглаживает дребезг. Симметрично «залипанию» эскалации зоны в ADR-016.

**Остаточная оговорка (принимается, симметрична ADR-016).** Провал `up` короче разрешения scrape (~15 с) может лечь между scrape-сэмплами и не породить `0`-сэмпл → сгладится. Нотификатор сознательно ловит **устойчивые** (≥ scrape) падения, а не микро-дребезг. TD не заводится.

### 3. Recovery-уведомление (`offline→online`)

Добавляется алерт «снова в сети» при переходе `prev.online == False → im.online == True`. Новый `AlertKind` **`recovered`** + `build_recovered(name, ip)` в `app/domain/notifications.py`, формат по стилю существующих (🟢 ВОССТАНОВЛЕНО, как у recovery AI-ключей):

```
🟢🟢🟢ВОССТАНОВЛЕНО🟢🟢🟢
Сервер "<name>"
IP <ip>

Сервер снова в сети
```

**Взаимодействие с alert-on-first-elevated (ADR-014).** Recovery срабатывает **только** при явном `prev.online == False`. Отсутствие строки (`prev is None`) трактуется как здоровый baseline `online=True` (ADR-014) → сервер, впервые увиденный online, **НЕ** даёт ложного recovery. Дедуп — через персист: после recovery `online=True` записывается в `notifier_server_state` → следующий опрос видит `base.online=True` → повторного recovery нет.

**Recovery + элевированный возврат.** Recovery определяется по факту `up` (`prev.online=False → im.online=True`), **независимо от наличия метрик**:
- если при возврате метрики уже доступны и сервер вернулся в yellow/red — шлётся recovery **плюс** зонный алерт (база сравнения = green, как и раньше по ADR-014 → нагрузка переалертивается). Порядок: **сначала `recovered`, затем `warning`/`critical`** (сначала «жив», потом «под нагрузкой»);
- если метрики ещё недоступны (`im.metrics is None`) — шлётся только recovery, зоны не оцениваются (`zone_* = NULL`).

**Симметрия с offline.** Пара офлайн/онлайн теперь полная: `🔴 Сервер не доступен` (`online→offline`) ↔ `🟢 Сервер снова в сети` (`offline→online`). Отменяет соответствующий пункт out-of-scope Этапа 1 ([modules/notifier](../modules/notifier/README.md)).

### 4. Durable-лог отправленных алертов (`notifier_alert_log`)

Новая таблица `notifier_alert_log` — **append-only** запись на **каждый отправленный серверный алерт** (одна строка на один вызов `TelegramClient.send_message`), с фактом доставки:

| Поле | Тип | Смысл |
|------|-----|-------|
| `id` | `bigint` GENERATED BY DEFAULT AS IDENTITY, PK | Монотонный суррогат = порядок вставки |
| `server_id` | `uuid` NULL, FK → `servers(id)` **ON DELETE SET NULL** | Сервер алерта; `NULL` после удаления сервера (лог переживает удаление) |
| `kind` | `text` NOT NULL, CHECK `IN ('offline','recovered','warning','critical')` | Тип алерта (= `AlertKind`) |
| `message` | `text` NOT NULL | Отправленный plain-текст (содержит имя/IP; **секретов нет** — токен/`chat_id` в текст не входят) |
| `delivered` | `boolean` NOT NULL | Результат `send_message` (успех доставки в Telegram) |
| `created_at` | `timestamptz` NOT NULL DEFAULT now() | Время попытки отправки |

**Ключевые решения:**
- **`bigint identity` PK, а не `uuid` (осознанное отклонение от конвенции).** Остальные таблицы на `uuid`, но это append-only ops-лог с неограниченным ростом: монотонный суррогат естественно задаёт порядок вставки, даёт компактный индекс и дешёвую вставку. `uuid gen_random_uuid()` не даёт ничего (на строки лога нет внешних FK, скрывать количество/непредсказуемость для внутреннего лога не нужно). Это верный инструмент для лог-таблицы.
- **`ON DELETE SET NULL` (не CASCADE).** Лог — audit/ops-запись «слался ли алерт». Удаление сервера **не должно стирать историю** его алертов (это ровно то, ради чего durable-лог и заводится). `server_id` обнуляется, `message` (с именем/IP) сохраняет идентичность. Отличается от `notifier_server_state` (CASCADE) намеренно: state — эфемерное состояние дедупа (после удаления бессмысленно), лог — история (ценна и после удаления).
- **Гранулярность — один алерт = одна строка.** `send_message` инкапсулирует до 3 HTTP-ретраев и возвращает один `bool`; логируется **логическая** попытка отправки (один `Alert`), а не каждый HTTP-ретрай. `delivered` = возвращённый флаг.
- **Scope — только серверные алерты этого модуля** (`offline/recovered/warning/critical`). Алерты AI-ключей (`AiKeyMonitorService`, ADR-010) — отдельный сервис, в этот лог **не** пишутся (возможный follow-up, [TD-027](../100-known-tech-debt.md)).
- **Секреты не логируются.** В `message` идёт только тело алерта; токен/`chat_id`/URL — никогда ([05-security.md](../05-security.md)).
- **Индекс** `ix_notifier_alert_log_created_at` на `(created_at DESC)` — под ретенцию-очистку и потенциальный будущий просмотр «последние алерты».
- **Запись — в финальной сессии итерации.** В `poll_once` собирается список `(server_id, kind, message, delivered)` по мере отправок; строки лога вставляются **в той же короткой сессии**, что и UPSERT состояния (в конце итерации), одним коммитом.

**Ретенция — отложена как TD.** Автоочистка старых строк лога не реализуется на этом этапе — [TD-027](../100-known-tech-debt.md) (объём ограничен числом серверов × частота алертов; на масштабе Этапа 1 незначим).

**API-эндпоинт просмотра лога — вне scope.** [04-api.md](../04-api.md) **не затрагивается** (нет нового endpoint). Возможный follow-up — read-only endpoint просмотра последних алертов.

## Обоснование

- **Закрывает корневые причины инцидента «Фотобудка».** (1) windowed `min_over_time(up)` перестаёт «просыпать» падения между опросами; (2) recovery-алерт закрывает информационный разрыв «сервер вернулся»; (3) durable-лог делает факт отправки проверяемым независимо от ротации stdout-логов.
- **Максимальное переиспользование.** Ни нового компонента, ни нового env: окно — то же `NOTIFIER_METRIC_WINDOW_SEC`; offline-детект — та же windowed-ветка маппинга; recovery — та же чистая `evaluate()` (плюс одна ветка и один `build_recovered`); доставка — тот же `TelegramClient`.
- **Нулевой blast radius для UI.** `min_over_time` живёт только в windowed-ветке (`window_sec` задан); мгновенный read-path карточек не тронут (как и в ADR-016).
- **Простота (NFR-1).** +1 таблица (append-only), +1 миграция, +1 `AlertKind`/builder, +1 PromQL-обёртка. Ни Alertmanager, ни recording-rules, ни Redis.

## Последствия

- (+) Короткие падения доступности между опросами больше не теряются → offline-алерт срабатывает.
- (+) Полная симметрия offline/online: оператор получает и «не доступен», и «снова в сети».
- (+) Факт отправки каждого серверного алерта и результат доставки — в durable-таблице, переживают ротацию логов и рестарт.
- (+) UI-карточки не меняются (мгновенный `up`/значения).
- (−) **Задержка recovery до ~W** (окно min): recovery-алерт приходит через ≤ 90 с + `poll_interval` после фактического возврата — принято (дебаунс флаппинга).
- (−) **Возможен offline-алерт на устойчивый (≥ scrape) кратковременный провал**, за которым быстро идёт recovery — приемлемо (лучше лишняя пара offline+recovered, чем пропущенный инцидент); дедуп ограничивает каждым переходом по одному алерту.
- (−) Микро-провалы `up` короче scrape (~15 с) всё ещё сглаживаются — сознательно (см. Решение п. 2).
- (−) +1 строка лога на каждый отправленный алерт; рост таблицы не ограничен до реализации ретенции ([TD-027](../100-known-tech-debt.md)) — на объёмах Этапа 1 незначим.
- (−) Небольшой рост стоимости windowed-запроса: `min_over_time(up[W])` добавляется к уже windowed usage-запросам; на ≤ десятков серверов под общим семафором — незначим.

## Альтернативы

- **Оставить мгновенный `up`** — отвергнуто: это и есть дефект (1). Симметрично отвергнутому в ADR-016 мгновенному сэмплингу зон.
- **Отдельный `NOTIFIER_OFFLINE_WINDOW_SEC`** — отвергнуто (для Этапа 1): идентичный инвариант и default с окном зон → дублирование конфигурации без функциональной разницы; вводится отдельным ADR при появлении требования к независимому тюнингу.
- **Recovery через отдельный «last-alerted» флаг** — отвергнуто: избыточно; переход `prev.online=False → im.online=True` уже полностью определяется зеркалом состояния `notifier_server_state.online` (ADR-014), дедуп получается из персиста бесплатно.
- **`uuid` PK для лога (ради конвенции)** — отвергнуто: для append-only лога `bigint identity` естественнее (порядок вставки, компактность); `uuid` не даёт выгод.
- **`ON DELETE CASCADE` для лога** — отвергнуто: стёрло бы историю алертов удалённого сервера, что противоречит цели durable-лога.
- **Писать лог в файл/внешний sink** — отвергнуто: PostgreSQL уже в стеке, даёт транзакционность (в одной сессии с UPSERT состояния) и запросность; файловый лог так же ротируется, как и текущий stdout.
