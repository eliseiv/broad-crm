# ADR-055 — Отдельные команды для СМС и Почты у пользователя (per-channel scope) + фильтр по командам от 2 команд на всех пяти экранах

- **Статус:** accepted
- **Дата:** 2026-07-14
- **Контекст-модули:** [auth](../modules/auth/README.md), [teams](../modules/teams/README.md), [mail](../modules/mail/README.md), [sms](../modules/sms/README.md), [ui](../modules/ui/README.md)

- **⚠️ Разворачивает действующие нормы (ADR-reversal, оформлено явно):**
  - **[ADR-036](ADR-036-sms-team-filter-admin-only.md)** («Фильтр «Все команды» на `/sms` рендерится **только** при admin-уровне видимости SMS») — **ОТМЕНЕНО в части гейта**: фильтр рендерится **любому**, у кого доступно **≥ 2 команды канала** (§6). Механизм ADR-036 (backend — единственный источник признака; фронт не дублирует `permissions_subset`) — **в силе и расширен**: backend отдаёт не только `sees_all_*_teams`, но и **сам список команд канала** (§5.1), чтобы фронт не тянул `GET /api/teams` под чужим гейтом `teams:view`.
  - **[ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.2** («селектор «Команда» на вкладке «Почты» рендерится **только** при `sees_all_mail_teams === true`») — **ОТМЕНЕНО**, тем же правилом «≥ 2 команды канала». Остальные части ADR-050 (поиск, личная прочитанность) **не затронуты**.
  - **[ADR-037](ADR-037-mini-app-tabs.md)** («фильтры «Все номера/команды» в Mini App **не показываются**») — **ЧАСТИЧНО ОТМЕНЕНО**: фильтр **«Команда»** появляется в **обеих** Mini App (`/tg/mail`, `/tg/sms`) по тому же порогу «≥ 2 команды канала». Фильтр **«Все номера»** в Mini App **по-прежнему не показывается** (эта часть ADR-037 в силе).
  - **[ADR-044](ADR-044-mail-full-merge-into-crm.md) §7 / [ADR-032](ADR-032-sms-visibility-admin-full-catalog.md)** — `MailScope`/`SmsScope` **расширяются**: `team_ids` собирается уже **не только** из `user_teams`, и добавляется поле `includes_unassigned` (§3). Предикат `sees_all_teams` и вся анти-энумерация — **без изменений**.
  - **[TD-058](../100-known-tech-debt.md)** («не-admin в нескольких командах не имеет фильтра по команде на вкладке «Почты»») — **ЗАКРЫВАЕТСЯ** этим ADR (§5.1 даёт фронту список команд канала без права `teams:view`; §6 рендерит фильтр).
  - **Правило «мутация бесхозного объекта (`team_id IS NULL`) → `403` всегда»** (`backend/app/services/mail_service.py:906`; `backend/app/services/sms_number_service.py:126-127` — `if number.team_id is None or number.team_id not in scope.team_ids: raise forbidden()`) — **РАЗВЁРНУТО** для носителя флага «Без команды» (§3, §3.1): при `includes_unassigned=true` не-админ получает **полные** права на бесхозные объекты канала (правка, синк, **удаление**, для СМС — **перенос**). Решение владельца: read-only-режим для бесхозных **отвергнут**.
  - **Правило «снятие команды с номера (`POST /sms/numbers/{id}/transfer` c `team_id=null`) доступно любому носителю `sms:transfer`, чей номер попал в scope»** (`backend/app/services/sms_number_service.py:89-99` — целевая команда валидируется **только на существование**, `:94-97`) — **РАЗВЁРНУТО** (§3.2): для не-админа `team_id=null` теперь допустим **только** при `sms_includes_unassigned=true`. **[TD-060](../100-known-tech-debt.md) — ЗАКРЫВАЕТСЯ** этой нормой.
  - **Отсутствие проверки целевой команды переноса номера по scope** (`sms_number_service.py:94-97`: `get_existing_ids` → `404 sms_team_not_found`; принадлежность целевой команды scope актора **не проверяется**) — **ЗАКРЫВАЕТСЯ** (§3.2): целевой `team_id` обязан ∈ scope актора, иначе `403 forbidden`. Иначе носитель `sms:transfer` с доп-командой мог бы вынести номер в **любую** существующую команду — ровно тот обходной путь, ради запрета которого перенос **ящика** оставлен admin-only (§3).
  - **[TD-050](../100-known-tech-debt.md)** («селектор «Команда» в форме ящика показывает опцию «Без команды» **всем**, не-admin по клику получает `403`») — **ЗАКРЫВАЕТСЯ** §6.3. Прод-баг 2026-07-14 (роль «Пользователь» с полным `mail:*`, но без `teams:view`): `GET /api/teams` под `require("teams","view")` (`backend/app/api/teams.py:27-31`) не отдал ролям список ⇒ в селекте формы ящика (`frontend/src/components/MailboxFormModal.tsx:325` → `useTeams`; `teamOptions` — `:73-77`) остался **единственный** вариант «Без команды», а он admin-only (`mail_service.py:878-881` — `if team_id is None: if not scope.sees_all_teams: raise forbidden()`) ⇒ **создание ящика невозможно вовсе**. §6.3 нормирует: источник команд канала на клиенте не-админу — **`GET /api/auth/me`**, а опция «Без команды» показывается **только** тому, кто вправе её выбрать.

- **НЕ затронуто (явно):** каталог прав (`CATALOG` — новых действий/страниц **нет**), предикат admin-уровня (`is_superadmin OR permissions_subset(full_catalog, permissions)`), анти-энумерация (чтение вне scope → **пустой результат**, не `403`), правило **«перенос почтового ящика между командами — только admin-уровень»** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4), правило **«создание ящика с `team_id = null` — только admin-уровень»** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4), личная прочитанность ([ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §2), Telegram-fan-out уведомлений (§7 — разбирается отдельно), таблица `user_teams` (структура не меняется).

## Контекст

Требование владельца (батч 2026-07-14, дословно):

> «При создании пользователя или при его редактировании необходимо добавить два блока СМС и ПОЧТЫ, которые по умолчанию будут свернуты внизу. При раскрытии одного из блоков будет дублироваться список команд из Поля Команды + значение "Без команды". По умолчанию в СМС и Почты будет галочка на команде из блока Команды. Но можно выбрать дополнительные команды для СМС и Почты отдельно. Таким образом Пользователь из "Команда Ивана" сможет смотреть почты по нескольким командам и СМС по нескольким командам.»

Уточнения, **подтверждённые владельцем**:
1. Дополнительные команды дают **и видимость, и действия** — в них пользователь работает наравне со своей основной командой (**не** read-only).
2. Фильтр по командам показывается **от 2 команд** (не от 3) — на `/mail` (обе вкладки), на `/sms` (вкладка «Сообщения») и **в обеих Mini App**. Порог **единый** для веба и Mini App (разный порог означал бы, что один и тот же пользователь на телефоне фильтр видит, а в браузере — нет).

**Факты по коду (`claims-from-code`, на 2026-07-14, дерево стабильно):**

- `MailScope(sees_all_teams: bool, team_ids: frozenset[UUID])` — `backend/app/domain/mail.py:18-31`; `SmsScope` с той же парой полей — `backend/app/domain/sms.py:24-38`.
- Обе фабрики берут `team_ids` **исключительно** из `user_teams`: `get_mail_scope` — `backend/app/api/deps.py:264-277`, `get_sms_scope` — `:235-249`. Предикат admin-уровня — `principal_sees_all_mail_teams` (`:252-262`) / `principal_sees_all_sms_teams` (`:223-233`).
- Мутация почтового ящика вне scope → `403`: `if not scope.sees_all_teams and account.team_id not in scope.team_ids: raise forbidden()` (`backend/app/services/mail_service.py:906`, `:915-916`) ⇒ **ящик без команды (`team_id IS NULL`) сегодня недоступен не-админу вообще** (`None not in frozenset` — всегда истина).
- Мутация SMS-номера вне scope → `403`: `_require_mutation_scope` — `backend/app/services/sms_number_service.py:122-127` (`number.team_id is None or number.team_id not in scope.team_ids`) ⇒ **бесхозный номер тоже недоступен не-админу**.
- **Перенос номера** (`transfer_number` — `backend/app/services/sms_number_service.py:78-108`): целевая команда валидируется **только на существование** (`get_existing_ids` → `sms_team_not_found`, `:94-97`) — принадлежность целевой команды scope актора **не проверяется**; снятие команды (`team_id=null`) доступно **любому** носителю `sms:transfer`, чей номер попал в scope. Обе бреши закрываются §3.2 ⇒ **[TD-060](../100-known-tech-debt.md) закрыт**.
- **Бесхозные номера рождаются массово и автоматически:** `POST /api/sms/numbers/sync` вставляет **все** входящие номера Twilio-аккаунта как unassigned (`team_id=NULL`) — `SmsNumberRepository.bulk_upsert_unassigned` (`backend/app/repositories/sms_number_repository.py:138-163`), вызов — `backend/app/services/sms_sync_service.py:57`. (Входящее SMS на неизвестный номер строку номера **НЕ** создаёт — `sms_ingest_service.py:120-130`: `find_by_phone` → `team_id=None` в снимке, номер не заводится.) Бесхозные **ящики**, напротив, заводит **только** админ (`mail_service.py:878-881`). Это и есть асимметрия §3.1.
- **Видимость SMS считается по ТЕКУЩЕЙ команде номера** (`sms_message_service.py:95-100` — множество видимых `to_number` из `list_by_teams(scope.team_ids)`), а не по снимку `sms_inbound.team_id` ⇒ флаг «Без команды» в СМС открывает и **сообщения** на все ещё не распределённые номера.
- **`GET /api/teams` гейтится `teams:view`** (`backend/app/api/teams.py:27-31` — `require("teams","view")`) ⇒ mail/sms-оператор без этого права получает **пустой** список команд: корень [TD-058](../100-known-tech-debt.md) и прод-бага «не-админ не может создать ящик» ([TD-050](../100-known-tech-debt.md), §6.3).
- `GET /api/auth/me` отдаёт `sees_all_sms_teams` / `sees_all_mail_teams` ([04-api.md §GET /api/auth/me](../04-api.md#get-apiauthme)), но **не отдаёт список команд пользователя** — именно поэтому фильтр «Команда» не-админу сегодня нечем наполнить ([TD-058](../100-known-tech-debt.md)).
- Каталог прав `mail` — `("view","create","edit","delete","sync","tags")` (`backend/app/domain/permissions.py:21`).
- Последняя миграция цепочки — **`0026_users_is_system`** ([03-data-model.md](../03-data-model.md#миграция-0026_users_is_system-концепт-adr-051-11)) ⇒ новая — **`0027`**.

**Суть задачи.** Сегодня видимость почты и SMS выводится из **одного** набора — членства в командах (`user_teams`). Владельцу нужно, чтобы **сотрудник одной команды** обрабатывал почты нескольких команд и СМС нескольких (**других**) команд, при этом наборы каналов **независимы**, а базовое членство остаётся тем, чем было.

## Решение

### 1. Модель: базовое членство + **дополнительные** наборы по каналам (union-семантика)

Вводится понятие **scope канала** (`channel ∈ {mail, sms}`):

```
effective_channel_teams(user, channel) = user_teams(user)  ∪  extra_channel_teams(user, channel)
effective_includes_unassigned(user, channel) = users.<channel>_includes_unassigned
```

- **Базовые команды (`user_teams`) входят в scope ОБОИХ каналов всегда** — их нельзя «отключить» в блоке канала. Это буквальное прочтение требования владельца («по умолчанию галочка на команде из блока Команды; **но можно выбрать дополнительные** команды»): блок канала задаёт **добавку**, а не замену.
- **Хранится только добавка.** Команда, входящая в `user_teams`, в таблицу добавок **не пишется** (см. §2.3) — поэтому исключение пользователя из команды в основном блоке **автоматически** убирает её из обоих каналов, без backfill и без «повисших» прав. Это и есть ответ на вопрос «как дефолтная галочка связана с изменением основного блока».
- **«Без команды»** — не команда, а **флаг** доступа к объектам с `team_id IS NULL` (ящики/номера без команды). Хранится булевой колонкой на `users` — по одной на канал (§2.2). У базового блока «Команды» такого значения нет и не появляется.

### 2. Данные ([03-data-model.md](../03-data-model.md))

#### 2.1 Таблица `user_channel_teams` (новая)

| Колонка | Тип | Ограничения | Комментарий |
|---|---|---|---|
| `user_id` | `uuid` | PK-часть, FK `users(id)` **`ON DELETE CASCADE`** | Пользователь |
| `channel` | `text` | PK-часть, **CHECK** `channel IN ('mail','sms')` | Канал |
| `team_id` | `uuid` | PK-часть, FK `teams(id)` **`ON DELETE CASCADE`** | **Дополнительная** команда канала |

- `PRIMARY KEY (user_id, channel, team_id)` — добавка не дублируется; префикс `(user_id, channel)` покрывает основную выборку «доп-команды пользователя по каналу».
- Индекс **`ix_user_channel_teams_team_id (team_id)`** — обязателен под `ON DELETE CASCADE` при удалении команды (иначе seq-scan) и под обратную выборку.
- `channel` — **`text` + CHECK**, а не PG-enum: канала два, добавление третьего не планируется, а enum потребовал бы `ALTER TYPE` в миграции (по образцу «свободных» CHECK'ов проекта — `ck_mail_tag_rules_type`, `ck_notifier_alert_log_kind`).

#### 2.2 Колонки на `users` (новые)

| Колонка | Тип | Ограничения | Комментарий |
|---|---|---|---|
| `mail_includes_unassigned` | `boolean` | `NOT NULL`, `DEFAULT false` | «Без команды» в блоке **«Почты»**: доступ к ящикам/письмам с `mail_accounts.team_id IS NULL` |
| `sms_includes_unassigned` | `boolean` | `NOT NULL`, `DEFAULT false` | «Без команды» в блоке **«СМС»**: доступ к номерам/сообщениям с `sms_phone_numbers.team_id IS NULL` |

**Почему флаг-колонки, а не строка с `team_id IS NULL` в `user_channel_teams`:** `NULL` в составе `PRIMARY KEY` PostgreSQL не допускает; вариант «`UNIQUE NULLS NOT DISTINCT` + суррогатный PK» (PG 16 это умеет) даёт таблицу, где одна и та же колонка значит то команду, то «отсутствие команды», и **каждый** scope-запрос всё равно вынужден писать `OR team_id IS NULL`. Флаг — на одно понятие одна колонка; контракт API (`*_includes_unassigned: boolean`, §5) при этом **зеркалит хранение 1:1**.

#### 2.3 Инвариант нормализации (нормативно, обеспечивает сервис — на ВСЕХ путях записи в `user_teams`)

**Инвариант:** в `user_channel_teams` **не хранятся** команды, входящие в `user_teams` того же пользователя.

**Он обязателен на КАЖДОМ пути записи в `user_teams`, а не только в users CRUD.** Путей записи в `user_teams` ровно **два** (сверено с кодом 2026-07-14):

| # | Путь | Точка записи в коде | Что обязан сделать сервис дополнительно |
|---|------|---------------------|------------------------------------------|
| **1** | **Users CRUD** — `POST /api/users`, `PATCH /api/users/{id}` (поле `team_ids`) | `UserRepository.set_membership` (`backend/app/repositories/user_repository.py:194-215`: `delete` выбывших → `insert` новых строк `user_teams`) | **Вычесть** базовый набор из присланной добавки: `extra := payload.<channel>_extra_team_ids − payload.team_ids` (для обоих каналов) и сохранить **разность**. Присланная «лишняя» базовая команда — **не ошибка** (не `422`), просто не сохраняется |
| **2** | **Teams CRUD** — `POST /api/teams` и `PATCH /api/teams/{id}` (поле `member_ids`) | `TeamRepository.create` → `_insert_members` (`backend/app/repositories/team_repository.py:100-118`, `:199-223`) и `TeamRepository.replace_members` (`:120-145`: `delete`/`insert` строк `user_teams`; **`user_channel_teams` не трогает**) | После приведения состава команды `T` к `member_ids` **удалить** строки `user_channel_teams (user_id ∈ member_ids, channel ∈ {mail, sms}, team_id = T)` — **в той же транзакции**. Т.е. пользователь, ставший **участником** команды, теряет её как **добавку** (она ему больше не нужна: базовое членство и так входит в scope обоих каналов) |

> **Зачем путь 2 (без него инвариант path-dependent и даёт «висящий» доступ).** Достижимая последовательность **до** этой нормы: (а) админ даёт `X` доп-команду `B` в блоке «Почты» → строка `(X, mail, B)`; (б) позже добавляет `X` в команду `B` участником на `/teams` (`replace_members` — сервис users **не вызывался**, инвариант нарушен); (в) админ исключает `X` из команды `B` на `/teams` → строка `user_teams` удалена, **строка добавки осталась** ⇒ `X` **сохраняет** доступ к почте команды `B` после исключения из неё. Ровно то, что §1/§2.3 объявляют невозможным. Гарантия «снял в основном блоке → потерял доступ в обоих каналах» обязана быть **безусловной**, а не зависеть от того, на какой странице админ правил членство.

- **Удаление команды/пользователя** нормализации не требует: обе таблицы каскадят (`ON DELETE CASCADE` — §2.1, `user_teams` — [03-data-model.md](../03-data-model.md#таблицы-teams-и-user_teams-crm-команды)).
- **Проверка инварианта — сервисная, не БД-триггером** (идиома проекта: инварианты обеспечивает сервис как единственная точка записи — ср. «если лидер задан — он ∈ участники», [ADR-026](ADR-026-teams-optional-leader-auto-transfer.md)).

#### 2.4 Миграция `0027_user_channel_teams` (23 симв. ≤ 32 — [правило id](../03-data-model.md#1-revision-id--не-длиннее-32-символов))

- `upgrade()`: `CREATE TABLE user_channel_teams` (+ `ix_user_channel_teams_team_id`); `ALTER TABLE users ADD COLUMN mail_includes_unassigned boolean NOT NULL DEFAULT false`, то же для `sms_includes_unassigned`.
- `downgrade()` (рабочий, обязателен): `DROP TABLE user_channel_teams;` + `ALTER TABLE users DROP COLUMN …` (обе).
- **Backfill НЕ выполняется и НЕ нужен** (ответ на вопрос миграции существующих пользователей): при пустых добавках эффективный scope канала **тождественно равен** `user_teams` — ровно то, что действует сегодня. **Регрессии видимости у существующих пользователей — ноль**, ни одной строки переносить не требуется.

### 3. Scope ([05-security.md](../05-security.md))

`MailScope` и `SmsScope` получают **третье** поле:

```
MailScope(sees_all_teams: bool, team_ids: frozenset[UUID], includes_unassigned: bool)
SmsScope (sees_all_teams: bool, team_ids: frozenset[UUID], includes_unassigned: bool)
```

- **`sees_all_teams`** — предикат **не меняется** (`is_superadmin OR permissions_subset(full_catalog, permissions)`). При `true` остальные поля не используются (видно всё, включая бесхозное).
- **`team_ids`** = `user_teams` **∪** `user_channel_teams[channel]` (одним запросом-`UNION`, а не двумя round-trip'ами).
- **`includes_unassigned`** = `users.<channel>_includes_unassigned`.

**Правила (нормативно, симметрично для обоих каналов):**

| Путь | Правило |
|---|---|
| **Чтение** ленты/каталога (`GET /api/mail/messages`, `/mail/mailboxes`, `/sms/messages`, `/sms/numbers`) при `sees_all_teams=false` | Предикат: `obj.team_id IN team_ids` **OR** (`includes_unassigned` **AND** `obj.team_id IS NULL`). Вне scope → **пустой результат** (анти-энумерация, не `403`/`404`). **Пустой `team_ids` И `includes_unassigned=false`** → пустая страница **без выборки** (существующий short-circuit сохраняется) |
| **Мутация/удаление/синк** ящика по `id`; правка/перенос/удаление номера по `id` | Объект обязан пройти **тот же** предикат, иначе `403 forbidden`. **Изменение против сегодняшнего:** бесхозный объект (`team_id IS NULL`) становится доступен не-админу — **только** при `includes_unassigned=true` (сегодня — `403` всегда, `mail_service.py:906`, `sms_number_service.py:124-127`) |
| **Reply / отметка прочитанности** письма | Тот же предикат; вне scope → **`404 mail_message_not_found`** (анти-энумерация, [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)) |
| **Создание ящика** (`POST /mail/mailboxes`, `POST /mail/mailboxes/oauth/authorize`) | `team_id` обязан ∈ `team_ids` (теперь — включая доп-команды) → `403` иначе. **`team_id = null` — по-прежнему ТОЛЬКО admin-уровень** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4 **НЕ разворачивается**): `includes_unassigned` даёт работу с **существующими** бесхозными ящиками, но **не право создавать новые вне командной модели** |
| **Перенос ящика** (`PATCH /mail/mailboxes/{id}` со сменой `team_id`) | **По-прежнему ТОЛЬКО admin-уровень** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4 **НЕ разворачивается**) — иначе доп-команды стали бы обходным путём выноса ящика из чужой команды |
| **Перенос номера** (`POST /sms/numbers/{id}/transfer`) | **Правило РАСШИРЕНО (§3.2)**: (1) сам номер обязан пройти предикат scope (иначе `403`); (2) **целевая** команда для не-админа обязана ∈ `team_ids` (иначе `403 forbidden`) — прежде проверялось **только существование** (`sms_number_service.py:94-97`); (3) `team_id = null` (снять команду) для не-админа — **только при `includes_unassigned=true`** (иначе `403`) |
| **Теги почты** | Глобальны, scope не применяется ([ADR-044](ADR-044-mail-full-merge-into-crm.md)) — **без изменений** |

**Граница безопасности — backend.** Фронт-гейтинг (§6) — только UX.

#### 3.1 Асимметрия «Без команды» между каналами (нормативно; последствие принято владельцем ОСОЗНАННО)

Флаг «Без команды» **формально симметричен** (одна колонка на канал), но **фактический объём доступа у каналов РАЗНЫЙ** — из-за того, как объекты становятся бесхозными (`claims-from-code`, 2026-07-14):

| Канал | Как появляется объект с `team_id IS NULL` | Объём того, что открывает флаг |
|---|---|---|
| **Почты** (`mail_includes_unassigned`) | **Только рукотворно и только админом:** создание ящика с `team_id=null` — admin-уровень (`mail_service.py:878-881`), перенос ящика между командами — тоже admin-уровень ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4, §3 выше). Автоматического источника бесхозных ящиков **нет** | Небольшой, **предсказуемый** набор ящиков, каждый из которых админ завёл вручную |
| **СМС** (`sms_includes_unassigned`) | **Массово и автоматически:** `POST /api/sms/numbers/sync` тянет **все** входящие номера Twilio-аккаунта и вставляет их как **unassigned** (`team_id=NULL`, `added_by_user_id=NULL`) — `SmsNumberRepository.bulk_upsert_unassigned` (`backend/app/repositories/sms_number_repository.py:138-163`), вызов — `sms_sync_service.py:57`. **Команда назначается позже**, вручную, через `transfer`. Плюс сюда же попадает номер, у которого команду **сняли** (`transfer` c `team_id=null`) | **ВЕСЬ ещё-не-распределённый поток номеров** аккаунта — и все SMS, пришедшие на них (видимость SMS считается по **текущей** принадлежности номера: `sms_message_service.py:95-100`) |

**Нормативное следствие (формулировка для админа, ставящего галочку):**

> **Галочка «Без команды» в блоке «СМС» открывает пользователю ВЕСЬ ещё не распределённый по командам поток номеров и их SMS — включая правку, УДАЛЕНИЕ номера и ПЕРЕНОС его в свою команду.** Каждый новый номер, подтянутый синхронизацией Twilio, попадает к нему **автоматически**, пока номеру не назначена команда. Это **не** «пара бесхозных объектов», как в почте.

- Права на бесхозные объекты — **полные** (решение владельца; read-only-вариант **отвергнут**): `sms:edit`/`sms:delete`/`sms:transfer` и `mail:edit`/`mail:delete`/`mail:sync` действуют на них наравне со своей командой. Это **разворот** действующего правила «мутация бесхозного объекта → `403` всегда» (`sms_number_service.py:126-127`, `mail_service.py:906`/`:915-916`) — оформлен явно (шапка ADR).
- **Не разворачивается** (остаётся admin-only): создание ящика с `team_id=null` и перенос **ящика** между командами ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §4) — иначе бесхозность стала бы **путём производства** бесхозных ящиков и обходом командной модели почты.
- **Митигация — UI, а не урезание прав:** чекбокс «Без команды» в блоке **«СМС»** формы пользователя несёт обязательную подсказку об объёме (§6.1, [08-design-system.md](../08-design-system.md#блоки-смс-и-почты-в-форме-пользователя-нормативно-adr-055)); риск зафиксирован в [05-security.md](../05-security.md#риск-флага-без-команды-в-смс-нормативно-adr-055-31).

#### 3.2 Перенос номера: целевая команда обязана быть в scope (нормативно)

Сегодня `transfer_number` (`backend/app/services/sms_number_service.py:78-108`) валидирует целевую команду **только на существование** (`self._teams.get_existing_ids({payload.team_id})` → иначе `sms_team_not_found`, `:94-97`); **принадлежность целевой команды scope актора не проверяется**. В связке с доп-командами это открывает ровно тот обходной путь, ради запрета которого перенос **ящика** оставлен admin-only: носитель `sms:transfer` мог бы вынести попавший в его scope номер в **любую** существующую команду — в том числе ту, в которой не состоит.

**Правила (нормативно, порядок проверок — прецеденция):**

1. **Номер** обязан пройти предикат scope (§3) → иначе `403 forbidden` (существующее поведение `_require_mutation_scope`, расширенное веткой `includes_unassigned`).
2. **`team_id = null` (снять команду):** admin-уровень — можно всегда; не-админ — **только при `includes_unassigned = true`**, иначе **`403 forbidden`**. Обоснование: (а) иначе актор безвозвратно **выбрасывает номер из собственного scope** и вернуть его не может (самоблокировка — прежний [TD-060](../100-known-tech-debt.md)); (б) при `includes_unassigned=true` бесхозный номер остаётся ему доступен ⇒ операция обратима.
3. **`team_id = <uuid>`:**
   - **admin-уровень:** команда обязана существовать → иначе **`404 sms_team_not_found`** (без изменений).
   - **не-админ:** целевая `team_id` обязана ∈ `SmsScope.team_ids` (базовые ∪ доп-команды) → иначе **`403 forbidden`**. Проверка scope идёт **ПЕРВОЙ**: несуществующая команда не-админу тоже даёт `403` (она заведомо вне scope) — **анти-энумерация**: не-админ не должен различать «команды нет» и «команда чужая». `404 sms_team_not_found` остаётся ответом **admin-уровня**.

**[TD-060](../100-known-tech-debt.md) закрывается** этой нормой (самоблокировка невозможна: снять команду вправе либо админ, либо носитель `includes_unassigned`, для которого номер остаётся видимым).

### 4. Права: доп-команда = **полноценная** команда канала (решение владельца)

Read-only-режим для доп-команд **отвергнут** (прямое уточнение владельца): пользователь в доп-команде отвечает на письма, правит ящики/номера, синкает, удаляет — под теми же правами роли (`mail:*` / `sms:*`), что и в своей команде. Новых действий в каталоге прав **не вводится**; «в какой команде можно» задаёт scope, «что можно» — роль.

> **Единственные два исключения** (оба — сохранение действующих норм, а не новое ограничение): создание ящика **без команды** и **перенос** ящика между командами остаются admin-уровнем (§3).

### 5. API ([04-api.md](../04-api.md))

#### 5.1 `GET /api/auth/me` += **эффективный** scope каналов

```json
{
  "username": "Никита", "role": "Оператор", "is_superadmin": false,
  "sees_all_sms_teams": false, "sees_all_mail_teams": false,
  "mail_teams": [ { "id": "d3f0…a1", "name": "Команда Ивана" }, { "id": "9b2c…7e", "name": "Business 2" } ],
  "mail_includes_unassigned": true,
  "sms_teams":  [ { "id": "d3f0…a1", "name": "Команда Ивана" } ],
  "sms_includes_unassigned": false,
  "permissions": { "mail": ["view"] }
}
```

| Поле | Тип | Семантика |
|---|---|---|
| `mail_teams` / `sms_teams` | `TeamRef[]` (`{id, name}`) | **Команды канала, доступные актору** — отсортированы по `name` (ru, ci). Не-админ: **ЭФФЕКТИВНЫЙ** scope = `user_teams ∪ добавки` (объединение, **не** только добавка). **Admin-уровень (`sees_all_<channel>_teams = true`): ВСЕ команды системы** (его scope не сужен ⇒ доступны все) — **`[]` НЕ отдаётся** |
| `mail_includes_unassigned` / `sms_includes_unassigned` | boolean | Виден ли пользователю «объект без команды» этого канала. При `sees_all_*_teams = true` → **`true`** (admin видит и бесхозные) |

> **Зачем в `/me`, а не через `GET /api/teams`:** у mail/sms-оператора может не быть права `teams:view` — именно это делало фильтр «Команда» не-админу нереализуемым ([TD-058](../100-known-tech-debt.md), [ADR-036](ADR-036-sms-team-filter-admin-only.md)). `/me` возвращает **ровно те** команды, в которых пользователь и так работает ⇒ новой утечки нет: имена команд, чьи объекты он и так видит.

> **Почему admin-уровню отдаётся ПОЛНЫЙ список, а не `[]` (правка редакции 2, нормативно).** Первая редакция предписывала `sees_all_* = true → []` («полный список фронт возьмёт из `GET /api/teams`»). Это давало **сломанное состояние в Mini App**: `GET /api/teams` из Mini App **запрещён** (§6.2 — единственный источник опций там `/me`), значит у актора admin-уровня, открывшего `/tg/mail` или `/tg/sms`, фильтр рендерился бы **без единой команды** — тот самый «мусорный контрол», который [ADR-036](ADR-036-sms-team-filter-admin-only.md) упразднил. Состояние **достижимо**: Telegram-линковка запрещена только **консольному** супер-админу (`backend/app/services/sms_telegram_link_service.py:82-83` — `if is_superadmin: raise forbidden()`), а **БД-пользователь с полным каталогом прав** линковку проходит и получает `sees_all_*_teams = true` (`backend/app/api/deps.py:223-233`, `:252-262`). Отдавая admin-уровню полный список, мы делаем `/me` **ЕДИНСТВЕННЫМ** источником опций команд канала для **всех** акторов и **всех** экранов: `GET /api/teams` для фильтра не нужен **никому**, ветвление «admin ↔ не-admin» из клиента исчезает. Утечки нет: admin-уровень видит все команды и так (`GET /api/teams` ему доступен).

#### 5.2 Users CRUD += **хранимые добавки**

Поля симметрично добавляются в `UserListItem` (ответ), `UserCreateRequest`, `UserUpdateRequest`. **Гейт эндпоинтов не меняется — `require_admin`.**

| Поле (ответ `UserListItem`) | Тип | Семантика |
|---|---|---|
| `mail_extra_teams` / `sms_extra_teams` | `TeamRef[]` | **ТОЛЬКО ДОБАВКА** (строки `user_channel_teams`), **без** базовых `teams` — то, что реально хранится. Может быть `[]` |
| `mail_extra_includes_unassigned` / `sms_extra_includes_unassigned` | boolean | Флаг «Без команды» канала |

| Поле (запрос `UserCreateRequest` / `UserUpdateRequest`) | Тип | Правила |
|---|---|---|
| `mail_extra_team_ids` / `sms_extra_team_ids` | `uuid[]`? | **опц.** `create`: default `[]`. `PATCH`: не передано → не менять; передано → **полностью заменяет** набор добавок канала (`[]` → снять все). Каждый id должен существовать → иначе `422 unprocessable` (`details[].field="mail_extra_team_ids"` / `"sms_extra_team_ids"`). Без дублей. **Пересечение с `team_ids` того же запроса ВЫЧИТАЕТСЯ сервисом** (§2.3) — не ошибка |
| `mail_extra_includes_unassigned` / `sms_extra_includes_unassigned` | boolean? | **опц.** `create`: default `false`. `PATCH`: не передано → не менять |

> **Именование разведено намеренно:** `*_extra_*` (users CRUD) = **хранимая добавка**; `mail_teams`/`sms_teams` (`/me`) = **эффективный** объединённый scope. Одинаковые имена для разных множеств породили бы ровно тот класс ошибок, ради которого этот раздел нормируется.

**Прецеденция ошибок `POST /api/users`** (дополняется): схемная валидация → существование `role_id`, всех `team_ids` **и всех** `*_extra_team_ids` (`422 unprocessable`) → `409 username_taken` → `409 telegram_taken`.

#### 5.3 Серверные фильтры лент += «Без команды»

`GET /api/mail/messages` и `GET /api/sms/messages` (обе ленты — **серверные** фильтры) получают параметр:

| Параметр | Тип | Правила |
|---|---|---|
| `no_team` | boolean? | `true` → **только** объекты без команды (`mail_accounts.team_id IS NULL` / `sms_phone_numbers.team_id IS NULL`). Отсутствует/`false` → фильтр не применяется. **Взаимоисключающ с `team_id`**: оба заданы → **`400 validation_error`**. Пересекается со scope: `no_team=true` у пользователя без `includes_unassigned` (не-admin) → **пустая страница** (анти-энумерация, не `403`) |

Нужен потому, что «Без команды» — легитимное значение фильтра (§6), а `team_id` (UUID) выразить «команды нет» не может. Вкладка **«Почты»** фильтрует **клиентски** ([ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.2 — каталог ящиков грузится целиком).

> **⚠️ Что именно «не меняется» у `GET /api/mail/mailboxes` (уточнение — двусмысленности быть не должно):**
> - **НЕ меняется КОНТРАКТ:** query-параметры (единственный — `is_active`), схема ответа `MailMailboxesResponse`/`MailMailbox`, отсутствие пагинации. Параметров `no_team`/`team_id`/`q` здесь **нет и не вводится** ([04-api.md](../04-api.md#get-apimailmailboxes)).
> - **МЕНЯЕТСЯ РЕЗУЛЬТАТ:** каталог фильтруется `MailScope`, а `MailScope` расширен (§3) ⇒ при `includes_unassigned = true` ответ **обязан включать бесхозные ящики** (`team_id IS NULL`) — иначе клиентский фильтр «Без команды» на вкладке «Почты» покажет пустоту у пользователя, которому эти ящики **доступны на мутацию**. Выборка каталога обязана применять **единый предикат §3**, а не `team_id IN team_ids`.

### 6. UI ([08-design-system.md](../08-design-system.md))

#### 6.1 Форма пользователя — два свёрнутых блока

- **Расположение:** **внизу** формы (после блока «Команды», перед кнопками), в порядке **«СМС»**, затем **«Почты»** (порядок из формулировки владельца). Оба **свёрнуты по умолчанию** — и в `add`, и в `edit`.
- **Примитив:** общего `ui/Collapsible` в ДС **нет** ([TD-057](../100-known-tech-debt.md)) → используется **тот же ручной паттерн**, что у существующих сворачиваемых блоков (кнопка-триггер с `aria-expanded`/`aria-controls`, `ChevronDown` c `rotate-180` при раскрытии, содержимое монтируется при раскрытии). **Новый примитив ДС этой задачей НЕ вводится**; блоки — ещё два потребителя [TD-057](../100-known-tech-debt.md).
- **Содержимое блока (нормативно):** список **всех** команд (`GET /api/teams` — страница «Пользователи» admin-only, гейт есть) чекбоксами (`ui/Checkbox`) + **последним** пунктом чекбокс **«Без команды»**.
- **Подсказка под чекбоксом «Без команды» — ОБЯЗАТЕЛЬНА и РАЗНАЯ по каналам (§3.1).** Админ, ставящий галочку, обязан понимать объём:
  - блок **«СМС»**: **«Даст доступ ко ВСЕМ номерам без команды — это весь ещё не распределённый поток из синхронизации Twilio, включая правку, удаление и перенос номера.»**
  - блок **«Почты»**: **«Даст доступ к ящикам без команды (их заводит только администратор), включая правку, синк и удаление.»**
- **Базовая команда (из блока «Команды») в блоке канала:** чекбокс **отмечен и `disabled`** (снять нельзя), рядом — вторичная подпись **«из блока «Команды»»**. **Реактивность обязательна:** снятие команды в блоке «Команды» немедленно превращает её чекбокс в блоке канала в обычный **неотмеченный** (доступный) — авто-простановки галочки при этом **не происходит** (потеря доступа — намеренное действие администратора).
- **Отправляется** только **добавка**: базовые (disabled) чекбоксы в `*_extra_team_ids` **не включаются**; «Без команды» → `*_extra_includes_unassigned`.
- **Свёрнутый заголовок показывает сводку** (чтобы не раскрывать блок ради проверки): «**СМС** · доп. команд: N» / «**Почты** · доп. команд: N» (при `includes_unassigned` к N прибавляется «+ Без команды»). При нулевой добавке — «доп. команд нет».

#### 6.2 Фильтр «Команда» — единое правило на **пяти** экранах (порог **2**)

Правило **одно** для **пяти** экранов: (1) `/mail` вкладка **«Сообщения»** (серверный), (2) `/mail` вкладка **«Почты»** (клиентский), (3) `/sms` вкладка **«Сообщения»** (серверный), (4) **Mini App `/tg/mail`**, (5) **Mini App `/tg/sms`** (вкладка «Сообщения»; оба — серверные).

**Опции фильтра канала — ОДИН источник для ВСЕХ акторов (нормативно):**

| Актор | Опции |
|---|---|
| **любой** (и admin-уровень, и не-админ) | «Все команды» + команды из **`me.<channel>_teams`** (`GET /api/auth/me`, §5.1 — у admin-уровня это **все** команды системы) + «Без команды» **только при** `me.<channel>_includes_unassigned === true` (у admin-уровня — всегда `true`) |

**Условие рендера (нормативно):**

```
options_count = me.<channel>_teams.length + (me.<channel>_includes_unassigned ? 1 : 0)   // «Все команды» НЕ считается
render_filter  =  options_count >= 2
```

- Порог **2** (не 3) — прямое уточнение владельца. Один вариант выбора фильтровать нечего ⇒ контрол не рендерится (не пустой, не `disabled` — **отсутствует**), как и требовал [ADR-036](ADR-036-sms-team-filter-admin-only.md) для мусорных контролов.
- **Отдельной ветки `sees_all_<channel>_teams === true → рендерить всегда` НЕТ** (правка редакции 2). Она была лишней и вредной: у admin-уровня `includes_unassigned = true` и виден **весь** список команд ⇒ при ≥ 1 команде в системе порог и так выполнен, а при **нуле** команд в системе форсированный рендер дал бы контрол с единственной опцией «Без команды» — снова мусорный контрол. Единое правило `options_count >= 2` покрывает всех акторов **без ветвления** и без сломанных состояний.
- **Порог един для веба и Mini App** — иначе один и тот же пользователь видел бы фильтр на телефоне и не видел в браузере.
- **Источник опций — ТОЛЬКО `GET /api/auth/me`** (§5.1), для **любого** актора, включая admin-уровень. Ходить за `GET /api/teams` ради фильтра **запрещено** — и из Mini App, и из веба, и под admin-уровнем: источник обязан быть **один** (это и есть корень [TD-058](../100-known-tech-debt.md); а для admin-уровня в Mini App `GET /api/teams` вообще не вариант — см. §5.1).
- Выбор «Без команды» → серверные ленты шлют **`no_team=true`** (§5.3); вкладка «Почты» фильтрует клиентски по `mailbox.team_id === null` (как сегодня).
- Прочие правила фильтров **не меняются**: AND-комбинируемость с фильтром почты/номера, сброс пагинации при смене серверного фильтра, «ни один фильтр не сбрасывает другие» ([ADR-052](ADR-052-mail-mailbox-combobox.md) §3.1а).

#### 6.3 Единый источник команд канала на клиенте — НЕ только фильтр (нормативно; закрывает [TD-050](../100-known-tech-debt.md) и прод-баг 2026-07-14)

**Правило шире фильтра «Команда».** Любой контрол, показывающий **список команд канала** — **любому** актору, включая admin-уровень, — наполняется из **`GET /api/auth/me`** (`mail_teams`/`sms_teams`, §5.1), а **не** из `GET /api/teams`. Причина — структурная, а не стилистическая: `GET /api/teams` гейтится правом **`teams:view`** (`backend/app/api/teams.py:27-31` — `require("teams","view")`), которого у mail/sms-оператора **нет**; список приходит **пустым**, и контрол молча деградирует. А в Mini App `GET /api/teams` не берётся **вовсе** (§6.2) ⇒ ветка «для admin — `GET /api/teams`» там дала бы **пустой фильтр у admin-уровня** (§5.1). Поэтому источник **один на всех**, ветвления по актору в клиенте **нет**.

**Прод-баг, доказавший необходимость (2026-07-14).** Роль «Пользователь» с **полным** `mail:*`, но без `teams:view`, **не могла создать ящик**: селектор «Команда» в `MailboxFormModal` (`frontend/src/components/MailboxFormModal.tsx:325` → `useTeams`; `teamOptions` — `:73-77` = «Без команды» + все команды из `GET /api/teams`) получал **пустой** список ⇒ единственным вариантом оставалась опция **«Без команды»**, а она admin-only (`mail_service.py:878-881`) ⇒ **любой** выбор гарантированно давал `403 forbidden` («Недостаточно прав»). Тупик: единственный доступный вариант заведомо запрещён. Владелец обошёл это, выдав роли право на страницу «Команды», — **лекарство не должно требовать лишнего права**.

**Нормативно (все места, где список команд виден не-админу):**

| Контрол | Источник опций (**один для всех акторов**) | Опция «Без команды» |
|---|---|---|
| Фильтр **«Команда»** (5 экранов, §6.2) | **`me.<channel>_teams`** (`GET /api/auth/me`) | показывается **только** при `me.<channel>_includes_unassigned === true` (у admin-уровня — `true`) |
| **Селектор «Команда» в форме ящика** (`MailboxFormModal`, режимы `add` **и** `edit`, включая OAuth-подключение) | **`me.mail_teams`** (у admin-уровня — все команды системы) | **не предлагается к выбору никому, кроме admin-уровня** (`me.sees_all_mail_teams === true`). Не-админу она **не рендерится**: он не вправе её выбрать (создание ящика с `team_id=null` — admin-only, §3), а показанная опция = гарантированный `403` по клику. ⚠️ Гейт этой опции — **`sees_all_mail_teams`**, а **не** `mail_includes_unassigned` (флаг даёт работу с существующими бесхозными ящиками, но **не** право создавать новые — §3). **Единственное исключение — «зеркало текущего состояния» §6.3.1** (режим `edit`, контрол `disabled`, редактируемый ящик уже бесхозный): опция **отображается**, но выбрать её нельзя |
| **Резолв имени команды** в таблице вкладки «Почты» (`MailboxesTab` — колонка «Команда», `frontend/src/components/MailboxesTab.tsx:101` `useTeams()`) | **`me.mail_teams`** (все видимые ящики по определению ∈ scope актора ⇒ имя резолвится) | `team_id === null` → надпись **«Без команды»** (не опция, а отображение) |
| **Дропдаун переноса ящика** (`MailboxesTab`, колонка «Команда» в режиме переноса) | **`me.mail_teams`** — контрол рендерится **только** admin-уровню (перенос ящика admin-only, §3), а у него `mail_teams` = все команды системы ⇒ отдельный `GET /api/teams` **не нужен** | — |
| **`Select` переноса номера** в таблице «Номера» `/sms` (`SmsNumberRow`, гейт `sms:transfer` — **доступен не-админу**; сегодня `frontend/src/pages/SmsPage.tsx:71` тянет `useTeams(seesAllTeams)` ⇒ у не-админа список **пуст**) | **`me.sms_teams`** (ровно те команды, куда перенос разрешён по §3.2 ⇒ опция ≠ `403`) + **опция-зеркало текущей команды номера**, если её вдруг нет в `me.sms_teams` (§6.3.1) | показывается **только** при `me.sms_includes_unassigned === true` (§3.2 п.2: иначе снятие команды → `403`) |

- **Принцип (нормативно, шире этих пяти строк):** **вариант, который пользователь не вправе выбрать, не ПРЕДЛАГАЕТСЯ к выбору.** Контрол не должен предлагать вариант, гарантированно оканчивающийся `403` — это ловушка, а не «защита сервером». Backend остаётся границей безопасности; UI обязан не производить заведомо запрещённых вариантов. **Отображение фактического текущего значения объекта в контроле, где этот вариант выбрать невозможно, предложением НЕ является** — точные границы в §6.3.1.
- **`GET /api/teams` на клиенте остаётся ТОЛЬКО на страницах «Пользователи» и «Команды»** (admin-only / `teams:view`): там нужен **каталог всех команд** как справочник (в т.ч. чекбоксы блоков «СМС»/«Почты» в форме пользователя, §6.1), и право у актора заведомо есть. Во **всех** контролах, показывающих **команды канала** (таблица выше), источник — **`/api/auth/me`**, **без** ветвления по актору.

#### 6.3.1 Исключение «зеркало текущего состояния» (нормативно, редакция 3, 2026-07-14)

**Зачем.** Принцип §6.3 в первой формулировке был **безусловным** («опция не показывается»), и это конфликтовало с правдивостью контрола: контролируемый `<select>`, чей `value` отсутствует в `options`, показывает **первую опцию списка** — то есть **ЧУЖУЮ команду** вместо фактического состояния объекта. Так и вышло в коде: не-админ с `mail_includes_unassigned=true` видит бесхозный ящик (§3) и открывает его на редактирование; селектор «Команда» в этом режиме **`disabled`** целиком (перенос ящика — admin-only, §3), но без опции «Без команды» он **соврал бы** о команде ящика (`frontend/src/components/MailboxFormModal.tsx:94` — `if (seesAllTeams || showCurrentNoTeam)`; `:311` — `teamSelectDisabled = isEdit && !seesAllTeams`; `:366` — `showCurrentNoTeam = teamSelectDisabled && initialTeam === NO_TEAM`; `:840` — `disabled={teamSelectDisabled || noTeamOptions}`). Врать пользователю о данных — не меньший дефект, чем ловушка `403`. Принцип **уточняется**, а не отменяется: запрещено **ПРЕДЛАГАТЬ К ВЫБОРУ** недоступный вариант; **отображать фактическое состояние** — можно.

**Норма.** Опция, которую актор не вправе выбрать, допускается в `options` контрола **только** при выполнении **ВСЕХ ТРЁХ** условий (кумулятивно; ни одно не отбрасывается):

1. **Зеркало, а не вариант.** `option.value === ТЕКУЩЕЕ значение редактируемого объекта` (форма ящика — `mailbox.team_id`; строка номера — `number.team?.id ?? null`). Опция, которой **нет** в текущем состоянии объекта, под исключение **не подпадает никогда**.
2. **Выбор физически не может привести к мутации.** Либо контрол в этом состоянии **`disabled` целиком**, либо обработчик изменения гарантирует **no-op** при `next === current` (ранний выход **без** запроса). Пути «клик → запрос → `403`» существовать не должно.
3. **Без опции контрол соврал бы.** Отсутствие опции приводит к `value ∉ options` ⇒ отображается чужое значение. Опция вводится ради **правдивости отображения**, а не ради выбора.

**Анти-лазейка (обязательно к соблюдению — иначе исключение съест правило).** Режим **`add`** под исключение **не подпадает НИКОГДА**: текущего значения объекта нет ⇒ условие 1 невыполнимо. Именно там жил прод-баг 2026-07-14: опция «Без команды» **предлагалась к выбору** в **enabled**-контроле и гарантированно вела в `403`. Показ недоступной опции как **выбираемой** в **enabled**-контроле — **дефект**, а не применение §6.3.1. Ссылаться на §6.3.1 без явной проверки всех трёх условий запрещено.

**Исчерпывающий перечень мест, где исключение действует сегодня:**

| Контрол | Состояние | Опция-зеркало | Почему безопасно (условие 2) |
|---|---|---|---|
| **Селектор «Команда» в форме ящика** (`MailboxFormModal`), режим **`edit`**, актор — **не-админ** (`sees_all_mail_teams === false`), редактируемый ящик **бесхозный** (`team_id = null`; виден по `mail_includes_unassigned`, §3) | контрол **`disabled` целиком** (перенос ящика — admin-only, §3) | **«Без команды»** | выбрать нельзя (`disabled`) ⇒ пути в `403` нет. Без опции селект показал бы **первую команду** из `me.mail_teams` — чужую |
| **`Select` переноса номера** (`SmsNumberRow`, enabled) | оборонительно: **текущая команда номера отсутствует в `me.sms_teams`** (`frontend/src/components/SmsNumberRow.tsx:89-91`) | текущая команда номера | выбор равен текущему значению ⇒ обработчик выходит **без запроса** (`SmsNumberRow.tsx:95` — `if (nextTeamId === currentTeamId) return;`) |

- Вторая строка — **страховка, а не штатный путь**: по §3/§3.2 видимый и мутируемый номер имеет команду ∈ `SmsScope.team_ids` = `me.sms_teams`, т.е. условие «текущей команды нет в опциях» **недостижимо**. Оборонительный код **допустим** (он же — единственная защита от рассинхрона `/me` и списка номеров), но **не даёт** права добавлять в этот `Select` какие-либо иные опции вне `me.sms_teams`.
- **Опция «Без команды» в `Select` переноса номера исключением НЕ покрывается**: там контрол **enabled** и выбор «Без команды» = **мутация** (снятие команды) ⇒ гейт остаётся строгим — `me.sms_includes_unassigned === true` (§3.2). Бесхозный номер не-админу без флага не виден вовсе ⇒ ситуация «зеркало «Без команды» без права» недостижима.
- **Отклонённая альтернатива (B): рендерить в disabled-случае не `<select>`, а статичный текст «Без команды».** Отклонена: (1) она расщепляет один нормативный контрол формы на два разных по разметке/a11y (лейбл «Команда» + `Select`, [08-design-system.md](../08-design-system.md)) ради одного состояния и всё равно требует новой нормы на текст-заменитель; (2) главное — она **не решает** проблему в общем виде: в `SmsNumberRow` контрол **enabled** (перенос доступен не-админу), текст-заменитель там невозможен, а зеркало текущего состояния всё равно нужно ⇒ понадобилось бы **второе** правило. §6.3.1 даёт **одно** правило на оба случая и сохраняет ловлю исходного прод-бага (условия 1 и 2 его не пропускают).

### 7. Telegram-уведомления (границы решения — нормативно)

Fan-out Telegram-уведомлений о письмах/СМС (`MailDispatcherService`, `mail_telegram_links`/`sms_telegram_links`, [ADR-044](ADR-044-mail-full-merge-into-crm.md) §6, [ADR-030](ADR-030-sms-module-full-merge.md)) **этим ADR НЕ меняется**: получатели по-прежнему определяются **базовым** членством (`user_teams`), доп-команды **не расширяют** рассылку. Владелец просил про **просмотр и работу** в нескольких командах, а не про удвоение потока уведомлений; расширение fan-out на доп-команды — отдельное решение (при появлении требования — новый ADR). **Это осознанная асимметрия**, и она зафиксирована явно, чтобы исполнитель не «дотянул» её по аналогии.

## Последствия

- **Плюс:** сотрудник одной команды обслуживает почты/СМС нескольких команд, при этом каналы независимы; базовое членство остаётся единственным источником «своей» команды и уведомлений.
- **Плюс:** закрыт [TD-058](../100-known-tech-debt.md) — фильтр по командам работает у не-админа (в т.ч. без `teams:view`) и появился в Mini App.
- **Плюс:** закрыт [TD-050](../100-known-tech-debt.md) и прод-баг «не-админ не может создать ящик» — §6.3 (источник команд канала — `/api/auth/me`; опция «Без команды» — только тому, кто вправе её выбрать).
- **Плюс (редакция 3, 2026-07-14):** принцип §6.3 («опция, которую нельзя выбрать, не показывается») **уточнён до проверяемого** (§6.3.1): запрещено **предлагать к выбору**, а **отображать фактическое состояние** в контроле, где выбор невозможен (`disabled`) или является no-op, — **разрешено** при **всех трёх** условиях. Без этого уточнения норма требовала от UI **врать** о команде бесхозного ящика (контролируемый `<select>` с `value ∉ options` показывает чужую команду). Ловля исходного прод-бага **сохранена**: режим `add` и любой **enabled**-контрол под исключение не подпадают.
- **Плюс (редакция 2):** источник опций команд канала на клиенте — **ОДИН** (`/api/auth/me`) для **всех** акторов и **всех** пяти экранов ⇒ ветвления «admin ↔ не-admin» в клиенте нет, `GET /api/teams` для фильтра не нужен **никому**, а запрет «ходить за `GET /api/teams` из Mini App» становится **непротиворечивым** (в первой редакции он вместе с правилом «`sees_all` → `[]`» давал admin-уровню в Mini App **пустой фильтр**).
- **Плюс:** закрыт [TD-060](../100-known-tech-debt.md) — снятие команды с номера гейтится (§3.2), самоблокировка невозможна; заодно закрыта **дыра шире описанной в TD-060**: перенос номера в **чужую** команду (целевая команда не проверялась по scope).
- **Плюс (миграция):** backfill не нужен, регрессии видимости у существующих пользователей нет (§2.4).
- **Минус:** ещё одна ось прав рядом с ролями и командами — конфигурация пользователя усложняется (митигация: блоки свёрнуты, сводка в заголовке, базовая команда неснимаема).
- **Минус:** `MailScope`/`SmsScope` перестают быть чистой проекцией `user_teams` ⇒ любая новая выборка обязана использовать **предикат из §3** (`team_id IN … OR (includes_unassigned AND team_id IS NULL)`), а не `team_id IN scope.team_ids`. Прямое сравнение с `team_ids` в новом коде — дефект.
- **Минус (принят владельцем осознанно, §3.1):** галочка **«Без команды» в блоке «СМС»** открывает пользователю **весь ещё не распределённый поток номеров** Twilio-аккаунта (каждый новый номер из `sync` попадает к нему автоматически, пока не назначена команда) — с **полными** правами, включая **удаление** номера и **перенос** его в свою команду. Асимметрия с почтой (там бесхозные ящики рукотворны и заводит их только админ) — **свойство модели, а не дефект**; митигация — обязательная UI-подсказка объёма (§6.1) и запись риска в [05-security.md](../05-security.md#риск-флага-без-команды-в-смс-нормативно-adr-055-31).
- **Минус (инвариант нормализации — на двух сервисах, а не на одном):** teams-сервис (`POST`/`PATCH /api/teams`) обязан удалять совпадающие добавки при добавлении участника (§2.3, путь 2). Забыть это = вернуть «висящий» доступ после исключения из команды. Проверяется тестом (сценарий «доп-команда → участник → исключён» ⇒ доступа нет).

## Альтернативы (отклонены)

1. **Заменять базовый набор per-channel набором** (scope канала = ровно то, что отмечено в блоке) — отклонено: снятие галочки с базовой команды тихо лишало бы пользователя доступа к почте **своей** команды, а изменение блока «Команды» требовало бы backfill'а обоих каналов, чтобы не рассинхронизироваться.
2. **Колонка `channel` в `user_teams`** — отклонено: базовое членство (лидер, авто-передача лидерства, группировка `/users`, fan-out уведомлений) не имеет канала; расщепление сломало бы `PRIMARY KEY (user_id, team_id)` и семантику «участник команды».
3. **Две таблицы `user_mail_teams` / `user_sms_teams`** — отклонено: две идентичные схемы и два комплекта репозиториев/миграций ради значения, которое отлично выражается колонкой `channel` с CHECK.
4. **Новое право `mail:view_all` / `sms:view_all`** — отклонено (тот же довод, что в [ADR-032](ADR-032-sms-visibility-admin-full-catalog.md)): нужна не «видимость всего», а **точечный список команд**; правом это не выражается.
5. **Read-only доп-команды** — отклонено прямым уточнением владельца (§4).
6. **Строка `team_id IS NULL` в `user_channel_teams` вместо флагов** — отклонено (§2.2): `NULL` в PK недопустим, а `UNIQUE NULLS NOT DISTINCT` + суррогатный PK даёт колонку с двойным смыслом.
7. **«Без команды» = только просмотр (read-only на бесхозные объекты)** — отклонено **прямым решением владельца** (2026-07-14) после разбора асимметрии §3.1: флаг даёт **просмотр И действия** — полные права на бесхозные объекты наравне со своей командой. Митигация объёма — UI-подсказка (§6.1) и запись риска в [05-security.md](../05-security.md), а не урезание прав.
8. **Инвариант нормализации только в users CRUD** (как было в первой редакции §2.3) — отклонено: у `user_teams` **два** пути записи, и teams CRUD (`replace_members`) обходил бы инвариант ⇒ «висящий» доступ к почте/СМС команды после исключения из неё (§2.3, разбор пути 2). Гарантия обязана быть path-independent.
9. **`/me` отдаёт `[]` при `sees_all_*`, а admin-уровню в Mini App разрешается `GET /api/teams`** (вариант «B» при разборе дефекта редакции 1) — отклонено: остаётся **два** источника опций и ветвление «admin ↔ не-admin» в **каждом** из пяти экранов (включая обе Mini App), причём именно в Mini App `GET /api/teams` и хотели запретить. Принятый вариант (§5.1: admin-уровню — полный список в `/me`) убирает второй источник целиком; утечки нет — admin-уровень видит все команды и так.
10. **Оставить перенос номера без проверки целевой команды** («номер и так в scope, куда переносит — его дело») — отклонено (§3.2): это обходной путь **выноса номера в чужую команду**, симметричный тому, ради запрета которого перенос **ящика** оставлен admin-only. Асимметрия «ящик нельзя, номер можно» была бы непреднамеренной.
