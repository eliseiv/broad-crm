# ADR-017 · Страница «Дашборд» (клиентская агрегация) + серверные фильтры «Почт» по ящику/команде

> **⚠️ Частично superseded — [ADR-038](ADR-038-mail-headless-integration.md) (2026-07-09).** Описанное ниже **взаимоисключение** фильтров `mail_account_id` × `group_id` (`400 field=filter`; «выбор одного сбрасывает другой» в UI) **отменено**: фильтры стали **AND-комбинируемыми** (mail-агрегатор ADR-0039 §3 supersede ADR-0037). Актуальная семантика — [04-api.md#mail](../04-api.md#mail), [08-design-system.md](../08-design-system.md#фильтры-ленты-с-тегами--почта--команда), ADR-038 §3. Остальное в этом ADR (дашборд, справочники, read-through) — **см. следующий абзац**.

> ⚠️⚠️ **MAIL-ЧАСТЬ ОТМЕНЕНА ЦЕЛИКОМ — [ADR-044](ADR-044-mail-full-merge-into-crm.md) (2026-07-10).** Фильтры ленты **не проксируются** во внешний API и **не используют `group_id`**; эндпоинта **`GET /api/mail/teams`** и схемы **`MailTeam`** (а также прокси-справочника команд) **НЕ СУЩЕСТВУЕТ**. Действующие фильтры — **`mail_account_id`** (повторяемый) и **`team_id` (UUID CRM-команды)**, применяются **в БД CRM**, AND-комбинируемы, пересекаются со `MailScope.team_ids`. Контракт — [04-api.md §Mail](../04-api.md#mail).

## Контекст

Две связанные потребности продукта:

1. **Обзорная страница «Дашборд».** Нужна стартовая страница-сводка со счётчиками по разделам (сколько почтовых ящиков активно/неактивно, сколько серверов онлайн/офлайн, сколько AI-ключей работает/не работает) и быстрым переходом в раздел кликом. Сейчас дефолтная страница после логина — «Почты» ([ADR-013](ADR-013-mail-newest-first-master-detail-inline-reply.md)); обзора нет.

2. **Серверные фильтры «Почт» по ящику и команде.** Внешний mail-агрегатор финализировал контракт **ADR-0037**, добавивший:
   - `GET /api/external/teams` → `{teams:[{id:int,name:str}]}` (команда = `groups`, `id`+`name`);
   - `GET /api/external/mailboxes` → `{mailboxes:[{id:int,email:str,display_name:str|null,group_id:int|null,is_active:bool}]}` (ящик → команда через `group_id`, `is_active` — статус ящика);
   - в `GET /api/external/messages` — **опциональные взаимоисключающие** параметры `mail_account_id:int(ge=1)` и `group_id:int(ge=1)` (оба одновременно → `400 validation_error`, `field=filter`); несуществующий/чужой/non-canonical `id` → **пустая страница** (не `404`); работают вместе с `order`/`since_id`/`before_id`/`limit`, курсоры не меняются.

   Это позволяет фильтровать ленту «Почт» по конкретному почтовому ящику **или** команде на стороне внешнего сервиса (по всему набору, а не только по загруженному) — то, чего не было при read-through без хранения ([TD-024](../100-known-tech-debt.md)). **Команда ≠ тег:** теги (`tags[]`) остаются как есть, полнотекстовый поиск по-прежнему отсутствует.

Нужно решить: (1) как строится «Дашборд» — вводить ли backend-агрегатор счётчиков; (2) как CRM-прокси отражает новые external-эндпоинты и фильтры; (3) навигацию и дефолтный маршрут.

## Решение

### 1. «Дашборд» — клиентская агрегация, БЕЗ backend-агрегатора

Страница «Дашборд» собирает счётчики **на фронте** из уже существующих list-эндпоинтов; отдельного backend-эндпоинта-агрегатора **не вводится** (NFR-1 — минимум компонентов):

- **«Почты»** → `/mail`: **Активные N / Неактивные M** = число почтовых ящиков (`mailboxes`) с `is_active=true`/`false` — клиентский подсчёт по `GET /api/mail/mailboxes`.
- **«Серверы»** → `/servers`: **online / offline** — клиентский подсчёт по `GET /api/servers` (`server.online`).
- **«ИИ-ключи»** → `/ai-keys`: **активные** (`check_status='working'`) / **неактивные** (`check_status='error'`); опц. `pending` как «проверяется» — по `GET /api/ai-keys`.

Каждый блок — **кликабельная карточка** (`Card interactive`), клик = навигация на страницу соответствующего раздела. Состояния карточки независимы: `loading` (skeleton) / `error` / `empty`. Структура блоков расширяема («и т.д.» — новые разделы добавляются тем же паттерном). Детали — [08-design-system.md · Страница «Дашборд»](../08-design-system.md#страница-дашборд).

**Почему клиентская агрегация, а не backend-агрегатор:** счётчики — производные от списков, которые фронт и так запрашивает на страницах разделов; отдельный агрегатор дублировал бы чтения (в т.ч. read-through к `postapp.store`) и добавил бы компонент без ценности. Клиентский подсчёт по существующим `GET`-эндпоинтам проще и не расширяет backend-контракт.

### 2. CRM-прокси отражает external ADR-0037: teams/mailboxes + фильтры messages

Модуль «Почты» ([ADR-012](ADR-012-mail-read-through-proxy.md)) остаётся **read-through-прокси без хранения**. Добавляются два прокси-эндпоинта и два параметра фильтра — 1:1 к external ADR-0037:

- `GET /api/mail/teams` → `MailTeamsResponse{teams:[MailTeam{id,name}]}` — прокси external `/teams`.
- `GET /api/mail/mailboxes` → `MailMailboxesResponse{mailboxes:[MailMailbox{id,email,display_name,group_id,is_active}]}` — прокси external `/mailboxes`.
- `GET /api/mail/messages` — новые опциональные **взаимоисключающие** параметры `mail_account_id:int|None` и `group_id:int|None` (`ge=1`), пробрасываются во внешний API. Оба одновременно → `400 validation_error` (`field=filter`) локально; внешний `400` также маппится в `400`. Несуществующий/чужой/non-canonical `id` → пустая страница (`200`, не `404`) — CRM проксирует ответ внешнего сервиса как есть.

Схемы `MailTeam`/`MailMailbox` — побуквенно по external DTO. Все эндпоинты — под `CurrentUser` (JWT); `MAIL_API_KEY` — **только** в заголовке `X-API-Key` исходящего запроса; `mail_enabled=false` → `503 mail_not_configured`; недоступность внешнего сервиса → `502 mail_unavailable`. Точные схемы/коды — [04-api.md#mail](../04-api.md#mail).

**Backend-агрегатор дашборда не вводится** (см. §1) — счётчики собирает фронт из `GET /api/mail/mailboxes` (и других list-эндпоинтов) клиентски.

### 3. Серверная фильтрация ленты «Почт» по ящику/команде (UI)

На странице «Почты» рядом с тумблером тегов добавляются **два дропдауна** (`Select`): **«Почта»** (опции из `GET /api/mail/mailboxes`, `value=id`, первая опция «Все почты») и **«Команда»** (опции из `GET /api/mail/teams`, `value=id`, первая опция «Все команды»). Выбор передаётся в `GET /api/mail/messages` (`mail_account_id`/`group_id`); лента **ре-запрашивается** с фильтром (полный набор внешнего сервиса, не только загруженное). Взаимоисключение почта↔команда в UI (выбор одного сбрасывает другой) — консистентно с backend `400`. Клиентский тумблер тегов (переименован **«Только с тегами» → «С тегами»**) остаётся **поверх** серверного набора (теги внешний API не фильтрует). Авто-выбор свежего письма и бесконечная лента при смене фильтра не ломаются. Детали — [08-design-system.md · Страница «Почты»](../08-design-system.md#фильтры-ленты-с-тегами--почта--команда).

**Это частично снимает [TD-024](../100-known-tech-debt.md):** серверная фильтрация по ящику/команде — **доставлена**; полнотекстовый **поиск** — по-прежнему нет (внешний API его не отдаёт).

### 5. Инварианты [ADR-012](ADR-012-mail-read-through-proxy.md) сохранены

Read-through без хранения (нет БД/миграций под teams/mailboxes/фильтры); `MAIL_API_KEY` только в заголовке `X-API-Key` исходящего запроса (не в SPA/логах/URL); `body_html` — только в sandbox-iframe (не относится к дашборду, но mail-страница цела); JWT на всех новых эндпоинтах. Дашборд наружу не ходит — только `/api/*` через `lib/api`.

## Последствия

**Плюсы:**
- Обзорная стартовая страница с быстрым переходом в разделы; счётчики без нового backend-компонента (NFR-1).
- Серверная фильтрация «Почт» по ящику/команде по всему набору (частично снят [TD-024](../100-known-tech-debt.md)).
- Минимальное расширение контракта: два прокси-list-эндпоинта + два параметра фильтра, 1:1 к external ADR-0037; ADR-012 не отменён.

**Минусы / принятый долг:**
- Дашборд-счётчики запрашивают те же списки, что и страницы разделов (нет общего кэша сводки) — приемлемо для админ-панели с одним пользователем; при росте — backend-агрегатор/кэш.
- Полнотекстовый **поиск** «Почт» по-прежнему недоступен — остаток [TD-024](../100-known-tech-debt.md) (серверные фильтры по ящику/команде из него исключены).
- Взаимоисключение почта↔команда: нельзя фильтровать одновременно по ящику И команде (ограничение external ADR-0037) — на Этапе 1 достаточно.

## Альтернативы (отклонены)

- **Backend-агрегатор счётчиков дашборда** (`GET /api/dashboard/summary`) — новый компонент/чтения-дубли (в т.ч. read-through к `postapp.store`) без ценности; против NFR-1. Клиентская агрегация по существующим спискам проще.
- **Клиентская фильтрация «Почт» по ящику/команде** (поверх загруженного набора, как тумблер тегов) — фильтровала бы только загруженное, а не весь набор внешнего сервиса; ADR-0037 даёт серверный фильтр по всей ленте — используем его.
- **Оставить дефолтным маршрутом `/mail`** — обзорная страница должна быть стартовой; «Дашборд» логично первым.
- **Одновременный фильтр по ящику И команде** — не поддержан external ADR-0037 (взаимоисключение → `400`); повторяем контракт в UI (выбор одного сбрасывает другой).
