# ADR-015 · CSP `img-src` — разрешение удалённых (remote https) изображений писем

- Статус: `accepted`
- Дата: 2026-07-06
- Контекст модулей: [mail](../modules/mail/README.md)
- Связанные: [ADR-012](ADR-012-mail-read-through-proxy.md) (read-through-прокси, sandbox-iframe), [ADR-013](ADR-013-mail-newest-first-master-detail-inline-reply.md), [05-security.md](../05-security.md#content-security-policy-spa-location-), [07-deployment.md](../07-deployment.md#reverse-proxy-nginx-требования)

## Контекст

После выката страницы «Почты» ([ADR-012](ADR-012-mail-read-through-proxy.md), [ADR-013](ADR-013-mail-newest-first-master-detail-inline-reply.md)) изображения в HTML-теле писем **не отрисовываются** (иконка «битое изображение»). Причина: CSP SPA (`location /`, nginx) содержит директиву **`img-src 'self' data:`** — она разрешает только изображения со своего origin и `data:`-URI. Подавляющее большинство писем встраивает картинки как **remote https URL** (`<img src="https://…">`), которые этой директивой блокируются.

Тело письма рендерится в **sandbox-iframe** (`srcDoc` + `sandbox` без `allow-scripts`/`allow-same-origin` — [ADR-012](ADR-012-mail-read-through-proxy.md)). Sandbox-iframe **наследует CSP родительского документа** (srcDoc-iframe получает политику встраивающей страницы) и **не может её ослабить** через собственный `<meta http-equiv="Content-Security-Policy">` — meta может только ужесточать, а не расширять унаследованную политику. Следовательно, разблокировать remote-картинки можно **только** правкой CSP самой SPA-страницы.

Нужно решить: расширять ли `img-src`, и если да — насколько, не ослабляя XSS-инвариант [ADR-012](ADR-012-mail-read-through-proxy.md).

## Решение

Расширить `img-src` в CSP SPA до:

```
img-src 'self' data: https:
```

Добавляется **только** источник `https:` (любой origin по HTTPS). **НЕ** добавляются `http:`, `*`, `'unsafe-inline'`/`'unsafe-eval'` и прочие послабления. Изменяется **исключительно** директива `img-src`; остальные директивы CSP (`script-src 'self'`, `frame-ancestors 'none'`, `connect-src 'self'` и т.д.) остаются без изменений.

Нормативное значение CSP (побайтово совпадает в [05-security.md](../05-security.md#content-security-policy-spa-location-), [07-deployment.md](../07-deployment.md#reverse-proxy-nginx-требования) и `frontend/nginx/default.conf`):

```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:; connect-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self'
```

### Сохраняемые инварианты (не ослабляются)

- **Скрипты остаются заблокированы.** Sandbox-iframe без `allow-scripts`/`allow-same-origin` — скрипты письма не исполняются, доступа к origin/куки/DOM/JWT CRM у письма нет. XSS-инвариант [ADR-012](ADR-012-mail-read-through-proxy.md) **не меняется** — расширяется только загрузка **пассивных** изображений (`<img>`), не активного контента.
- **`script-src 'self'` не трогается** — SPA-скрипты по-прежнему только из собранной статики.
- **Referrer не утекает.** На iframe тела письма выставляется `referrerPolicy="no-referrer"` (и глобальный `Referrer-Policy: no-referrer`), поэтому при загрузке remote-картинки отправителю **не** передаётся URL/путь CRM.
- **Ключ почты, JWT, connect-src** — без изменений: фронт наружу ходит только за `/api` (`connect-src 'self'`), картинки грузит браузер напрямую по `<img>` (не fetch/XHR).

## Последствия

**Плюсы:**
- Изображения писем отрисовываются — страница «Почты» работает как ожидается от почтового клиента.
- Минимальная правка: одна директива CSP, без нового кода, без бэкенд-прокси, без зависимостей (NFR-1).
- XSS-инвариант [ADR-012](ADR-012-mail-read-through-proxy.md) полностью сохранён.

**Минусы / принятый компромисс:**
- **Трекинг-пиксели.** Разрешение remote-картинок допускает трекинг-пиксели: отправитель письма может узнать факт и время **открытия** письма (загрузка `1x1`-картинки с уникальным URL). Это **стандартный компромисс всех почтовых клиентов** без «блокировки картинок по умолчанию»; на служебной ленте Этапа 1 приемлемо. Митигация утечки контекста — `referrerPolicy=no-referrer` (URL CRM отправителю не уходит; уходит лишь сам факт запроса картинки).
- **`https:` — широкий источник.** Разрешён любой HTTPS-origin для `<img>`. Сузить до white-list доменов невозможно (картинки писем — с произвольных CDN). Риск ограничен пассивными изображениями (не скрипты/не fetch): утечка — только «письмо открыто».
- **`http:` намеренно не разрешён** — mixed-content и незашифрованный трекинг исключены; картинки только по HTTPS.
- **Остаточное ограничение — `cid:`-изображения** (инлайн-вложения, `<img src="cid:…">`) в read-through **не отрисуются**: внешний DTO вложений не отдаёт ([modules/mail](../modules/mail/README.md#out-of-scope-этап-1)), данных для резолва `cid:` нет. Зафиксировано как [TD-026](../100-known-tech-debt.md).

## Альтернативы (отклонены / отложены)

- **Бэкенд-прокси картинок со срезанием трекинга** (backend скачивает и переотдаёт `<img>` со своего origin, переписывая `src`): убрал бы трекинг-пиксели (отправитель не видит открытие) и оставил бы `img-src 'self'`. **Отклонено/отложено:** существенно сложнее (переписывание HTML тела, стрим-прокси картинок, кэш), вносит **SSRF-риск** (backend ходит по произвольным URL из письма) и egress-нагрузку — против NFR-1 и модели «read-through без хранения» [ADR-012](ADR-012-mail-read-through-proxy.md). Возможный путь при появлении требования приватности/анти-трекинга.
- **`img-src *` / `http:` / `data:` без ограничений** — избыточно; `https:` покрывает картинки писем, `http:` внёс бы mixed-content и незашифрованный трекинг. Отклонено.
- **Оставить `img-src 'self' data:` (не показывать remote-картинки)** — исходное состояние; письма с картинками отображаются «сломанными», ухудшает UX почтового клиента. Отклонено пользователем (баг-репорт).
