# ADR-020 · Страница «Бэки» — реестр сервисов + отдельный in-backend монитор healthcheck по домену + Telegram-алерты

- Статус: accepted
- Дата: 2026-07-07
- **Амендирован:** [ADR-040](ADR-040-backend-relations-secrets-reverse-lookup.md) — бэк перестал быть «без секрета/связей»: добавлены опциональные связи `server_id`/`ai_key_id` (FK, `ON DELETE SET NULL`) и **секреты** `api_key`/`admin_api_key` (Fernet at-rest, reveal под `backends:edit`), миграция `0019`. Пункт 3 контекста ниже («секрета у сущности нет») действует лишь для исходной модели Спринта 2. · [ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md) — страница «Бэки»: клиентский поиск + пустое состояние без текста.

## Контекст

Появляется новая сущность — **бэк** (backend-сервис), который администратор добавляет в CRM, чтобы следить за его доступностью по HTTP-эндпоинту здоровья. Требования:

1. Хранить реестр бэков (**Код** / **Название** / **Домен**) и показывать статус доступности каждого. **Код уникален** — повторное добавление того же кода запрещено (конфликт).
2. Периодически проверять доступность каждого бэка запросом `GET https://{домен}/health` и уведомлять администратора в Telegram, когда бэк **перестал отвечать** (🔴) и когда **снова заработал** (🟢 recovery).
3. Секрета у сущности нет (в отличие от прокси/AI-ключей) — только публичные поля; шифрование не требуется.

Уже существует близкий образец — **монитор доступности прокси** ([ADR-019](ADR-019-proxies-availability-monitor.md), [modules/proxies](../modules/proxies/README.md)), который сам построен по образцу монитора AI-ключей ([ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)): модель со статусом в БД + отдельная фоновая asyncio-задача + собственный `TelegramClient` + прямая сетевая проверка (без `unknown`-исхода). «Бэки» — тот же паттерн, но **проще**: секрета нет (нет Fernet), проверка — прямой `GET https://{домен}/health` (без прокси-туннеля), сущность идентифицируется уникальным `code`.

Серверный нотификатор ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md), [ADR-016](ADR-016-notifier-max-over-window-zone.md), [ADR-018](ADR-018-notifier-windowed-offline-recovery-alert-log.md)) устроен иначе — он оценивает per-metric зоны CPU/RAM/SSD и доступность через **Prometheus** (`up`, `max_over_time`/`min_over_time`), а не прямым HTTP-запросом; его модель состояния бэкам не подходит.

Ограничения проекта: монолит, команда 1–2 человека (NFR-1, [ADR-001](ADR-001-stack-i-monolit.md)); брокеры/новые контейнеры на Этапе 1 не вводятся.

## Решение

1. **Отдельный in-backend asyncio-монитор `BackendMonitorService`** — новая фоновая задача в процессе backend, устроенная по образцу `ProxyMonitorService` ([ADR-019](ADR-019-proxies-availability-monitor.md)) / `AiKeyMonitorService` ([ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)), а **НЕ** расширение серверного нотификатора. Причина: у бэка один атомарный статус (жив/не жив), состояние естественно ложится в БД (`backends.check_status`), проверка — прямой HTTP-запрос к `https://{домен}/health`, а не разбор Prometheus-метрик. Интервал — `BACKEND_CHECK_INTERVAL_SEC` (default 60 с). При создании бэка (`POST /api/backends`) и при изменении домена (`PATCH`) запускается немедленная фоновая проверка (`asyncio.create_task` + сильная ссылка). Спецификация — [modules/backends](../modules/backends/README.md).

2. **Состояние переходов — в БД (`backends.check_status`), а не in-memory.** Как у прокси/AI-ключей, монитор вычисляет переходы `pending|working → error` (алерт 🔴) и `error → working` (recovery 🟢) на основе персистентного `check_status`. Переходы **переживают рестарт backend** — сломанный бэк не «переоткрывается» после рестарта (нет дубль-алерта), recovery корректно отрабатывает между рестартами.

3. **Секрета нет — шифрование не применяется.** Бэк задаётся только публичными полями **Код / Название / Домен**. В отличие от прокси (`password_encrypted`, Fernet) и AI-ключей (`key_encrypted`), у бэка нет секрета: домен и код — не секретны, возвращаются в API как есть. `crypto.py`/`FERNET_KEY` не задействованы.

4. **`code` — уникальный бизнес-идентификатор.** На таблице `backends` — `UNIQUE(code)`. Повтор кода при создании (`POST`) или смене кода на уже занятый (`PATCH`) → **`409 backend_code_taken`** (детерминированный конфликт, как `409 server_conflict` по `ip` серверов). `code` возвращается в API. Прецеденция: схемная валидация Pydantic (`400`/`422`) выполняется **до** проверки уникальности (`409`).

5. **Проверка доступности — `GET https://{домен}/health`, строго 2xx.** Путь `/health` **фиксирован** (не конфигурируется). Домен нормализуется (принять с/без схемы, с завершающим `/` — итог хранится «голым»: `host[:port]`, без схемы и пути); URL проверки собирается как `https://{домен}/health`. `httpx.AsyncClient(timeout=BACKEND_CHECK_TIMEOUT_SEC, verify=True, follow_redirects=False)` → `GET`. Ответ **`2xx`** → `working`; **любой иной исход** (таймаут / сетевая-DNS-TLS-ошибка / не-2xx ответ, включая `3xx`/`4xx`/`5xx`, после ограниченных ретраев) → `error` с русской причиной, содержащей код статуса/причину. Схема фиксирована `https://` (health по TLS) — редиректы не следуются (`3xx` = ошибка здоровья).

6. **Нет исхода `unknown` (как у прокси, осознанное отличие от AI-ключей).** Недоступность бэка **и есть** отслеживаемое событие: провал проверки = бэк сейчас не отвечает → `error` + алерт. Чтобы не флапать на единичном сетевом всплеске, единичная проверка делает **ограниченные ретраи внутри себя** (backoff, образец `app/infra/ai_provider.py` / проверки прокси) и только затем заключает `error`. Компромисс: редкий транзиентный сбой сети backend может дать ложную пару 🔴/🟢 — приемлемо на Этапе 1; при шумности — windowed-детект в будущем ([TD-029](../100-known-tech-debt.md)).

7. **Telegram-отправка опциональна, статус в БД — всегда.** Отправка алертов гейтится тем же `settings.notifier_enabled` (`TELEGRAM_BOT_TOKEN`+`TELEGRAM_CHAT_ID`); монитор использует **собственный экземпляр** `TelegramClient` (как прокси/AI-монитор), передаваемый как `None` при отключённом боте. `check_status` в БД обновляется всегда, и **монитор стартует всегда** — UI-статус бэка работает независимо от бота.

## Обоснование

- **Единый паттерн с прокси/AI-ключами.** Модель со статусом + фоновый монитор + собственный `TelegramClient` + чистая функция перехода `evaluate_transition` уже доказаны ([ADR-019](ADR-019-proxies-availability-monitor.md), [ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)); переиспользование снижает риск и когнитивную нагрузку (NFR-1). «Бэки» — упрощённый вариант прокси-монитора без секрета/туннеля.
- **Персистентный статус переживает рестарт** — нет дубль-алертов, recovery не теряется.
- **Без новой инфраструктуры.** Нет нового контейнера/крона, нет Fernet — соответствует NFR-1.
- **Опциональность Telegram.** UI-функция (список/статусы бэков) не зависит от наличия бота.
- **`code` UNIQUE** даёт детерминированный `409` и однозначную идентификацию сервиса.

## Последствия

- (+) Переиспользован проверенный паттерн прокси/AI-ключей; переходы статуса переживают рестарт; UI-статусы работают без Telegram; точный контроль формата алертов.
- (+) Нет секрета → нет Fernet/маскирования → модуль проще прокси.
- (−) Один процесс — нет горизонтального масштабирования (то же ограничение, что у прокси/нотификатора/AI-монитора).
- (−) Фиксированный интервал проверки и фиксированный путь `/health`; нет ручного «проверить сейчас» и per-backend health-path ([TD-029](../100-known-tech-debt.md)).
- (−) Нет windowed-детекта — редкий транзиентный сетевой сбой backend может дать ложную пару 🔴/🟢 (смягчено ретраями внутри проверки); при шумности — будущий windowed-детект ([TD-029](../100-known-tech-debt.md)).
- (−) Best-effort доставка Telegram (at-least-once, наследуется от `TelegramClient`).

## Альтернативы

- **Расширить серверный нотификатор** — отвергнуто: он завязан на Prometheus (`up`, зоны, `max_over_time`/`min_over_time`); модель состояния (per-metric зоны + online на сервер) бэкам не подходит; бэк проверяется прямым HTTP-запросом к `/health`, а не метрикой.
- **Расширить `proxies` (общая таблица «endpoints»)** — отвергнуто: у прокси есть тип/секрет/логин и проверка через туннель; у бэка — уникальный `code` и проверка по фиксированному `/health`. Смешение усложнило бы обе модели; отдельная таблица чище.
- **Конфигурируемый путь/схема healthcheck** — отвергнуто на Этапе 1: фиксированный `https://{домен}/health` покрывает требование; per-backend path — [TD-029](../100-known-tech-debt.md).
- **Исход `unknown` как у AI-ключей** — отвергнуто (как для прокси): недоступность бэка и есть отслеживаемое событие; транзиентность гасится ретраями внутри проверки.
- **Не требовать уникальности `code`** — отвергнуто: код — бизнес-идентификатор сервиса, дубликаты порождают неоднозначность в алертах и UI.

## Будущее
