# ADR-019 · Страница «Прокси» — реестр HTTP/SOCKS-прокси + отдельный in-backend монитор доступности + Fernet для пароля

- Статус: accepted
- Дата: 2026-07-07

> **Амендмент [ADR-027](ADR-027-proxies-alert-grace.md) (2026-07-08).** Модель алерта недоступности прокси изменена с **немедленной** на **grace-порог 30 мин** (по образцу бэков [ADR-024](ADR-024-monitor-hard-deadline-backend-alert-grace.md)): `check_status→error` — сразу, но 🔴 — только после непрерывной недоступности ≥ `PROXY_ALERT_AFTER_SEC=1800`; поля `error_since`/`alert_sent` (миграция `0014`), time-aware `evaluate_transition`. Устраняет ложные срабатывания. Тело этого ADR ниже (описывающее немедленный алерт) сохранено как есть; актуальная модель алерта — [ADR-027](ADR-027-proxies-alert-grace.md).

## Контекст

Появляется новая сущность — **прокси** (HTTP/HTTPS/SOCKS5), которые администратор добавляет в CRM. Требования:

1. Хранить реестр прокси (название, тип, хост, порт, опциональные логин/пароль) и показывать статус доступности каждого.
2. Пароль прокси хранить безопасно (не в открытом виде), но **обратимо** — он нужен целиком, чтобы собрать URL и выполнить проверочный запрос через прокси.
3. Периодически проверять доступность прокси и уведомлять администратора в Telegram, когда прокси **перестал работать** и когда **снова заработал** (recovery). Никогда не возвращать пароль в API-ответах/логах.

Уже существует близкий образец — **монитор AI-ключей** ([ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md), [modules/ai-keys](../modules/ai-keys/README.md)): модель со статусом в БД + отдельная фоновая asyncio-задача + собственный `TelegramClient` + Fernet-шифрование секрета. Серверный нотификатор ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md), [ADR-016](ADR-016-notifier-max-over-window-zone.md), [ADR-018](ADR-018-notifier-windowed-offline-recovery-alert-log.md)) устроен иначе — он оценивает per-metric зоны CPU/RAM/SSD и доступность через **Prometheus** (`up`, `max_over_time`/`min_over_time`), а не через прямой сетевой запрос, и его модель состояния (три зоны + online на сервер) прокси не подходит.

Ограничения проекта: монолит, команда 1–2 человека (NFR-1, [ADR-001](ADR-001-stack-i-monolit.md)); брокеры/новые контейнеры на Этапе 1 не вводятся; Fernet-инфраструктура (`FERNET_KEY`, `app/infra/crypto.py`) уже есть.

## Решение

1. **Отдельный in-backend asyncio-монитор `ProxyMonitorService`** — новая фоновая задача в процессе backend, устроенная по образцу `AiKeyMonitorService` ([ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)), а **НЕ** расширение серверного нотификатора. Причина: у прокси один атомарный статус (жив/не жив), состояние естественно ложится в БД (`proxies.check_status`), проверка — прямой HTTP-запрос **через прокси** к эталонному URL, а не разбор Prometheus-метрик. Интервал — `PROXY_CHECK_INTERVAL_SEC` (default 60 с). При создании прокси (`POST /api/proxies`) запускается немедленная фоновая проверка (`asyncio.create_task` + сильная ссылка). Спецификация — [modules/proxies](../modules/proxies/README.md).

2. **Состояние переходов — в БД (`proxies.check_status`), а не in-memory.** Как у AI-ключей, монитор вычисляет переходы `pending|working → error` (алерт 🔴) и `error → working` (recovery 🟢) на основе персистентного `check_status`. Переходы **переживают рестарт backend** — сломанный прокси не «переоткрывается» после рестарта (нет дубль-алерта), recovery корректно отрабатывает между рестартами.

3. **Fernet-шифрование пароля тем же `FERNET_KEY`.** Пароль прокси шифруется при создании и хранится в БД как `proxies.password_encrypted bytea` (`NULL`, если пароль не задан). Переиспользуются `encrypt_secret`/`decrypt_secret` из `app/infra/crypto.py` ([ADR-007](ADR-007-shifrovanie-fernet.md)). Пароль **обратимо** шифруется, т.к. нужен целиком для сборки URL проверки; расшифровка — только в памяти монитора непосредственно перед запросом. В API пароль **никогда** не возвращается — вместо него производный флаг `has_password: bool` (= `password_encrypted IS NOT NULL`). Логин (`username`) — **не секрет**, возвращается как есть.

4. **Отдельные поля ввода, а не единая URL-строка.** Прокси задаётся полями **Название / Тип (`http`\|`https`\|`socks5`) / Хост / Порт / Логин (опц.) / Пароль (опц.)**, а не строкой `scheme://user:pass@host:port`. Причина: (а) валидация каждого поля (тип из enum, порт `1..65535`, длины); (б) редактирование без ре-парсинга; (в) маскирование пароля (флаг `has_password`) без риска утечки секрета в общей строке; (г) сборка URL проверки — детерминированная, секрет не хранится в открытой строке. URL `scheme://[user:pass@]host:port` собирается **только** в памяти монитора перед запросом.

5. **Проверка доступности — через эталонный URL.** `httpx.AsyncClient(proxy="<scheme>://[user:pass@]host:port", timeout=PROXY_CHECK_TIMEOUT_SEC)` → `GET` эталонного `PROXY_CHECK_URL` (default `https://www.gstatic.com/generate_204`). Ответ `2xx`/`3xx` (в т.ч. `204`) → `working`; таймаут / сетевая ошибка / прочий исход (после ограниченных ретраев) → `error` с русской причиной. Эталонный URL выбран лёгким (204 No Content, без тела, высокодоступный) — надёжная проба связности через прокси, не завязанная на конкретный внешний сервис.

6. **Нет исхода `unknown` (осознанное отличие от AI-ключей).** У AI-ключей транзиентная недоступность провайдера → `unknown` (статус не флипается, алерт не шлётся), потому что «сеть провайдера легла» ≠ «ключ отозван». У прокси **недоступность и есть отслеживаемое событие**: провал проверки = прокси сейчас не работает → `error` + алерт. Чтобы не флапать на единичном сетевом всплеске, единичная проверка делает **ограниченные ретраи внутри себя** (backoff, образец `app/infra/ai_provider.py`) и только затем заключает `error`. Компромисс: редкий транзиентный сбой сети backend может дать ложную пару 🔴/🟢 — приемлемо на Этапе 1; при шумности — windowed-детект в будущем ([TD-028](../100-known-tech-debt.md)).

7. **Telegram-отправка опциональна, статус в БД — всегда.** Отправка алертов гейтится тем же `settings.notifier_enabled` (`TELEGRAM_BOT_TOKEN`+`TELEGRAM_CHAT_ID`); монитор использует **собственный экземпляр** `TelegramClient` (как AI-монитор), передаваемый как `None` при отключённом боте. `check_status` в БД обновляется всегда, и **монитор стартует всегда** — UI-статус прокси работает независимо от бота.

8. **SOCKS требует доп. зависимости.** `httpx` проксирует HTTP/HTTPS из коробки, но для `socks5://` нужен экстра `httpx[socks]` (транзитивно `socksio`). На момент решения `backend/pyproject.toml` объявляет `httpx>=0.27,<0.28` **без** экстры socks — её нужно добавить (`httpx[socks]`), иначе проверка SOCKS5-прокси падает. Пометка для backend/devops — [modules/proxies](../modules/proxies/README.md#backend--ориентиры-реализации-структура--на-усмотрение).

## Обоснование

- **Единый паттерн с AI-ключами.** Модель со статусом + фоновый монитор + собственный `TelegramClient` + Fernet уже доказаны на AI-ключах ([ADR-010](ADR-010-ai-key-monitor-vnutri-backend.md)); переиспользование снижает риск и когнитивную нагрузку (NFR-1).
- **Персистентный статус переживает рестарт** — как у AI-ключей: нет дубль-алертов, recovery не теряется.
- **Единая инфраструктура шифрования.** `FERNET_KEY`/`crypto.py` уже есть; отдельный ключ/схема усложнили бы управление секретами без выгоды.
- **Без новой инфраструктуры.** Нет нового контейнера/крона — соответствует NFR-1, как решения по провижинингу/нотификатору/AI-ключам.
- **Отдельные поля безопаснее URL-строки** — пароль не растворяется в общей строке, маскируется флагом, каждое поле валидируется.
- **Опциональность Telegram.** UI-функция (список/статусы прокси) не зависит от наличия бота.

## Последствия

- (+) Переиспользован проверенный паттерн AI-ключей; переходы статуса переживают рестарт; UI-статусы работают без Telegram; точный контроль формата алертов.
- (+) Пароль зашифрован at-rest, никогда не в ответах/логах; логин виден (не секрет) для UX.
- (−) Один процесс — нет горизонтального масштабирования (то же ограничение, что у провижининга/нотификатора/AI-монитора).
- (−) Фиксированный интервал проверки; нет ручного «проверить сейчас» ([TD-028](../100-known-tech-debt.md)).
- (−) Нет windowed-детекта — редкий транзиентный сетевой сбой backend может дать ложную пару 🔴/🟢 (смягчено ретраями внутри проверки); при шумности — будущий windowed-детект ([TD-028](../100-known-tech-debt.md)).
- (−) Best-effort доставка Telegram (at-least-once, наследуется от `TelegramClient`).
- (−) Добавляется рантайм-зависимость `httpx[socks]`/`socksio` ради SOCKS5.

## Альтернативы

- **Расширить серверный нотификатор** — отвергнуто: он завязан на Prometheus (`up`, зоны CPU/RAM/SSD, `max_over_time`/`min_over_time`), его модель состояния (per-metric зоны + online на сервер) прокси не подходит; прокси проверяется прямым HTTP-запросом через прокси, а не метрикой. Смешение двух разнородных источников усложнило бы нотификатор.
- **Единое поле URL (`scheme://user:pass@host:port`)** — отвергнуто: сложнее валидировать/редактировать/маскировать; пароль в общей строке легче утекает в лог/ответ.
- **Хранить пароль в открытом виде** — отвергнуто: секрет at-rest, тот же класс, что SSH-пароли/AI-ключи; Fernet ([ADR-007](ADR-007-shifrovanie-fernet.md)) уже покрывает обратимое шифрование.
- **Поддержать только прокси без авторизации** — отвергнуто: многие прокси требуют логин/пароль; ограничение сделало бы фичу неполной.
- **Исход `unknown` как у AI-ключей** — отвергнуто для прокси: недоступность прокси и есть отслеживаемое событие (в отличие от «сеть провайдера легла ≠ ключ отозван»); транзиентность гасится ретраями внутри проверки.

## Будущее

- Ручной триггер «проверить сейчас», настраиваемый интервал per-proxy ([TD-028](../100-known-tech-debt.md)).
- Windowed-детект доступности (по образцу [ADR-018](ADR-018-notifier-windowed-offline-recovery-alert-log.md)) при шумных ложных срабатываниях.
- Карточка «Прокси» на «Дашборде» (сводный счётчик Работает/Не работает) — по образцу остальных разделов ([ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md)).
- Расширение типов (SOCKS4, per-proxy health-URL) — добавлением значений в enum/конфиг.
