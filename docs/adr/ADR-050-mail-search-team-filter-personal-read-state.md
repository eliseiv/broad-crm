## Контекст

Требования владельца (батч 2026-07-13):

1. **`/mail`, вкладка «Почты»:** «добавить поле поиска рядом с кнопками Все/Активные/Неактивные. Поиск ищет по номеру, приложению или самой почте. Также добавить сортировку по командам как на вкладке Сообщения» (= **такой же селектор фильтра по команде**, как на вкладке «Сообщения»).
2. **`/mail`, вкладка «Сообщения»:** прочитанные / непрочитанные письма. Решения владельца, **обязательные**:
   - прочитанность **ЛИЧНАЯ у каждого пользователя** (не общая на команду);
   - письмо помечается прочитанным **ПРИ ОТКРЫТИИ** — и в веб-UI, и в Telegram Mini App `/tg/mail`;
   - отдельный вопрос владельца: «сможем ли пометить прочитанным при открытии из Mini App?» — ответ фиксируется нормативно (§2.6).

**Что уже есть в коде (сверено, `claims-from-code`):**

- **Список ящиков грузится ЦЕЛИКОМ, без пагинации.** `GET /api/mail/mailboxes` → `MailMailboxesResponse { mailboxes: list[MailMailbox] }` (`backend/app/schemas/mail.py:127`) — ни курсора, ни `limit`. Единственный query-параметр — `is_active` (`IsActive = Annotated[bool | None, Query()]`, `backend/app/api/mail.py:56`). Фронт: `useMailboxesManage(isActive)` (`frontend/src/features/mail/hooks.ts`). ⇒ **клиентская фильтрация здесь корректна и достаточна.**
- **Фильтр активности** — не общий примитив, а три `<button>` в `div role="group" aria-label="Фильтр активности"` (`frontend/src/components/MailboxesTab.tsx`), лейблы `Все` / `Активные` / `Неактивные`; поиска на вкладке **нет**.
- **Team-фильтр вкладки «Сообщения»** рендерится под условием `{seesAllMailTeams && (<Select aria-label="Команда" …/>)}` (`frontend/src/pages/MailPage.tsx:329`), где `seesAllMailTeams = useSeesAllMailTeams()` ← `me.sees_all_mail_teams` ← backend-предикат `principal_sees_all_mail_teams` = `principal.is_superadmin or permissions_subset(full_catalog_permissions(), principal.permissions)` (`backend/app/api/deps.py:247-254`). **Словарь команд для не-admin даже не запрашивается:** `useTeams(seesAllMailTeams)` (`frontend/src/pages/MailPage.tsx:172`) — источник опций `GET /api/teams` гейтится `require("teams","view")` (`backend/app/api/teams.py:30`), которого у mail-оператора может не быть.
- **Лента — одна схема на список И на деталь.** `MailMessage` (`backend/app/schemas/mail.py:46-62`) уже несёт **полное тело** (`body_text`/`body_html`/`body_present`/`body_truncated`). **Отдельного `GET /api/mail/messages/{id}` НЕ СУЩЕСТВУЕТ** (`backend/app/api/mail.py` — эндпоинта нет). «Открытие письма» — **чисто клиентский выбор** уже загруженного объекта: веб — `handleSelect(id)` → `setSelectedId` (`frontend/src/pages/MailPage.tsx:233-236`) → `<MailDetail message={selected}/>`; Mini App — `MailMiniAppCard onOpen` → `setSelected` → `MailMiniAppDetail` (`frontend/src/pages/MailMiniAppPage.tsx`). ⇒ **пометка «прочитано при открытии» физически требует НОВОГО явного эндпоинта** — «на чтении детали» её повесить не на что.
- **Лента — компаундный keyset.** `MailMessageRepository.list_feed` (`backend/app/repositories/mail_message_repository.py:63`): `ORDER BY internal_date DESC, id DESC`, предикат курсора `(internal_date, id) < (d0, id0)`, `limit+1`. Курсор — opaque (`encode_mail_cursor`/`decode_mail_cursor`, `backend/app/domain/mail.py`).
- **Mini App несёт ОБЫЧНЫЙ CRM access-JWT, а не особый «SSO-принципал».** `MailTelegramService.auth` резолвит строку `users` по Telegram-id и выпускает токен: `issue_access_token(sub=resolved.username, role=resolved.role.name, superadmin=False, uid=str(resolved.id))` (`backend/app/services/mail_telegram_service.py:140-143`); фронт шлёт его `Bearer`-заголовком в **те же** эндпоинты (`listMail(..., getMailMiniAppToken())`, `frontend/src/features/mail/miniAppHooks.ts`). ⇒ в Mini App `principal.user_id` **всегда непуст**.
- **Супер-админ из `.env` НЕ ИМЕЕТ строки в `users`** (факт на 2026-07-13, **устарел** — [ADR-051](ADR-051-superadmin-db-anchor-personal-state.md) завёл ему системную строку-якорь): `Principal.user_id` — «`uid` из claim… ТОЛЬКО у БД-пользователя; супер-админ → `None`» (`backend/app/api/deps.py:83-85`). Прецедент обработки на тот момент: `GET`/`PATCH /api/mail/me/settings` → `if principal.user_id is None: raise forbidden()` (`backend/app/api/mail_me.py:31-33` и `:47-49`), т.е. **`403`**.
- **Прочитанности сейчас НЕТ нигде** — ни колонки, ни таблицы, ни поля контракта (grep `unread|is_read|read_at|seen` по mail-коду — пусто). Переиспользовать нечего.
- **`mail_messages.id` — `BigInteger`** (`backend/app/models/mail_message.py:45`); `users.id` — `uuid` (`backend/app/models/user.py:49`).

## Решение

### §1. Фикс 1 — вкладка «Почты»: клиентский поиск + клиентский фильтр по команде

#### §1.1. Поиск — КЛИЕНТСКИЙ (нормативно)

- **Клиентский**, над уже загруженным `GET /api/mail/mailboxes`. Обоснование: список ящиков возвращается **целиком** (пагинации в контракте нет; на проде 121 ящик) — ровно тот же случай, что поиск по бэкам ([ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md)) и поиск по номерам на `/sms`. **Серверный поиск и новый query-параметр НЕ вводятся.**
- **Поля поиска (нормативно, ровно три):** **`number`** («Номер»), **`app_name`** («Приложение»), **`email`** (сам адрес почты). Совпадение — **подстрока, регистронезависимо**, по `trim()`-нутому запросу; пустой запрос = фильтр не применяется. Поля `display_name`/`team_id`/статус в поиск **не входят** (`display_name` — производная склейка `number`+`app_name` ([ADR-047](ADR-047-mail-fix-pack.md) §3.3): включать её значит искать по тем же данным дважды).
- **Контрол — существующий `ui/Input`** с `trailing={<Search/>}` (lucide), по образцу `/backends` (`frontend/src/pages/BackendsPage.tsx`). **Новый примитив не вводится.**
- **Нормативные строки:** плейсхолдер **«Поиск по почтам…»** (символ `…` — один символ, не три точки), `aria-label` **«Поиск по почтам»**. Образец — «Поиск по бэкам…» / «Поиск по номеру…» ([08-design-system.md](../08-design-system.md#локализация-страницы-почты)).
- **Расположение:** в одной строке-тулбаре **рядом** с сегментом «Все / Активные / Неактивные» (требование владельца «рядом с кнопками»). При нехватке ширины контролы **переносятся** (`flex-wrap`), значимый текст не обрезается (правило [CLAUDE.md](../../CLAUDE.md)).
- **Пустой результат:** активный непустой запрос без совпадений → **«Ничего не найдено»** (уже существующая нормативная строка, [08 §Локализация «Бэки»](../08-design-system.md#локализация-страницы-бэки)). Это **отличается** от «Почт пока нет» (пустой каталог) — не путать.

#### §1.2. Фильтр по команде — КЛИЕНТСКИЙ, гейт `sees_all_mail_teams` (нормативно)

- **Контрол — `ui/Select`**, `aria-label` **«Команда»**, первая опция **«Все команды»** (сброс). Опции — из **`GET /api/teams`** (`value = team.id`, UUID; лейбл — `name`) — **тот же источник, что у team-фильтра вкладки «Сообщения»** (`frontend/src/pages/MailPage.tsx`). Дополнительная последняя опция — **«Без команды»** (ящики с `team_id = null`, они существуют — `MailMailbox.team_id: uuid | null`); без неё unassigned-ящики нельзя было бы отфильтровать вовсе.
- **Условие рендера — ТО ЖЕ, что на вкладке «Сообщения»: `me.sees_all_mail_teams === true`.** Для прочих ролей селектор **не рендерится вовсе** (не пустой, не disabled — отсутствует). Решение (владелец просил «такой же селектор, как на вкладке Сообщения» — включая его гейт):
  - **(а) Источник опций гейтится ЧУЖИМ правом.** Опции берутся из `GET /api/teams` под `require("teams","view")` (`backend/app/api/teams.py:30`), которого у mail-оператора может не быть → у не-admin дропдаун рендерился бы **пустым** — ровно тот «мусорный контрол», который [ADR-036](ADR-036-sms-team-filter-admin-only.md) и упразднил. В коде фронта это уже закреплено: `useTeams(seesAllMailTeams)` — словарь команд для не-admin **не запрашивается**.
  - **(б) Единая норма.** Ввести на вкладке «Почты» правило, отличное от вкладки «Сообщения» и от `/sms`, значит расщепить одну норму на три — при том, что предикат один (`sees_all_mail_teams`).
  - **(в) Scope и так сужает.** Не-admin видит только ящики своих команд (`MailScope`, [ADR-044](ADR-044-mail-full-merge-into-crm.md) §7) — как и на «Сообщениях».
  - **Остаточное ограничение (принимается):** не-admin, состоящий в **нескольких** командах, не получает быстрого фильтра по команде на этой вкладке. Смягчение: колонка **«Команда»** в таблице ящиков остаётся (значение читается полностью, [ADR-047](ADR-047-mail-fix-pack.md) §4), и поиск (§1.1) покрывает основной сценарий. Долг — [TD-058](../100-known-tech-debt.md).
- **Фильтрация — КЛИЕНТСКАЯ, по `mailbox.team_id`** (сравнение с выбранным UUID; «Без команды» → `team_id === null`). **Серверный параметр `team_id` у `GET /api/mail/mailboxes` НЕ вводится** — список уже загружен целиком (в контракте эндпоинта есть только `is_active`, `backend/app/api/mail.py:56`).
  > **⚠️ Это осознанное отличие от вкладки «Сообщения», где team-фильтр СЕРВЕРНЫЙ.** Причина — в природе данных, а не в непоследовательности: лента писем **курсорная** (загружена частично) — там клиентский фильтр над загруженным набором сломал бы догрузку (известный дефект тумблера «С тегами», [08 §Тумблер «С тегами»](../08-design-system.md#тумблер-с-тегами-клиентский-переименован)); каталог ящиков загружен **целиком** — клиентский фильтр здесь полон и точен. **Backend по этому фиксу НЕ МЕНЯЕТСЯ ВООБЩЕ.**

#### §1.3. Порядок применения фильтров (нормативно)

`GET /api/mail/mailboxes?is_active=…` (**серверный** сегмент «Все/Активные/Неактивные» — как сейчас) → **клиентский поиск** (§1.1) → **клиентский фильтр по команде** (§1.2) → рендер строк. Все три комбинируются (AND); ни один не сбрасывает другой.

### §2. Фикс 2 — ЛИЧНАЯ прочитанность писем

#### §2.1. Данные: отдельная таблица связи `mail_message_reads` (миграция `0025`)

**Решение — таблица связи «пользователь × письмо», а не колонка в `mail_messages`.** Прочитанность **личная** (требование владельца): одно письмо видят все члены команды, и у каждого своё состояние ⇒ признак принадлежит **паре** (пользователь, письмо), а не письму. Колонка `mail_messages.is_read` физически не может выразить личное состояние.

| Колонка | Тип | Ограничения | Смысл |
|---------|-----|-------------|-------|
| `user_id` | `uuid` | `NOT NULL`, FK → `users(id)` **`ON DELETE CASCADE`** | Пользователь CRM. Удалили пользователя — его отметки уходят |
| `message_id` | `bigint` | `NOT NULL`, FK → `mail_messages(id)` **`ON DELETE CASCADE`** | Письмо (`mail_messages.id` — `BigInteger`, `backend/app/models/mail_message.py:45`). Удалили письмо/ящик — отметки уходят |
| `read_at` | `timestamptz` | `NOT NULL`, `server_default = now()` | Когда пользователь открыл письмо (диагностика/аналитика; в контракт **не отдаётся**) |

- **PK — составной `(user_id, message_id)`** (имя `pk_mail_message_reads`). Он же — **рабочий индекс** обоих горячих путей: батч-лукап «какие из письмa страницы прочитаны мной» и анти-джойн фильтра `unread=true`. Отдельный индекс по `(user_id)` **не нужен** (PK ведёт с `user_id`).
- **Индекс `ix_mail_message_reads_message_id (message_id)` — ОБЯЗАТЕЛЕН.** Причина конкретная: PK ведёт с `user_id`, поэтому поиск по `message_id` его **не использует**, а `ON DELETE CASCADE` со стороны `mail_messages` при удалении ящика (`mail_accounts` → CASCADE → `mail_messages` → CASCADE → `mail_message_reads`, [ADR-044](ADR-044-mail-full-merge-into-crm.md)) выполнял бы **seq scan** по всей таблице отметок на каждое письмо. Без этого индекса удаление ящика со 100k писем становится неприемлемо долгим.
- **Существование строки = «прочитано».** Отсутствие = «не прочитано». Отдельного булева поля нет (оно было бы избыточным).
- **Идентификатор миграции:** `revision = "0025_mail_message_reads"` — **23 символа** ≤ **32** (жёсткий предел `alembic_version.version_num VARCHAR(32)`, норматив [ADR-047](ADR-047-mail-fix-pack.md) §3.5). `down_revision = "0024_mail_accounts_num_app_name"` (31 символ) — фактическая голова цепочки (`backend/alembic/versions/0024_mail_accounts_number_app_name.py:37`). `downgrade()` — рабочий (`drop_table`).
- **Backfill не требуется:** пустая таблица = все письма непрочитаны для всех. Это корректное начальное состояние.

#### §2.2. Контракт: `MailMessage += is_unread`, параметр `unread`, два эндпоинта отметки

Точные схемы — [04-api.md §Mail](../04-api.md#mail). Нормативно:

- **`MailMessage` (схема ленты И детали — она одна) `+= is_unread: boolean`** (не nullable). `true` ⇔ для **текущего принципала** строки `mail_message_reads(user_id=<принципал>, message_id=<письмо>)` **нет**. Поле — **производное и персональное**: один и тот же `id` письма у разных пользователей даёт разные `is_unread`. `read_at` наружу **не отдаётся** (не нужен UI, а «когда коллега прочитал» — лишнее раскрытие).
  > **Почему `is_unread`, а не `is_read`:** имя поля совпадает с тем, что показывает UI (индикатор «непрочитано», фильтр «Непрочитанные») и с именем query-параметра `unread` — одно понятие, одно слово, без инверсий на границах.
- **`GET /api/mail/messages` `+= unread: boolean?`** — серверный фильтр. `unread=true` → только непрочитанные **текущим принципалом**. Отсутствие параметра / `unread=false` → фильтр не применяется (`false` **не означает** «только прочитанные» — такого сценария владелец не просил, и семантика «фильтр включён/выключен» проще). AND-комбинируется с `mail_account_id`/`team_id` и пересекается с `MailScope`.
- **`POST /api/mail/messages/{message_id}/read` → `204 No Content`.** Пометить прочитанным. **Идемпотентно** (`INSERT … ON CONFLICT (user_id, message_id) DO NOTHING`): повторный вызов — тоже `204`, `read_at` **не обновляется** (первое открытие важнее последнего). Тела запроса и ответа нет.
- **`DELETE /api/mail/messages/{message_id}/read` → `204 No Content`.** Вернуть в «непрочитано». **Идемпотентно**: строки не было — всё равно `204`.
- **Новых кодов ошибок НЕТ.** Оба эндпоинта: `401` (нет JWT), `403 forbidden` (нет `mail:view`; **супер-админ из `.env` — НЕ исключение, у него отметка работает**, [ADR-051](ADR-051-superadmin-db-anchor-personal-state.md); прежнее исключение §2.5 отменено), **`404 mail_message_not_found`** — письмо не существует **или** вне `MailScope` (анти-энумерация; **тот же код и та же семантика, что у reply**, [04-api.md](../04-api.md#mail)). Чужое письмо **неотличимо** от несуществующего.
- **⚠️ Гонка «письмо удалено между scope-проверкой и записью» (нормативно).** Между проверкой видимости письма и `INSERT … ON CONFLICT` письмо может быть удалено (`DELETE` ящика → CASCADE `mail_messages` → CASCADE `mail_message_reads`, [ADR-044](ADR-044-mail-full-merge-into-crm.md)). Тогда `INSERT` нарушит FK `message_id` → драйвер поднимет `IntegrityError`, и без обработки клиент получил бы **`500`** вместо контрактного `404`. **Норма:** `IntegrityError` по FK **`message_id`** в `POST …/read` **транслируется в `404 mail_message_not_found`** (тот же код, что и «письма нет» — семантически это ровно оно). **Новых кодов ошибок не вводить.** Нарушение FK по `user_id` в этот `404` **НЕ транслируется** (пользователь удалён посреди собственного запроса — не «письмо не найдено»; это штатная `500`-ситуация, маскировать её нельзя). У `DELETE …/read` гонки нет: удаление несуществующей строки — но-оп, `204` (идемпотентность §2.2).

#### §2.3. RBAC: гейт — `mail:view` (нормативно, нового права НЕ вводится)

- Оба эндпоинта отметки гейтятся **`require("mail","view")`**. Обоснование: прочитанность — **личный артефакт ЧТЕНИЯ**, а не мутация домена: она не меняет ни письмо, ни ящик, ни то, что видят другие. Кто вправе **прочитать** письмо, тот вправе и отметить **своё** прочтение.
- **Прецедент в этом же модуле:** «`reply` — под `view`» ([04-api.md §Mail RBAC](../04-api.md#mail)) — действие, которое даже **отправляет письмо наружу**, уже живёт под `view`. Требовать `mail:edit` для отметки собственного прочтения было бы строже, чем для ответа адресату.
- **`CATALOG["mail"]` не меняется** (`("view","create","edit","delete","sync","tags")`, `backend/app/domain/permissions.py`). Ни нового действия, ни миграции прав.
- **Scope — существующий `MailScope`** ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §7), без изменений: письмо обязано принадлежать ящику с `team_id ∈ MailScope.team_ids` (для не-admin), иначе `404`. **Отметить чужое письмо нельзя** — и по этому факту нельзя узнать о его существовании.

#### §2.4. Производительность ленты (нормативно — обязательные требования к реализации)

Лента — горячий путь (бесконечный скролл + догрузка при каждом раскрытии страницы). Ниже — **нормы**, а не рекомендации.

> **⚠️ Факт по коду (`claims-from-code`): ЛЕНТА НЕ АВТОПОЛЛИТСЯ.** `useMailFeed` — `useInfiniteQuery` **без `refetchInterval`** и с **`refetchOnWindowFocus: false`** (`frontend/src/features/mail/hooks.ts` → `useMailFeed`). Периодический `refetchInterval: env.pollIntervalMs` стоит **только у каталога ящиков** (`useMailMailboxes`, `useMailboxesManage`, там же), т.е. поллится **вкладка «Почты»**, а **не** лента писем. Фоновый poll новых писем в ленте — **не реализован** ([TD-025](../100-known-tech-debt.md)). Это **предпосылка нормы §2.8** — см. врезку там.

- **`is_unread` вычисляется ВТОРЫМ, БАТЧЕВЫМ запросом по уже отобранной странице — НЕ JOIN'ом в keyset-запрос.**
  `MailMessageRepository.list_feed` (`backend/app/repositories/mail_message_repository.py:63`) — **не трогать**: его `ORDER BY internal_date DESC, id DESC` + предикат курсора обслуживаются индексом ленты, и подмешивание `LEFT JOIN` ради булева флага ломает план без нужды. После отбора страницы (≤ `limit` строк) выполняется **один** запрос:
  `SELECT message_id FROM mail_message_reads WHERE user_id = :uid AND message_id = ANY(:page_ids)` → множество прочитанных; `is_unread = (id NOT IN <множество>)`.
  Это **один дополнительный запрос на батч** (≤ 200 ключей, поиск строго по PK `(user_id, message_id)`) — **не N+1**, и стоимость не зависит от размера ленты.
- **Фильтр `unread=true` — НАОБОРОТ, применяется ВНУТРИ keyset-запроса** (`MailMessageRepository.list_feed`), анти-джойном:
  `NOT EXISTS (SELECT 1 FROM mail_message_reads r WHERE r.message_id = mail_messages.id AND r.user_id = :uid)`.
  **Клиентская фильтрация непрочитанных ЗАПРЕЩЕНА:** лента курсорная, фильтр над загруженным набором ломает догрузку и даёт «страницы по 2 письма» — это в точности известный дефект тумблера «С тегами» ([08 §Тумблер «С тегами»](../08-design-system.md#тумблер-с-тегами-клиентский-переименован)), повторять его нельзя. Порядок `internal_date DESC, id DESC`, семантика и формат opaque-курсора при этом **не меняются**.
- **Счётчик непрочитанных (badge «N») НЕ ВВОДИТСЯ.** Он потребовал бы `COUNT(*)` по всей видимой ленте пользователя — и, чтобы оставаться актуальным, ещё и регулярного пере-запроса (которого у ленты сегодня нет, см. врезку выше). Владелец просил **индикатор** и **фильтр** — они дают тот же UX без агрегата. Вне scope, не долг.

#### §2.5. ~~Супер-админ из `.env` — личного состояния НЕТ~~ — **ОТМЕНЁН ([ADR-051](ADR-051-superadmin-db-anchor-personal-state.md))**

> **⛔ Не реализовывать.** Норма отменена целиком 2026-07-13 решением [ADR-051](ADR-051-superadmin-db-anchor-personal-state.md) по требованию владельца (он работает на проде под консольным `admin`). Действующее поведение: супер-админу заводится **системная строка-якорь** в `users` с константным `SUPERADMIN_USER_ID` (невидимая для `/api/users`, `/api/teams`, `/api/roles`; вход по-прежнему только из `.env`), `Principal.user_id` становится **не-опциональным** ⇒ личная прочитанность работает и под супер-админом: `POST`/`DELETE …/read` → **`204`**, `is_unread` — **реальное личное значение**, `unread=true` — **обычный серверный фильтр**, UI-контролы (индикатор / фильтр «Непрочитанные» / «Отметить непрочитанным») **рендерятся** всем с `mail:view`. Что осталось от §2.5 и почему `403` на `/api/mail/me/settings` **сохранён** (уже по security-основанию) — [ADR-051 §2](ADR-051-superadmin-db-anchor-personal-state.md) и [§1.6](ADR-051-superadmin-db-anchor-personal-state.md).

Историческое обоснование (на момент 2026-07-13, **более не действует**): супер-админ не был строкой в `users` (`Principal.user_id is None`, `backend/app/api/deps.py:83-85`), поэтому FK `mail_message_reads.user_id → users(id)` для него был невыполним, и норма отдавала `403` по прецеденту `GET`/`PATCH /api/mail/me/settings`. Общий принцип «граница — backend, UI-скрытие — только UX» **остаётся в силе**.

> **Опечатка (зафиксирована, `claims-from-code`):** env-переменные супер-админа называются **`ADMIN_USER`/`ADMIN_PASSWORD`** (`backend/app/config.py:55-56`), а не `ADMIN_USERNAME` — как ошибочно было в исходной редакции этого параграфа.

#### §2.6. Пометка ПРИ ОТКРЫТИИ — веб и Mini App (нормативно; прямой ответ на вопрос владельца)

**Ответ на вопрос «сможем ли пометить прочитанным при открытии из Mini App» — ДА, тем же эндпоинтом, без спец-логики и без второго контракта.** Обоснование по коду: Mini App **не имеет** отдельного «SSO-принципала» — `MailTelegramService.auth` резолвит **реального пользователя CRM** по Telegram-id и выпускает **обычный CRM access-JWT** с `uid = str(resolved.id)` (`backend/app/services/mail_telegram_service.py:140-143`), который фронт Mini App шлёт `Bearer`-заголовком в **те же** `/api/mail/*` (`frontend/src/features/mail/miniAppHooks.ts` → `listMail(..., getMailMiniAppToken())`). Значит в Mini App `principal.user_id` **всегда непуст** (супер-админа из `.env` там быть не может — SSO поднимает строку `users`), и per-user прочитанность работает **идентично** вебу.

| Поверхность | Что считается «открытием» | Действие |
|-------------|---------------------------|----------|
| **Веб `/mail`, вкладка «Сообщения»** | **смена выбранного письма** (`handleSelect(id)`, `frontend/src/pages/MailPage.tsx:233-236`) — как кликом по `MailListItem`, так и **авто-выбором самого свежего письма** при открытии вкладки/смене серверного фильтра ([08 §Layout](../08-design-system.md#layout-master-detail): «по умолчанию выбрано и показано самое свежее письмо») | `POST /api/mail/messages/{id}/read` |
| **Mini App `/tg/mail`** | клик по `MailMiniAppCard` → открытие detail-оверлея (`onOpen` → `setSelected`, `frontend/src/pages/MailMiniAppPage.tsx`) | тот же `POST /api/mail/messages/{id}/read` (SSO-JWT) |

- **Авто-выбранное свежее письмо ТОЖЕ помечается прочитанным** (нормативно — чтобы исполнитель не гадал): его тело **полностью отрендерено** в правой панели, т.е. оно фактически открыто и показано. Обратное («авто-выбор не считается») дало бы противоречие «письмо прочитано глазами, но горит непрочитанным». Именно поэтому обязателен откат (§2.7).
- **Триггер — СМЕНА выбранного письма, а не каждый рендер.** Повторные рендеры/ре-фетчи ленты при неизменном `selectedId` **не** шлют `POST` повторно. Иначе (а) шторм запросов при каждом ре-рендере/догрузке страницы, (б) кнопка «Отметить непрочитанным» (§2.7) стала бы бесполезной — её эффект тут же затирался бы авто-пометкой того же открытого письма.
- **Отметка — best-effort, лента не ломается.** Ошибка `POST …/read` (сеть/`5xx`) **не** блокирует показ письма и **не** даёт toast-спама: письмо открыто и читается; максимум — **индикатор остаётся гореть** (письмо так и числится непрочитанным). Отдельного состояния/ретрая UI не заводит.
- **Локальное обновление состояния — ТОЛЬКО ПО УСПЕШНОМУ `204` (нормативно, развилки нет).** Получив `204`, фронт помечает письмо прочитанным **в уже загруженном кэше ленты** (точечная правка `is_unread` у одного элемента). **Оптимистичное обновление ДО ответа — ЗАПРЕЩЕНО:** оно прямо противоречит норме best-effort выше (при ошибке индикатор погас бы, хотя письмо осталось непрочитанным на сервере) — а корректный оптимистичный вариант потребовал бы обязательного отката с онбордингом ошибки, т.е. ровно того состояния/ретрая, которого мы избегаем. Задержка в один round-trip здесь несущественна: письмо уже отрендерено, гаснет только индикатор.
- **Полный инвалидэйт ленты после отметки ЗАПРЕЩЁН** — он перезапрашивал бы все страницы бесконечного скролла на каждый клик по письму (и сломал бы §2.8).

#### §2.7. Возврат в «непрочитано» — ДА (решение с обоснованием)

**Решение: возможность вернуть письмо в «непрочитано» ВВОДИТСЯ** (`DELETE /api/mail/messages/{id}/read`).

Обоснование (владелец оставил выбор за архитектором):
1. **Пометка автоматическая и частично непроизвольная** (§2.6: авто-выбор самого свежего письма помечает его прочитанным без осознанного клика). Без отката **любой** промах необратим — а необратимость, введённая автоматикой, это дефект, а не фича.
2. **«Оставить на потом» — базовый почтовый workflow.** Оператор открыл письмо, понял, что не может ответить сейчас, и возвращает его в очередь. Без отката единственный способ не потерять письмо — держать вкладку открытой.
3. **Цена — околонулевая:** `DELETE` по PK в уже созданной таблице; эндпоинт-близнец `POST`, тот же гейт, тот же scope, тот же `404`, без новых кодов ошибок.

**UI (нормативно):** в **шапке детали письма** (веб `/mail`, правая панель) — кнопка/иконка **«Отметить непрочитанным»**; рендерится **только** когда письмо уже прочитано (`is_unread === false`) — **у любого пользователя с `mail:view`, включая супер-админа** ([ADR-051](ADR-051-superadmin-db-anchor-personal-state.md) §3; прежняя оговорка «только у не-супер-админа» отменена вместе с §2.5). После клика: `DELETE …/read` → `204` → письмо в ленте загорается индикатором, **но остаётся открытым** (деталь не закрывается, авто-пометка повторно **не** срабатывает — см. §2.6 «триггер = смена письма»). **В Mini App кнопки отката НЕТ** (владелец не просил; экран узкий, read-only-поверхность — [08 §Mini App почты](../08-design-system.md#telegram-mini-app-почты-tgmail-нормативно)).

#### §2.8. UI ленты: индикатор и фильтр (нормативно; точный рендер — [08-design-system.md](../08-design-system.md))

- **Непрочитанное письмо в списке (`MailListItem`):** **тема (`subject`) — полужирным** + **точка-индикатор акцентного цвета (`--accent`)** в начале строки. Прочитанное — обычный вес, точки нет. **Не полагаться только на цвет/вес** (a11y, NFR-8): у элемента — sr-only/`aria-label` текст **«Непрочитано»**.
  - **Новый примитив ДС НЕ вводится.** Точка — простой элемент строки списка, залитый `--accent`. **`Badge` с `dot` НЕ переиспользуется:** его тон-палитра — **статусная** (`green`/`red`: здоровье ящика/сервера), а непрочитанность — не статус здоровья; вешать на неё зелёный/красный значит сломать семантику цвета в дизайн-системе.
- **Фильтр «Непрочитанные»** — toggle-кнопка в **том же тулбаре** ленты, рядом с «С тегами» (`Button`, toggle-стиль, `aria-pressed`). Нормативный лейбл — **«Непрочитанные»**.
  - **Он СЕРВЕРНЫЙ** (в отличие от клиентского «С тегами»): включение → `unread=true` в `GET /api/mail/messages` → **сброс пагинации** и новый первый запрос **без `before`** (та же механика, что у «Почта»/«Команда» — [08 §Дропдауны](../08-design-system.md#дропдауны-почта-и-команда-серверные-select)). Комбинируется (AND) с «Почта»/«Команда» и с клиентским «С тегами».
  - **Пустой результат:** **«Непрочитанных писем нет»** (новая нормативная строка).
  - **⚠️ Открытие письма при активном фильтре «Непрочитанные» НЕ УДАЛЯЕТ его из текущего списка** (нормативно): письмо помечается прочитанным, индикатор гаснет, но строка **остаётся на месте до следующего ре-запроса ленты** (смена фильтра / ручной refetch). Иначе выбранное письмо исчезало бы **из-под курсора пользователя** в момент открытия — деталь схлопывалась бы сама. Это прямое следствие запрета на инвалидэйт из §2.6.
  - **⚠️ ПРЕДПОСЫЛКА этой нормы (нормативно, не удалять): ЛЕНТА НЕ АВТОПОЛЛИТСЯ.** Норма выше держится на том, что список не перезапрашивается сам собой: сегодня `useMailFeed` — `useInfiniteQuery` **без `refetchInterval`**, с **`refetchOnWindowFocus: false`** (`frontend/src/features/mail/hooks.ts`), а фоновый poll новых писем не реализован ([TD-025](../100-known-tech-debt.md)). **Автополлинг/`refetchOnWindowFocus` на ленте ВВОДИТЬ ЗАПРЕЩЕНО без пересмотра этой нормы:** любой фоновый refetch при активном `unread=true` выбросит только что открытое (уже прочитанное) письмо из ответа сервера — и оно исчезнет из-под курсора, а деталь схлопнется. Если поллинг ленты когда-либо понадобится (закрытие [TD-025](../100-known-tech-debt.md)), он обязан идти **отдельным ADR**, который решит эту коллизию явно (напр. «pin» открытого письма в наборе поверх серверного ответа). Реализовать поллинг «заодно», опираясь на §2.4, — **дефект**.
- **Mini App `/tg/mail`:** индикатор непрочитанного в `MailMiniAppCard` — **добавляется** (тот же визуальный язык: жирная тема + точка `--accent` + sr-only «Непрочитано»). **Фильтра «Непрочитанные» в Mini App НЕТ** (владелец не просил; в Mini App вообще нет тулбара фильтров — `frontend/src/pages/MailMiniAppPage.tsx`).

- **Одна миграция `0025_mail_message_reads`** — чисто аддитивная (новая таблица), без изменения существующих. `downgrade()` рабочий. Backfill не нужен: пусто = «всё непрочитано», корректное начальное состояние. На проде (2874+ письма, 121 ящик) — мгновенная.
- **Контракт `/api/mail` расширяется аддитивно:** `MailMessage += is_unread`, `GET /api/mail/messages += unread`, два новых эндпоинта отметки. Существующие поля, коды ошибок, курсор, порядок — **не меняются**. Ломающих изменений нет.
- **Вкладка «Почты» (фикс 1) — правка ТОЛЬКО frontend.** Backend не трогается: поиск и team-фильтр клиентские, новых query-параметров нет.
- **Прочитанность работает в Mini App «бесплатно»** — потому что Mini App с самого начала ходит под обычным CRM-JWT с `uid` ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §6). Ни второго контракта, ни спец-принципала.
- **Производительность ленты не деградирует:** +1 батч-запрос по PK на страницу (не N+1); фильтр `unread` — анти-джойн по PK внутри существующего keyset-запроса; счётчика-агрегата нет.
- ~~**Супер-админ из `.env` теряет функцию** (индикатор/фильтр/отметка)~~ — **ОТМЕНЕНО** ([ADR-051](ADR-051-superadmin-db-anchor-personal-state.md)): владелец работает на проде под консольным `admin`, поэтому супер-админу заведена системная строка-якорь в `users`, и прочитанность работает и под ним (личная, как у всех). `403` на `/api/mail/me/settings` при этом **сохранён** — по security-основанию ([ADR-051 §1.6](ADR-051-superadmin-db-anchor-personal-state.md)), а не из-за отсутствия строки.
- **Модуль mail теряет один non-goal:** «пометка прочитано/непрочитано» уходит из [Out of scope](../modules/mail/README.md#out-of-scope); «папки» и «архив» **остаются** вне scope.
- **Новый долг [TD-058](../100-known-tech-debt.md):** не-admin в нескольких командах не имеет team-фильтра на вкладке «Почты» (следствие §1.2).
- **QA:** покрыть — идемпотентность `POST`/`DELETE …/read`; `404` на чужое письмо (анти-энумерация); ~~`403` супер-админу~~ → **отметка супер-админом работает** (`204`, персональный `is_unread`) — [ADR-051](ADR-051-superadmin-db-anchor-personal-state.md); `is_unread` персонален (два пользователя, одно письмо — разные значения); `unread=true` не ломает курсор (страница 2 не теряет/не дублирует письма); CASCADE-удаление ящика чистит отметки.

## Альтернативы (отклонены)

1. **Колонка `mail_messages.is_read` (общая на команду).** Отклонено — **прямо противоречит требованию владельца** («прочитанность личная у каждого пользователя»): один флаг на письмо означал бы, что прочтение коллегой гасит индикатор всем.
2. **`mail_message_reads.user_id` как `TEXT` (username) вместо FK на `users(id)`** — чтобы вместить супер-админа из `.env`. Отклонено: отказ от FK (мусор при удалении пользователя, нет `ON DELETE CASCADE`) ради bootstrap-учётки. **Схема (FK на `users`) осталась верной и после [ADR-051](ADR-051-superadmin-db-anchor-personal-state.md)** — он решил задачу иначе (дал супер-админу строку-якорь в `users`), не трогая `0025`; развёрнутый разбор этой альтернативы — [ADR-051 §Альтернативы п.2](ADR-051-superadmin-db-anchor-personal-state.md).
3. **Хранить прочитанность на клиенте** (`localStorage`). Отклонено: не переживает смену устройства/браузера, а Mini App и веб — **разные** хранилища у одного человека; владелец просил единое личное состояние в обеих поверхностях.
4. **Вычислять `is_unread` через `LEFT JOIN` прямо в keyset-запросе ленты.** Отклонено: подмешивает join в запрос, чей план держится на индексе `(internal_date DESC, id DESC)`, ради булева флага — при том, что батч-лукап по PK на ≤200 ключей стоит околоноля. (Для **фильтра** `unread=true` — наоборот, анти-джойн внутри запроса **обязателен**: клиентская фильтрация ломает курсор.)
5. **Клиентский фильтр «Непрочитанные»** (по образцу тумблера «С тегами»). Отклонено: тумблер «С тегами» — **известный дефект** ленты (фильтрует только загруженное, даёт «Нет писем с тегами среди загруженных»); тиражировать его на непрочитанные значит осознанно копировать баг.
6. **Отдельный эндпоинт пометки для Mini App** (`POST /api/mail/telegram/messages/{id}/read`). Отклонено: Mini App ходит под **обычным** CRM-JWT в **те же** эндпоинты (`backend/app/services/mail_telegram_service.py:140-143`) — второй контракт был бы дублем с нулевой выгодой.
7. **Не давать возможности вернуть «непрочитано».** Отклонено — см. §2.7: пометка частично непроизвольна (авто-выбор свежего письма), необратимость превратила бы это в дефект.
8. **Badge-счётчик непрочитанных.** Отклонено: `COUNT(*)` по всей видимой ленте пользователя + необходимость регулярного пере-запроса, чтобы число не «застывало»; владелец просил индикатор и фильтр — они закрывают задачу без агрегата.
9. **Серверный поиск по ящикам** (новый query-параметр `q`). Отклонено: каталог грузится целиком (пагинации в контракте нет), клиентский фильтр полон и точен; серверный дал бы новый параметр, новую валидацию и новый класс ошибок ради нуля выгоды. Тот же выбор, что в [ADR-039](ADR-039-ui-server-inline-edit-backends-search-empty-sms-label.md).
10. **Показывать team-фильтр вкладки «Почты» всем ролям** (в отличие от вкладки «Сообщения»). Отклонено: источник опций `GET /api/teams` гейтится `teams:view`, которого у mail-оператора может не быть → пустой дропдаун (тот самый мусорный контрол из [ADR-036](ADR-036-sms-team-filter-admin-only.md)); плюс расщепление одной нормы на три страницы. Остаточное ограничение — [TD-058](../100-known-tech-debt.md).
