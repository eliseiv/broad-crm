# ADR-002 · Двухшаговый вход + JWT HS256

- Статус: accepted
- Дата: 2026-06-28

## Контекст

ТЗ требует двухшаговый вход: сначала экран логина, затем экран пароля; проверка против `ADMIN_USER`/`ADMIN_PASSWORD` из `.env`, после успеха — JWT. ТЗ оставляет выбор: два endpoint'а (`/auth/login` + `/auth/password`) или единый. Один админ, один backend-сервис.

## Решение

1. **Двухшаговость — это UX страницы входа, а не два серверных запроса.** Шаг 1 (ввод логина) — клиентский переход без обращения к API. Шаг 2 (ввод пароля) отправляет единый запрос `POST /api/auth/login {username, password}`.
2. **Единый серверный endpoint** `POST /api/auth/login` проверяет логин+пароль constant-time сравнением и выдаёт JWT.
3. **JWT — HS256** (симметричный, `JWT_SECRET` из `.env`), TTL по умолчанию 60 мин _(обновлено: 1440 мин / 24 ч — см. [Амендмент](#амендмент-2026-07-07--ttl-jwt-24-ч))_, без refresh-токенов (по истечении — повторный вход).
4. Единое сообщение об ошибке `invalid_credentials` для неверного логина и/или пароля.

## Обоснование

- Отсутствие серверного шага 1 **исключает user-enumeration** (нельзя по ответу узнать, существует ли логин).
- Один endpoint проще в реализации/тестировании, меньше состояний (не нужен промежуточный «login_token» между шагами).
- HS256 достаточно: единственный сервис и выпускает, и валидирует токен; асимметрия (RS256) избыточна (NFR-1).

## Последствия

- (+) Нет промежуточного состояния между шагами, нет утечки информации о пользователях.
- (+) Простая реализация и тесты.
- (−) При появлении нескольких независимых сервисов-валидаторов потребуется переход на RS256 (новый ADR).
- (−) Нет «тихого» продления сессии — пользователь повторно логинится после TTL (приемлемо для админ-панели).

## Альтернативы

- **Два серверных endpoint'а** (`/auth/login` возвращает временный токен, `/auth/password` обменивает на JWT) — отвергнуто: лишнее состояние, риск user-enumeration на шаге 1, без выгод для одного админа.
- **Session-cookie вместо JWT** — отвергнуто: SPA + stateless API, JWT проще без серверного стора сессий.
- **RS256** — отложено до многосервисной валидации.

## Амендмент 2026-07-07 · TTL JWT 24 ч

Исходное решение (п.3) — TTL по умолчанию **60 мин** — оставлено историческим. По запросу пользователя дефолт `JWT_EXPIRES_MIN` повышен до **1440 мин (24 ч)**, чтобы снизить частоту релогина в админ-панели. Trade-off (более длинное окно валидности токена) принят: токен хранится только в памяти SPA, refresh-токенов нет, значение — env-параметр и при необходимости ужесточается без изменения кода. Актуальное нормативное значение и обоснование — [05-security.md](../05-security.md#jwt).
