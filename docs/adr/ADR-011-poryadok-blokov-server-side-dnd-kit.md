# ADR-011 · Порядок блоков — server-side `position` + @dnd-kit, drag по задержке

- Статус: `accepted`
- Дата: 2026-07-01
- Контекст модулей: [servers](../modules/servers/README.md), [ai-keys](../modules/ai-keys/README.md), [ui](../modules/ui/README.md)

## Контекст

Пользователь запросил три связанные фичи для страниц «Серверы» и «ИИ - ключи»:

1. **Редактирование блока по клику** — короткий клик по карточке открывает ту же модалку, что «Добавить», в режиме edit (сервер — только `name`; AI-ключ — `name`/`provider`/`key`).
2. **Группировка AI-ключей** по провайдерам (секции OpenAI / Anthropic).
3. **Drag-and-drop перестановка** карточек, порядок сохраняется **на сервере**.

Ключевой конфликт взаимодействия: карточка одновременно должна быть (а) кликабельной для открытия редактирования и (б) «хватом» для перетаскивания. Нужно развести эти жесты и выбрать способ хранения порядка.

Требования-ограничения: простота (NFR-1), тёмная enterprise-эстетика, один админ, до ~50 карточек на странице, персистентность порядка между сессиями и устройствами.

## Решение

### 1. Порядок хранится на сервере (`position` в БД)

В таблицы `servers` и `ai_keys` добавляется колонка `position integer NOT NULL DEFAULT 0`. Списки (`GET /api/servers`, `GET /api/ai-keys`) сортируются `ORDER BY position ASC, created_at DESC, id` (тай-брейк — новые выше при равных `position`). Порядок задаётся отдельными endpoint'ами `PATCH /api/servers/order` и `PATCH /api/ai-keys/order`, принимающими полный упорядоченный список `id`; backend присваивает `position = индекс` в одной транзакции. Детали — [04-api.md](../04-api.md#перестановка-порядок-карточек), [03-data-model.md](../03-data-model.md#колонка-position-порядок-карточек).

**Почему server-side, а не localStorage:** порядок должен переживать смену устройства/браузера и очистку кэша; это свойство реестра, а не UI-настройка. Один админ → нет конфликтов конкурентного редактирования порядка. Стоимость — одна колонка + два тонких endpoint'а, что дешевле, чем объяснять пользователю «почему порядок разный на ноутбуке и на рабочем ПК».

**Отклонённая альтернатива — дробный/float `position` (LexoRank-подобный):** позволяет двигать один элемент без переписи всех. Отклонена как оверинжиниринг для ≤50 карточек: полная перезапись `position = 0..N-1` при каждом reorder тривиальна и детерминирована, не копит дрейф точности, не требует ребаланса.

### 2. AI-ключи переставляются ТОЛЬКО внутри своей провайдер-группы

Провайдер у ключа при перестановке фиксирован (между секциями OpenAI/Anthropic не таскаем — сменить провайдера можно только через edit-модалку). `PATCH /api/ai-keys/order` принимает `{provider, ids}` и переставляет только ключи указанного провайдера. `position` уникален и непрерывен **в пределах группы** (0..M-1); глобальная уникальность `position` между провайдерами не требуется — frontend сначала группирует по `provider`, затем внутри секции сортирует по `position`. Серверы — единый список, свободная перестановка.

### 3. @dnd-kit + активация drag по задержке (delay), а не отдельная ручка

Добавляется зависимость **@dnd-kit** (`@dnd-kit/core`, `@dnd-kit/sortable`, `@dnd-kit/utilities`) — [02-tech-stack.md](../02-tech-stack.md#frontend). Вся карточка — область хвата (нет отдельной «ручки»/grip-иконки). Жесты разводятся через `PointerSensor` с `activationConstraint: { delay: ~200ms, tolerance: ~5px }`:

- **короткий клик (< 200 мс, без сдвига)** → срабатывает `onClick` карточки → открывается edit-модалка (фича 1);
- **зажать (~200 мс) + движение** → стартует drag → перестановка.

Кнопка **Удалить** на карточке вызывает `stopPropagation` и не инициирует ни drag, ни открытие edit-модалки.

**Почему drag-по-задержке, а не отдельная ручка:** пользователь явно попросил «вся карточка активна как хват», короткий клик = редактирование. Ручка (drag-handle) заняла бы место в композиции карточки (референс-дизайн её не предусматривает) и добавила бы лишний элемент. Delay-активация — штатный паттерн @dnd-kit, надёжно разводит tap и drag без визуального шума.

**Почему @dnd-kit, а не react-dnd / нативный HTML5 DnD / SortableJS:** @dnd-kit — современная, headless, accessibility-ориентированная библиотека без внешних зависимостей, с готовым `sortable`-пресетом, delay-сенсорами и поддержкой клавиатуры (`KeyboardSensor`). Нативный HTML5 DnD плохо кастомизируется и слаб на тач-устройствах; react-dnd тяжелее и требует backend-провайдер; SortableJS — императивный, хуже дружит с React-рендер-моделью.

## Последствия

**Плюсы:**
- Порядок персистентный и одинаковый на всех устройствах.
- Единый жест «клик = редактирование, зажатие = перетаскивание» без лишних элементов управления.
- Полная перезапись `position` детерминирована и проста в тестировании.

**Минусы / принятый долг:**
- Reorder переписывает `position` всех карточек списка/группы (для ≤50 — незначимо).
- Оптимистичная перестановка на фронте + инвалидация запроса после ответа (TanStack Query); при ошибке сохранения — откат к серверному порядку.
- Клавиатурная перестановка (dnd-kit `KeyboardSensor`) на Этапе 1 — опциональна; если не реализована, фиксируется как [TD-022](../100-known-tech-debt.md) (a11y-гэп reorder для клавиатуры, NFR-8). Редактирование и удаление остаются полностью доступными с клавиатуры.
- Delay-активация (~200 мс) добавляет крошечную задержку старта drag — осознанный компромисс ради разведения с кликом.
