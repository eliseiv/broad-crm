# ADR-022 · CRM-команды, категоризированная навигация и каталог прав `roles`/`teams`

- Статус: accepted
- Дата: 2026-07-08
- Спринт: A (backend) + B (frontend)
- Расширяет / уточняет: [ADR-021](ADR-021-rbac-users-roles.md) (амендмент §1 — каталог прав), [ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md) (дефолт-маршрут: «Дашборд» больше не стартовая вкладка), [ADR-008](ADR-008-admin-iz-env.md) (супер-админ — граница эскалации)

## Контекст

После Спринта 3 ([ADR-021](ADR-021-rbac-users-roles.md)) есть пользователи, роли и RBAC на все страницы. Требуется расширение:

1. **Доменная сущность «Команды»** (CRM teams) — группировка пользователей вокруг лидера. Пользователь может входить в 0..N команд.
2. **Пользователи += опциональный email**; форма создания пользователя расширяется (Логин, Почта, Пароль, Роль, Команды).
3. **Страница «Роли»** выносится в отдельную от «Пользователей»; список ролей показывает число носителей роли (`user_count`).
4. **Каталог прав** пополняется страницами `roles` и `teams` (амендмент [ADR-021](ADR-021-rbac-users-roles.md) §1); управление ролями переводится с `require_admin` на матричные права `roles:*`. Страница «Пользователи» **остаётся** admin-only (вне матрицы).
5. **Навигация** переходит с плоского ряда вкладок на **3 категории-дропдауна**; «Дашборд» убирается из меню (маршрут остаётся по прямому URL).

### Дизамбигуация: CRM-команды ≠ mail-«команды» (нормативно)

В проекте уже есть сущность «Команда» в модуле «Почты» — это `groups` внешнего сервиса `postapp.store` ([ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md), external ADR-0037): схема [`MailTeam`](../04-api.md#схема-mailteam), эндпоинт `GET /api/mail/teams`, дропдаун-фильтр «Команда» на странице «Почты». Это **read-through-прокси без хранения** — в CRM не персистится.

Вводимые здесь **CRM-команды** — **отдельная** доменная сущность:

| Аспект | mail-«команды» (существующие) | CRM-команды (новые, этот ADR) |
|--------|-------------------------------|--------------------------------|
| Неймспейс API | `/api/mail/teams` | `/api/teams` |
| Схема | `MailTeam { id: integer, name }` | `TeamListItem` (uuid, лидер, участники) |
| Хранение | нет (прокси к внешнему) | таблицы `teams` + `user_teams` в БД CRM |
| Смысл | группа почтовых ящиков внешнего сервиса | группа пользователей CRM вокруг лидера |
| Идентификатор | `integer` (внешний) | `uuid` (CRM) |

Сущности **не пересекаются** и не связаны. Слова «лидер»/«участник» в CRM-контексте свободны. При коммуникации всегда уточнять «CRM-команда» vs «команда почт».

## Решение

### 1. Модель данных (см. [03-data-model.md](../03-data-model.md#таблицы-teams-и-user_teams-crm-команды))

- `teams(id uuid PK, name text UNIQUE, leader_id uuid NOT NULL FK→users(id) ON DELETE RESTRICT, created_at, updated_at)`. `name` — уникален; CHECK «свободного» инварианта по образцу `users.username` (длина 1–64, `btrim`, без control-символов); полное правило набора символов — на Pydantic.
- `user_teams(user_id uuid FK→users(id) ON DELETE CASCADE, team_id uuid FK→teams(id) ON DELETE CASCADE, PK(user_id, team_id))` — **первая M2M-таблица в проекте** (все прежние связи — FK-колонки).
- **Инвариант: лидер ∈ участники** — обеспечивается сервисом (лидер всегда добавляется в `user_teams` команды). БД инвариант не форсирует (простота; сервис — единственная точка записи).
- Индексы: `ix_teams_leader_id`, `ix_user_teams_team_id` (обратный доступ к PK-порядку `(user_id, team_id)`).
- **`users` += `email text NULL`** с частичным уникальным индексом `uq_users_email … WHERE email IS NOT NULL` (UNIQUE-when-present). Формат email валидируется на Pydantic → `422`; дубликат → `409 email_taken`.
- Миграции: **`0009_create_teams`** (`down_revision=0008_create_users_roles`) — `teams` + `user_teams` + индексы; **`0010_add_user_email`** (`down_revision=0009_create_teams`) — `ALTER users ADD email` + частичный уникальный индекс, а также **обновление seed-роли `admin`** до полного нового каталога (иначе БД-админ не получит прав `roles`/`teams`).

### 2. Каталог прав — амендмент [ADR-021](ADR-021-rbac-users-roles.md) §1

Каталог `app/domain/permissions.py::CATALOG` пополняется. **Порядок ключей каталога = порядок строк матрицы прав в UI:**

| # | Страница (`page`) | Действия (`actions`) |
|---|-------------------|----------------------|
| 1 | `dashboard` | `view` |
| 2 | `servers` | `view`, `create`, `edit`, `delete` |
| 3 | `ai-keys` | `view`, `create`, `edit`, `delete` |
| 4 | `proxies` | `view`, `create`, `edit`, `delete` |
| 5 | `backends` | `view`, `create`, `edit`, `delete` |
| 6 | `mail` | `view` |
| 7 | `roles` | `view`, `create`, `edit`, `delete` |
| 8 | `teams` | `view`, `create`, `edit`, `delete` |

- Страница **`users` в каталог НЕ входит** — управление пользователями остаётся под `require_admin` (создание/удаление юзеров, сброс паролей — admin-only, вне матрицы).
- Seed-роль `admin` в БД обновляется миграцией `0010` до полного нового каталога (включая `roles:[view,create,edit,delete]`, `teams:[…]`). Супер-админ (`.env`) получает полный каталог из `full_catalog_permissions()` в рантайме (в БД не пишется).

### 3. Гейтинг API (нормативно)

| Роутер / эндпоинт | Прежний гейт | Новый гейт |
|-------------------|--------------|------------|
| `/api/teams` (GET/POST/PATCH/DELETE) | — (новый) | `require("teams", <action>)` |
| `/api/roles` (GET/POST/PATCH/DELETE) | `require_admin` | `require("roles", <action>)` |
| `GET /api/permissions/catalog` | `require_admin` | `require("roles", "view")` |
| `/api/users` (GET/POST/PATCH/DELETE) | `require_admin` | **`require_admin`** (без изменений) |

- `/api/roles` переводится на матричные права `roles:*` (было `require_admin`). Каталог прав нужен редактору роли для построения матрицы, поэтому `GET /api/permissions/catalog` меняет гейт на `require("roles","view")` (носитель `roles:view` строит матрицу).
- Супер-админ и роль `admin` проходят любой `require(...)` (полный каталог), поэтому доступ администраторов не сужается.
- **`/api/users` остаётся строго `require_admin`** — управление учётными записями не выводится в матрицу (см. §4в).

Маппинг метод→действие для новых роутеров (дополняет таблицу [04-api.md](../04-api.md#rbac-и-enforcement-прав)):

| Роутер | GET (list) | POST | PATCH `/{id}` | DELETE |
|--------|------------|------|----------------|--------|
| `teams` | `teams:view` | `teams:create` | `teams:edit` | `teams:delete` |
| `roles` | `roles:view` | `roles:create` | `roles:edit` | `roles:delete` |

### 4. Security-инвариант эскалации привилегий (обязательно, реализует backend)

Перевод редактирования ролей под матрицу (`roles:create/edit`) несёт риск эскалации: носитель `roles:edit`, не будучи админом, мог бы выдать роли (в т.ч. своей) права сверх собственных. Сервер (`403`) — **единственная граница безопасности**; UI-гейтинг не защищает. ADR фиксирует три правила (проверяются в handler после прохождения гейта `require("roles",…)`):

- **(а) Subset-инвариант.** Для актора, который **не** супер-админ и **не** роль `admin`, при `POST`/`PATCH /api/roles`: `permissions` создаваемой/изменяемой роли ⊆ `permissions` актора (для каждой `page` набор `actions` — подмножество `actions` актора по этой `page`). Нарушение → `403 forbidden`. Супер-админ и роль `admin` (полный каталог) проходят всегда.
- **(б) Защита встроенной роли `admin`.** Роль с `name == "admin"` может изменять (`PATCH`) или удалять (`DELETE`) **только** актор с `is_superadmin || role == "admin"`. Иначе → `403 forbidden` (даже при наличии `roles:edit`/`roles:delete`).
- **(в) Пользователи — вне матрицы.** Страница «Пользователи» (создание/удаление пользователей, сброс паролей, смена ролей/статуса) остаётся строго `require_admin` — управление учётками и назначение ролей не выдаётся через `roles:*`/`teams:*`. Это замыкает эскалацию: даже выдав себе `roles:edit`, не-админ не может назначить усиленную роль пользователю (назначение роли — только admin).

**Прецеденция ошибок `POST`/`PATCH /api/roles` (нормативно):** схемная/каталожная валидация (`400`/`422 unprocessable` — `permissions` вне каталога) → subset-инвариант и защита `admin` (`403 forbidden`) → уникальность `name` (`409 role_name_taken`).

### 5. Навигация — 3 категории-дропдауна (реализация — Спринт B)

Плоский ряд вкладок заменяется тремя категориями-дропдаунами:

| Категория | Пункты (в порядке) |
|-----------|--------------------|
| **Агрегатор** | Почты (`mail`) |
| **Мониторинг** | Серверы (`servers`), ИИ-ключи (`ai-keys`), Прокси (`proxies`), Бэки (`backends`) |
| **Пользователи** | Пользователи (`users`, admin-only), Роли (`roles`), Команды (`teams`) |

- **«Дашборд» убирается из меню** (вкладки/пункта нет), но маршрут `/dashboard` **остаётся доступен по прямому URL** под page-level view-guard `dashboard:view` (заглушка «нет доступа к разделу» при отсутствии права). `dashboard:view` остаётся в каталоге.
- **Плоский порядок листьев навигации** (для резолва дефолт-маршрута): `mail, servers, ai-keys, proxies, backends, users, roles, teams` — **без `dashboard`**.
- **Дефолт после логина, index `/`, fallback `*`** → **первая доступная по правам вкладка** этого порядка (permission-aware). «Дашборд» **больше не является дефолтом** (амендмент [ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md)/[ADR-021](ADR-021-rbac-users-roles.md), где дефолтом был `/dashboard`). Если нет ни одного доступного листа — заглушка «нет ни одного раздела» ([08-design-system.md](../08-design-system.md#гейтинг-навигации-и-действий-по-правам-rbac-нормативно)).
- **Видимость категории:** категория (её дропдаун-триггер) видна ⇔ доступен хотя бы один её пункт. Пункт `users` виден по `is_superadmin || role=="admin"`; `roles`/`teams`/ресурсные — по `<page>:view` (или супер-админ).
- **Page-level view-guard** ([ADR-021](ADR-021-rbac-users-roles.md) §Последствия) распространяется на новые страницы: `roles` → `roles:view`, `teams` → `teams:view` (прямой URL без права → заглушка «нет доступа к разделу»).
- UI-примитив дропдаун-меню (в проекте отсутствует) — детали в [08-design-system.md](../08-design-system.md#навигация-категории-дропдауны-applayout); граница безопасности — серверный `403`, категории/пункты — только UX.

## Обоснование

- **CRM-команды в БД, отдельный неймспейс `/api/teams`** — не смешивать с эфемерными mail-`groups` (внешний сервис, `/api/mail/teams`); коллизия имени разрешена явной дизамбигуацией.
- **M2M `user_teams`** — пользователь в 0..N команд; лидер — обязательная одиночная ссылка `teams.leader_id` (RESTRICT, чтобы у команды всегда был лидер и лидера нельзя было удалить, не разобравшись с командой). Инвариант «лидер ∈ участники» — на сервисе (единственная точка записи), без БД-триггеров (простота, NFR-1).
- **email опционален, UNIQUE-when-present** — частичный уникальный индекс не мешает множеству пользователей без email; формат — на Pydantic (по образцу `username`), БД не зависит от locale.
- **Роли под матрицей + subset-инвариант** — даёт делегирование управления ролями без риска эскалации; `users` вне матрицы замыкает границу (назначение ролей — только admin). Супер-админ-fallback ([ADR-008](ADR-008-admin-iz-env.md)) исключает lockout.
- **Категории вместо плоского ряда** — 8 листьев в одном ряду перегружают шапку; группировка по доменам (Агрегатор/Мониторинг/Пользователи) читаемее. «Дашборд» как обзорная страница остаётся по прямому URL, но не занимает место в меню.

## Последствия

- (+) Команды с лидером и участниками; пользователь с email и членством в нескольких командах; роли с числом носителей; делегируемое управление ролями без эскалации.
- (+) Навигация масштабируется: новые страницы добавляются в существующие категории без переполнения шапки.
- (=) **Дефолт-маршрут больше не `/dashboard`.** После логина/`/`/`*` пользователь попадает на первую доступную вкладку нового порядка (Почты → … → Команды). `/dashboard` доступен только по прямому URL. Это амендмент [ADR-017](ADR-017-dashboard-client-aggregation-mail-server-filters.md) (дашборд-как-стартовая) и правила дефолт-редиректа [ADR-021](ADR-021-rbac-users-roles.md).
- (=) Первая M2M-таблица (`user_teams`) — задаёт образец для будущих many-to-many связей.
- (−) `GET /api/permissions/catalog` и `/api/roles` меняют гейт (`require_admin` → `roles:*`) — деплой не ломает админов (полный каталог), но роль без `roles:view` теряет доступ к каталогу/списку ролей (ожидаемо).
- (−) Роль `admin` — по-прежнему зарезервированное имя; вдобавок к гейту «Пользователей» ([ADR-021](ADR-021-rbac-users-roles.md)) теперь защищена от правки/удаления не-админом (§4б).
- Аудит-лог действий с командами/ролями — вне scope ([TD-001](../100-known-tech-debt.md)).

## Альтернативы

- **Переиспользовать mail-`groups` как CRM-команды** — отвергнуто: разные жизненные циклы, хранение (внешний прокси vs БД CRM), типы id (int vs uuid) и семантика (ящики vs пользователи); связывание породило бы зависимость CRM-RBAC от доступности внешнего сервиса.
- **Инвариант «лидер ∈ участники» на уровне БД (триггер/CHECK на M2M)** — отвергнуто ради простоты: сервис — единственная точка записи, БД-триггеры избыточны при NFR-1.
- **Оставить роли под `require_admin`** — отвергнуто: пользователь просил управление ролями через матрицу прав; риск эскалации закрыт subset-инвариантом (§4).
- **Хранить членство как массив uuid в `users`/`teams` (jsonb) вместо M2M** — отвергнуто: теряются ссылочная целостность (FK/CASCADE) и запросы «участники команды»/«команды пользователя»; классическая M2M-таблица корректнее.
- **Оставить «Дашборд» дефолтом и первой категорией** — отвергнуто: пользователь явно просил убрать дашборд из меню; маршрут сохранён по прямому URL.
