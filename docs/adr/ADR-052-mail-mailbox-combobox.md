# ADR-052 — Единый combobox выбора/поиска почты на обеих вкладках `/mail` (примитив `ui/Combobox`)

- **Статус:** accepted
- **Дата:** 2026-07-13
- **Контекст-модули:** [mail](../modules/mail/README.md), [ui](../modules/ui/README.md)

Дропдаун **«Команда»** остаётся `ui/Select` — **не затронут**.

## Контекст

Требование владельца (батч 2026-07-13, дословно):

> «На странице /mail во вкладке Сообщения можно фильтровать сообщения по почтам, а во вкладке Почты проводить поиск по почтам. Необходимо изменить оба этих фильтра. в обоих вкладках будет поиск по почтам, но при нажатии на поле будет выпадать список из всех почт, где можно будет выбрать определенную почту или ввести текст, который будет отфильтровывать почты из выпадающего списка. Поиск так же должен происходить по Номеру, Приложению и самой почте»

Уточнение владельца (на прямой вопрос): на вкладке **«Почты»** ввод текста **фильтрует ТАБЛИЦУ** по всем совпадениям (как сейчас), а выпадающий список нужен для **быстрого перехода к конкретной почте**. Таблица **не схлопывается** до одной строки при вводе.

**Факты по коду (`claims-from-code`, на 2026-07-13):**

- Вкладка **«Сообщения»**: фильтр по почте — `ui/Select` (нативный `<select>`), `aria-label="Почта"`, опции из `useMailMailboxes()` (**полный** каталог, `listMailboxes({})` — `frontend/src/features/mail/hooks.ts:253-262`), первая опция **«Все почты»** (`value=''`), лейбл опции — `mb.display_name ? \`${mb.display_name} ${mb.email}\` : mb.email` (`frontend/src/pages/MailPage.tsx:197-206`). Выбор → `setMailAccountId` → серверный фильтр ленты (`useMailFeed({mailAccountId, teamId, unread})`, `frontend/src/pages/MailPage.tsx:181-182`). **Поиска в списке нет** — нативный `<select>` умеет только type-ahead по первым буквам.
- Вкладка **«Почты»**: поиск — `ui/Input` + `trailing={<Search/>}`, `useState('')` (`frontend/src/components/MailboxesTab.tsx:82`, `:180-188`); фильтрация **клиентская**, ровно по трём полям `number`/`app_name`/`email` (подстрока, ci — `frontend/src/components/MailboxesTab.tsx:109-127`). Каталог — `useMailboxesManage(toIsActive(filter))`, т.е. **уже суженный** серверным `is_active` (`frontend/src/components/MailboxesTab.tsx:95`).
- **Примитива combobox в дизайн-системе НЕТ.** `frontend/src/components/ui/` содержит ровно: `Badge`, `Button`, `Card`, `Checkbox`, `Input`, `Modal`, `MultiSelect`, `Pill`, `Select`, `Spinner`, `Textarea`. Ни `Combobox`, ни `Autocomplete`.
- `MailMailbox` несёт все три поля поиска: `number: str | None`, `app_name: str | None`, `email: str`; `display_name: str | None` — **производное** (`"<number> <app_name>"`, считает сервер — `backend/app/schemas/mail.py:125-134`, [ADR-047](ADR-047-mail-fix-pack.md) §3.3).

**Суть задачи.** Два разных контрола (нативный `<select>` без поиска; текстовое поле без списка) заменяются **одним** примитивом, дающим оба режима: список всех почт по клику **и** фильтрацию списка вводом. Семантика **выбора** при этом на вкладках **разная** (серверный фильтр ленты vs фильтр таблицы каталога) — и это отличие нормируется явно, чтобы исполнитель не унифицировал его «по аналогии».

## Решение

### §1. Новый примитив дизайн-системы — `ui/Combobox`

**Имя — `ui/Combobox`** (`frontend/src/components/ui/Combobox.tsx`), консистентно с `ui/Select`/`ui/Input`/`ui/MultiSelect` и **с уже зафиксированным в docs именем** ([TD-056](../100-known-tech-debt.md): «ввести примитив **`ui/Combobox`**»). Полная спецификация (props, поведение, клавиатура, a11y, состояния, визуал) — **[08-design-system.md §Компонент `ui/Combobox`](../08-design-system.md#компонент-uicombobox-нормативно-adr-052)**; ADR не дублирует её, а фиксирует решения.

**Ключевые решения примитива:**

1. **Реализация — своя (не нативный `<select>`), БЕЗ новой зависимости.** Нативный `<select>` **физически не умеет** фильтровать опции вводом — требование владельца им невыполнимо. Библиотека combobox (Downshift/Headless UI/Radix) в проект **не добавляется** ([02-tech-stack.md](../02-tech-stack.md#frontend) — тот же выбор, что у `Select`/`MultiSelect`/`Textarea`: примитивы пишутся руками на нативных элементах + Tailwind). Цена — собственный a11y-контракт, поэтому он **нормирован поэлементно** (§1.2).
2. **Контролируемый примитив с ДВУМЯ независимыми состояниями:** `value` (выбранная опция) и `query` (текст поля). Оба — снаружи (страница владеет семантикой), примитив владеет только `open`/`activeIndex`/`dirty`.
3. **Два режима — `mode: 'select' | 'search'`** (единственный переключатель семантики; см. §1.1). Ровно они покрывают две вкладки, и ровно они не дают исполнителю «унифицировать» разную семантику вкладок.
4. **Фильтрация опций — ВНУТРИ примитива**, по `ComboboxOption.keywords` (по умолчанию `[label]`). Страница отдаёт **полный** список опций. Это позволяет **вкладке «Почты» использовать ОДИН И ТОТ ЖЕ предикат** для списка и для таблицы (§3).
5. **Портал НЕ вводится.** Панель — `absolute` внутри обёртки поля (`z-20`, `mt-1`, `w-full`, `max-h-64` + внутренний скролл со `scrollbar-none`). Обоснование: оба места использования — тулбары в верхней части своих контейнеров; панель высотой ≤16rem помещается. Портал потребовал бы позиционирования/ресайз-логики без выгоды.

#### §1.1. Режимы (нормативно — ровно два, разница между вкладками живёт ЗДЕСЬ)

| | `mode='select'` (вкладка «Сообщения») | `mode='search'` (вкладка «Почты») |
|---|---|---|
| Что значит **текст** | **Эфемерный** — только фильтрует выпадающий список. Наружу семантики не несёт | **Значимое состояние** — страница фильтрует им таблицу |
| Текст при **закрытии без выбора** | **Возвращается** к лейблу выбранной опции (`onQueryChange(labelOf(value))`). **⚠️ Если `labelOf(value) === undefined`** (выбранной опции **нет в текущем наборе**) — **`query` НЕ трогается** (`onQueryChange` не вызывается): поле продолжает показывать **ТЕКУЩИЙ текст** — лейбл, если пользователь не печатал; **напечатанный текст**, если печатал (в `mode='select'` ввод выбор не сбрасывает). Поле **не пустеет**. Обнулять текст при живом `value` **ЗАПРЕЩЕНО** (дало бы разрыв «фильтр есть — поле его не показывает»; на «Сообщениях» `placeholder` не задан, поле просто опустело бы) | **Сохраняется** как есть |
| **Ввод текста** | Выбор **НЕ сбрасывает** (серверный фильтр ленты не должен слетать от того, что пользователь листает список) | **СБРАСЫВАЕТ выбор** (`onChange(null)`) — текст и выбор взаимоисключающи |
| **Выбор опции** | `onChange(value)`; текст → лейбл опции; список закрывается | `onChange(value)`; текст → лейбл опции; список закрывается |

**Общее правило открытия списка (нормативно, оба режима):** при **открытии** список показывается **ЦЕЛИКОМ** (все опции) — даже если в поле уже стоит текст (лейбл выбранной опции в `select`; сохранённый запрос в `search`). Фильтрация списка включается с **первого введённого/изменённого пользователем символа** (внутренний флаг `dirty`).

**Жизненный цикл `dirty` (нормативно — ПОЛНЫЙ перечень точек сброса, их ЧЕТЫРЕ):** `dirty = true` — только при вводе/изменении текста пользователем; `dirty = false` — при **(1) открытии** панели **кликом/фокусом/шевроном/`↓`/`↑`/`Home`/`End`** (открытие **вводом** `dirty` **НЕ** сбрасывает — фильтрация включается с первого же напечатанного символа), **(2) выборе** опции, **(3) ЗАКРЫТИИ** панели **любым** способом (`Escape`-при-открытом / `Tab` / клик вне / выбор) и **(4) ОЧИСТКЕ** (клик по `X` / `Escape` при закрытом списке) — иначе `X` остался бы отрисован после собственного клика. **Сброс на закрытии обязателен:** иначе «напечатал → закрыл без выбора» оставляет `dirty = true` при вернувшемся к лейблу тексте, и кнопка `X` отрисуется там, где **сбрасывать нечего** — ровно аномалия, запрещённая §1.1а.

Правило открытия буквально реализует требование владельца («при нажатии на поле будет выпадать список **из всех почт**») и снимает противоречие «в поле уже есть текст → список схлопнут до одной опции».

> **⚠️ Список ≠ таблица (нормативно, вкладка «Почты»).** Выпадающий **список** фильтруется текстом только после ввода (правило выше), а **таблица** остаётся отфильтрованной сохранённым `query` всегда (это состояние страницы, а не панели). Расхождение намеренное: панель — навигационный список «куда перейти», таблица — результат поиска.

#### §1.1а. Очистка и состояние «нет выбора» — ОДНО состояние, не два (нормативно)

**Проблема, которую закрывает этот параграф:** «нет фильтра» не должно быть представимо **двумя** разными состояниями (`value=''` с текстом «Все почты» **против** `value=null` с пустым полем) — иначе `Escape`/`X` и явный выбор опции «Все почты» давали бы **одинаковый фильтр при разном виде поля**, а кнопка `X` висела бы всегда и «очищала» состояние в эквивалентное самому себе.

**Норма — очистка зависит от наличия pinned-опции сброса в `options`:**

| В `options` есть `pinned`-опция сброса | Очистка (`X` / `Escape` при закрытом списке) | Кто применяет |
|---|---|---|
| **ДА** (вкладка «Сообщения»: `{value:'', label:'Все почты', pinned:true}`) | ≡ **ВЫБОР этой опции**: `onChange(resetOption.value)` + `onQueryChange(resetOption.label)`. **`onChange(null)` в этом случае НЕ эмитится** | примитив |
| **НЕТ** (вкладка «Почты») | `onChange(null)` + `onQueryChange('')` | примитив |

- **Кнопка `X` рендерится ТОЛЬКО когда есть что сбрасывать:** выбрана **не-`pinned`** опция **ИЛИ** поле `dirty` (пользователь изменил текст **и панель ещё открыта** — на закрытии `dirty` сбрасывается, §1.1) **ИЛИ** (`mode='search'` и `query !== ''`). **При выбранной `pinned`-опции сброса и не-`dirty` поле `X` НЕ рендерится** — очищать нечего.
  > **⚠️ Это ЕДИНСТВЕННОЕ нормативное условие видимости `X`.** Прежняя формулировка «рендерится, когда поле непусто» **ОТМЕНЕНА** (в т.ч. в §1.3 «Визуал» и в 08 §Визуал): на «Сообщениях» поле **никогда не пусто** (в нём лейбл выбранной опции) ⇒ по старому условию `X` висел бы **всегда**. Второй формулировки условия быть не должно.
- **`onChange(null)` = «нет выбранной опции».** Страница, у которой есть `pinned`-сброс, обязана трактовать `null` как `value` опции сброса (`v ?? ''`) — оборонительно; примитив ей `null` не шлёт.
- **Инициализация страницы (нормативно, вкладка «Сообщения»):** `value = mailAccountId != null ? String(mailAccountId) : ''` (сегодня `mailAccountId: number | undefined = undefined`, `frontend/src/pages/MailPage.tsx:169`), `query` инициализируется **лейблом опции сброса — «Все почты»** (сегодня тот же смысл несёт `<select value=''>`, `frontend/src/pages/MailPage.tsx:371`). ⇒ **«нет фильтра» на «Сообщениях» = ровно одно состояние:** `value=''`, `query='Все почты'`, `X` не отрисован. **Пустого поля с плейсхолдером на этой вкладке не бывает** — `placeholder` там **не задаётся**.
- На вкладке **«Почты»** `pinned`-опции нет ⇒ «нет выбора и нет запроса» = `value=null`, `query=''`, видны плейсхолдер «Поиск по почтам…» и **никакой** `X`.

#### §1.2. A11y и клавиатура — ОБЯЗАТЕЛЬНЫ (поэлементный перечень)

Виджет интерактивный и не-нативный ⇒ доступность даёт **только** наш код. Полный перечень ARIA-атрибутов и **каждой** клавиши (`ArrowDown`/`ArrowUp`/`Home`/`End`/`Enter`/`Escape`/`Tab`/печать символа) + поведение клика вне — **[08-design-system.md §Компонент `ui/Combobox`](../08-design-system.md#компонент-uicombobox-нормативно-adr-052)**, подразделы «ARIA-контракт» и «Клавиатура». Перечень **нормативный и полный**: каждая строка обязана быть реализована и покрыта тестом (`qa`). Реализация «мышью работает, клавиатурой нет» = дефект, а не долг.

**Активная опция при открытии МЫШЬЮ/фокусом — ЕЁ НЕТ** (`aria-activedescendant` не задан): активная опция появляется **только** от `↓`/`↑`/`Home`/`End` и **не выводится из `value`** — иначе при выбранной опции **вне набора** (§3.1а) атрибут сослался бы на **несуществующий** `optionId`.

**Мышь (нормативно, [08 §Клавиатура](../08-design-system.md#клавиатура-нормативно--перечень-полный-каждая-строка-обязательна) — буллеты под таблицей):** клик **вне** поля/панели → закрыть без выбора; клик **по опции** ≡ `Enter` на ней; клик **по самому полю при уже открытой панели** → **панель ОСТАЁТСЯ открытой** (не toggle — пользователь ставит курсор в текст, закрытие потеряло бы контекст); **`ChevronDown` — КЛИКАБЕЛЕН, toggle** (закрыта → открыть, открыта → закрыть без выбора; `<button type="button" tabIndex={-1} aria-hidden="true">` — вне таб-порядка, `aria-expanded` не дублирует). Отличие от `ui/Select`, где шеврон декоративен (`pointer-events-none`): там клик обрабатывает нативный `<select>`, здесь поле — обычный `<input>`.

**Клик по кнопке `X` (нормативно — состояние достижимо ВСЕГДА, когда `X` виден по dirty-ветке):** очистка по правилу §1.1а **НЕЗАВИСИМО от того, открыта панель или нет**; это **НЕ** «клик вне». **Панель ЗАКРЫВАЕТСЯ — единое правило для ОБОИХ режимов** (обоснование: при `pinned`-опции очистка **есть выбор** этой опции (§1.1а), а выбор опции закрывает список (§1.1) — два имени одного действия обязаны вести себя одинаково; разное поведение по режимам запрещено). **`dirty = false`** (точка (4)), **фокус ОСТАЁТСЯ в `<input>`** (`onMouseDown` → `preventDefault()`), после клика `X` **исчезает**.

**Прокрутка активной опции (нормативно):** активная опция всегда прокручивается в видимую область панели (`scrollIntoView({ block: 'nearest' })`) при изменении от `↓`/`↑`/`Home`/`End`. Панель — `max-h-64` со скроллом, каталог на проде — **121** ящик: без этого `End`/длинный проход `↓` уводят активную опцию за границу скролла.

**⚠️ Из этого следует ШТАТНОЕ состояние «панель открыта, активной опции НЕТ» — клавиши в нём нормированы отдельной колонкой** таблицы [08 §Клавиатура](../08-design-system.md#клавиатура-нормативно--перечень-полный-каждая-строка-обязательна): `↓` → **первая** опция; `↑` → **последняя** (симметрично колонке «список закрыт» — правило не зависит от способа открытия); `Enter` → **выбора НЕТ** (`onChange` не вызывается), **панель остаётся открытой**, `preventDefault()` применяется. Без этой колонки перечень клавиш был бы неполон (из «активировать следующую/предыдущую» и «выбрать активную» поведение при отсутствии активной опции не выводится).

#### §1.3. Визуал — существующие токены, ноль новых

Поле — **побайтово те же** классы, что у `ui/Input`/`ui/Select`: `h-10`, `rounded-[10px]`, `border-border-strong`, `bg-surface-2`, `text-sm text-text-primary`, focus-ring `focus-visible:border-accent focus-visible:ring-2 focus-visible:ring-accent/40`, `disabled:opacity-60`. Трейлинг — `ChevronDown` (как у `ui/Select`) + кнопка очистки `X`, чьё **условие видимости задано РОВНО В ОДНОМ месте — §1.1а** (здесь **не** дублируется; формулировка «при непустом поле» **отменена**). **Новых цветовых токенов нет.**

### §2. Поведение на вкладке «Сообщения» (нормативно)

- `ui/Select` (`aria-label="Почта"`) **заменяется** на `ui/Combobox` **`mode='select'`**, `aria-label` **«Почта»** (строка не меняется).
- **Опции** — из **`useMailMailboxes()`** (полный каталог, без `is_active`) — **источник не меняется**. Первая опция — **«Все почты»** (`value=''`), **`pinned: true`**: она **не фильтруется** вводом и всегда видна первой (сброс обязан оставаться достижимым при любом запросе). **Она же — опция сброса:** «нет фильтра» = `value=''` + `query='Все почты'` (**ровно одно** состояние; `X` при нём не рендерится; `Escape`-при-закрытом ≡ выбор этой опции — см. **§1.1а**). Лейбл прочих опций — **`display_name ? \`${display_name} ${email}\` : email`** — **строка лейбла не меняется** (ровно та, что сегодня в `<option>`).
- **Семантика выбора НЕ меняется:** выбор почты → `mail_account_id` → **СЕРВЕРНЫЙ** фильтр ленты: входит в `queryKey` `useMailFeed`, **сбрасывает пагинацию** (первый запрос без `before`), заново выполняется авто-выбор самого свежего письма ([ADR-044](ADR-044-mail-full-merge-into-crm.md) §7, [08 §Дропдауны](../08-design-system.md#дропдауны-почта-и-команда-серверные)). **Комбинируемость (AND) с «Команда» и «Непрочитанные» — сохраняется**; выбор одного не сбрасывает другой.
- **Ввод текста ленту НЕ фильтрует** — только выпадающий список. **Серверный поиск по письмам не вводится** (владелец не просил).
- **Запрос без совпадений (нормативно — предикат при наличии `pinned`):** «нет совпадений» считается **среди НЕ-`pinned` опций** (`pinned` — не результат поиска, а сброс, и в предикате **не участвует**). ⇒ панель показывает **«Все почты»** (она видна всегда) **И** строку **«Ничего не найдено»** под ней. **Вариант «список никогда не пуст, поэтому „Ничего не найдено“ недостижимо» — ОТКЛОНЁН:** одинокая опция «Все почты» без пояснения читается как **результат** поиска, хотя она — сброс; пользователь не узнаёт, что его запрос не нашёл ничего.
- Фильтр **«Команда»** рядом — **остаётся `ui/Select`** и по-прежнему рендерится **только** при `me.sees_all_mail_teams === true` ([ADR-036](ADR-036-sms-team-filter-admin-only.md)). Гейт **не меняется**.
- **Загрузка каталога** (`mailboxesQuery.isLoading`) → `loading` у комбобокса (в панели — **«Загрузка…»**), поле активно.

### §3. Поведение на вкладке «Почты» (нормативно — ГИБРИДНАЯ семантика, решение владельца)

`ui/Input` + `Search` **заменяется** на `ui/Combobox` **`mode='search'`**, `aria-label` **«Поиск по почтам»**, плейсхолдер **«Поиск по почтам…»** (строки **не меняются**, [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1).

- **Опции** — из **уже загруженного каталога вкладки** (`useMailboxesManage(is_active)`, `frontend/src/components/MailboxesTab.tsx:95`), т.е. **из того же набора, что рендерит таблица**. Следствие: в сегментах «Активные»/«Неактивные» список показывает **только** ящики этого сегмента — **в момент выбора** ящика вне текущего сегмента в списке нет. Опции сброса («Все почты») здесь **НЕТ** — сброс делает кнопка `X` / `Escape`.
- **⚠️ Набор опций меняется СЕРВЕРНО при смене сегмента — выбор при этом СОХРАНЯЕТСЯ (нормативно, §3.1а).** Последовательность «выбрал ящик в сегменте «Все» → переключил на «Неактивные»» **достижима** и оставляет `selectedMailboxId`, указывающий на ящик **вне** нового набора. Это **штатное** состояние (пустое пересечение фильтров), а не «невозможное» — см. **§3.1а**.
- **Лейбл опции — тот же, что на «Сообщениях»:** `display_name ? \`${display_name} ${email}\` : email`. Единый визуальный идентификатор почты на обеих вкладках.

#### §3.1. Что фильтрует таблицу (нормативно — точный предикат, развилки нет)

Состояние вкладки: `selectedMailboxId: number | null` (выбор из списка) и `query: string` (текст). Они **взаимоисключающи** (ввод сбрасывает выбор — §1.1, `mode='search'`).

| Состояние | Таблица показывает |
|---|---|
| `selectedMailboxId !== null` | **ровно один** ящик — `mailbox.id === selectedMailboxId` («быстрый переход к конкретной почте») |
| иначе `query.trim() !== ''` | **все совпадения** по правилу поиска §3.3 (**текущее поведение [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1 сохранено — таблица НЕ схлопывается до одной строки при вводе**) |
| иначе | все ящики набора |

- **Сброс:** кнопка `X` в поле / `Escape` при закрытом списке → `selectedMailboxId = null` **и** `query = ''` → таблица возвращается к полному набору (`pinned`-опции сброса на этой вкладке нет — §1.1а).
- **Почему выбор = сужение до одной строки, а не скролл/подсветка:** (а) владелец просил «быстрый переход к **определённой** почте» — сужение даёт его детерминированно; (б) это **та же ментальная модель, что на «Сообщениях»** («выбор почты = фильтр по этой почте»), т.е. одна норма на две вкладки, а не две разные; (в) scroll-to-row + подсветка потребовали бы новой механики (refs/`scrollIntoView`/временное состояние подсветки) без прецедента в ДС — при 121 ящике на проде и уже работающем сужении это цена без выгоды. Отклонённый вариант — [Альтернативы п.2](#альтернативы-отклонены).

#### §3.1а. Смена сегмента активности (и любое изменение набора опций) — выбор и текст СОХРАНЯЮТСЯ (нормативно)

Сегмент «Все/Активные/Неактивные» — **серверный** фильтр: он меняет сам набор (`useMailboxesManage(toIsActive(filter))`). Поэтому выбранный ранее ящик может **выпасть** из набора.

**Норма (развилка закрыта, вариант «сохраняем»):**

- При смене сегмента (и при любом обновлении набора опций — рефетч каталога, поллинг) **`selectedMailboxId` и `query` НЕ сбрасываются**.
- Если выбранный ящик **отсутствует** в новом наборе → таблица **пуста** → **«Ничего не найдено»**. Поле продолжает показывать **лейбл выбранного ящика** (состояние честно видно пользователю), а в выпадающем списке этого ящика уже **нет** (список = набор).
- **Выход из состояния — только явный:** пользователь жмёт `X` / `Escape` (сброс выбора) **или** возвращает сегмент. **Авто-сброса нет.**

**Обоснование (почему НЕ сбрасывать):**

1. **Прямое требование §3.2:** «ни один фильтр не сбрасывает другие». Авто-сброс выбора при смене сегмента — ровно такой сброс, и он завёл бы **исключение** в норму, действующую на обеих вкладках.
2. **Это тот же класс состояния, что уже нормирован** как штатный: выбранный ящик, не проходящий по фильтру **команды**, тоже даёт пустую таблицу и «Ничего не найдено» (§3.2). Разное поведение для двух пересекающихся фильтров (один сбрасывает выбор, другой — нет) было бы непоследовательным.
3. **Молчаливый сброс хуже пустого результата:** пользователь не увидел бы, что его выбор исчез. Пустая таблица + видимый лейбл в поле + `X` — состояние наблюдаемое и обратимое одним кликом.

> **Формулировка «расхождения „выбрал, а строки нет“ не возникает» — ОТОЗВАНА** (она была неверна: состояние достижимо). Действует норма выше.

#### §3.2. Комбинируемость (AND) — сохраняется

**Порядок применения (амендирует [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.3):**

> серверный `is_active` (сегмент «Все/Активные/Неактивные») → **клиентский combobox (выбор ИЛИ текст, §3.1)** → **клиентский фильтр по команде** → рендер строк.

Все три комбинируются **AND**; ни один **не сбрасывает** другие — **в обе стороны**: выбор почты из списка не сбрасывает фильтр по команде и сегмент активности, и **смена сегмента/команды не сбрасывает выбор почты и текст** (§3.1а).

**Пустое пересечение — штатное состояние в ЛЮБОЙ комбинации** (выбранный ящик не проходит по команде; выбранный ящик выпал из сегмента; текст без совпадений): таблица пуста → **«Ничего не найдено»**, **авто-сброса чужих фильтров НЕТ**, выход — явный (`X`/`Escape` или возврат фильтра).

- **Пустое пересечение — штатно:** выбран ящик, не проходящий по фильтру команды → таблица пуста → **«Ничего не найдено»** (существующая строка). **Авто-сброса других фильтров НЕТ** (молчаливый сброс чужого фильтра — худшее из зол; пользователь видит пустой результат и сбрасывает сам).
- Строка **«Почт пока нет»** — по-прежнему **только** при пустом каталоге без активных фильтров (не путать, [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1).
- Фильтр **«Команда»** — остаётся `ui/Select`, клиентский, гейт `sees_all_mail_teams` ([ADR-036](ADR-036-sms-team-filter-admin-only.md), [TD-058](../100-known-tech-debt.md)) — **не меняется**.

#### §3.3. Правило поиска — ЕДИНОЕ для всех трёх мест (нормативно)

**Поля — ровно три: `number`, `app_name`, `email`.** Совпадение — **подстрока, регистронезависимо**, по `trim()`-нутому запросу; пустой запрос → фильтр не применяется. **`display_name` НЕ входит** (производная склейка `number`+`app_name`, [ADR-047](ADR-047-mail-fix-pack.md) §3.3 — искать по ней значит искать по тем же данным дважды). Норма [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1 **сохраняется без изменений** и **распространяется на все три места**:

| # | Место | Что фильтрует |
|---|-------|---------------|
| 1 | Выпадающий список, вкладка **«Сообщения»** | опции комбобокса |
| 2 | Выпадающий список, вкладка **«Почты»** | опции комбобокса |
| 3 | **Таблица** ящиков, вкладка **«Почты»** | строки таблицы |

- **Единый источник истины предиката — ОДИН модуль (нормативно):** `frontend/src/features/mail/mailboxSearch.ts`, экспортирующий (а) `mailboxSearchKeywords(mb: MailMailbox): string[]` → `[mb.number ?? '', mb.app_name ?? '', mb.email]` (ровно три элемента) и (б) `matchesMailboxQuery(mb: MailMailbox, query: string): boolean`. Модуль импортируют **все три** места: опции комбобокса на обеих вкладках отдают `keywords: mailboxSearchKeywords(mb)`, таблица `MailboxesTab` фильтруется `matchesMailboxQuery`. **Дублировать предикат в двух файлах ЗАПРЕЩЕНО** — прецедент и причина нормы: раздвоенный билдер `srcDoc` («починили в одном месте, забыли в другом», [ADR-047](ADR-047-mail-fix-pack.md) §6).
- **Осознанное ограничение (не дефект):** поля матчатся **по отдельности**, поэтому запрос, пересекающий границу полей (напр. `«12345 What»` при `number="12345"`, `app_name="WhatsApp"`), совпадения **не даёт** — хотя визуально в лейбле опции строка выглядит слитной. Так и было в [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1 (владелец перечислил ровно три поля и менять правило не просил); склейка полей в поисковый ключ = поиск по `display_name`, что нормой прямо запрещено. Долга не заводим — правило нормативно и намеренно.

## Последствия

- **Backend, БД, миграции, RBAC, контракт API — НЕ меняются.** Правка чисто frontend (как и фикс 1 [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)).
- **В ДС появляется первый не-нативный интерактивный примитив** с собственным a11y-контрактом (`ui/Select`/`ui/MultiSelect`/`ui/Textarea` доступность получали от нативных элементов). Это осознанная плата за требование владельца; поэтому ARIA/клавиатура нормированы **поэлементно** и обязаны быть покрыты тестами.
- **Единая ментальная модель на обеих вкладках:** «клик по полю → все почты; ввод → фильтр списка; выбор → фильтр по этой почте». Разница остаётся ровно одна и она нормирована: на «Сообщениях» фильтруется **лента (серверно)**, на «Почтах» — **таблица каталога (клиентски)**; и **только** на «Почтах» текст фильтрует ещё и таблицу.
- **Правило поиска (три поля) стало единым и физически одним куском кода** (`mailboxSearch.ts`) — расхождение «в списке ищет иначе, чем в таблице» структурно невозможно.
- **`ui/Select` не мигрируется массово** — остаётся дефолтным дропдауном ДС там, где поиск по опциям не нужен (в т.ч. «Команда» на обеих вкладках `/mail`). Массовая миграция была бы разворотом [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md)/[ADR-047](ADR-047-mail-fix-pack.md) без запроса владельца.
- **[TD-056](../100-known-tech-debt.md) уточняется, но НЕ закрывается:** примитив `ui/Combobox` теперь **существует** (его отсутствие было основной ценой долга), но in-row селектор **команды** на вкладке «Почты» остаётся `ui/Select` и по-прежнему не переносит длинное имя команды. Долг сузился до «мигрировать этот селектор на `ui/Combobox`».
- **Новых долгов нет.** Ограничение «поиск не пересекает границу полей» — намеренная норма §3.3, не долг.
- **QA (покрыть):** клавиатура комбобокса поэлементно (↓/↑/Home/End-открыт/Home/End-закрыт — **не перехватываются**/Enter/Escape-открыт/Escape-закрыт/Tab/печать); открытие показывает **все** опции при непустом поле; `pinned` «Все почты» не исчезает при вводе; `mode='select'` — ввод не сбрасывает серверный фильтр, закрытие без выбора возвращает текст к лейблу; `mode='search'` — ввод сбрасывает выбор; таблица «Почт»: ввод → **все** совпадения (не одна строка), выбор → ровно одна строка, `X`/Escape → полный набор; AND с сегментом активности и фильтром команды (в т.ч. пустое пересечение → «Ничего не найдено»); единый предикат (одинаковый результат в списке и в таблице по одному запросу).
- **QA — развилки, закрытые по замечаниям ревью (обязательны отдельными кейсами):**
  - **§1.1а (одно состояние «нет фильтра»):** на «Сообщениях» при монтировании — `value=''`, текст «Все почты», **`X` не отрисован**; `Escape` при закрытом списке → **остаётся** «Все почты» (`onChange('')`), поле **НЕ** пустеет; выбор опции «Все почты» и `Escape` дают **идентичное** состояние поля и ленты. На «Почтах» (без `pinned`) — `X`/`Escape` → поле **пусто**, виден плейсхолдер.
  - **§3.1а (смена сегмента):** выбрать ящик в сегменте «Все» → переключить на «Неактивные» (ящик активен) → таблица **пуста**, «Ничего не найдено», **лейбл ящика остаётся в поле**, выбор **НЕ сброшен**; `X` → полный набор нового сегмента; возврат сегмента «Все» → снова **одна** строка выбранного ящика.
  - **§1.1 (`dirty` сбрасывается на ЗАКРЫТИИ):** на «Сообщениях» при выбранной «Все почты» напечатать текст → закрыть список **без выбора** (`Escape` / `Tab` / клик вне) → текст вернулся к «Все почты» и **кнопка `X` НЕ отрисована** (аномалия «X при нечего сбрасывать» отсутствует).
  - **§1.2 (клавиши из состояния «панель открыта, активной опции НЕТ»):** открыть **мышью** → `↓` → активна **первая** опция; открыть мышью → `↑` → активна **последняя**; открыть мышью → `Enter` → **выбор не изменился**, `onChange` **не вызван**, **панель осталась открытой**, сабмита формы нет.
  - **§1.2 (мышь):** клик по полю при **открытой** панели → панель **не закрылась**; клик по `ChevronDown` при открытой → **закрылась без выбора**, при закрытой → **открылась**; `ChevronDown` **вне таб-порядка** (`Tab` из поля уходит на следующий контрол, а не на шеврон).
  - **§1.2 (клик по `X` при ОТКРЫТОЙ панели — единственный способ увидеть `X` в дефолте на «Сообщениях»):** открыть панель → напечатать текст (`X` появился) → клик по `X` → текст очищен по правилу режима (на «Сообщениях» — вернулся к «Все почты», `value=''`; на «Почтах» — поле пусто), **панель ЗАКРЫЛАСЬ**, **фокус остался в `<input>`**, `X` **исчез**, `dirty=false`.
  - **§1.2 (прокрутка активной опции):** при 121 опции нажать `End` → активная опция (последняя) **видима** в панели (проскроллена), не за границей `max-h-64`; то же для длинного прохода `↓`.
  - **§2 (нет совпадений при `pinned`):** на «Сообщениях» ввести заведомо несовпадающий запрос → в панели видны **И** опция **«Все почты»**, **И** строка **«Ничего не найдено»** (не одинокая «Все почты» без пояснения).
  - **§1.1 (`dirty` при открытии ВВОДОМ):** при закрытой панели напечатать символ → панель открылась **и список уже отфильтрован** этим символом (первый символ **не теряется**).
  - **§1.1 (`value` вне набора, `mode='select'`):** выбран ящик → он **исчез из каталога** опций (удаление/рефетч) при живом `mailAccountId` → открыть и закрыть список без выбора → **поле продолжает показывать лейбл** (НЕ пустеет), фильтр ленты сохранён; `aria-activedescendant` при открытии мышью **не задан** (нет ссылки на несуществующий `optionId`).

## Альтернативы (отклонены)

1. **Оставить `ui/Select` и добавить рядом поле поиска.** Отклонено: владелец просил **одно** поле, в котором «при нажатии выпадает список из всех почт», а ввод фильтрует этот список. Два контрола рядом — ровно то состояние, которое он просил изменить.
2. **На вкладке «Почты» выбор из списка НЕ фильтрует таблицу, а скроллит к строке и подсвечивает её.** Отклонено: (а) вводит новую механику (refs/`scrollIntoView`/временная подсветка) без прецедента в ДС и без нормы; (б) расщепляет ментальную модель «выбор = фильтр по этой почте», действующую на «Сообщениях»; (в) при 121 ящике сужение до одной строки даёт «быстрый переход» дешевле и детерминированнее. Требование владельца «ввод фильтрует таблицу» при этом **соблюдено** — сужение делает **выбор**, а не ввод (§3.1).
3. **Ввод текста на вкладке «Почты» фильтрует ТОЛЬКО выпадающий список (как на «Сообщениях»), таблица меняется лишь по выбору.** Отклонено: **прямо противоречит уточнению владельца** («ввод фильтрует таблицу по всем совпадениям, таблица не схлопывается») и был бы регрессом действующего поведения [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1.
4. **Добавить combobox-библиотеку** (Downshift / Headless UI / `@radix-ui/react-select`). Отклонено: [02-tech-stack.md](../02-tech-stack.md#frontend) — все примитивы ДС написаны руками на нативных элементах + Tailwind (`Select`, `MultiSelect`, `Textarea`, `Modal`-исключение — Radix уже был). Новая зависимость ради одного контрола с ~150 строками логики не оправдана; a11y-контракт всё равно пришлось бы нормировать и тестировать.
5. **Серверный поиск по ящикам (`q` у `GET /api/mail/mailboxes`).** Отклонено — повторно, по тем же причинам, что в [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) (Альтернатива 9): каталог грузится **целиком** (пагинации в контракте нет), клиентский фильтр полон и точен; серверный дал бы новый параметр, валидацию и класс ошибок ради нуля выгоды.
6. **Включить `display_name` в поля поиска** (чтобы запрос мог пересекать границу «Номер»/«Приложение»). Отклонено: `display_name` — производная склейка тех же двух полей ([ADR-047](ADR-047-mail-fix-pack.md) §3.3); её включение = поиск по одним и тем же данным дважды и разворот нормы [ADR-050](ADR-050-mail-search-team-filter-personal-read-state.md) §1.1, которую владелец менять не просил (он перечислил ровно три поля: Номер, Приложение, почта).
7. **Мигрировать на `ui/Combobox` все `ui/Select` в проекте** (ради консистентности). Отклонено: разворот действующих норм на ~6 страницах без запроса владельца; `ui/Select` остаётся корректным выбором там, где опций мало и поиск не нужен. Точечная миграция in-row селектора команды — отдельный долг ([TD-056](../100-known-tech-debt.md)).
