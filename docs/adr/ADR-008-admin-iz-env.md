# ADR-008 · Учётка администратора только из `.env`

- Статус: accepted
- Дата: 2026-06-28

## Контекст

Этап 1 — один администратор. ТЗ: проверка против `ADMIN_USER`/`ADMIN_PASSWORD` из `.env`. Нужно решить, хранить ли учётку в БД (таблица users) или только в окружении.

## Решение

Учётная запись администратора хранится **только в `.env`** (`ADMIN_USER`, `ADMIN_PASSWORD`), без таблицы пользователей в БД. Backend сравнивает введённые креды с переменными окружения constant-time. JWT выпускается с `sub = ADMIN_USER`.

## Обоснование

- Один пользователь на Этапе 1 — таблица users и RBAC избыточны (NFR-1).
- Меньше поверхности атаки (нет хранилища учёток в БД).
- Простая ротация: смена `.env` + перезапуск.

## Последствия

- (+) Минимальная модель данных и кода аутентификации.
- (+) Нет риска утечки учёток из дампа БД.
- (−) Нельзя добавить второго пользователя без доработки (миграция на таблицу users + RBAC — будущий этап).
- (−) Смена пароля требует доступа к `.env`/деплою (нет UI смены пароля).
- Опционально хранить `ADMIN_PASSWORD_HASH` (bcrypt) вместо plaintext — [Q-SEC-1](../99-open-questions.md).

## Альтернативы

- **Таблица users в БД + хэши** — отвергнуто на Этапе 1: преждевременно для одного админа; запланировано при многопользовательском режиме.
- **Внешний IdP/OAuth** — вне scope Этапа 1.

## Амендмент (2026-07-07, Спринт 3 — [ADR-021](ADR-021-rbac-users-roles.md))

ADR-008 остаётся **`accepted`** — `.env`-учётка не отменяется. С введением RBAC ([ADR-021](ADR-021-rbac-users-roles.md)) её роль уточняется:

- `.env`-учётка (`ADMIN_USER`/`ADMIN_PASSWORD`) более не «единственный пользователь», а **несменяемый супер-админ (bootstrap)**: всегда полный доступ, роль `admin` неявно (`superadmin=true` в JWT), **в таблицу `users` не попадает**, из UI **не редактируется и не удаляется**.
- Сравнение кредов супер-админа — по-прежнему **plaintext constant-time** (`secrets.compare_digest`); bcrypt-хэш применяется только к БД-пользователям (`users.password_hash`, [ADR-021](ADR-021-rbac-users-roles.md)). Опция `ADMIN_PASSWORD_HASH` из [Q-SEC-1](../99-open-questions.md) не вводится — для супер-админа plaintext-сравнение сохранено осознанно (bootstrap вне БД).
- Дополнительные пользователи и роли теперь **есть в БД** (`users`/`roles`) — прежнее ограничение «нельзя добавить второго пользователя без доработки» снято [ADR-021](ADR-021-rbac-users-roles.md).
- Смена пароля супер-админа — по-прежнему только через `.env`/деплой (UI-смены нет by design); UI-управление паролями появляется для БД-пользователей → сужает [TD-009](../100-known-tech-debt.md).
- **Fallback-инвариант сохранён:** супер-админ вне БД → систему нельзя залочить через данные (даже при пустой/битой таблице ролей вход супер-админом работает).
