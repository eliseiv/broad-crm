# ADR-065 — Страница «Пользователи»: плоский список с чипами команд вместо группировки по командам

- **Статус:** accepted
- **Дата:** 2026-07-21
- **Область:** frontend / страница `/users` (`UsersPage`)
- **Связи:** разворачивает норму отображения списка из [ADR-021](ADR-021-rbac-users-roles.md)/[ADR-022](ADR-022-teams-nav-categories.md) («список сгруппирован по CRM-командам»); контракт `GET /api/users` ([04-api.md](../04-api.md#users)) и модалки add/edit **не меняются**.

## Контекст

Сейчас `/users` группирует пользователей по командам: функция `groupByTeams` строит секции с заголовками-названиями команд, и **пользователь, входящий в несколько команд, дублируется в каждой** секции; пользователи без команды — в отдельном бакете «Без команды». Backend при этом отдаёт **плоский список** (`GET /api/users` → `UserListResponse.items: UserListItem[]`; каждый элемент несёт `teams: TeamRef{id,name}[]`) — вся группировка **чисто клиентская**.

Дублирование одной учётки в нескольких секциях мешает восприятию «сколько всего пользователей» и «где конкретный пользователь», а принадлежность к командам и так доступна в самом элементе (`user.teams`).

## Решение (утверждено с пользователем)

Страница `/users` отображает **плоский список пользователей без группировки и без дублирования**:

1. **Один элемент на пользователя.** Секции-заголовки команд и бакет «Без команды» как контейнер — **упразднены**. `groupByTeams` больше не применяется к раскладке списка.
2. **Принадлежность к командам — чипами в строке пользователя.** Массив `user.teams` (`TeamRef{id,name}`) рендерится набором чипов с названиями команд. При **пустом** массиве — единичный фолбэк-чип/подпись **«Без команды»** (вторичным цветом).
3. **Сортировка списка — по `username`, алфавитная** (`localeCompare`, локаль `ru`).
4. **Сохраняется без изменений** всё остальное наполнение строки и поведение страницы: название роли (`role_name`), тристатус-бейдж по производному `status` (Активен/Ожидает входа/Неактивен, [ADR-028](ADR-028-user-status-first-login.md)), индикатор **«Без пароля»** при `has_password=false` ([ADR-025](ADR-025-passwordless-users-login-identifier-open-first-login.md)), клик по строке → модалка редактирования (`PATCH`), кнопки add/edit. Супер-админ по-прежнему исключён из ответа сервера ([ADR-051](ADR-051-superadmin-db-anchor-personal-state.md) §1.4).
5. **`useTeams` остаётся** на странице — он нужен модалке `AddUserModal` (мультивыбор команд), а **не** раскладке списка. Убирать его нельзя.
6. **Backend-контракт и модалки не меняются** — задача чисто presentational на клиенте.

### Чипы команд — примитив

Чипы команд — presentational, с названием команды. `ui/Pill` работает только с семантическими tone-токенами ([ui/Pill](../08-design-system.md#компонент-uipill)); чип команды не требует произвольного цвета, поэтому допустим либо `ui/Pill` (нейтральный/`accent` tone), либо существующая чип-идиома строки — выбор реализации за frontend, **новый примитив ДС не вводится**. Визуал — существующие токены, новых токенов нет.

## Последствия

- Восприятие списка упрощается: одна строка = один пользователь; команды видны компактно, без прокрутки секций.
- Никаких изменений API, БД, модалок, прав — риск локализован в раскладке `UsersPage`.
- Прежняя норма «список сгруппирован по командам, пользователь дублируется в каждой секции» ([ADR-021](ADR-021-rbac-users-roles.md), [ADR-022](ADR-022-teams-nav-categories.md)) — **отменена** в части раскладки; в [08-design-system.md](../08-design-system.md#страница-пользователи) заголовок раздела и текст приведены к плоскому списку с явной пометкой отмены.

## Почему отдельный ADR, а не поправка/фикс-пакет с ADR-064

ADR-064 — foundational-изменение движка токенов ДС с долгоживущим правилом формата на все будущие токены; данный ADR — page-level презентационное решение одной страницы. Темы не связаны. Репо-практика фикс-паков ([ADR-046](ADR-046-ui-infra-fix-pack.md)/[ADR-047](ADR-047-mail-fix-pack.md)) объединяет **связанные** элементы; здесь связи нет, поэтому — отдельный ADR по одной теме (как [ADR-057](ADR-057-mail-send-contract-fix.md)/[ADR-061](ADR-061-documents-sidebar-two-panel-nav.md)).
