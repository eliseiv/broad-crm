# ADR-010 · Периодическая проверка AI-ключей внутри backend + Fernet-шифрование ключей

- Статус: accepted
- Дата: 2026-07-01

## Контекст

Появляется новая сущность — **AI-ключи** (API-ключи провайдеров OpenAI/Anthropic), которые администратор добавляет в CRM. Требования:

1. Хранить секрет ключа безопасно (не в открытом виде), но **обратимо** — ключ нужен целиком, чтобы обратиться к провайдеру и проверить его валидность.
2. Периодически проверять «живость» ключа (не отозван, не заблокирован, не исчерпана квота) и уведомлять администратора в Telegram, когда ключ **перестал работать** и когда **снова заработал** (recovery).
3. Никогда не возвращать полный ключ в API-ответах/логах; в UI показывать маску.

Варианты реализации проверки:

1. **Внешний воркер/крон** (отдельный процесс/контейнер), опрашивающий провайдеров и пишущий статус в БД.
2. **Фоновая asyncio-задача внутри backend-процесса**, аналогично провижинингу ([ADR-006](ADR-006-async-provisioning-bez-brokera.md)) и Telegram-нотификатору ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md)).

Варианты шифрования: тот же **Fernet** ([ADR-007](ADR-007-shifrovanie-fernet.md)), что и для SSH-паролей, либо отдельная схема/ключ.

Ограничения проекта: монолит, команда 1–2 человека (NFR-1, [ADR-001](ADR-001-stack-i-monolit.md)); брокеры/новые контейнеры на Этапе 1 не вводятся; Fernet-инфраструктура (`FERNET_KEY`, `app/infra/crypto.py`) уже есть.

## Решение

1. **Fernet-шифрование ключа тем же `FERNET_KEY`.** Секрет ключа шифруется при создании и хранится в БД как `ai_keys.key_encrypted bytea` (переиспользуется `encrypt_password`/`decrypt_password` из `app/infra/crypto.py`, [ADR-007](ADR-007-shifrovanie-fernet.md)). Дополнительно хранятся **plaintext-фрагменты для маски** — `key_prefix` (первые 4 символа) и `key_last4` (последние 4 символа); полный ключ восстанавливается только расшифровкой в памяти перед запросом к провайдеру.

2. **In-backend asyncio-монитор `AiKeyMonitorService`** — отдельная фоновая задача в процессе backend (НЕ переиспользует state-машину нотификатора серверов). Интервал `AI_KEY_CHECK_INTERVAL_SEC` (default 900 с). При создании ключа (`POST /api/ai-keys`) запускается немедленная фоновая проверка (`asyncio.create_task` + сильная ссылка — как в `POST /api/servers`). Alertmanager/внешний воркер **не вводятся**. Спецификация — [modules/ai-keys](../modules/ai-keys/README.md).

3. **Состояние переходов — в БД (`ai_keys.check_status`), а не in-memory.** В отличие от нотификатора серверов ([ADR-009](ADR-009-in-backend-notifier-vs-alertmanager.md), state in-memory, [TD-019](../100-known-tech-debt.md)), монитор ключей вычисляет переходы `pending|working → error` (алерт) и `error → working` (recovery) на основе персистентного `check_status`. Поэтому переходы **переживают рестарт backend** — сломанный ключ не «переоткрывается» после рестарта и не шлёт дубль-алерт; recovery корректно отрабатывает даже если backend перезапускался между проверками.

4. **Проверка — только валидность, без траты токенов.** Лёгкий `GET /v1/models` провайдера (OpenAI и Anthropic оба поддерживают этот read-only эндпоинт). Транзиентные ошибки (сеть/таймаут/5xx) → исход `unknown`: статус НЕ меняется, алерт НЕ шлётся. Флип в `error` — только на 4xx-ошибках авторизации/квоты.

5. **Telegram-отправка опциональна, статус в БД — всегда.** Отправка алертов гейтится тем же `settings.notifier_enabled` (`TELEGRAM_BOT_TOKEN`+`TELEGRAM_CHAT_ID`), что и нотификатор серверов; переиспользуется `TelegramClient` ([modules/notifier](../modules/notifier/README.md)). Но `check_status` в БД для UI обновляется всегда, и **монитор стартует всегда** (даже без Telegram) — UI-статус работает независимо от бота.

## Обоснование

- **Единая инфраструктура шифрования.** `FERNET_KEY` и `crypto.py` уже есть; отдельный ключ/схема усложнили бы управление секретами без выгоды.
- **Без новой инфраструктуры.** Нет нового контейнера/крона — соответствует NFR-1 (простота), как и решения по провижинингу и нотификатору.
- **Персистентный статус лучше, чем у нотификатора серверов.** Здесь состояние естественно ложится в БД (у ключа один атомарный статус, в отличие от per-metric зон серверов), поэтому мы сразу берём БД как источник состояния и не наследуем ограничение in-memory ([TD-019](../100-known-tech-debt.md)).
- **Экономия токенов.** `GET /v1/models` не расходует биллинговые токены — можно опрашивать периодически без затрат.
- **Опциональность Telegram.** UI-функция (список/статусы ключей) не зависит от наличия бота.

## Последствия

- (+) Один источник шифрования, нет нового инфраструктурного компонента, точный контроль формата алертов.
- (+) Переходы статуса переживают рестарт (персистентный `check_status`) — нет дубль-алертов и потерянных recovery после рестарта.
- (+) UI-статусы ключей работают без Telegram.
- (−) **Точный баланс/остаток средств недоступен** по обычному API-ключу (провайдеры не отдают биллинг по ключу) — детектируем только валидность и признак исчерпания квоты (`insufficient_quota`/429) ([TD-020](../100-known-tech-debt.md)).
- (−) Фиксированный интервал проверки, нет ротации ключей и ручного «проверить сейчас» ([TD-021](../100-known-tech-debt.md)).
- (−) Best-effort доставка Telegram с семантикой at-least-once (наследуется от `TelegramClient`, [modules/notifier](../modules/notifier/README.md#доставка-в-telegram)) — редкий дубликат вместо потери алерта.
- (−) Не горизонтально масштабируется (один процесс — то же ограничение, что у провижининга/нотификатора).

## Альтернативы

- **Внешний воркер/крон-контейнер** — отвергнуто: новый деплой-юнит и копия доступа к БД/секретам; противоречит монолиту (NFR-1).
- **Отдельная схема шифрования/ключ для AI-ключей** — отвергнуто: усложняет управление секретами; Fernet ([ADR-007](ADR-007-shifrovanie-fernet.md)) уже покрывает требование обратимого at-rest шифрования.
- **Проверка через биллинговый/платный эндпоинт (реальный запрос к модели)** — отвергнуто: тратит токены и деньги на каждую проверку; `GET /v1/models` достаточно для «жив/отозван/квота».
- **Переиспользовать in-memory state-машину нотификатора серверов** — отвергнуто: state ключей должен переживать рестарт (алерт «ключ не работает» критичен), а per-metric-эскалация серверов здесь не нужна — модель статуса другая.

## Будущее

- Ручной триггер «проверить сейчас», настраиваемый интервал per-key, ротация ключей ([TD-021](../100-known-tech-debt.md)).
- Более точная диагностика лимитов/остатка при появлении провайдерских API биллинга ([TD-020](../100-known-tech-debt.md)).
- Расширение списка провайдеров (Gemini, Mistral и т.п.) — добавлением значений в enum `provider` и адаптеров проверки.
