# ADR-043 — Ленивый провижининг почтовой группы «команда = почтовая группа»: авто-создание группы при первой почте, `team_id` в write-контракте ящика, `MailScope.team_ids`

Статус: `accepted` · Дата: 2026-07-10

Амендмент [ADR-038](ADR-038-mail-headless-integration.md) (§2 backfill / §3 `MailScope`), [ADR-022](ADR-022-teams-nav-categories.md) / [ADR-026](ADR-026-teams-optional-leader-auto-transfer.md) (форма команды). Парный ADR в mail-агрегаторе — `ADR-0042` (external team create + guarded delete). Частично закрывает [TD-036](../100-known-tech-debt.md).

## Context

[ADR-038](ADR-038-mail-headless-integration.md) связал CRM-команду с группой агрегатора полем `teams.mail_group_id INTEGER NULL UNIQUE` (1:1) и предписал **ручное** сопоставление администратором (`PATCH /api/teams/{id}`, селектор «Почтовая группа» из `GET /api/mail/teams`). Существующие команды получают `mail_group_id = NULL` — до ручного сопоставления участник видит **пустую ленту** ([TD-036](../100-known-tech-debt.md)).

Владелец продукта пересмотрел модель: **команда и почтовая группа — одна сущность**, вручную сопоставлять не нужно. Дословно: «если при создании почты пользователь из Команды Андрея добавит почту, то только тогда создастся группа». То есть:
1. Селектор «Почтовая группа» **убирается из создания** команды.
2. Группа в агрегаторе создаётся **лениво — по первому добавлению почты** в команду с `mail_group_id IS NULL` (имя группы = имя команды; см. пары в проде: имена CRM-команд и групп агрегатора **не** совпадают → связь только по `id`).
3. В external API агрегатора добавляется создание группы (mail-агрегатор `ADR-0042`).

## Decision

### §1. Создание команды — без привязки к почте; правка — сохраняет ручную привязку

- **`POST /api/teams` (`TeamCreateRequest`): поле `mail_group_id` УДАЛЕНО.** Новая команда всегда стартует `mail_group_id = NULL`; группа провижинится лениво (§2). Селектор **«Почтовая группа» убирается из `AddTeamModal` в режиме создания** (`mode='add'`).
- **`PATCH /api/teams/{id}` (`TeamUpdateRequest`): поле `mail_group_id` СОХРАНЯЕТСЯ** (presence-семантика, как в ADR-038). Ручная (пере)привязка к **существующей** группе агрегатора остаётся — это нужно, чтобы прицепить команду к **уже существующей группе с ящиками** (напр. `Business 2 → 8`) вместо создания новой пустой, и как механизм backfill из ADR-038 / [TD-036](../100-known-tech-debt.md). Селектор «Почтовая группа» **остаётся в `AddTeamModal mode='edit'`**. `409 team_mail_group_taken` теперь возникает **только** на `PATCH`.
- `teams.mail_group_id INTEGER NULL UNIQUE` (миграция `0018`, ADR-038) — **без изменений**; меняется только способ заполнения (лениво-авто vs вручную). Схема БД не мигрирует.

### §2. Ленивый провижининг группы при write-операции над ящиком; `team_id` вместо `group_id`

**Контракт write-эндпоинтов ящика ссылается на команду по `team_id` (UUID CRM), а не по `group_id` (int агрегатора).**

- **`POST /api/mail/mailboxes`** (`MailMailboxCreateRequest`): поле `group_id` → **`team_id: string(uuid)`** (команда CRM, `TeamListItem.id`; `null` — персональный ящик без команды).
- **`PATCH /api/mail/mailboxes/{id}`** (перенос): поле `group_id` → **`team_id: string(uuid) | null`** (целевая команда; `null` — открепить). presence-семантика сохраняется.
- Поля **чтения** не меняются: `MailMailbox.group_id` (int агрегатора) по-прежнему отдаётся (идентичность ящика в агрегаторе, нужна резолву «Почты команды» на `/teams`). Меняется только **запись**.

**Резолв команды → группы (нормативно), общий хелпер `ensure_team_mail_group(team_id) -> group_id`.** Атомарность инварианта «одна команда — ровно одна группа» обеспечивается **compare-and-set (CAS) на `teams.mail_group_id`**, БЕЗ удержания row-lock через сетевой вызов (§3/§4):
1. `team_id` существует в CRM (иначе `404 team_not_found`); авторизация — §5.
2. Прочитать `teams.mail_group_id` (без лока). `IS NOT NULL` → `group_id = mail_group_id`; перейти к шагу 5.
3. `IS NULL` → создать группу в агрегаторе `POST /api/external/teams { name: team.name }` (mail-агрегатор `ADR-0042` §2) → `g` (**вне** какого-либо CRM-лока — сетевой вызов не держит соединение пула под локом).
4. **CAS-запись:** `UPDATE teams SET mail_group_id = :g WHERE id = :team_id AND mail_group_id IS NULL RETURNING mail_group_id` (атомарный переход `NULL → значение`, единственная точка записи привязки):
   - **rowcount = 1** (выиграли гонку) → `group_id = g`;
   - **rowcount = 0** (параллельная ветка уже записала) → `SELECT mail_group_id` (теперь `IS NOT NULL` = `g_other`) → `group_id = g_other`; **наша `g` осиротела** (пустая, не привязана) → best-effort `DELETE /api/external/teams/{g}` (реклейм, `ADR-0042` §3; `204` для пустой). Ящик заводится в `g_other`.
5. Вызвать агрегатор `POST /api/external/mailboxes { group_id, ...creds }` (или `PATCH` при переносе).

**Обоснование замены `group_id → team_id` (осознанный контракт-брейк):**
- Ленивое создание требует **якоря-команды**: `group_id` не может выразить «команда, у которой группы ещё нет» — а именно это исходное состояние (`mail_group_id IS NULL`);
- Реализует «команда = группа»: клиент (единственный — SPA CRM) **никогда не оперирует id групп агрегатора** на записи, только командами;
- Авторизация становится **CRM-нативной** (членство в команде), работает и для команд без группы (§5);
- Модуль почты **не хранит данных** (read-through-прокси, ADR-012/038) и SPA — единственный потребитель → брейк локализован (форма создания ящика; селектор «Почтовая группа» из создания команды и так убран).

### §3. Атомарность распределённой операции (два сервиса) — порядок + компенсация

Ключ: **CAS-запись `mail_group_id` (§2 шаг 4) коммитит привязку ДО создания ящика**; сетевые вызовы агрегатора идут **вне** какого-либо CRM-лока.

- **«Группа создалась, ящик — нет»** (после CAS-коммита; ящик упал на `409 mail_conflict`/`422`/`502`): команда уже привязана к группе с 0 ящиков — это **валидное** промежуточное состояние. Повтор пользователя попадает в §2 шаг 2 (`IS NOT NULL`) → создаёт ящик в **существующей** группе. **Сироты нет**, ретрай идемпотентен. Компенсация не требуется.
- **«Группа создалась, привязка не записалась»** (CAS проиграл гонку — §4, либо CRM `UPDATE`/commit упал, либо ответ агрегатора потерян read-timeout'ом): наша только что созданная группа **осиротела** (пустая, ни с какой командой не связана). Best-effort `DELETE /api/external/teams/{g}` (`ADR-0042` §3) реклеймит её сразу; если DELETE не прошёл — остаётся **безвредная** пустая сирота (0 ящиков ⇒ 0 видимости/доставки), убираемая позже ([TD-037](../100-known-tech-debt.md) CRM / `TD-048` агрегатор). Осознанно принимаем редкую пустую сироту **вместо** 2PC/саги: (а) идемпотентного ключа без миграции нет; (б) сирота безвредна и не нарушает «одна команда — одна группа» (она **не** привязана к команде); (в) реюз по имени отвергнут владельцем («связывать по имени нельзя, только по id»).
- **Ретрай-политика клиента:** `POST /api/external/teams` — мутирующий, ретраится **только** на `ConnectError`/`ConnectTimeout` (запрос заведомо не ушёл; ADR-038 §1, **подтверждено кодом** `backend/app/infra/mail_client.py:233-252` — read-timeout/`5xx` на write сразу бросает `MailUnavailable`). → нет двойного создания из ретрай-слоя.

### §4. Гонка «двое из одной команды добавляют первую почту одновременно» — инвариант «одна команда — ровно одна группа»

Требование владельца: «два человека из одной команды не могут создавать разные группы» — гонка **обязана** привести к тому, что проигравшая ветка **использует группу, созданную выигравшей**. Механизм — **CAS на `teams.mail_group_id`** (§2 шаг 4), а не долгий лок:

- Обе ветки видят `mail_group_id IS NULL` и создают в агрегаторе по группе (`g1`, `g2`) — это два **несвязанных** объекта, ещё ничьи.
- Обе выполняют `UPDATE ... WHERE id=:team AND mail_group_id IS NULL`. Условие `mail_group_id IS NULL` — **атомарный compare-and-set**: ровно **одна** ветка получает `rowcount=1` (пусть `g1`), вторая — `rowcount=0`.
- Проигравшая ветка перечитывает `mail_group_id = g1`, **использует `g1`** (ящик заводится в `g1`), а свою `g2` реклеймит best-effort `DELETE`.
- **Итог:** `teams.mail_group_id` записан **ровно один раз** (`NULL → g1`, этот путь никогда не меняет значение на другое) → у команды **ровно одна** привязанная группа `g1` во **всех** сценариях. `g2` **не** привязана к команде (транзиентная пустая сирота, удаляется) — инвариант «одна команда — одна группа» не нарушается ни на мгновение (речь о **привязанной** группе).
- `teams.mail_group_id UNIQUE` (миграция `0018`) — дополнительный DB-guard (одна группа ↔ максимум одна команда); в связке с CAS-условием `IS NULL` даёт серверную сериализацию **без** удержания соединения пула через сетевой вызов (§3/Alternatives).

**Взрыв-радиус гонки:** CAS-условие `WHERE id=:team AND mail_group_id IS NULL` затрагивает **только** конкурентное добавление **первой** почты в **ту же** команду; после первой привязки путь провижининга не исполняется (шаг 2 отдаёт готовый `group_id`). Разные команды и повторные добавления не конкурируют.

### §5. `MailScope` += `team_ids`; create по членству, **transfer — только admin-уровень**

`MailScope(sees_all_teams, group_ids)` → **`MailScope(sees_all_teams, group_ids, team_ids)`** (`domain/mail.py`; `get_mail_scope`, `api/deps.py`):
- `team_ids: frozenset[UUID]` — команды пользователя из `user_teams` (все его команды, включая с `mail_group_id IS NULL`);
- `group_ids` (непустые `mail_group_id` этих команд) и `sees_all_teams` — **без изменений** (лента/чтение резолвятся как в ADR-038 §3).

Авторизация (граница безопасности — backend):
- **Создание ящика по `team_id`** (`POST /mailboxes`, не-admin): требует `team_id ∈ MailScope.team_ids` (участник команды), иначе `403 forbidden` (`errors.forbidden()`, `code="forbidden"`). Проверка по `team_ids` (а не `group_ids`) — для команды без группы `group_id` ещё нет; членство — корректный CRM-нативный критерий «своей команды» (владелец: «человек из Команды Андрея создаёт только почты своей группы»). `sees_all_teams=true` → любая команда.
- **Перенос ящика между командами** (`PATCH .../{id}` со сменой `team_id`) — **ТОЛЬКО admin-уровень** (`MailScope.sees_all_teams=true`), **НЕ** членство: даже участник двух команд переносить не вправе → `403 forbidden` при `sees_all_teams=false` (владелец: «переносить из одной команды в другую может только админ»).
  - **Выбор механизма — admin-предикат, а НЕ новое действие `mail:transfer`** (обосновано): (1) владелец требует «только админ», а `sees_all_teams` = `is_superadmin OR permissions_subset(full_catalog, permissions)` — тот же admin-level, что «видеть все почты»; отдельное грантуемое право `mail:transfer` можно было бы выдать не-админской роли, что противоречит требованию; (2) симметрично «view all» ([ADR-032](ADR-032-sms-visibility-admin-full-catalog.md)/[ADR-036](ADR-036-sms-team-filter-admin-only.md)) — «смотреть все + переносить» связаны одним admin-предикатом; (3) не раздувает `CATALOG["mail"]` (остаётся `view/create/edit/delete/sync/tags`, ADR-038 §4 — без изменений). Право `mail:edit` — необходимое, но недостаточное условие переноса.
- **Мутация существующего ящика по `id`** (`PATCH` кредов/`is_active`, `DELETE`, `sync`): без изменений — целевой ящик ∈ `MailScope.group_ids`, иначе `403` (ADR-038 §3).
- **Просмотр всех почт** — только `sees_all_teams` (ADR-038 §3, без изменений); вне scope чтение → пустая страница без вызова внешнего API.
- Зафиксировано в [05-security.md](../05-security.md#каталог-прав-канон-на-сервере) (каталог) и таблице анти-энумерации.

### §6. Пустое состояние команды без `mail_group_id` — actionable

С ленивым провижинингом участник с правом `mail:create` может **сам** создать первую почту (это и создаст группу). Пустая лента заменяется на **actionable-пустое-состояние** с CTA. Нормативные строки — [08-design-system.md](../08-design-system.md#пустое-состояние-команды-без-почты-нормативно):
- вкладка «Почты» при отсутствии привязанной почты и наличии `mail:create` — заголовок + подсказка «команда и почтовая группа создадутся автоматически» + кнопка добавления почты;
- без `mail:create` — пассивная подсказка обратиться к администратору;
- секция «Почты команды» на `/teams` при `mail_group_id=NULL` — «Почты не привязаны» + подсказка о ленивом создании.

## Consequences

- **[TD-036](../100-known-tech-debt.md) в основной части закрыт:** ручное сопоставление при создании команды больше не требуется — группа провижинится по первой почте. Ручная (пере)привязка к **существующей** группе сохранена в `PATCH /api/teams/{id}` (селектор в edit-модалке) как осознанная опция (прицепить команду к готовой группе с ящиками). TD-036 переводится в «частично закрыт» (остаток — опц. cleanup-report непривязанных команд).
- Клиент оперирует командами (`team_id`), а не id групп агрегатора, на всех write-путях почты — согласовано с «команда = группа».
- **Инвариант «одна команда — ровно одна группа» гарантирован** CAS-записью `mail_group_id` (§4): у команды всегда ≤1 **привязанной** группы; гонка не создаёт второй привязки, проигравшая ветка использует группу победителя.
- **Перенос ящика между командами — только admin-уровень** (`sees_all_teams`); участник (даже нескольких команд) переносить не вправе. Просмотр всех почт — тоже только admin-уровень (§5).
- Новый **редкий** остаток: **пустая** осиротевшая группа в агрегаторе (проигравшая CAS-ветка / read-timeout между созданием и CAS) — безвредна (0 ящиков), best-effort-реклейм `DELETE`, окончательная уборка — `TD-048` (агрегатор). Отлично от [TD-036](../100-known-tech-debt.md) (**команда без группы**): это **группа без команды**.
- `MailScope` расширен `team_ids` (аддитивно; лента/`group_ids`-инъекция не тронуты).
- Контракт-брейк `group_id → team_id` на 2 write-эндпоинтах ящика — локализован (нет хранения, SPA — единственный клиент).
- Удаление CRM-команды с непустой `mail_group_id` **не** трогает агрегатор в этом ADR (ящики продолжают работать, группа становится непривязанной = группа без команды) — заведено как [TD-037](../100-known-tech-debt.md) (не путать с TD-036: там команда без группы).

## Alternatives considered

- **Оставить `group_id` в теле create.** Отклонён: не выражает «команда без группы» — не может запустить ленивое создание; и заставляет клиента знать id групп агрегатора (против «команда = группа»).
- **Идемпотентность/реюз группы по имени (на стороне агрегатора).** Отклонён владельцем прямо («связывать по имени нельзя, только по id») и по коду: `groups.name` не уникально, имена штатно коллизируют — реюз слил бы несвязанные команды (mail-агрегатор `ADR-0042` §2).
- **Eager: создавать группу сразу при создании команды.** Отклонён: плодит пустые группы для команд, которые почтой не пользуются; владелец явно хочет **лениво** («только тогда создастся группа»).
- **Row-lock `SELECT ... FOR UPDATE` на строке команды, удерживаемый через внешний HTTP-вызов.** Отклонён (был в черновике, снят по ревью). Количественно: mail-API — **async**-сервис, обслуживающий много одновременных запросов из общего пула соединений SQLAlchemy (не Ansible-провижининг «один воркер» из `00-vision.md:37`, и NFR-1 `00-vision.md:75` про «монолит без брокера» — к latency пула неприменимы). Удержание Postgres-соединения на весь внешний вызов = `MAIL_API_TIMEOUT_SEC` (10 с) + connect-ретраи с backoff `(0.2, 0.5)` → до ~10.7 с на операцию. При `K` одновременных провижинингов `K` соединений заняты ~10 с; при пуле `pool_size+max_overflow` (типично 5+10=15) уже `K≈15` совпавших первых-добавлений исчерпывают пул → прочие запросы блокируются/таймаутят. Антипаттерн «сетевой I/O под БД-локом». **CAS-вариант (принят)** держит соединение лишь на суб-миллисекундный `UPDATE`; цена — чуть выше вероятность пустой сироты (проигравшая ветка создаёт группу до CAS), но сирота **пустая и безвредная** (0 ящиков), реклеймится `DELETE`; вероятность самой гонки мала (два члена одной команды жмут «добавить первую почту» в окне ~создания-группы — редкое совпадение при нечастом заведении ящиков). Инвариант «одна команда — одна группа» CAS выполняет строго (§4).
- **2PC / сага с компенсацией удаления группы при сбое ящика.** Отклонён: избыточно — сирота пустая и безвредная; порядок «CAS-привязка до ящика» уже делает mailbox-fail самоизлечимым на ретрае, а единственный остаток (пустая сирота) дешевле реклеймить (best-effort `DELETE`), чем тащить распределённую транзакцию.
- **Автор-хилинг stale `mail_group_id`** (если привязка указывает на удалённую группу — пересоздать). Отклонён в этом ADR: редкий кейс; при внешнем `group_not_found` во время create отдаётся `404 mail_group_not_found` (диагностируемо), авто-пересоздание — потенциальный TD.
