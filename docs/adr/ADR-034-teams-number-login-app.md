# ADR-034 · Detail-панель `/teams` показывает `login`/`app_name` номеров команды (частичный разворот сужения ADR-030 §8)

- Статус: accepted
- Дата: 2026-07-09
- Амендмент: [ADR-030](ADR-030-sms-module-full-merge.md) §8 (авторизационное сужение полей `TeamNumberItem`)

## Контекст

По [ADR-030](ADR-030-sms-module-full-merge.md) §8 эндпоинт `GET /api/teams/{id}/numbers` (гейт `teams:view`) был **сужен по полям**: держатель `teams:view` видел **состав** номеров команды, но не чувствительный учётный контекст (`login`/`app_name`/`note`/`label`). Схема `TeamNumberItem` содержала только `{id, phone_number, team}`. Полный контекст (`login`/`app_name`/`note`) отдавался исключительно на странице «СМС» под матрицей `sms:*` и SMS-scope.

Продуктовое требование изменилось: в detail-панели команды на `/teams` список номеров должен показывать **Логин** и **Приложение** каждого номера — эти поля нужны менеджерам команд (держателям `teams:view`) для идентификации, к какому аккаунту/приложению привязан номер, без перехода на страницу «СМС» (которая может быть недоступна по правам).

Это осознанный **разворот** части сужения ADR-030 §8: два поля (`login`, `app_name`) переклассифицируются из «доступно только под `sms:*`» в «доступно под `teams:view`».

## Решение

1. **`TeamNumberItem` расширяется двумя nullable-полями:** `login: string | null` и `app_name: string | null`. Новая схема:
   ```json
   { "id": 42, "phone_number": "+13105551234", "team": { "id": "d3f0…a1", "name": "Продажи" }, "login": "acme", "app_name": "WhatsApp" }
   ```
   Значения берутся из `sms_phone_numbers.login`/`app_name` номера (те же колонки, что отдаёт `SmsNumberItem`). Новых колонок БД/миграций нет — только расширение read-контракта.

2. **`note` и `label` остаются сужёнными** — в `TeamNumberItem` их **нет**. Полный учётный контекст с `note` (свободная заметка) и системным `label` (Twilio `friendly_name`) по-прежнему доступен только на странице «СМС» под `sms:*` и SMS-scope.

3. **Гейт эндпоинта не меняется** — `teams:view` (не `sms:view`). Набор возвращаемых **строк** (номера команды `{id}`) не меняется — меняется только набор **полей** в каждой строке (+`login`, +`app_name`).

4. **UI (detail-панель `/teams`):** строка номера теперь рендерит **Номер + Логин + Приложение** (пилюли `Логин: {value}` / `Приложение: {value}`, идиома пилюль страницы «СМС»). При пустых значениях пилюля показывает `-`. `note`/`label` в панели по-прежнему не показываются.

## Security-трейдофф (осознанный)

- **Что раскрывается:** держатель `teams:view` теперь видит `login` и `app_name` номеров **любой** команды (эндпоинт не сужает строки по scope — он под `teams:view`, а не под SMS-scope). Ранее эти поля были доступны только под `sms:*` + SMS-scope (только свои команды для не-admin).
- **Обоснование приемлемости:** `login`/`app_name` — **слабо-чувствительный** идентифицирующий контекст (какой аккаунт/приложение привязаны к номеру), а не секрет: они не дают доступа к аккаунту сами по себе (нет пароля/токена). Управление командами уже де-факто admin-ориентировано ([ADR-022](ADR-022-teams-nav-categories.md) §3), а `teams:view` — доверенная роль уровня менеджмента команд.
- **Что НЕ раскрывается:** `note` (свободный текст, может содержать произвольные пометки, в т.ч. чувствительные) и `label` — остаются за матрицей `sms:*`. Секреты (пароли/ключи) к номерам не относятся — у номера секрета нет.
- **Асимметрия с SMS-scope сохраняется для `note`:** таким образом `teams:view` даёт идентифицирующий минимум (`login`/`app_name`), а полный контекст (`note`) — только SMS-права. Это сознательный компромисс между продуктовой потребностью и минимизацией раскрытия.

## Альтернативы (отклонены)

- **Полный разворот (`+note`/`+label`):** отклонён — пользователь запросил только Логин + Приложение; `note` может нести чувствительный текст, `label` — системное поле не для этого контекста.
- **Гейт на `sms:view` вместо `teams:view`:** отклонён — сломал бы саму цель (видеть Логин/Приложение в панели `/teams` без SMS-прав); эндпоинт концептуально принадлежит `/teams`.
- **Новое отдельное право `teams:view_numbers_context`:** отклонён (NFR-1, простота) — избыточно для двух слабо-чувствительных полей.
